## 商业价值
1. 降本：降低机器成本与人力成本
手写一个ROCm用例需数小时，Agent一句话自动闭环。效率提升10倍以上，工程师转为最终审核者，专家时间释放至高ROI工作，高质量的
测试用例，有效减少机器测试时间。

2. 提质：用例从经验驱动到资产驱动
RAG知识库保障用例贴合官方规范、风格统一、语法可校验。一次性通过率提升，团队知识沉淀为可执行资产，消除人员离职造成的知识断层。

3. 提效：验证周期从天到分钟
Agent自动接管"生成→执行→失败修复"全流程。端到端验证从以天计压缩至分钟级，加速模型与算子交付。

4. 增值：质量保障从人力线性到算力弹性
Agent可并行批量生成用例，测试覆盖率随版本快速规模化。降低因覆盖不足导致的线上故障风险与品牌损失。

核心指标：ROI提升、迭代加速、故障风险降低。




## 项目定位

基于 **LangGraph + LangChain** 和 Langraph/deepagent 构建的智能体，自动生成、执行、修复 AMD ROCm 及 AI Model（运行于 ROCm）的测试用例。



---

## 核心功能

### 1. 生成测试用例

- 用户以自然语言描述测试需求，Agent 生成 **pytest 格式可执行代码** + 简要说明
- 覆盖范围：
  - ROCm 底层算子：rocBLAS、MIOpen、rocSOLVER 等
  - AI Model 推理：vLLM / PyTorch 推理服务

### 2. （结果驱动）修复用例

- AIDevOps 返回测试失败 → Agent 自动分析失败原因 → 修复用例

### 3. 生成并执行

- 在功能 1 基础上，自动调用 **AIDevOps Agent（已有外部系统）** 执行生成的测试用例，返回执行结果

### 4. （用户驱动）修复用例

- 用户描述用例问题 → Agent 分析 → 修复

## 核心流程

```
用户需求
   │
   ▼
┌──────────┐    ┌──────────┐    ┌──────────┐
│  规划器   │ → │  生成器   │ → │  验证器   │
│ 拆解任务  │   │ 生成代码  │   │ 校验格式  │
└──────────┘    └──────────┘    └──────────┘
       ↑              │
       │    ┌─────────┘
       │    ▼
       │  ┌──────────┐    ┌──────────────┐
       │  │ 执行用例  │ → │ AIDevOps Agent │ (已有外部系统)
       │  └──────────┘    └──────────────┘
       │        │
       │        ▼ (失败)         ▼ (通过)
       │  ┌──────────┐      ┌──────────┐
       └──│ 自动修复  │      │  完成 ✅  │
          └──────────┘      └──────────┘
               │
               └──→ 修复后自动重新提交执行（最多 3 轮）
```

**修复循环规则**：
- AIDevOps 返回失败 → Agent 分析失败日志 → 修复用例 → 自动重新提交
- 最多重试 **3 轮**，超过后交由人工处理
- 通过后结束流程，用例文件落盘保存

---

## Agent 间通信

本项目 Agent 与 AIDevOps Agent 之间采用 **A2A 协议**：

- **Test Case Agent** 通过 A2A 向 AIDevOps 发起任务委托（提交 pytest 用例 → 请求环境执行）
- **AIDevOps** 执行完成后通过 A2A 回传结果（pass/fail + 日志 + 性能数据）
- A2A 提供能力发现、任务委托、流式反馈能力，无需底层协议适配

第一个大坑是token消耗。AI Agent在处理复杂任务时，会消耗大量的token，成本很快就上去了。我的解决方案是：

1. 优化prompt，减少不必要的输出
2. 使用更便宜的模型处理简单任务
3. 实现token使用量监控和限制
第二个问题是模型幻觉。有时候Agent会给出看起来很合理但实际上错误的信息，这在运维场景下是很危险的。我的应对策略是：


3. 建立知识库，让Agent基于已知信息回答问题

第三个问题是稳定性。API服务偶尔会出现故障或限流，影响Agent的正常工作。我的解决方案包括：

## 来源文件：`测试集.md`

 25 条自然语言输入，可直接拿来测试 TestCaseAgent。覆盖 ROCm 基础栈、HIP/ROCK、环境前置、WSL 工具、框架模型、LLM、ComfyUI 等层次。

为 ROCm 基础环境生成一个 pytest 测试用例，验证当前机器能正常执行 rocminfo，并检查输出中至少包含一个 GPU agent、设备名称和 ISA 信息。如果 rocminfo 不存在，应标记为 skipped，而不是直接 fail。

生成一个测试用例，验证 rocm-smi 可用并能读取 GPU 基础信息，包括 GPU 数量、温度、显存使用和功耗字段。测试需要区分命令不可用、无 GPU、字段缺失三类失败原因。

我想测试 ROCm agent 枚举能力，请生成 pytest 用例，对比 rocminfo 和 rocm_agent_enumerator 的输出，确认至少一个 GPU target name 能被两个工具一致识别。

生成一个 ROCm bandwidth smoke test，用 pytest 调用 rocm-bandwidth-test 或等价命令，验证 GPU 带宽测试能启动、退出码为 0，并从日志中提取带宽结果，结果为空时 fail。

为 KFD 生成一个测试用例，检查 /dev/kfd 是否存在且当前用户具备访问权限；如果缺少设备节点，输出清晰的 skipped 原因；如果权限不足，输出修复建议。

生成一个 kernel/driver 环境测试用例，验证 amdgpu 内核模块已加载，dmesg 中没有最近的 GPU reset、ring timeout、memory fault 等关键错误。需要保存匹配到的异常日志片段。

给 environment suite 设计一个 pytest 测试，检查当前系统是否存在 OOT amdgpu-dkms 驱动、当前 kernel 版本、initramfs 工具链状态，并输出一份结构化环境摘要。

生成一个 WSL 工具测试用例，检查当前是否运行在 WSL 环境，读取 GPU/硬件信息，验证 /mnt/wsl 或相关挂载路径存在，并给出不满足条件时的 skip 原因。

为 install 层设计一个测试用例，验证 ROCm driver 安装后的关键命令都可用：rocminfo、rocm-smi、hipcc。要求每个命令单独断言，失败时明确指出缺哪个组件。

生成 HIP smoke test，用 pytest 创建并编译一个最小 HIP 程序，执行简单 vector add，验证结果正确。如果 hipcc 不存在则 skip；如果编译失败，需要打印编译日志。

为 HIP memory copy 设计测试用例，验证 host-to-device、device-to-host 拷贝路径可用，输入数组大小覆盖 1、1024、1048576 三种规模，并检查结果一致性。

生成一个 HIP 多 GPU 检测测试，用 pytest 查询可见 GPU 数量；如果 GPU 数量大于等于 2，则执行一次简单跨设备选择测试；如果只有 1 张 GPU，则 skip 多 GPU 部分但整体不失败。

为 hipBLASLt 生成测试用例，验证 GEMM 基础路径可运行。要求测试包含 fp16 或 fp32 矩阵乘法，检查输出矩阵 shape 和数值有限性，并记录执行耗时。

生成一个 RCCL 通信库测试用例，检查 rccl-tests 或 all_reduce_perf 是否存在；如果存在，运行最小 all-reduce smoke test；如果不存在，返回 skipped 并说明依赖缺失。

为 ROCK suite 设计一个 rocBLAS 测试用例，执行最小矩阵乘 benchmark 或自测命令，验证退出码为 0，并从日志中检查没有 FAILED、Segmentation fault、Memory access fault。

生成一个 rocFFT 测试用例，验证 ROCm FFT 组件可用。测试应运行一个小规模 FFT smoke case，检查命令成功、日志无 core dumped，并记录输入规模和执行时间。

为 rocRAND 生成测试用例，验证随机数生成库可用，运行最小 sample 或测试命令，检查输出中存在随机数统计结果，并确认没有 NaN 或异常退出。

生成一个 PyTorch ROCm 测试用例，验证 torch.cuda.is_available() 为 True，创建 1024x1024 tensor 放到 GPU 上执行矩阵乘法，并检查结果 shape、dtype 和无 NaN。

为 PyTorch Llama 推理生成一个 pytest 测试，目标类似 wsl_pytorch_rocm/test_pyt_rocm_llama-3-8b_inf_perf.sh：检查模型路径或下载缓存是否存在，执行一次短 prompt 推理，提取 tokens/s 或 inference time。

生成一个 JAX ROCm 测试用例，验证 JAX 能识别 GPU backend，执行一个简单矩阵乘法或 XLA 编译函数，检查结果正确，并记录设备列表。

为 TensorFlow ROCm 生成测试用例，验证 TensorFlow 能发现 GPU，运行一个最小 BERT 或 MNIST inference smoke，检查日志中没有 OOM、segmentation fault、aborted。

生成 ONNX Runtime ROCm 测试用例，加载一个 DistilGPT2 或 BERT ONNX 模型，使用 ROCMExecutionProvider 运行一次 inference，检查 provider 生效且输出 tensor 不为空。

为 text_generation suite 生成一个 vLLM 测试用例，使用小 prompt 和小 num_prompts 做 smoke benchmark，要求解析 throughput、latency 或生成 token 数；如果模型需要 HF token，则明确 skipped。

生成 llama.cpp / llama-bench 风格的 pytest 测试，用于验证一个 GGUF 模型在 ROCm 后端上能完成短文本生成。要求检查模型文件存在、命令退出码、输出 token 数，并保存 benchmark 日志。

为 ComfyUI text-to-image 生成测试用例，验证 ROCm PyTorch 环境下可以加载 Flux/Stable Diffusion 工作流，执行一张低分辨率图片生成，检查输出图片文件存在、大小大于 0，并收集 ComfyUI 日志中的错误关键字。
