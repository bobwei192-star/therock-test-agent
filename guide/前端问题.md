# 前端问题排查与解决方案

## 问题列表

### 问题 1: Invalid DOM property `clip-path`
**错误信息**:
```
Invalid DOM property `clip-path`. Did you mean `clipPath`?
```

**根因**: React JSX 中 SVG 属性需要使用驼峰命名法

**解决方案**:
- 文件: `etc/agent-chat-ui/src/components/icons/langgraph.tsx`
- 修改: 将 `<g clip-path="...">` 改为 `<g clipPath="...">`

### 问题 2: 前端能复制内容但不显示
**现象**: 在 UI 里发消息，看不到智能体回复，但能 ctrl+a 全选并复制到内容

**根因**: 后端 LLM 输出包含大量 prompt 模板内容（意图选项列表、用户需求复述等），导致前端渲染混乱

**解决方案**:
- 文件: `src/agent/utils/clean_llm_output.py`
- 修改: 新增规则移除意图选项列表和多余的 prompt 模板内容

### 问题 3: assistant_id 不匹配（核心问题）

#### 现象
- 前端配置: `NEXT_PUBLIC_ASSISTANT_ID=agent`
- 后端日志显示: `assistant_id=fe096781-5601-53d2-b2f6-0d3403f7e9ca` (UUID)
- 即使将后端 langgraph.json 图名改为 UUID，前端又变成新的 UUID

#### 根因分析

**1. LangGraph SDK 的设计机制**

`@langchain/langgraph-sdk` 的 `client.js` 中，`RunsClient.stream()` 方法强制使用 `assistant_id` 参数：

```javascript
// node_modules/@langchain/langgraph-sdk/dist/client.js
async *stream(threadId, assistantId, payload) {
    const json = {
        // ...
        assistant_id: assistantId,  // 强制使用 assistant_id
        // ...
    };
    // ...
}
```

**2. 后端 API 的验证机制**

后端 LangGraph API 要求 `assistant_id` 必须是 UUID 格式：

```
GET /assistants/agent
→ {"detail":"Invalid assistant ID: must be a UUID"}
```

**3. 前后端配置不匹配**

| 组件 | 配置 | 说明 |
|------|------|------|
| 后端 `langgraph.json` | `"agent": "./src/agent/app.py:graph"` | 图名是字符串 `agent` |
| 前端 `.env` | `NEXT_PUBLIC_ASSISTANT_ID=agent` | 配置为字符串 `agent` |
| SDK 发送请求 | `assistant_id: "agent"` | 被后端拒绝 |

**4. 问题的本质**

LangGraph 平台在部署时会为每个 graph 自动生成一个 UUID 作为 `assistant_id`：
- 后端 `langgraph.json` 配置的是 `graph_id`（字符串，如 `agent`）
- 但 LangGraph API 要求使用 `assistant_id`（UUID）
- 前端 SDK 强制发送 `assistant_id`，且后端要求必须是 UUID
- 这就导致了前后端无法匹配

#### 详细解决过程

**使用正确的 API 获取 assistant_id（成功！）**

最终解决方案：

```bash
# 1. 启动后端服务
make serve

# 2. 搜索已创建的 assistants（关键步骤！）
curl -s -X POST http://localhost:2024/assistants/search \
  -H "Content-Type: application/json" \
  -d '{}'

# 3. 从返回结果中获取 assistant_id
# 返回格式：
# [{"assistant_id":"d8070331-8eb2-5a91-9f0b-95bcc149f24c","graph_id":"fe096781-5601-53d2-b2f6-0d3403f7e9ca",...}]

# 4. 将 assistant_id 配置到前端 .env
NEXT_PUBLIC_ASSISTANT_ID=d8070331-8eb2-5a91-9f0b-95bcc149f24c

# 5. 重启前端服务
make clean-frontend
make chat
```

**验证结果**：

```bash
# 检查后端日志
tail -f /home/zs/TestCaseAgent/test_case_agent.log | grep "assistant_id"

# 输出：
# "assistant_id": "d8070331-8eb2-5a91-9f0b-95bcc149f24c"
```

✅ **前后端 assistant_id 匹配，问题解决！**

#### 最终配置

| 配置项 | 值 | 来源 |
|--------|-----|------|
| **assistant_id** | `d8070331-8eb2-5a91-9f0b-95bcc149f24c` | `/assistants/search` API 返回 |
| **graph_id** | `fe096781-5601-53d2-b2f6-0d3403f7e9ca` | `langgraph.json` 配置 |

#### 关键发现

1. **`GET /assistants` 返回 `Method Not Allowed`**，需要使用 `POST /assistants/search`
2. **`assistant_id` 是运行时动态生成的**，每次后端重启可能生成新的 UUID
3. **`graph_id` 和 `assistant_id` 是不同的概念**：
   - `graph_id`: 在 `langgraph.json` 中配置的图名
   - `assistant_id`: 系统为每个 graph 实例自动生成的 UUID

### 问题 4: 队列积压严重

#### 现象
```
n_pending=39 n_running=3
pending_runs_wait_time_max_secs=183470.181375 (约 50 小时!)
```

#### 根因分析

**1. 真正的持久化存储位置**

LangGraph 使用 `PersistentDict` 将内存数据**定期刷新到文件**：
- 存储位置：`/home/zs/TestCaseAgent/.langgraph_api/`
- 文件格式：`.pckl` (pickle)
- 文件列表：
  - `.langgraph_checkpoint.1.pckl` - 检查点数据
  - `.langgraph_checkpoint.2.pckl` - 检查点数据
  - `.langgraph_checkpoint.3.pckl` - 检查点数据
  - `.langgraph_ops.pckl` - 操作数据
  - `store.pckl` - 存储数据

**2. 为什么 `make clean-state` 无效**

```bash
$ make clean-state
🧹 清理 LangGraph 状态存储...
rm -rf .langgraph/state/* 2>/dev/null || true  # 错误：目录不存在
rm -rf .langgraph/*.db 2>/dev/null || true    # 错误：目录不存在
✅ 状态已清理
```

问题：
- 清理的是 `.langgraph/` 目录（不存在）
- 真正的持久化存储在 `.langgraph_api/` 目录
- 重启后从 `.langgraph_api/` 恢复队列状态

**3. CLI 能跑但前端不能跑的原因**

| 方式 | 执行路径 | 队列状态 |
|------|----------|----------|
| `make gen` (CLI) | 直接调用 Python 代码 | 不经过队列，直接执行 |
| 前端 Chat UI | LangGraph API 服务器 | 请求进入队列，需要 Worker 处理 |

**4. 队列积压的深层原因**

- `n_pending=39`: 队列中有 39 个待处理请求
- `n_running=3`: 只有 3 个工作线程
- Worker 卡死或异常，无法处理新请求
- 所有请求卡在 `busy` 状态，`values=null`

#### 解决方案

**方案 1: 删除持久化存储并重启（推荐）**

```bash
# 1. 停止后端进程
pkill -f "langgraph" 2>/dev/null || true

# 2. 删除持久化存储文件（关键步骤！）
rm -rf .langgraph_api/

# 3. 重启后端
make serve
```

```bash
# 1. 停止后端服务
pkill -f "langgraph" 2>/dev/null || true

# 2. 等待进程完全停止
sleep 2

# 3. 重新启动后端
make serve
```

**方案 2: 清理所有线程**

```bash
# 清理所有线程（谨慎使用，会删除所有历史记录）
curl -s -X POST http://localhost:2024/threads/prune \
  -H "Content-Type: application/json" \
  -d '{"before": "2026-06-03T00:00:00Z"}'
```

**方案 3: 修改 Makefile**

```makefile
clean-state:
	@echo "🧹 清理 LangGraph 状态存储..."
	# 停止后端进程（内存队列会被清空）
	pkill -f "langgraph" 2>/dev/null || true
	# 清理前端缓存
	rm -rf etc/agent-chat-ui/.next/* 2>/dev/null || true
	@echo "✅ 状态已清理，请重新启动后端服务"
```

#### 验证

```bash
# 检查队列状态
curl -s http://localhost:2024/info | grep -E "n_pending|n_running"

# 预期结果（重启后）
# n_pending=0 n_running=0
```

## 一键获取 assistant_id 的方法

### 方法 1: 使用 curl 命令

```bash
# 启动后端后执行
curl -s -X POST http://localhost:2024/assistants/search \
  -H "Content-Type: application/json" \
  -d '{}' | jq -r '.[0].assistant_id'
```

### 方法 2: 添加到 Makefile

可以在 `Makefile` 中添加以下命令：

```makefile
get-assistant-id:
	@echo "🔍 获取 assistant_id..."
	@curl -s -X POST http://localhost:2024/assistants/search \
		-H "Content-Type: application/json" \
		-d '{}' | jq -r '.[0].assistant_id'
	@echo ""
	@echo "📝 请将上述 assistant_id 配置到 etc/agent-chat-ui/.env"
```

然后使用：
```bash
make get-assistant-id
```


## 常见问题 FAQ

### Q1: 为什么 assistant_id 是 UUID 而不是字符串？

**A**: LangGraph 服务层在设计上为每个 graph 实例分配了一个稳定的 UUID，用于区分同一 graph 的不同部署（例如多环境、多版本）。这是 LangGraph 的设计规范，不是 bug。

### Q2: 每次重启后端，assistant_id 会变吗？

**A**: 取决于状态存储：
- 如果状态存储被清理（`make clean-state`），assistant_id 会重新生成
- 如果状态存储保留，assistant_id 会保持不变

### Q3: 如何避免每次手动获取 assistant_id？

**A**: 有两种方式：
1. 在 `install.sh` 中自动获取并配置
2. 使用 LangGraph Cloud 部署，平台会自动处理

### Q4: 使用 LangGraph 是否需要付费？

**A**: LangGraph OSS 是完全开源免费的。只有使用 LangGraph Cloud 托管服务时才需要付费。本地开发不需要任何费用。
