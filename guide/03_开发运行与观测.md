第一个大坑是token消耗。AI Agent在处理复杂任务时，会消耗大量的token，成本很快就上去了。我的解决方案是：

1. 优化prompt，减少不必要的输出
2. 使用更便宜的模型处理简单任务
3. 实现token使用量监控和限制
第二个问题是模型幻觉。有时候Agent会给出看起来很合理但实际上错误的信息，这在运维场景下是很危险的。我的应对策略是：

1. 对关键操作增加人工确认环节
2. 使用多个Agent交叉验证重要信息
3. 建立知识库，让Agent基于已知信息回答问题
第三个问题是稳定性。API服务偶尔会出现故障或限流，影响Agent的正常工作。我的解决方案包括：

1. 配置多个API服务作为备份
2. 实现请求重试机制
3. 添加降级处理逻辑

## 来源文件：`Agent开发规范.md`

对于AI Agent开发，目前业界并没有一个统一的“官方规范”，而是呈现出“国家级标准制定顶层框架、行业联盟/协会细化具体领域、技术社区沉淀主流实践”的多元格局。好消息是，标准化进程非常快，已经有不少可以落地参考的文件和指南。

下面我为你梳理了当前的核心规范体系与实践指南。

🏛️ 官方与行业标准：把握宏观框架与合规底线
“官方规范”主要体现在国家及行业机构发布的标准文件中，为AI智能体的互联互通、安全合规和特定领域应用提供了顶层设计。

国家层级：奠定互联与治理基础

《人工智能 智能体互联》系列国家标准（AIP标准）：即将正式发布，是AI智能体领域的核心国标。它致力于解决不同智能体之间的可信接入、身份认证、协作和交互问题，相当于给每个智能体一个可互相识别的“数字身份证”。

《智能体规范应用与创新发展实施意见》：由国家网信办、发改委、工信部联合发布，从政策层面明确了在19个典型场景推动智能体应用，并对智能体互联协议（AIP）等关键标准的应用提出了要求。

行业/团体标准：聚焦特定领域

T/SIA 065—2025《智能体行为安全要求》：由中国软件行业协会发布，是国内少有的专门针对智能体行为安全的标准。它确立了智能体在设计、开发和运营中应遵循的安全原则，如告知同意、权限申请、用户权益保障等，是所有商业级智能体必须关注的合规基线。

AIIA / T 0219-2025《开发智能体》：由中国信通院牵头，是国内首个针对软件开发场景的智能体技术规范。它从技术能力（感知、记忆、规划、执行）和服务能力（编码、测试、修复等）两大维度，为开发类智能体提供了能力建设指导和技术选型参考。

值得关注的趋势：业界普遍认为，建立AI Agent的标准化体系已成为产业共识。未来几年，围绕AI Agent核心功能构建标准化框架，将是推动行业有序发展的关键。

🛠️ 主流实践与行业共识：掌握具体开发方法
除了正式的官方文件，一些经过大量实践验证的开发范式和流程，已经成为事实上的“行业规范”。

全球通行的技术协议标准
这些协议定义了智能体与工具、模型之间交互的“语言”，是目前开发中最主流的底层标准。

MCP (Model Context Protocol)：可以理解为AI模型与外部工具、数据源交互的“万能插头”。它统一了接口格式，让开发者无需为每个模型单独开发适配层，极大地提升了开发效率和跨模型迁移能力。

A2A (Agent-to-Agent Protocol)：专注解决智能体之间的协作通信问题。

AIP (Agent Interconnection Protocol)：前述的国标，旨在解决更广泛的跨域、可信互联问题，与MCP、A2A形成互补。

广泛采用的生命周期与开发范式
这指的是经过大量项目验证的开发流程和设计模式。

ADLC (Agentic Development Lifecycle) 架构：一种专门为AI智能体设计的开发流程，涵盖了从目标定义、工具集成、记忆系统设计到安全评估、持续进化的全生命周期。

规范驱动开发 (Spec-driven Development)：这是被GitHub等平台推崇的开发范式。它强调在编码前，先由人类和AI共同创建一份详尽、结构化的规范文档，将其作为项目唯一的“事实来源”，从而避免开发跑偏。

高价值实践指南：编写Agent规范的“六大军规”
这是对GitHub上2500个案例进行分析后总结出的实战经验，很有参考价值，核心思想是“像管理实习生一样管理AI”。

结构清晰：规范应像专业文档，明确包含命令、测试方法、项目结构、代码风格、Git工作流、安全边界六大模块。

从“愿景”开始：先给出高层级目标，让AI辅助生成详细规范，而非一上来就陷入技术细节。

规划优先：使用“只读模式”先让AI制定和评审计划，反复推敲确认无误后，再允许其开始编写代码。

任务分治：将宏大目标拆解为可独立实现和测试的微小任务，避免AI因上下文过长而“失忆”或混乱。

📋 落地参考：一份完整的开发与上线流程清单
结合上述所有信息，一个较为稳妥的AI Agent开发与上线路径如下：

定义与建模：明确智能体的角色、目标和边界，完成架构选型（如单Agent还是多Agent协作）。

开发与集成：依据项目类型遵循相应标准（如开发类Agent参考AIIA标准）；为Agent挂载RAG知识库、API工具，并使用MCP等标准协议。

评测与安全：建立专门的评测数据集，多维度评估行为；设置“人在回路”审批节点，并遵循《智能体行为安全要求》等标准构建安全护栏。

上线与合规：

算法备案：具有舆论属性或社会动员能力的服务，务必完成算法备案。

内容安全：生成内容需打上AI标识，并接入敏感词过滤系统。

数据脱敏：确保用户隐私数据不直接参与模型训练。

监控与演进：上线后全链路追踪，收集真实数据，并以此为基础形成“评测->优化->发布”的持续迭代闭环。

from typing import List, TypedDict, Annotated
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from langgraph.graph import StateGraph, END
from langgraph.prebuilt import ToolNode
import json

# ======================
# 1. 定义状态（工作流数据）
# ======================
class TestCaseState(TypedDict):
    query: str                  # 用户需求
    rocm_component: str         # ROCm 组件（rocBLAS / rocRNN / MIOpen 等）
    test_type: str              # 功能 / 性能 / 稳定性 / 边界值
    analysis: str                # 需求解析结果
    test_spec: str               # ROCm 官方规范
    test_cases_raw: str          # 生成的原始用例
    test_cases_final: str        # 格式化后的最终用例
    error: str                   # 错误信息

# ======================
# 2. 初始化 LLM
# ======================
llm = ChatOpenAI(model="gpt-4o", temperature=0.1)  # 低温度 = 稳定、规范

# ======================
# 3. 核心节点（Agent 大脑）
# ======================

# 节点1：解析需求，提取 ROCm 组件 + 测试类型
def analyze_query(state: TestCaseState):
    prompt = ChatPromptTemplate.from_template("""
你是 ROCm 测试专家，解析用户需求，输出 JSON：
{{
  "rocm_component": "rocBLAS/rocRNN/MIOpen/rocSOLVER/rocm-smi 等",
  "test_type": "功能/性能/稳定性/边界值/压力测试",
  "analysis": "详细解析"
}}

用户需求：{query}
""")
    chain = prompt | llm
    res = chain.invoke({"query": state["query"]})

核心机制：.with_config() 挂载 Callback
LangGraph 编译后的图对象支持 .with_config() 方法，它返回一个新的 CompiledGraph，自动把 callback 注入到每次调用中，不需要每次 invoke() 时手动传 ：
Python
Copy
# src/graph.py
from langgraph.graph import StateGraph, END
from langfuse.langchain import CallbackHandler
from src.config import settings

# 1. 初始化 Langfuse handler（从环境变量读取 key）
langfuse_handler = CallbackHandler(
    public_key=settings.LANGFUSE_PUBLIC_KEY,
    secret_key=settings.LANGFUSE_SECRET_KEY,
    host=settings.LANGFUSE_HOST
)

# 2. 定义你的 StateGraph
builder = StateGraph(SelfHealState)
builder.add_node("jenkins", jenkins_node)
builder.add_node("error", error_node)
builder.add_node("gitlab", gitlab_node)
# ... 条件边 ...

# 3. 编译图，并挂载 Langfuse callback
# 这是关键：.with_config() 返回的新图，每次 invoke/stream 都会自动带上 callback
graph = builder.compile(checkpointer=checkpointer).with_config(
    {"callbacks": [langfuse_handler]}
)

## 来源文件：`langraph概念.md`

Step 1：理解 State（状态）—— Agent 的"共享内存"
1.1 什么是 State？
State 是 TypedDict，专门用来定义"整个工作流中所有节点共享的数据结构"。
你可以把它想象成一块全局白板：
每个 Node（节点）都可以读取白板上的全部内容
每个 Node 返回时，只更新白板上自己想改的部分
LangGraph 会自动把更新合并到白板上


在 LangGraph 里，节点的本质就是一个 Python 函数（同步或异步），接收状态并返回状态更新。所以除了调用 LLM，你可以把任何逻辑封装成节点。常见的节点类型可以这样划分：

LLM 节点
调用大模型生成回复或做出决策，比如直接用 ChatOpenAI 的 invoke。

Tool 节点（工具执行节点）
LangGraph 内置了 ToolNode，它能自动解析 LLM 的 tool_calls 并执行对应函数（如搜索、计算器、API 工具）。

python
from langgraph.prebuilt import ToolNode
tool_node = ToolNode(tools)
纯代码/函数节点
完全自定义的 Python 逻辑：数据转换、格式校验、状态清洗、条件计算等。这就是你说的“纯代码算 node”。

外部服务节点
封装任何外部调用：HTTP API、数据库读写、消息队列、文件存储。写一个 async def call_external_service(state) 然后 add_node 即可。

人工介入节点（Human-in-the-loop）
在节点里调用 interrupt()，暂停图运行，等待人工审批或输入后再继续。常和 Command(resume=...) 配合。

子图/子 Agent 节点
把另一个编译好的 StateGraph（或 create_react_agent 返回的 agent）直接当作一个节点加入父图。这样就能实现多 Agent 层级协作，比如一个“规划 Agent”调用多个“执行子 Agent”。

python
sub_agent = create_react_agent(model, tools)
builder.add_node("assistant", sub_agent)   # 子Agent就是节点
分支/路由节点
通常用条件边完成，但也可以写一个节点专门负责决策并返回下一步要去的节点名（例如在 Command(goto=...) 中）。

并行/Send 节点
不是通过 add_node 产生新类型，而是用 Send API 动态生成多个并行任务，比如对列表每项展开处理。

“Skill” 节点
“Skill”一般是封装好的特定能力（比如：处理 PDF、翻译文本、情感分析），实现上就是一个 Python 函数节点或者用 @tool 装饰后被 ToolNode 调用。它不是 LangGraph 的特殊节点类型，只是业务上的抽象。

其他内置/预构建节点
比如 AgentExecutor 模式的节点、ChatAgent 等，但这些本质还是函数节点或子图节点的封装。

总结一下：
没有任何限制，你能写成函数的都能是节点——纯代码、外部服务、人工、子 Agent、多模态处理、记忆更新，通通都可以。这种灵活性正是 LangGraph 用来构建复杂、有状态 Agent 的核心设计。

## 运行态 Graph 能不能 Dump

可以。静态图只能告诉你有哪些节点和边；运行态 trace 可以告诉你每一步实际走到了哪个节点、当时的 `State` 是什么、失败发生在哪一步。

LangGraph 的 checkpoint 机制就是为这个设计的：

- 实时看：使用 `graph.stream()` 读取节点执行事件流。
- 任意时刻看当前状态：使用 `graph.get_state(config)`。
- 事后回放：使用 `graph.get_state_history(config)`。
- 生产可视化：接入 LangSmith。

要使用 `get_state()` 和 `get_state_history()`，编译 graph 时建议配置 checkpointer，例如 `MemorySaver` 或 `PostgresSaver`。

## 实时看：Stream 事件流

`graph.stream()` 会在节点执行时持续返回事件。每个事件一般形如：

```python
{
    "coder": {
        "messages": [...],
        "code": "...",
        "retry": 0,
    }
}
```

示例：

```python
from langgraph.checkpoint.memory import MemorySaver

from src.agent.graph import build_graph


graph = build_graph(model=model, tools=tools)

config = {
    "configurable": {
        "thread_id": "test-case-agent-demo-001",
    }
}

initial_state = {
    "messages": [
        {
            "role": "user",
            "content": "为 vLLM Llama-3.1-8B 生成推理性能测试",
        }
    ],
    "code": "",
    "retry": 0,
}

for event in graph.stream(initial_state, config):
    for node_name, node_state in event.items():
        if node_name == "__end__":
            continue

        print(f"[{node_name}] 执行完成")
        print(f"  State keys: {list(node_state.keys())}")

        snapshot = graph.get_state(config)
        print(f"  Next: {snapshot.next}")
```

注意：如果要完整保存历史，需要 graph 本身配置 checkpointer。只调用 `stream()` 可以看实时事件，但没有 checkpointer 时事后回放能力有限。

## 事后看：Checkpoint 历史回放

执行结束后可以读取完整 checkpoint 历史：

```python
history = list(graph.get_state_history(config))

for i, state in enumerate(history):
    print(f"Step {i}: {state.next}")
    print(f"  Values: {list(state.values.keys())}")
    print(f"  Metadata: {state.metadata}")
```

常见输出：

```text
Step 0: ('coder',)
  Values: ['messages', 'code', 'retry']
Step 1: ('checker',)
  Values: ['messages', 'code', 'retry']
Step 2: ()
  Values: ['messages', 'code', 'retry']
```

这些历史状态可用于：

- 失败复盘。
- 自动修复前后的 state 对比。
- 统计每个节点耗时和调用次数。
- 归档每次用例生成过程。
- Debug 路由为什么走到 `retry` 或 `giveup`。

## 推荐封装：Dump 完整执行轨迹

可以把下面函数放到 `test_debug.py`、`cli.py`，或后续单独做成 `src/agent/tracing.py`。

```python
import json
import os
from datetime import datetime
from pathlib import Path
from typing import Any


def _safe_json_value(value: Any, max_length: int = 2000) -> Any:
    """把复杂对象转成适合写入 JSON 的值。"""
    if isinstance(value, (str, int, float, bool)) or value is None:
        text = value
    else:
        text = str(value)

    if isinstance(text, str) and len(text) > max_length:
        return text[:max_length] + "...<truncated>"
    return text


def dump_execution_trace(graph, config, output_dir: str = "./traces") -> None:
    """Dump 完整执行轨迹，便于事后回放和审计。"""
    trace_dir = Path(output_dir)
    trace_dir.mkdir(parents=True, exist_ok=True)

    # 1. 保存静态图结构
    graph_mmd = graph.get_graph(xray=True).draw_mermaid()
    (trace_dir / "graph.mmd").write_text(graph_mmd, encoding="utf-8")

    # 2. 保存每步 State
    history = list(graph.get_state_history(config))
    for index, state in enumerate(history):
        next_name = "_".join(state.next) if state.next else "end"
        filename = trace_dir / f"step_{index:03d}_{next_name}.json"

        payload = {
            "step": index,
            "next_node": state.next,
            "values": {
                key: _safe_json_value(value)
                for key, value in state.values.items()
            },
            "metadata": state.metadata,
        }
        filename.write_text(
            json.dumps(payload, ensure_ascii=False, indent=2, default=str),
            encoding="utf-8",
        )

    # 3. 保存摘要
    summary = {
        "total_steps": len(history),
        "nodes_visited": [state.next for state in history],
        "final_state_keys": list(history[-1].values.keys()) if history else [],
        "timestamp": datetime.now().astimezone().isoformat(),
    }
    (trace_dir / "summary.json").write_text(
        json.dumps(summary, ensure_ascii=False, indent=2, default=str),
        encoding="utf-8",
    )

    print(f"Trace saved to {trace_dir}")
    print("  - graph.mmd")
    print(f"  - step_*.json ({len(history)} checkpoints)")
    print("  - summary.json")
```

调用方式：

```python
thread_id = "test-case-agent-demo-001"
config = {
    "configurable": {
        "thread_id": thread_id,
    }
}

for _event in graph.stream(initial_state, config):
    pass

dump_execution_trace(
    graph,
    config,
    output_dir=f"traces/{thread_id}",
)
```

生成内容：

- `graph.mmd`：静态图结构。
- `step_000_*.json`：每个 checkpoint 的状态快照。
- `summary.json`：执行摘要和时间线。

## Checkpointer 选择

| Checkpointer | 适用场景 | 特点 |
|---|---|---|
| `MemorySaver` | 本地开发、单进程调试 | 零外部依赖，进程退出后丢失 |
| `SqliteSaver` | 本地持久化调试 | 文件级持久化，适合 CLI |
| `PostgresSaver` | 生产环境、多进程服务 | 可持久化、可共享、适合审计 |
| LangSmith | 团队可视化、生产追踪 | UI 友好，能看 LLM 调用、token、时间线 |

P0 建议先用 `MemorySaver` 或本地 dump JSON。等 Agent 稳定后，再接 Postgres 或 LangSmith。

生产环境推荐接 LangSmith。只需要在运行前配置环境变量：


然后正常执行：

```python

```

可以在 LangSmith 页面看到：

- 节点执行时间线。
- 每次 LLM 调用输入输出。
- token 消耗。
- tool call 参数。
- 错误栈。
- 消息流和中间状态。

注意：LangSmith 会上传 trace 数据。涉及 AMD 内部路径、token、模型 URL、日志内容时，需要先确认数据合规策略。

## 核心 API 速查

| API | 作用 | 调用时机 |
|---|---|---|
| `graph.get_graph().draw_mermaid()` | 导出静态 Mermaid 图 | 开发/评审 |
| `graph.get_graph().draw_mermaid_png()` | 导出 PNG | 开发/文档 |
| `graph.stream(state, config)` | 实时事件流 | 执行时 |
| `graph.get_state(config)` | 获取当前完整 State | 执行中或 interrupt 后 |
| `graph.get_state_history(config)` | 获取历史 State | 事后复盘 |
| `graph.update_state(config, values)` | 手动修改 State | 人工介入或恢复 |
| `state.next` | 下一个要执行的节点 | 判断进度 |
| `state.values` | 当前 State 数据 | dump / debug |
| `state.metadata` | checkpoint 元数据 | 审计 / 统计 |

## 建议落地顺序

1. 静态图：先用 `src.agent.visualize` 导出 `agent_architecture.mmd`。
2. 本地实时调试：用 `graph.stream()` 打印节点事件。
3. 本地事后审计：实现 `dump_execution_trace()`，保存 `traces/<thread_id>/`。
4. 多轮修复闭环：每轮修复前后 dump state，用于对比。
5. 生产追踪：接 LangSmith 或 PostgresSaver。

## 结论

LangGraph 的运行态是可以透明追踪的：

- 想实时看，用 `stream()`。
- 想看当前完整状态，用 `get_state(config)`。
- 想事后回放，用 `get_state_history(config)`。
- 想团队可视化，用 LangSmith。

对 `Test Case Agent` 来说，推荐先做本地 JSON trace dump。这样每次“需求 -> planning -> generate -> validate -> execute -> repair”的状态都能归档，后续排查 hallucination、失败修复逻辑和 token 成本会更容易。

Token 预算应该怎么设？
既然 Input/Output 价格不同，预算不能只看总数，建议分别监控：
Python
Copy
class TokenBudget:
    input_budget: int = 100_000      # 输入上限
    output_budget: int = 20_000    # 输出上限（更严格，因为更贵）
    input_used: int = 0
    output_used: int = 0

    def check(self, usage: dict):
        # usage 来自 LLM API 返回的 usage_metadata
        self.input_used += usage["prompt_tokens"]
        self.output_used += usage["completion_tokens"]
        
        if self.input_used >= self.input_budget:
            raise BudgetExceeded("Input token 预算耗尽")
        if self.output_used >= self.output_budget:
            raise BudgetExceeded("Output token 预算耗尽，停止生成")
为什么 Output 预算要更紧？
Output 单价高 4 倍
Agent 容易陷入"废话循环"（反复解释不修改代码）
代码生成任务中，Output 长度直接决定费用
一句话总结
Input 是喂进去的资料（便宜但容易膨胀），Output 是吐出来的结果（贵且直接决定成本）。Agent 省钱的关键是：压缩 Input 上下文 + 限制 Output 长度 + 减少无效生成轮次。

Input token 怎么算？
Input = 你发给模型的所有内容，按文本分词后的 token 数累加。
具体包括：
Table
内容	是否算 Input	说明
System Prompt	✅	你的角色定义、风格指南、ROCm 规范
User Message	✅	用户说的"生成一个 rocblas sgemm 测试"
历史对话	✅	之前轮次的 User + Assistant 内容（Agent 多轮修复时越滚越大）

Function Schema	✅	你定义的 tools/functions 的 JSON Schema 描述

def generator_node(state):
    response = llm.invoke(state["messages"])
    
    # 累加预算
    state["token_input_used"] += response.usage_metadata["input_tokens"]

实际开发中的关键细节
1. 流式输出（streaming）也能拿到
即使你用 stream() 或 astream()，最后一块 chunk 通常附带 usage：
Python
Copy
async for chunk in llm.astream("..."):
    print(chunk.content, end="")
    
# 最后一块
# chunk.usage_metadata = {'input_tokens': 100, 'output_tokens': 50}
2. 工具调用时 Output 被"吃掉"很多
如果模型输出的是：
JSON
Copy
{"tool_calls": [{"name": "write_file", "arguments": {"path": "test.py", "content": "import pytest\n..."}}]}
整个 JSON 字符串都算 Output token，包括字段名 tool_calls、name、arguments。
3. Reasoning 模型要特别小心
Claude 3.7 Sonnet（Thinking 模式）、DeepSeek-R1、o1 系列：
Input：你的问题
Output：思考过程（可能 3000 tokens）+ 最终答案（可能 500 tokens）
计费：按总 Output 收，思考过程不免费
建议： 对 Test Case Agent 的代码生成任务，如果不需要深度推理，关闭 CoT/Thinking 模式，能省 50%+ Output token。
一句话总结
Input 是你喂进去的（上下文越大越贵），Output 是模型吐出来的（单价更高）。API 响应的 usage.completion_tokens 就是 Output 数，LangChain 里通过 response.usage_metadata["output_tokens"] 直接取，务必在每次 LLM 调用后累加到预算计数器里。
