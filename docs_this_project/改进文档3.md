# TheRock Test Agent 全自动 Debug/Repair Loop 设计与施工方案

## 1. 目标

本方案要求 **全自动**，不是半自动。

目标是把当前 TheRock Test Agent 从：

```text
后台测试执行 + 失败集重跑
```

升级为：

```text
后台测试执行 + 自动读日志 + 自动分析 bug + 自动修复可修项 + 自动记录证据 + 自动进入下一轮验证
```

也就是完整的：

```text
Test Loop + Debug Loop + Repair Loop
```

每一轮测试结束后，runner 不应立即机械进入下一轮，而应自动执行：

```text
round_end
  -> collect_round_failures
  -> read_global_state
  -> read_progress
  -> read_failure_reports
  -> read_stdout_stderr_logs
  -> classify_failures
  -> generate_round_analysis
  -> generate_repair_plan
  -> apply_safe_repairs
  -> record_repair_actions
  -> continue_next_round
```

用户不需要手工执行 `/therock-debug-round`、`/therock-repair-round`、`/therock-resume` 才能进入下一轮。这些命令可以保留，但只作为人工复盘、重跑某个阶段或排障入口。

## 2. 背景问题

实测 run `20260710_153107_1747ce36` 证明当前能力已经包括：

- 后台运行正常。
- `logs/` 已经保存原始 stdout/stderr。
- `progress.jsonl` 能记录 round/task 事件。
- `global_state.json` 能记录结果和 loop 状态。
- Loop Engineering 生效，失败集能从 6 个 runnable task 收缩到 2 个失败 task。

但当前两轮之间只有状态机动作：

```text
写 round_end
更新 failed_task_history
比较失败集是否稳定
更新 stable_failed_count
把失败集放进下一轮 pending
立即开始下一轮
```

缺少：

- 自动读取失败组件日志。
- 自动识别 `prettytable`、`lit` 这类依赖缺失。
- 自动修复可修项。
- 自动判断修复后是否继续重跑。
- 自动记录修复命令、输出、diff 和原因。
- 自动把 debug/repair 结论写入 run 目录。

本次 run 的真实问题是：

```text
libhipcxx_hiprtc-quick -> missing_dependency: lit
sanity-quick -> ModuleNotFoundError: No module named 'prettytable'
```

这类问题应该自动修复后继续下一轮，而不是盲目重复 4 轮，也不是直接最终 blocked。

## 3. 采纳的评审建议与修正

除“修复前必须依赖 OpenCode ask 权限确认”的半自动建议外，其余建议基本有效，但要改写成全自动版本。

### 3.1 日志按轮次保存是硬性前提

自动 debug/repair 必须依赖精确的 round 级日志。日志文件必须按轮次命名：

```text
logs/<task_id>.round<N>.stdout.log
logs/<task_id>.round<N>.stderr.log
```

例如：

```text
logs/sanity-quick.round1.stdout.log
logs/sanity-quick.round1.stderr.log
logs/sanity-quick.round2.stdout.log
logs/sanity-quick.round2.stderr.log
```

如果日志不带 round，第二轮会覆盖第一轮，自动分析将失去依据。

当前 runner 已经使用 `logs/<task_id>.round<N>.*.log`，后续实现必须把这作为不可回退的不变量。

### 3.2 全自动状态不能使用 `waiting_for_debug`

评审中提到的 `waiting_for_debug` 适合半自动模式，但不符合本方案的全自动目标。

全自动模式下，round 结束后状态应短暂进入内部阶段：

```text
auto_debug_running
auto_repair_running
```

这些阶段只写入 `global_state.json` 和 `progress.jsonl`，不暂停等待用户输入。状态机应当自动完成：

```text
round_end
  -> auto_debug_running
  -> auto_repair_running
  -> next_round_after_repair
```

如果自动分析或修复失败，才进入：

```text
needs_manual_approval
```

### 3.3 修复可能引入回归，需要记录而非阻塞

自动修复可能改变 Python 环境或测试脚本，例如安装 `prettytable`、`lit`，或 patch `build_tools/**`。这可能影响已经 pass 的任务。

MVP 阶段接受“下一轮只重跑失败/修复相关 task”的取舍，但必须记录：

- 修复动作。
- 修复命令。
- 修复前后环境摘要。
- 修复涉及的 task。
- 是否建议后续做全量回归。

报告中应明确：

```text
repair changed environment; only failed tasks were revalidated
full regression recommended if this run is used as release evidence
```

### 3.4 结构化失败列表必须由 runner 直接生成

自动 debug 不应临时从多份文件猜失败集。runner 在每个 round 结束时必须直接写出本轮失败列表：

```json
{
  "schedule": {
    "failed_tasks_this_round": ["sanity-quick"],
    "blocked_tasks_this_round": ["libhipcxx_hiprtc-quick"],
    "round_pending_tasks": [],
    "round_failed_tasks": ["sanity-quick", "libhipcxx_hiprtc-quick"]
  }
}
```

同时生成：

```text
round_analysis/round<N>_inputs.json
debug/round<N>_failure_index.json
```

这样 debug/repair 模块可以直接读取结构化输入，不需要从聊天上下文或日志文件名反推。

## 4. 总体架构

全自动闭环由两层组成：

```text
OpenCode overlay 层
  ├── agents
  ├── commands
  ├── skills
  └── markdown 规则

Python runner 层
  ├── deterministic test execution
  ├── state machine
  ├── log collection
  ├── failure classification
  ├── repair execution
  └── audit/report generation
```

### 4.1 OpenCode 原生能力如何利用

OpenCode 原生能力通过项目 overlay 体现：

- `.opencode/agents/*.md`：定义 TheRock debug/repair 角色、权限和行为边界。
- `.opencode/commands/*.md`：提供人工触发和复盘入口。
- `.opencode/skills/*.md`：沉淀日志分类、修复策略、ROCm/TheRock 领域知识。
- Markdown 规则：让 OpenCode agent 清楚知道 run 目录结构、失败分类、允许修改范围和记录要求。

但全自动主链路不能依赖后台 Python 进程去“聊天式调用 OpenCode”。原因：

- OpenCode agent 是交互式协调者，不适合被长时间后台进程阻塞调用。
- 权限审批、上下文、工具调用生命周期较复杂。
- 需要无人值守时，OpenCode 必须能确定性编排 debug/repair；runner 只负责测试执行、日志和失败索引。

因此最终设计是：

```text
OpenCode 执行全自动 debug/repair 编排
Python runner 执行测试、checkpoint、日志和失败输入索引
```

这不是放弃 OpenCode，而是把职责拆清：

- skill 文档
- command 文档
- agent 行为约束
- run 目录结构
- debug/repair 报告格式

runner 按这些规则执行测试并在失败轮次后交出结构化输入；OpenCode debugger/repairer 按这些规则生成分析、建议和可见审计的 safe repair。

### 4.2 自动主链路

默认全自动流程：

```text
/therock-run
  -> start-kv
  -> _run-existing
  -> run_loop
      round 1 execute tasks
      round 1 failed set generated
      write round inputs and failure index
      status=waiting_for_opencode_debug
  -> OpenCode /therock-debug-round run_id=<run_id> round=1
      round_analysis generated by OpenCode
  -> OpenCode /therock-repair-round run_id=<run_id> round=1 apply=safe
      safe_auto repair actions recorded
  -> /therock-resume or next verification run
      round 2 execute repaired failed set
      ...
      until pass / stable failure / max rounds / unrecoverable
```

## 5. 新增运行状态

`global_state.json` 增加：

```json
{
  "debug_repair": {
    "mode": "auto",
    "max_repair_rounds": 3,
    "current_repair_round": 0,
    "last_analysis": "round_analysis/round1.json",
    "last_repair_plan": "repairs/round1_repair_plan.json",
    "last_repair_status": "success",
    "unrepairable_tasks": [],
    "repaired_tasks": ["sanity-quick", "libhipcxx_hiprtc-quick"]
  }
}
```

新增 CLI 参数：

```text
debug_repair=<auto|off>
max_repair_rounds=<n>
repair_policy=<safe|ask|off>
```

默认建议：

```text
debug_repair=auto
repair_policy=safe
max_repair_rounds=3
```

含义：

- `debug_repair=auto`：每轮失败后自动分析和修复。
- `debug_repair=off`：保持旧行为，只重跑失败集。
- `repair_policy=safe`：只执行安全修复，例如 pip 安装缺失包、patch `build_tools/**`。
- `repair_policy=ask`：生成修复计划但不执行，等待用户确认。此模式作为排障备用，不是默认。
- `repair_policy=off`：只分析，不修复。

## 7. 自动 Debug/Repair 状态机

### 7.1 当前状态机

当前流程：

```text
round_start
  -> execute tasks
  -> round_end
  -> compare failed set
  -> next round
```

### 7.2 新状态机

改为：

```text
round_start
  -> execute tasks
  -> round_end
  -> write_round_inputs
  -> waiting_for_opencode_debug
  -> OpenCode therock-debugger
  -> OpenCode therock-repairer apply=safe
  -> record_repair_actions
  -> decide_next_round
  -> next round
```

伪代码：

```python
while current_loop < max_rounds:
    round_no = current_loop + 1
    run_round(round_no)

    failed_set = collect_failed_set(round_no)
    if not failed_set:
        final_status = "passed"
        break

    if debug_repair == "opencode":
        write_round_failure_index(state, round_no, failed_set)
        final_status = "waiting_for_opencode_debug"
        break

    next_tasks = failed_set

    update_stable_failed_count(failed_set)
    if stable_failed_count >= stable_threshold:
        final_status = "failed"
        break
```

关键点：

- 失败后 runner 只生成 OpenCode 分析所需输入。
- 问题分析、debug 建议和 `round_analysis` 由 OpenCode debugger 完成。
- 修复执行由 OpenCode repairer 完成，并写 `repairs/**` 审计。
- OpenCode repairer 修复成功后必须进入下一轮验证。
- 只有不可修复且失败集稳定，才最终 failed。
- `blocked` 不等于最终失败，先尝试修复。

## 8. OpenCode 分类规则

runner 只提取 evidence signals，例如缺失模块、关键日志片段、路径硬编码迹象；最终 `classification`、`root_cause`、`next_action`、`repair_policy` 由 OpenCode debugger 根据 skill 和上下文生成。

| 分类 | 识别信号 | 自动修复 | 下一步 |
|------|----------|----------|--------|
| `missing_python_dependency` | `ModuleNotFoundError: No module named X` | `python3 -m pip install X` | 重跑 |
| `missing_dependency` | `missing_dependency: X` | 判断是否 Python 包；能 pip 则 pip | 重跑 |
| `network_transient` | `Connection timed out` / `Temporary failure in name resolution` / `Could not resolve host` / `HTTP 5xx` / `Connection reset by peer` / TLS 临时失败 | 不 patch；允许有限次数 transient retry | 重跑，若重复出现则标记环境问题 |
| `cmake_configuration_error` | `CMake Error` / `Could not find package` / `find_package` / `CMAKE_PREFIX_PATH` / `No CMAKE_* found` / `ninja: error` | 只允许修 wrapper/env/build_tools 参数；不改源码或产物 | 生成修复计划，安全范围内才修 |
| `missing_runtime_library` | `cannot open shared object file` | 检查 `LD_LIBRARY_PATH` / artifacts | 重跑或 manual |
| `runtime_path_error` | `No such file or directory` 指向 ROCm/runtime 路径、`LD_LIBRARY_PATH`、`ROCM_PATH`、`HIP_PATH` | 只允许修 wrapper/env/build_tools 参数 | 重跑或 manual |
| `script_bug` | wrapper 正常、脚本参数/路径错误 | 若在 `build_tools/**`，自动 patch | 重跑 |
| `permission_or_sudo` | sudo unavailable / permission denied | 检查 askpass/cache | 重跑或 manual |
| `disk_space_error` | `No space left on device` / `Disk quota exceeded` / `ENOSPC` | 不 patch；记录并提示清理空间 | 清理空间后重跑 |
| `timeout` | `TimeoutExpired` / `timed out` / `timeout after` / `SIGTERM` | 不直接扩大超时；先判定是否 transient 或 hang risk | 重跑或 manual |
| `gpu_runtime_error` | `HSA_STATUS_ERROR` / `PERMISSION_FAULT` | 不直接修 | quarantine/manual |
| `gpu_hang_risk` | ring timeout / GPU reset | 不自动扩大 | quarantine/manual |
| `test_assertion_failure` | `AssertionError` / `assert ... failed` / gtest `FAILED` / pytest failure | 不自动 patch 组件源码 | retry until stable，生成 issue/debug report |
| `real_component_failure` | 组件测试断言、结果错误或稳定复现失败 | 不自动 patch 组件源码 | retry until stable |

### 8.1 `next_action` 生成规则

Debug 阶段必须为每个失败 task 生成明确的 `next_action`，供后续 repair policy 使用：

| 分类 | `next_action` | 说明 |
|------|---------------|------|
| `missing_python_dependency` | `repair_then_retry` | 可进入自动 pip 安装策略 |
| `missing_dependency` | `repair_then_retry` 或 `inspect_dependency_then_retry` | 只有确认是 Python 包时自动安装 |
| `network_transient` | `retry_transient` | 不改文件；有限次数自动重试 |
| `cmake_configuration_error` | `plan_wrapper_or_build_tools_repair` | 只生成计划；仅 wrapper/env/build_tools 范围可修 |
| `missing_runtime_library` / `runtime_path_error` | `inspect_runtime_path_then_retry` | 优先检查 `LD_LIBRARY_PATH`、`ROCM_PATH`、artifacts 路径 |
| `permission_or_sudo` | `check_sudo_policy_then_retry` | 检查 `none/cache/askpass` 和 sudo agent |
| `disk_space_error` | `needs_environment_cleanup` | 不自动删除文件 |
| `timeout` | `retry_or_timeout_review` | 不盲目扩大超时 |
| `gpu_runtime_error` / `gpu_hang_risk` | `quarantine_or_manual` | 不自动扩大 GPU 风险范围 |
| `test_assertion_failure` / `real_component_failure` | `retry_until_stable` | 稳定后生成问题报告 |

### 8.2 Repair policy 分层

自动 repair 只能执行低风险动作：

| 层级 | 允许动作 | 例子 |
|------|----------|------|
| `safe_auto` | Python 依赖安装、明确 transient retry、写 run 目录产物 | `pip install prettytable`、网络抖动重试 |
| `safe_plan_only` | 生成修复计划，但不直接改源码或产物 | CMake/env/runtime path 诊断 |
| `safe_patch_limited` | 仅 patch `build_tools/**` 或 wrapper/env 生成逻辑，并记录 diff | 修正测试脚本参数、路径拼接 |
| `manual_required` | 涉及 ROCm 源码、组件源码、编译产物、GPU hang 风险扩大 | 不自动执行 |

当前实现阶段先完成分类和 `next_action`，只为后续 repair policy 提供结构化输入；不在 debug 阶段直接执行修复。

## 9. 自动修复权限边界

### 9.1 允许自动修复

允许：

- 安装 Python 依赖：
  - `python3 -m pip install prettytable`
  - `python3 -m pip install lit`
- 写 run 目录：
  - `round_analysis/**`
  - `repairs/**`
  - `debug/**`
  - `memory/**`
- 修改 TheRock 测试脚本：
  - `/home/zs/TheRock/build_tools/**`
  - `/home/zx/TheRock_CI测试流程/TheRock/build_tools/**`
- 修改本项目 overlay：
  - `.opencode/**`
  - `.opencode/tools/therock_agent/**`
  - `docs_this_project/**`

### 9.2 禁止自动修复

默认禁止：

- 修改 `dist/rocm/bin`
- 修改 `dist/rocm/lib`
- 修改 ROCm 子模块源码
- 修改 TheRock 组件源码
- 修改编译产物
- 写 sudo 密码或敏感配置
- 强行执行 `gpu_hang_risk=true` 任务

如必须修改这些内容，自动流程应标记为：

```text
needs_manual_approval
```

并写入 `round_analysis/round<N>.md`，不自动执行。

## 10. 如何利用 OpenCode 原生能力

全自动主链路由 runner 执行，但 OpenCode 原生能力仍然要沉淀并可复用：

### 10.1 新增 Skill

```text
.opencode/skills/therock-debugging/SKILL.md
```

内容包括：

- run 目录结构。
- 如何读取 `global_state.json`。
- 如何从 `progress.jsonl` 找失败 task。
- 如何读取 `failures/` 与 `logs/`。
- 失败分类规则。
- 修复权限边界。
- 修复动作记录规范。
- issue/词汇库对照规则。

### 10.2 新增 Agent

```text
.opencode/agents/therock-debugger.md
```

用途：

- 事后复盘自动生成的 round analysis。
- 用户要求人工分析时，读取同样的 run 产物。
- 人工介入修复时，遵守同样权限边界。

### 10.3 新增 Command


这些命令不再是主流程必须步骤，而是：

## 11. OpenCode 自动修复执行与审计

自动修复执行交给 OpenCode 原生能力完成，而不是继续塞进后台 runner。

职责边界：

```text
Python runner
  -> 执行测试
  -> 保存 round 日志
  -> 生成 round_analysis
  -> 输出 classification / next_action / repair_policy

OpenCode debugger
  -> 读取 round inputs / failure index / logs
  -> 生成 round_analysis
  -> 生成 debug log excerpt

OpenCode repairer
  -> 读取 round_analysis
  -> 生成 repair plan
  -> 按 repair_policy 执行 safe_auto 动作
  -> 写 repairs 审计产物
  -> 提示用户 resume 或重跑验证
```

runner 侧只保留测试执行和输入索引生成，不保留 debug 分析器。

不新增 runner 侧 `debug_round.py` / `repair.py` 作为默认实现，避免后台 Python 进程绕过 OpenCode 权限、上下文和人工可见审计。

### 11.1 OpenCode `therock-debugger`

职责：

- 读取 runner 生成的 `round_analysis/round<N>_inputs.json`。
- 读取 runner 生成的 `debug/round<N>_failure_index.json`。
- 读取 failure report 与 round 日志。
- 生成 `round_analysis/round<N>.json`。
- 生成 `round_analysis/round<N>.md`。
- 生成 `debug/round<N>_log_excerpt.md`。
- 输出分类、根因、`next_action`、`repair_policy`。

### 11.2 OpenCode `therock-repairer`

职责：

- 读取 `round_analysis/round<N>.json`。
- 读取 `debug/round<N>_log_excerpt.md` 和 failure report。
- 生成 `repairs/round<N>_repair_plan.json`。
- 生成 `repairs/round<N>_repair_plan.md`。
- 当用户通过 `/therock-repair-round ... apply=safe` 触发时，执行 `repair_policy=safe_auto` 动作。
- 写 `repairs/round<N>_actions.jsonl`。
- 写 `repairs/round<N>_tool_calls.jsonl`。
- 写 `repairs/round<N>_environment_changes.md`。
- 如修改 `build_tools/**`，写 diff patch。

OpenCode repairer 只允许执行：

- `missing_python_dependency` 的明确 Python 包安装。
- `network_transient` 的有限重试建议或触发 resume。
- `safe_patch_limited` 范围内的 `build_tools/**` 或 overlay wrapper/env 逻辑修复。

以下只生成计划，不执行：

- `cmake_configuration_error`
- `missing_runtime_library`
- `runtime_path_error`
- `permission_or_sudo`
- `disk_space_error`
- `timeout`
- `gpu_runtime_error`
- `gpu_hang_risk`
- `test_assertion_failure`

## 16. 具体施工步骤

### Phase 1：Runner 基础不变量

这是最高优先级。没有稳定的 round 级日志和结构化失败列表，后续 debug/repair 没有可靠输入。

1. 保证日志不可覆盖：
   - `logs/<task_id>.round<N>.stdout.log`
   - `logs/<task_id>.round<N>.stderr.log`
2. 在 round end 写入结构化失败列表：
   - `schedule.failed_tasks_this_round`
   - `schedule.blocked_tasks_this_round`
   - `schedule.round_failed_tasks`
3. 生成 debug 输入索引：
   - `round_analysis/round<N>_inputs.json`
   - `debug/round<N>_failure_index.json`
4. 在 `progress.jsonl` 写入：
   - `round_failure_index_written`
   - `waiting_for_opencode_debug`
   - `opencode_debug_written`
   - `opencode_repair_start`
   - `opencode_repair_end`
5. 确保 resume 不破坏 round 历史：
   - 已完成 task 不重跑。
   - 上一轮失败/blocked task 作为下一轮候选。
   - 已生成日志、analysis、repair 产物不被覆盖。

### Phase 2：OpenCode 全自动分析模块

1. 新增/更新 `.opencode/agents/therock-debugger.md` 与 `.opencode/commands/therock-debug-round.md`。
2. OpenCode debugger 优先读取结构化输入：
   - `round_analysis/round<N>_inputs.json`
   - `debug/round<N>_failure_index.json`
   - `global_state.json`
   - `progress.jsonl`
3. 按索引读取：
   - `failures/*.json`
   - `logs/*.round<N>.stdout.log`
   - `logs/*.round<N>.stderr.log`
4. 由 OpenCode 实现分类和建议：
   - `ModuleNotFoundError`
   - `missing_dependency`
   - `ImportError`
   - `cannot open shared object file`
   - GPU/runtime 关键字
5. 输出：
   - `round_analysis/round<N>.json`
   - `round_analysis/round<N>.md`
   - `debug/round<N>_log_excerpt.md`

### Phase 3：OpenCode repairer 自动修复执行

1. 不新增 runner 侧 `repair.py`；修复执行由 `.opencode/agents/therock-repairer.md` 和 `/therock-repair-round` 完成。
2. 支持 Python 包安装：
   - `lit`
   - `prettytable`
   - 其他从 `ModuleNotFoundError` 自动识别的包
3. 支持 `build_tools/**` patch 计划与记录：
   - 修复前 diff
   - 修复后 diff
   - patch 文件
4. 记录环境变更：
   - Python 版本。
   - pip freeze 摘要。
   - git diff 摘要。
   - 修复后只验证失败任务的风险提示。
5. 输出：
   - `repairs/round<N>_repair_plan.json`
   - `repairs/round<N>_repair_plan.md`
   - `repairs/round<N>_actions.jsonl`
   - `repairs/round<N>_tool_calls.jsonl`
   - `repairs/round<N>_environment_changes.md`

### Phase 4：集成 run_loop

在 `run_loop` 的 round end 后加入 OpenCode 等待点，不在 runner 内执行 debug 或 repair：

```python
if current_failed_sorted and debug_repair == "opencode":
    write_round_failure_index(state, round_no, current_failed_sorted)
    state.final_status = "waiting_for_opencode_debug"
    # OpenCode continues:
    # /therock-debug-round run_id=<run_id> round=<N>
    # /therock-repair-round run_id=<run_id> round=<N> apply=safe
    # /therock-resume run_id=<run_id>
```

注意：

- OpenCode repairer 修复成功后必须提示 `/therock-resume` 或重新进入下一轮验证。
- 修复失败但仍可重试的任务继续进入 retry。
- 不可修任务进入 `unrepairable_tasks`。
- 不进入 `waiting_for_debug`；只有自动流程无法安全继续时才进入 `needs_manual_approval`。

### Phase 5：OpenCode overlay

新增：

```text
.opencode/skills/therock-debugging/SKILL.md
.opencode/agents/therock-debugger.md
.opencode/agents/therock-repairer.md
.opencode/commands/therock-debug-round.md
.opencode/commands/therock-repair-round.md
```

这些主要用于：

- 人工复盘自动分析。
- 手工重跑 debug/repair。
- 让 OpenCode 原生理解 TheRock run 目录和 debug/repair 语义。

### Phase 6：报告、安装与文档

1. `summary.json` / `failures.json` 提供 reporter 所需结构化字段。
2. `therock-reporter` 生成的 Markdown 增加 round analysis 和 repair 摘要链接。
3. failure JSON 增加 runner evidence；failure Markdown 增加 OpenCode 分类、关键日志片段、修复历史。
3. `install.sh` 校验新增 agent/command/skill。
4. README 增加全自动 debug/repair 说明。
5. `docs_this_project/需求.md` 同步新状态机。
6. 测试文档补充依赖修复和回归风险说明。

## 19. 风险与缓解

| 风险 | 说明 | 缓解 |
|------|------|------|
| pip install 改变全局环境 | 可能影响其他测试 | 记录所有 pip action；后续支持 venv |
| 修复引入回归但未被本轮发现 | 下一轮默认只重跑失败或修复相关 task，已通过任务不会自动全量回归 | 在报告中标记验证范围；记录环境变更；release 证据需要额外全量回归 |
| 自动安装包失败 | 网络或权限问题 | 记录失败，任务进入 retry/needs_manual_approval |
| 自动 patch build_tools 误改 | 可能影响测试脚本 | 只允许 `build_tools/**`，必须记录 diff |
| GPU hang 类问题不可自动修 | 自动修复风险高 | quarantine/manual，不扩大范围 |
| 日志过大 | 分析上下文过多 | 只提取失败 task 关键片段 |
| 修复后仍失败 | 可能是真实组件问题 | 继续 retry until stable，生成 issue/debug report |

## 20. 最终结论

本方案改为 **全自动 Debug/Repair Loop**。

最终形态：

```text
执行测试
  -> 保存日志
  -> runner 提取 evidence JSON
  -> OpenCode debugger 自动分类
  -> OpenCode repairer 自动修复可修项
  -> 自动记录所有动作
  -> 自动进入下一轮验证
  -> 直到通过 / 稳定失败 / 不可修复
```

`/therock-debug-round` 和 `/therock-repair-round` 仍保留，但只作为人工复盘工具，不再是主流程必需步骤。
