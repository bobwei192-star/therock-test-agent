极简版 Loop 方案（Zero 回归，Pure 重试）
核心逻辑（一句话）
每次只跑上一轮失败的用例，直到连续 3 轮失败的用例集合完全一样，则停止。

输入
30 个组件 × 2 种测试类型 = 60 个测试任务

每个任务一个唯一 ID（如 rocblas-quick, miopen-standard）

执行流程（4 步循环）
graph TD
    A[开始: 初始任务集 = 全部60个] --> B[执行当前任务集]
    B --> C[收集本轮失败的任务列表]
    C --> D{本轮失败数 == 0?}
    D -->|是| E[🎉 全部通过，Loop终止]
    D -->|否| F{连续3轮失败列表完全一致?}
    F -->|是| G[🛑 停止Loop，报告顽固失败]
    F -->|否| H[更新任务集 = 本轮失败列表]
    H --> I[重置连续一致计数器? 不，需判断]
    I --> B

关键判定规则：

每一轮执行后，记录本轮失败的任务 ID 集合（如 {A, B, C}）。

比较本轮与上一轮的失败集合是否完全相同（大小相等且元素一致）。

如果相同，则 stable_count += 1；否则 stable_count = 0。

当 stable_count == 3 时，停止循环，认为剩下的失败是“顽固失败”。

特别注意：如果本轮失败数变为 0，直接停止（全部通过）。

伪代码（最简单粗暴版本）
python
failed_set = all_60_tasks          # 初始任务
stable_count = 0
last_failed_set = None

while True:
    # 1. 执行当前 failed_set
    results = run(failed_set)
    
    # 2. 提取本轮失败的任务
    current_failed_set = {t for t in results if t.status == FAIL}
    
    # 3. 判断是否全部通过
    if len(current_failed_set) == 0:
        print("🎉 所有任务通过！")
        break
    
    # 4. 判断是否稳定（连续3轮完全相同）
    if current_failed_set == last_failed_set:
        stable_count += 1
    else:
        stable_count = 0
    
    if stable_count >= 3:
        print(f"🛑 顽固失败集已收敛：{current_failed_set}")
        break
    
    # 5. 准备下一轮：只跑当前失败的任务
    last_failed_set = current_failed_set
    failed_set = current_failed_set
    
    print(f"🔄 下一轮仅重试 {len(failed_set)} 个任务...")

轮次	执行任务数	本轮失败数	失败集合	是否连续一致？	稳定计数	决策
1	60	10	{A, B, C, D, E, F, G, H, I, J}	-	0	下一轮只跑这 10 个
2	10	5	{A, C, F, H, J}	否（集合变了）	0	收缩到 5 个
3	5	3	{A, F, J}	否	0	再收缩
4	3	3	{A, F, J}	是（与上轮相同）	1	继续
5	3	3	{A, F, J}	是	2	继续

最终结果：通过 57 个，顽固失败 3 个。
总轮次：6 轮（但其实第 4、5、6 轮只跑了 3 个任务，成本极低）。

为什么这个方案够简单、够粗暴？
无需回归验证：已通过的任务永远不再运行，确保测试总时间快速收敛。

停止条件直观：连续 3 轮相同的失败集合，高度可信地说明这些是“真·失败”（非 flaky）。

工程实现极简：只需维护一个失败列表和计数器，几乎没有状态管理复杂度。

天然符合你的 Phase 2 报告：像 rocalution 这种每次必败的，会稳居失败集，3 轮后即被识别为顽固失败，后续 CI 可直接跳过。

存在的“已知风险”（但这就是你要求的“简单粗暴”）
可能漏掉回归：如果第 2 轮修好了 A，但导致原本通过的 B 在第 3 轮崩溃，你的循环永远不会去重跑 B，所以会错过这个回归。

但是，你明确说了 “测试通过的下一轮对上一轮测试通过的组件就别管了”，说明你愿意接受这个代价，换取极致的执行效率。
