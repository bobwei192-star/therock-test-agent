够一部分，但不是我说的“深度利用”的那一部分。

你现在的 install.sh 把 .opencode/ 和 docs_this_project/ 复制到 TheRock 目录下，确实能让原生安装的 OpenCode 自动识别项目级配置。这一点是对的，也已经算是利用了 OpenCode 原生 overlay 能力：

TheRock/
  .opencode/
    commands/
    agents/
    skills/
    tools/
  docs_this_project/
OpenCode 启动后能识别 /therock-run、agent、skill、权限规则。这已经满足“不要 fork OpenCode、不要改 OpenCode 源码、用项目级 overlay 接入”的要求。

但“不够”的地方在于：OpenCode 识别了配置，不等于 OpenCode 原生知道你的测试任务语义。

现在链路大概是：

OpenCode 识别 /therock-run
  -> agent 根据 markdown 指令拼 shell 命令
  -> 执行 .opencode/tools/therock_agent.sh
  -> Python runner 自己记录状态、审计、报告
也就是说，OpenCode 看到的是一次 bash 调用；真正的 run_id、task、loop round、失败集、wrapper、resume 状态，主要都在 runner 自己的文件里。OpenCode 原生层没有结构化工具 schema，也没有原生 tool metadata。

我说的“更深”是变成：

OpenCode 识别 therock_run 原生 tool
  -> tool schema 校验 artifacts/gpu/components/testTypes
  -> OpenCode tool context 记录 sessionID/messageID/agent/worktree
  -> tool 内部调用 Python runner
  -> tool 返回结构化 run_id/output/status/counts
所以区别是：

你现在：OpenCode 能识别这个项目有命令和 agent。
更深：OpenCode 能结构化理解并审计 TheRock 测试动作本身。
不过要强调：你现在的方式不是错的。对于 MVP，它已经够合理了。是否继续做 plugin，要看你目标：

如果目标是“先把 TheRock 测试跑稳”，现在的 overlay + runner 就够，优先补 resume、GPU health、quarantine、timeout。

如果目标是“证明最大限度复用 OpenCode 原生能力”，那就需要再加 plugin/tool/session metadata 这一层。我的建议是：当前 install.sh 这层保留，它是基础；plugin 是下一层增强，不是替代它。
