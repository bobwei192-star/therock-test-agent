# therock-test-agent 需求说明

## 1. 项目目标

基于 `opencode` 打造一个 CLI 级别的 TheRock 终端测试智能体：`therock-test-agent`。

目标仓库：

- `opencode`：https://github.com/anomalyco/opencode.git
- `TheRock`：https://github.com/ROCm/TheRock.git

核心目标：

- 自动串行执行 TheRock 组件测试。
- 支持 quick / standard / comprehensive / full 四类测试从低到高逐步执行。
- 使用 Loop Engineering 策略，只重跑失败任务，让失败集逐轮收敛。
- 自动生成汇总报告、测试日志、单组件失败详情报告。
- 全过程记录智能体行为，用于后续评估效率和生产率提升。

## 2. 典型使用场景

用户打开 `therock-test-agent` 终端工具，输入必要参数后，智能体自动完成：

1. 准备 TheRock 仓库和 ROCm artifacts。
2. 根据 GPU 型号和组件列表生成测试任务。
3. 按固定顺序串行执行测试。
4. 根据失败结果进行 loop 重试。
5. 输出测试报告、失败报告、日志和过程审计记录。

## 3. 用户输入

| 参数 | 是否必填 | 说明 |
|------|:--------:|------|
| TheRock repo 地址 | 否 | 如果不填写，智能体自动下载 `ROCm/TheRock`。 |
| ROCm artifacts path | 是 | 使用者必须提供，指向本地 ROCm 构建产物或测试产物路径。 |
| GPU model | 是 | 使用者必须提供，例如 `gfx1151`。 |
| TheRock 组件列表 | 否 | 如果不填写，默认测试 `docs_this_project/component_sort_order.json` 中定义的全部组件。 |
| 测试类型 | 否 | 默认按 quick → standard → comprehensive → full 递增执行。 |
| 是否测试 GPU reset 高风险组件 | 否 | 默认否。若开启，则允许运行已知可能触发 GPU reset / ring timeout / 桌面崩溃的组件或测试类型。 |
| GPU reset 高风险组件策略 | 否 | 默认 `skip`。可选：`skip` / `include` / `quarantine`。`quarantine` 表示放到最后单独执行，并在执行前再次确认。 |



## 4. 测试任务定义

一个测试任务由以下字段唯一确定：

```text
task_id = test_component + test_type
```

示例：

- `rocblas-quick`
- `rocblas-standard`
- `miopen-comprehensive`
- `rocsparse-full`

任务维度：

- 组件：来自用户输入或 `component_sort_order.json`。
- 测试类型：quick / standard / comprehensive / full。
- 执行顺序：先按测试类型递增，再按组件排序文件排序。

## 5. 组件排序规则

排序规则来源：

```text
docs_this_project/component_sort_order.json
```

规则：

1. 如果用户传入组件列表，则只测试这些组件。
2. 如果用户未传入组件列表，则默认测试排序文件中的全部组件。
3. 所有组件都按照 `component_sort_order.json` 中定义的顺序排队。
4. 测试类型按以下顺序执行：

```text
quick → standard → comprehensive → full
```

## 6. 全局状态文件

智能体必须维护全局状态文件：

```text
global_state.json
```

建议记录内容：

| 字段 | 说明 |
|------|------|
| run_id | 本次测试运行唯一 ID。 |
| start_time | 测试开始时间。 |
| end_time | 测试结束时间。 |
| therock_repo_path | TheRock 仓库路径。 |
| artifacts_path | ROCm artifacts 路径。 |
| gpu_model | GPU 型号。 |
| output_dir | 本轮输出目录。 |
| task_queue | 初始任务队列。 |
| current_loop | 当前 loop 轮次。 |
| current_task | 当前正在执行的任务。 |
| task_results | 所有任务结果。 |
| failed_task_history | 每轮失败任务集合。 |
| stable_failed_count | 失败集合连续一致次数。 |
| include_gpu_reset_risk_tasks | 是否允许测试 GPU reset 高风险任务。 |
| gpu_reset_risk_policy | GPU reset 高风险任务处理策略：skip / include / quarantine。 |
| skipped_gpu_reset_risk_tasks | 因 GPU reset 风险被跳过的任务。 |
| interrupted_task | 如果进程中断，记录中断时正在执行的任务。 |
| last_checkpoint_time | 最近一次状态落盘时间。 |
| resume_count | 本次 run 被 resume 的次数。 |
| final_status | running / passed / failed / stopped / interrupted。 |

状态文件更新时机：

- 初始化参数后更新。
- 排序生成任务队列后更新。
- 创建输出目录后更新。
- 每个任务开始前更新。
- 每个任务结束后更新。
- 每轮 loop 结束后更新。
- 最终报告生成后更新。

## 8. 测试执行流程

### 8.1 初始化

1. 读取用户输入。
2. 校验 `artifacts_path` 是否存在。
3. 校验 GPU 型号。
4. 如果未提供 TheRock repo，则自动下载 git clone --depth 1 https://github.com/ROCm/TheRock.git。
5. 创建 `run_id`。
6. 创建输出目录。
7. 初始化 `global_state.json`。

### 8.2 生成任务队列

1. 读取组件列表。
2. 按 `component_sort_order.json` 排序。
3. 按 quick → standard → comprehensive → full 生成测试任务。
4. 根据 GPU reset 高风险组件策略过滤或隔离任务。
5. 写入 `global_state.json`。

### 8.3 执行测试循环

按照 `docs_this_project/loop_engineering.md` 的极简 Loop 方案执行：

1. 第一轮执行全部任务。
2. 收集失败任务集合。
3. 如果失败集合为空，停止，结论为全部通过。
4. 如果失败集合与上一轮完全相同，则 `stable_failed_count += 1`。
5. 如果失败集合变化，则 `stable_failed_count = 0`。
6. 当 `stable_failed_count >= 3` 时停止，剩余失败任务判定为顽固失败。
7. 下一轮只执行上一轮失败任务。

### 8.4 GPU reset 高风险组件策略

部分组件或测试类型可能触发 GPU ring timeout / GPU reset / 桌面崩溃。一旦发生，当前 shell、opencode 会话和测试进程都可能被系统终止，导致整个工作流中断。

因此必须把这类任务作为显式策略处理，而不是默认混在普通任务中执行。

建议新增配置文件：

```text
docs_this_project/gpu_reset_risk_tasks.json
```

用于记录已知高风险任务，例如：

```json
[
  {
    "task_id": "rocsparse-full",
    "reason": "full 模式曾触发 GPU ring timeout / GPU reset"
  },
  {
    "task_id": "miopenprovider-comprehensive",
    "reason": "长时间 GPU 卷积测试曾触发 ring timeout"
  },
  {
    "task_id": "miopen-comprehensive",
    "reason": "长时间 GPU 卷积测试风险高"
  }
]
```

处理策略：

| 策略 | 行为 | 适用场景 |
|------|------|----------|
| `skip` | 默认跳过高风险任务，结果标记为 `skip` 或 `blocked`。 | 日常测试、开发调试、需要保证工作流不中断。 |
| `include` | 与普通任务一起执行。 | 用户明确接受 GPU reset 风险。 |
| `quarantine` | 普通任务全部完成后，再单独执行高风险任务。执行前二次确认，并建议 headless / SSH 环境。 | 需要完整覆盖，但希望降低对主流程影响。 |

高风险任务执行前必须做：

1. 写入 checkpoint。
2. 刷新 `global_state.json` 到磁盘。
3. 提醒用户可能发生 GPU reset。
4. 记录当前桌面 / SSH / root / 普通用户环境。
5. 可选插入 GPU cooldown。

### 8.5 中断、关机和 resume 策略

如果测试进行中发生关机、重启、GPU reset、桌面崩溃或 opencode 会话退出，不能假设 opencode 原会话能自动恢复正在运行的进程。

正确恢复方式必须依赖 agent 自己的持久化状态：

```text
runs/<run_id>/global_state.json
```

恢复入口建议：

```bash
therock-test-agent --resume <run_id>
```

恢复流程：

1. 读取 `runs/<run_id>/global_state.json`。
2. 检查 `final_status` 是否为 `running` 或 `interrupted`。
3. 读取 `current_task` 和 `last_checkpoint_time`。
4. 对中断时正在运行的任务标记为 `interrupted` 或 `unknown`。
5. 执行 GPU / ROCm 健康检查，例如 `rocminfo`、`rocm-smi`、`dmesg` 摘要。
6. 如果健康检查通过，从中断任务开始重跑。
7. 如果健康检查失败，停止恢复并生成 `resume_blocked_report.md`。
8. 恢复后 `resume_count += 1`，并继续原 loop。

中断任务处理规则：

| 中断位置 | 恢复策略 |
|----------|----------|
| 任务开始前 | 直接继续该任务。 |
| 任务执行中 | 将该任务标记为 `interrupted`，恢复时重跑该任务。 |
| 任务结束后但状态未写入 | 根据日志和返回码判断；无法判断则重跑该任务。 |
| loop 轮次结束前 | 重算本轮已完成任务结果，未完成任务继续执行。 |
| 报告生成中 | 重新生成报告。 |

结论：

- 切回 opencode 原 session 只能恢复聊天上下文，不等于恢复已被系统杀掉的测试进程。
- 真正可恢复的前提是 `global_state.json`、日志、任务结果和 checkpoint 都已经落盘。
- 所以智能体必须支持 `--resume`，否则遇到关机或 GPU reset 后只能人工重新开始。

## 9. 测试结果定义

每个任务结果至少包含：

| 字段 | 说明 |
|------|------|

| component | 组件名。 |
| test_type | 测试类型。 |
| status | pass / fail / skip / blocked / timeout / flaky。 |

## 10. 输出报告

智能体最终需要生成三类报告：

### 10.1 汇总测试报告

文件：

```text
summary_report.md
```

内容：

- 总任务数。
- 通过数。
- 失败数。
- 跳过数。
- blocked 数。
- flaky 数。
- 每轮 loop 的失败集合变化。
- 最终顽固失败任务列表。
- Native / WSL 或 GPU 型号等环境信息。

### 10.2 测试日志

目录：

```text
logs/
```

每个任务应保存：

- stdout log。
- stderr log。
- 执行命令。
- 环境变量摘要。

### 10.3 单组件失败详细报告

目录：

```text
failures/
```

每个最终失败任务生成一份：

```text
<task_id>_failure_report.md
```

内容建议参考单组件问题模板：

- 组件名。
- 测试类型。
- 失败命令。
- 失败日志。
- 复现结果。
- 根因分类。
- 是否稳定复现。
- 建议处理方式。

## 11. 权限与 sudo 策略

需求中提出：是否需要用户一开始提供 sudo 密码。

建议策略：

1. 默认不保存 sudo 明文密码。
2. 启动阶段做权限预检查。
3. 如果某些组件必须 root 权限，例如 amdsmi Fan / Power / Overdrive 测试，则提前提示用户执行授权。
4. 支持两种模式：
   - 普通模式：不使用 sudo，遇到需 root 的测试自动标记为 skip / blocked。
   - 授权模式：用户手动输入 sudo，智能体只在必要命令上使用。
5. 不建议让智能体长期持有 sudo 密码。

## 12. 文件修改安全边界

需求中提出：是否需要内置“禁止修改文件清单”。

建议必须提供安全边界。

### 12.1 默认禁止修改

智能体默认不得修改：

- TheRock 开发源码。
- ROCm 组件源码。
- 构建产物中的核心库文件。
- 用户未授权的配置文件。

### 12.2 允许修改

默认允许写入：

- `runs/<run_id>/` 输出目录。
- `global_state.json`。
- 测试报告。
- 测试日志。
- 智能体自己的缓存和记忆文件。

### 12.3 需要确认后才能修改

以下内容必须用户确认：

- 测试脚本。
- 环境配置文件。
- CI 配置文件。
- workaround 脚本。
- symlink 或路径修复。

### 12.4 建议新增清单文件

建议新增：

```text
docs_this_project/file_write_policy.json
```

用于定义：

- allowlist：允许写入路径。
- denylist：禁止写入路径。
- require_confirmation：需要用户确认的路径。

## 15. 智能体自身测试方案

需求中提出：智能体开发完成后如何验证。

建议分三层测试。

### 15.1 Dry-run 测试

不真正执行 TheRock 测试，只验证：

- 参数解析。
- 任务生成。
- 排序规则。
- 输出目录创建。
- `global_state.json` 更新。
- 报告生成。

### 15.2 小规模真实测试

在原 Ubuntu 26.04 环境上选 3-5 个轻量组件测试：

- sanity
- amdsmi
- rocdecode
- rocjpeg
- hipfile

验证：

- 测试命令能执行。
- 日志能采集。
- 失败能进入下一轮 loop。
- 报告能生成。

### 15.3 全量回归测试

在已经跑过全量测试的 Ubuntu 26.04 + gfx1151 环境上再跑一次。

目的：

- 对比人工执行和智能体执行结果是否一致。
- 验证 loop 是否减少重复测试成本。
- 验证顽固失败识别是否准确。
- 评估效率和生产率提升。

## 16. MVP 范围

第一版建议只做以下能力：

1. CLI 参数输入。
2. 组件排序。
3. 任务队列生成。
4. 串行执行测试。
5. `global_state.json` 状态维护。
6. 失败集收缩 loop。
7. 日志采集。
8. 汇总报告。
9. 单组件失败报告。
10. 工具调用和文件修改审计。
11. `--resume <run_id>` 中断恢复。
12. GPU reset 高风险任务 skip / include / quarantine 策略。

暂不做：

- Web Dashboard。
- 并行调度。
- 自动修改 TheRock 源码。
- 自动提交 PR。
- 复杂根因自动修复。
