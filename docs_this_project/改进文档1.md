## 问题 1：`test_runner.py` 调用方式错误

当前 agent 曾这样调用：

```bash
python3 test_runner.py --component hiprand --test-type quick
```

这是错误的。TheRock 的 `test_runner.py` 不通过 `--component` / `--test-type` 接收参数，而是在模块加载时读取环境变量：

```python
TEST_COMPONENT = os.getenv("TEST_COMPONENT")
TEST_TYPE = os.getenv("TEST_TYPE") or "quick"
AMDGPU_FAMILIES = os.getenv("AMDGPU_FAMILIES")
THEROCK_BIN_DIR = os.getenv("THEROCK_BIN_DIR")
```

其中 `TEST_COMPONENT` 是必填项。未设置时会直接退出：

```text
ERROR: TEST_COMPONENT environment variable is required but not set.
```

### 解决方法

下一版 agent 必须改为环境变量驱动：

```bash
env \
  TEST_COMPONENT=hiprand \
  TEST_TYPE=quick \
  AMDGPU_FAMILIES=gfx1151 \
  THEROCK_AMDGPU_FAMILIES=gfx1151 \
  THEROCK_AMDGPU_TARGETS=gfx1151 \
  THEROCK_BIN_DIR=<rocm_dist>/bin \
  OUTPUT_ARTIFACTS_DIR=<rocm_dist> \
  ROCM_PATH=<rocm_dist> \
  HIP_PATH=<rocm_dist> \
  LD_LIBRARY_PATH=<rocm_dist>/lib:<rocm_dist>/lib64:$LD_LIBRARY_PATH \
  python3 build_tools/github_actions/test_executable_scripts/test_runner.py
```

实现要求：

- 删除对 `test_runner.py --component` / `--test-type` 的调用。
- 将 `TEST_COMPONENT required` 分类为 `entrypoint_error`，这是 agent 调用错误，不计入组件失败。
- 在测试用例中断言 `test_runner` 路径必须包含 `TEST_COMPONENT` 环境变量。



## 问题 2：artifacts 路径不是真实 ROCm 安装树

真实测试要求 artifacts path 能解析到 ROCm 安装目录。agent 需要把用户输入规范化为：

```text
build_root = <...>/build
rocm_dist = <build_root>/dist/rocm
therock_bin_dir = <rocm_dist>/bin
```

如果用户传入空目录，或目录下没有 `dist/rocm/bin` / `dist/rocm/lib`，测试不应继续。

### 解决方法

runner 开始前必须做 artifacts preflight：

```text
<rocm_dist>/bin exists
<rocm_dist>/lib or <rocm_dist>/lib64 exists
```

失败时：

- 标记为 `blocked`。
- 不生成组件失败结论。
- 写入 `runs/_audit/agent_invocations.jsonl`。
- 在 summary 中说明 `missing_artifacts`。



## 问题 3：所有组件强行走同一个入口

TheRock 测试脚本分两类：

- 通用 `test_runner.py`
- 专用 `test_<component>.py`

很多组件不能简单交给 `test_runner.py`，例如：

- `sanity`
- `amdsmi`
- `tensilelite`
- `libhipcxx_hipcc`
- `libhipcxx_hiprtc`
- `rocsparse`
- `rocsolver`
- `rocrtst`
- `hiptests`
- `hipdnn`
- `hipdnn_integration`
- `hipkernelprovider`
- `rocprofiler_sdk`
- `rocprofiler_systems`
- `aqlprofile`

### 解决方法

使用独立索引文件决定每个组件怎么运行：

```text
docs_this_project/component_env_script_index.json
```

职责划分：

| 文件 | 职责 |
|------|------|
| `component_sort_order.json` | 决定跑哪些 task、执行顺序、耗时、状态、GPU hang 风险 |
| `component_env_script_index.json` | 决定每个组件使用哪个脚本、哪些环境变量、哪些依赖和 known issue 分类 |

runner 解析顺序：

1. 读取 `component_sort_order.json` 生成 task 队列。
2. 用 task 的 `component` 查询 `component_env_script_index.json`。
3. 合并 `defaults`、组件配置和 `mode_overrides`。
4. 根据 `entrypoint_type` 决定执行：
   - `test_runner`
   - `dedicated_python`
   - `none`



## 问题 4：缺少组件到脚本和环境变量的 index

只靠 `component_sort_order.json` 不够。它能告诉 agent “先跑哪个”，但不能告诉 agent “怎么跑”。

### 解决方法

已新增并通过 JSON 校验：

```text
docs_this_project/component_env_script_index.json
```

当前版本：

```text
version = 0.1
last_updated = 2026-07-09
```

结构：

```text
path_variables
env_profiles
defaults
components
mode_overrides
runner_rules
```

关键字段语义：

| 字段 | 说明 |
|------|------|
| `entrypoint_type` | `test_runner` / `dedicated_python` / `none` |
| `test_component` | 传给 `TEST_COMPONENT` 的值，仅 `test_runner` 需要 |
| `script` | 专用 Python 脚本文件名 |
| `env_profiles` | 需要合并的环境变量 profile |
| `requires` | 组件额外依赖，例如 `lit`、`joblib`、`OUTPUT_ARTIFACTS_DIR` |
| `known_issue_category` | 已知失败或风险分类 |
| `retry_policy` | 重试策略 |
| `gpu_hang_risk` | 是否默认受 GPU reset 风险策略控制 |
| `timeout_hint_seconds` | 后续 per-component timeout 建议值 |
| `default_status` | `entrypoint_type=none` 或官方排除时的默认状态 |

示例：

```json
"hiprand": {
  "entrypoint_type": "test_runner",
  "test_component": "hiprand",
  "env_profiles": ["test_runner"]
}
```

实际执行应是：

```bash
TEST_COMPONENT=hiprand TEST_TYPE=quick python3 test_runner.py
```

而不是：

```bash
python3 test_runner.py --component hiprand --test-type quick
```



## 问题 5：环境变量不完整导致路径、动态库、工具找不到

手动报告中已有多个组件因为环境变量缺失导致失败或超时：

- `rocrtst`
- `hipdnn`
- `hipdnn_integration`
- `hipkernelprovider`
- `sanity`

常见缺失：

```text
THEROCK_BIN_DIR
OUTPUT_ARTIFACTS_DIR
ROCM_PATH
HIP_PATH
LD_LIBRARY_PATH
PATH
```

### 解决方法

每个 task 执行前统一构造基础环境：

```bash
export TEST_TYPE=<test_type>
export AMDGPU_FAMILIES=<gfx>
export THEROCK_AMDGPU_FAMILIES=<gfx>
export THEROCK_AMDGPU_TARGETS=<gfx>
export THEROCK_BIN_DIR=<rocm_dist>/bin
export OUTPUT_ARTIFACTS_DIR=<rocm_dist>
export ROCM_PATH=<rocm_dist>
export HIP_PATH=<rocm_dist>
export PATH=<rocm_dist>/bin:$PATH
export LD_LIBRARY_PATH=<rocm_dist>/lib:<rocm_dist>/lib64:$LD_LIBRARY_PATH
export SHARD_INDEX=1
export TOTAL_SHARDS=1
```

如果是 `test_runner` 入口，再追加：

```bash
export TEST_COMPONENT=<component>
```

实现要求：

- 增加 `build_task_env(task, entrypoint)`。
- 按 `env_profiles` 合并变量。
- 展开 `{rocm_dist}`、`{test_type}`、`{amdgpu_families}`、`{test_component}` 等占位符。
- `PATH` / `LD_LIBRARY_PATH` 使用 prepend 策略，不要覆盖用户已有值。



## 问题 6：Python 依赖未安装

手动报告明确记录，本地测试前如果未安装依赖，会出现：

| 组件 | 缺失依赖 | 典型错误 |
|------|----------|----------|
| `tensilelite` | `joblib` | `ModuleNotFoundError: No module named 'joblib'` |
| `libhipcxx_*` | `lit` | `Could not find libcudacxx_LIT` |
| pytest 组件 | `pytest` / `pytest-xdist` | pytest 无法运行或无法并行 |

### 解决方法

增加 dependency preflight：

```bash
python3 - <<'PY'
import importlib.util
for mod in ["pytest", "joblib", "lit"]:
    print(mod, bool(importlib.util.find_spec(mod)))
PY
```

根据 `component_env_script_index.json` 中的 profile 做有选择检查：

| profile | 检查 |
|---------|------|
| `pytest_dependency_sensitive` | `pytest`、`joblib` |
| `libhipcxx` | `lit`、`OUTPUT_ARTIFACTS_DIR` |

缺依赖时：

- 标记 `blocked`。
- 写入 `global_state.json`。
- 生成失败报告，但分类为 `missing_dependency`。
- 不归类为组件真实失败。

手动报告中 `sanity` 曾被误判为环境缺失，实际是 `THEROCK_BIN_DIR` 没指向正确产物目录。

## 问题 9：硬编码构建路径导致组件失败

手动报告中这些组件出现过路径硬编码问题：

- `hipdnn`
- `hipdnn_integration`
- `hipkernelprovider`

典型问题：

- `CTestTestfile.cmake` 中存在 `/therock/src/...`。
- Docker Python 路径 `/opt/python/cp312-cp312/bin/python3.12` 在本地不存在。
- `libhipdnn_backend.so` 找不到。

### 解决方法

当前已采用 wrapper 策略：

1. 每个真实 task 执行前在 run 目录生成 shell wrapper。
2. wrapper 只负责 `cd` 到 TheRock 仓库、`export` 本次 task 所需环境变量、执行原始测试脚本。
3. wrapper 不修改 ROCm 产物、组件源码和 TheRock 子模块。
4. wrapper 与环境变更写入：
   - `wrappers/<task_id>.round<N>.sh`
   - `wrapper_changes.jsonl`
   - `agent_activity.jsonl`
   - `tool_calls.jsonl`
   - `global_state.json`
   - `summary_report.md`
   - 单组件 failure report
5. 测试失败后扫描 stdout / stderr，识别硬编码路径特征。
6. 不允许默认修改：
   - 组件源码
   - ROCm 子模块源码
   - `dist/rocm/lib`
   - `dist/rocm/bin`

失败分类建议：

```text
path_hardcode
missing_runtime_library
invalid_embedded_python_path
```

wrapper 示例：

```bash
#!/usr/bin/env bash
set -euo pipefail

# Generated by TheRock Test Agent. Do not edit artifacts or component sources.
cd /path/to/TheRock
export ROCM_PATH=/path/to/build/dist/rocm
export HIP_PATH=/path/to/build/dist/rocm
export THEROCK_BIN_DIR=/path/to/build/dist/rocm/bin
export OUTPUT_ARTIFACTS_DIR=/path/to/build/dist/rocm
export LD_LIBRARY_PATH=/path/to/build/dist/rocm/lib:/path/to/build/dist/rocm/lib64:$LD_LIBRARY_PATH

exec python3 /path/to/TheRock/build_tools/github_actions/test_executable_scripts/test_hipdnn.py
```

如果日志命中 `/therock/src/`、`/opt/python/cp312-cp312/bin/python3.12`、`libhipdnn_backend.so` 等特征，agent 会把结果写入 `path_hardcode_detection` 字段，并在报告中显示“硬编码路径检测”。



## 问题 10：sudo / root 权限导致测试误判

`amdsmi` 在 root 权限下 50/50 通过，包含 Fan / Power / Overdrive 等读写操作。

如果没有 sudo，部分测试可能失败，但这不代表组件功能失败。

### 解决方法

当前已实现策略：

- 不保存 sudo 密码。
- `.env` 只保存 `THEROCK_SUDO_POLICY`。
- `THEROCK_SUDO_POLICY=cache` 时要求用户先执行：

```bash
sudo -v
```

runner 检查：

```bash
sudo -n true
```

如果 task 命中 `sudo_sensitive` profile：

- `THEROCK_SUDO_POLICY=none`：直接 `blocked`。
- `THEROCK_SUDO_POLICY=ask`：当前自动 loop 中也直接 `blocked`，避免脚本卡在交互输入。
- `THEROCK_SUDO_POLICY=cache`：执行 `sudo -n true` 检查 cache。
- cache 不可用时标记为 `blocked`。
- 不归类为组件失败。
- 写入 `global_state.json`、`agent_activity.jsonl`、`tool_calls.jsonl` 和 failure report。

当前含 `sudo_sensitive` 的典型组件：

```text
amdsmi
```



## 问题 11：GPU reset / ring timeout 高风险任务可能拖垮系统

手动报告中 `rocsparse full` 触发过：

```text
ring gfx_0.0.0 timeout
MODE2 reset
GPU reset
```

高风险任务还包括：

- `rocsparse full`
- `rocrtst comprehensive/full`
- `hiptests standard/comprehensive/full`

### 解决方法

继续保持 GPU risk 策略：

| 策略 | 行为 |
|------|------|
| `skip` | 默认跳过 `gpu_hang_risk=true` |
| `include` | 直接执行高风险任务 |
| `quarantine` | 普通任务先跑，高风险任务最后单独跑 |

检测到以下日志时立即停止 loop：

```text
ring gfx
MODE2 reset
GPU reset
MES failed
PERMISSION_FAULT
HSA_STATUS_ERROR
```

停止后：

- 当前任务标记为 `gpu_reset_risk` 或 `interrupted`。
- 写 `resume_blocked_report.md`。
- 提示用户检查 `dmesg`、`rocminfo`、`rocm-smi`。
- 不自动继续跑高风险任务。



## 问题 12：官方排除或无入口组件被误跑

手动报告中明确不应默认执行：

| 组件 | 原因 |
|------|------|
| `hipsparselt` | 官方 `exclude_family gfx1151` |
| `rocroller` | 官方 `exclude_family gfx1151` / 架构枚举不识别 |
| `rocprofiler_compute` | 无独立测试入口 |

### 解决方法

官方排除规则现在放在独立文件：

```text
docs_this_project/official_exclude.json
```

示例：

```json
{
  "component": "rocprofiler_compute",
  "test_types": ["quick", "standard", "comprehensive", "full"],
  "amdgpu_families": ["*"],
  "status": "skip",
  "known_issue_category": "no_independent_entrypoint",
  "reason": "当前无独立测试入口，runner 不应强行猜测脚本。"
}
```

runner 处理：

- 先读 `official_exclude.json`。
- 命中官方排除后直接 skip / blocked。
- 排除优先级高于 `component_sort_order.json` 和 `component_env_script_index.json`。
- `rocprofiler_compute` 这类无入口组件不猜脚本。
- 默认生成 skip / blocked 记录。
- `global_state.json`、summary 和 failure report 中记录 `official_exclude` 命中原因。



## 问题 13：稳定失败任务被反复重试浪费时间

手动报告中稳定失败或不宜反复重试：

| 组件 / 模式 | 现象 | 建议 |
|-------------|------|------|
| `rocalution` quick/standard/comprehensive | CG / IDR / FGMRES 数值问题 | known fail，减少重试 |
| `rocprofiler_systems` | Python 3.14 兼容 + pytest collection 阻塞 | Python 3.14 下 blocked 或只跑 CTest 子集 |
| `tensilelite full` | Python 3.14 pathlib 路径 bug | full 模式谨慎 |
| `rocprofiler_sdk` standard/comprehensive/full | rocpd 缺失 / profiler 集成失败 | known issue |

### 解决方法

在 index 中使用：

```json
"known_issue_category": "stable_numeric_failure_gfx1151",
"retry_policy": "do_not_retry_many_times"
```

runner 策略：

- known fail 不参与无限 retry。
- 生成 known fail 报告。
- `global_state.json` 记录 `known_issue_category`。
- `summary_report.md` 列出 known fail 分类。



## 问题 14：flaky 失败和真实失败没有区分

手动报告中有重跑后通过的组件：

- `rocfft`
- `rocsparse quick`

### 解决方法

增加失败稳定性分类：

| 分类 | 行为 |
|------|------|
| `flaky_retry_ok` | 可少量重试 |
| `stable_numeric_failure` | 不反复重试 |
| `gpu_reset` | 停止 loop |
| `missing_env` | blocked |
| `missing_dependency` | blocked |

loop 收敛时：

- 普通失败按失败集重跑。
- known fail 不超过低次数重试。
- GPU reset 直接停止。
- blocked 不计入组件真实失败。

## 问题 17：下一版 agent 的实施顺序不够明确

当前 runner 已完成核心执行链路，但仍需要按优先级继续补齐真机验证和少量策略增强。

### 解决方法

当前已完成：

1. 读取并校验 `component_env_script_index.json`。
2. 增加 `resolve_entrypoint(component, test_type)`：
   - 合并 `defaults`。
   - 读取 `components[component]`。
   - 应用 `mode_overrides[test_type][component]`。
3. 增加 `build_task_env(task, entrypoint)`：
   - 合并 `env_profiles`。
   - 展开 `{rocm_dist}`、`{test_type}`、`{amdgpu_families}`、`{test_component}`。
   - 设置 PATH / LD_LIBRARY_PATH prepend。
4. 修改 `render_command()`：
   - `test_runner` 只执行 `test_runner.py`，不再传 `--component` / `--test-type`。
   - `dedicated_python` 执行 index 中的专用脚本。
   - `none` 直接 skip / blocked。
5. 修改 `run_task()`：
   - 使用 entrypoint command。
   - 使用 task env。
   - 生成 wrapper。
   - 写入 entrypoint metadata。
6. 增加 preflight：
   - artifacts
   - Python 依赖
   - sudo cache
   - 必要环境变量
7. 增加官方排除：
   - 读取 `official_exclude.json`。
   - 命中后直接 skip / blocked。
8. 更新测试用例：
   - `test_runner` 路径必须包含 `TEST_COMPONENT`。
   - `dedicated_python` 路径必须使用 `script`。
   - sudo_sensitive 默认 blocked。
   - official exclude 默认 skip。
   - wrapper 生成和硬编码路径检测。

下一步建议：

cd /home/zx/TheRock_CI测试流程/TheRock
