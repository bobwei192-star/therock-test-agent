On Ubuntu 16.04.3, R9-Nano, ROCm 1.6-180,
It seems to me there's a case of bad code generation.

The code is trimmed down from an NTT FFT that I develop (an FFT with modular arithmetic), I prepared a small test case here (with build instructions included):
https://github.com/preda/gpuowl/tree/master/bug

Let me explain why I think there's a problem:

in the kernel, commenting or un-commenting the barrier() right after the shufl() changes the result. IMO, that barrier is not needed and should not affect the result. The different outputs with/without the barrier indicates that at least one of the outputs is wrong.
```
kernel __attribute__((reqd_work_group_size(256, 1, 1))) void bug(global T2 *io, global T2 *trig) {
  local uint lds[8 * 256];
  uint me = get_local_id(0);
  T2 u[8];
  for (int i = 0; i < 8; ++i) { u[i] = io[256 * i + me]; }
  fft8(u);
  shufl(lds, u, 8, 32);

  // Comment or un-comment the next barrier() to observe different behavior.
  // barrier(CLK_LOCAL_MEM_FENCE);

  tabMul(trig, u, 8, 32);
  for (int i = 0; i < 8; ++i) { io[256 * i + me] = u[i]; }  
}
```

Now, for why the barrier() after shufl() isn't needed: after the last write to LDS there is a barrier(). So no thread changes LDS anymore after that.

After the final reading from LDS, each thread uses only what it itself read, thus no need for another barrier() after LDS read.

Please consider this, to me it clearly looks like it's P0 material.
