# ROCm 3.5.0 leaves Vega 56 stuck in P-state 3

- **Issue #:** 1149
- **State:** closed
- **Created:** 2020-06-15T14:59:53Z
- **Updated:** 2021-01-12T11:05:15Z
- **URL:** https://github.com/ROCm/ROCm/issues/1149

I'm on Arch Linux, using [`rocm-opencl-runtime`](https://aur.archlinux.org/packages/rocm-opencl-runtime/) as well as [`hip-rocclr`](https://aur.archlinux.org/packages/hip-rocclr/) from the Arch AUR, and after the first run of any OpenCL or HIP related program, ROCm leaves the Vega 56 stuck in P-state 3 at idle. The GPU will happily go up to higher levels, but then when there is no load it ends up sitting at P-state 3:
```bash
$ cat /sys/class/drm/card0/device/pp_dpm_sclk
0: 852Mhz 
1: 991Mhz 
2: 1138Mhz 
3: 1269Mhz *
4: 1312Mhz 
5: 1474Mhz 
6: 1538Mhz 
7: 1590Mhz 
```

This only happens once the ROCm stack is used - I can play games just file and this does not occur, nor does it during normal system usage. It seems to be the exact same issue that @Bengt had here:
- https://github.com/ROCmSoftwarePlatform/tensorflow-upstream/issues/621
- https://github.com/Oblomov/clinfo/issues/43

Restricting the power states with rocm-smi does not help, and the GPU stays in state 3. I've tried both:
```bash
$ sudo rocm-smi --setsclk 0 1 2 3 4 5 6 7


========================ROCm System Management Interface========================
GPU[0]          : Successfully set sclk frequency mask to Level 0 1 2 3 4 5 6 7
==============================End of ROCm SMI Log ==============================
```
and:
```bash
$ sudo rocm-smi --setsclk 0 1 6 7            


========================ROCm System Management Interface========================
GPU[0]          : Successfully set sclk frequency mask to Level 0 1 6 7
==============================End of ROCm SMI Log ==============================
```

However the GPU still stays the same:
```bash
$ cat /sys/class/drm/card0/device/pp_dpm_sclk
0: 852Mhz 
1: 991Mhz 
2: 1138Mhz 
3: 1269Mhz *
4: 1312Mhz 
5: 1474Mhz 
6: 1538Mhz 
7: 1590Mhz 
```

While in state 3 it makes a very high-pitched electrical whine, which makes this completely unsuitable for desktop use (which is what the GPU is in right now). `clpeak` and `HIP-Examples` all run successfully. Nothing useful appears in `dmesg`. I'm on kernel `5.7.2-arch1-1` and mesa `20.1.1-1`. The only way I have found that fixes the issue is rebooting the machine. OpenGL and Vulkan do not seem to trigger this issue.
