# ROCm 3.1 OpenCL bad compilation

- **Issue #:** 1039
- **State:** closed
- **Created:** 2020-03-10T01:57:17Z
- **Updated:** 2021-05-10T04:17:44Z
- **URL:** https://github.com/ROCm/ROCm/issues/1039

ROCm 3.1 OpenCL, Linux kernel 5.4.24, RadeonVII

This issue refers to this commit:
https://github.com/preda/gpuowl/commit/79ae868be3fa0f97eaa11a762d1a4bcf2926fb82

To repro, please create dump folders named e.g. "good" and "bad", and run:

./gpuowl -prp 94080289 -dump good
./gpuowl -prp 94080289 -dump bad -use ENABLE_ROCM_BUG

The only difference between the two is controlled by the define ENABLE_ROCM_BUG in gpuowl.cl (see the commit linked above), i.e.:
```
#if ENABLE_ROCM_BUG
  out += MIDDLE * (WIDTH * SIZEX * gx + WG * gy);
#else
  out += MIDDLE * WIDTH * SIZEX * gx + MIDDLE * WG * gy;
#endif
```

In the repro case above, those values are:
```
MIDDLE = 10
WIDTH = 1024
SIZEX = 32
WG = 256
```
(in particular, there should be no overflow on the additions or multiplications).

In this situation, the two expressions used in the two branches
```
MIDDLE * (WIDTH * SIZEX * gx + WG * gy);
MIDDLE * WIDTH * SIZEX * gx + MIDDLE * WG * gy;
```
should be equivalent. Yet one works and the other doesn't.

What's more, if in the "bad" case we enable "DEBUG" which consists in just sprinkling a few printf() in the code (that are not even executed), the "bad" case starts working, see e.g.:
./gpuowl -prp 94080289 -use ENABLE_ROCM_BUG,DEBUG

The bug is isolated to a single kernel "fftMiddleOut", this should make it easier to localize.

As this bug at first look seems to concern a bad compilation of simple arithmetic on a pointer, it seems rather severe (i.e. high priority) to me.
