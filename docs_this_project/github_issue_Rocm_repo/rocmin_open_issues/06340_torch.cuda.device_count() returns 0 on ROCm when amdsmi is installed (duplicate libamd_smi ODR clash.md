# torch.cuda.device_count() returns 0 on ROCm when amdsmi is installed (duplicate libamd_smi ODR clash); HIP reports 1

### Problem Description

### 🐛 Describe the bug

On ROCm, `torch.cuda.device_count()` returns **0** (so any `cuda:0` access raises `AssertionError: Invalid device id`) whenever the standalone `amdsmi` Python package is importable — even though the HIP runtime correctly reports 1 device.

**Root cause: a duplicate `libamd_smi` ODR/symbol clash.** The ROCm torch wheel bundles its own `libamd_smi.so.26` inside `_rocm_sdk_core`, while the `amdsmi` package ships its own `libamd_smi.so` (a *different* build). Once `import torch` loads the bundled copy first (RTLD_GLOBAL), the amdsmi binding's `amdsmi_get_processor_handles()` is hijacked by the duplicate symbols and returns **0**. Since `_device_count_amdsmi()` returns `0` (not a negative value), `device_count()` trusts it and never falls back to `torch._C._cuda_getDeviceCount()`, which returns the correct `1`.

Note: `torch/cuda/__init__.py` already contains a `_amdsmi_cdll_hook` intended to avoid exactly this ODR violation, but it does not help when torch is installed from the pip ROCm wheel: the two copies have different filenames (`libamd_smi.so` vs `libamd_smi.so.26`), so the hook cannot dedupe them.

**Minimal reproduction (clean conda env, no Docker):**

```bash
conda create -n rocmtest python=3.12 -y && conda activate rocmtest
pip install --index-url https://repo.amd.com/rocm/whl/gfx1150/ torch==2.9.1+rocm7.13.0
python -c "import torch; print(torch.cuda.device_count())"   # -> 1  (amdsmi not installed)
pip install /opt/rocm/share/amd_smi                          # install amdsmi 26.4.0
python -c "import torch; print(torch.cuda.device_count())"   # -> 0  (BUG)
```

**The internal divergence:**

```python
import torch, torch.cuda as c
print(torch._C._cuda_getDeviceCount())  # 1   (HIP, correct)
print(c._raw_device_count_amdsmi())     # 0   (amdsmi, wrong after `import torch`)
print(torch.cuda.device_count())        # 0   (trusts amdsmi -> WRONG)
```

**`import torch` alone breaks amdsmi enumeration** (no CUDA call, and you never `import amdsmi` yourself — torch imports it for `_HAS_PYNVML`):

```python
# fresh process
import amdsmi; amdsmi.amdsmi_init()
len(amdsmi.amdsmi_get_processor_handles())   # 1

# after import torch
import torch
import amdsmi; amdsmi.amdsmi_init()
len(amdsmi.amdsmi_get_processor_handles())   # 0
```

`/proc/self/maps` confirms two different `libamd_smi` builds are mapped after `import torch`:

- `.../amdsmi/libamd_smi.so` (Build ID `087fa6e8...`, 5.0 MB)
- `.../_rocm_sdk_core/lib/libamd_smi.so.26` (Build ID `5cd81c7b...`, 5.5 MB)

`LD_PRELOAD`-ing only `_rocm_sdk_core/lib/libamd_smi.so.26` into a fresh `amdsmi`-only process (no torch) reproduces the drop from 1 -> 0, isolating it to that bundled library.

**Also affected:** `python -m torch.utils.collect_env` itself crashes with the same `AssertionError: Invalid device id` in `get_gpu_info()`.

**Reproduces on the latest available wheel too:** `torch==2.11.0+rocm7.13.0` (triton 3.6.0) — identical behavior.

### Expected behavior

`device_count()` should return 1. When the amdsmi-based count is `0` but `_cuda_getDeviceCount()` (HIP) reports `> 0`, PyTorch should not report fewer devices than HIP — e.g. treat amdsmi's `0` as "unavailable" and fall back to the HIP count, and/or extend `_amdsmi_cdll_hook` to also dedupe the wheel-bundled `_rocm_sdk_core/lib/libamd_smi.so.26`.

### Workaround

Initialising amdsmi before importing torch makes the correct copy win and restores `device_count() == 1`:

```python
import amdsmi; amdsmi.amdsmi_init()
import torch
```

### Versions
