## Summary

`test_linux_jax_wheels.yml` currently creates a temporary `/opt/rocm -> /opt/rocm-<version>` symlink in CI so that JAX wheels with hardcoded `/opt/rocm/lib` RUNPATH entries can resolve ROCm libraries during testing.

This is a CI workaround only. It is not the long-term fix for JAX wheel packaging.

## Why this exists

The current JAX wheel test environment exposes two issues:

1. TheRock installs ROCm to `/opt/rocm-<version>/`, while the tested JAX wheels currently expect `/opt/rocm/lib` in their RUNPATH.
2. Without strict backend enforcement, JAX tests may appear to pass on CPU even when ROCm backend initialization is broken.

The symlink helps make CI reflect the wheel's current runtime expectation, but it should be removed once the wheel-side packaging issues are fixed.

## Follow-up needed

Remove the `/opt/rocm` symlink workaround from `test_linux_jax_wheels.yml` after the wheel-side fixes are fully landed and validated.

## Removal criteria

Before removing the workaround, confirm all of the following:

- JAX wheels no longer rely on a hardcoded `/opt/rocm/lib` assumption for CI/runtime correctness.
- Required ROCm library paths, including any needed sysdeps coverage, are handled correctly by wheel packaging / RUNPATH.
- JAX wheel CI passes on GPU runners with `JAX_PLATFORMS=rocm` and without the `/opt/rocm` symlink workaround.
- Logs confirm ROCm backend initialization instead of CPU fallback.
