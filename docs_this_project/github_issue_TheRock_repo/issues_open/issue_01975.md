# #1975 - [Discussion]: Mechanism to omit rocm-libraries components when the requested GPU target is unsupported

**State**: open  
**Author**: bstefanuk  
**Created**: 2025-10-30T19:41:24Z  
**Updated**: 2025-10-30T23:46:50Z  
**URL**: https://github.com/ROCm/TheRock/issues/1975  

---

Components with no supported GPU architectures are still built in CI, resulting in packages containing incompatible code objects for the target architecture. **Current behavior:** when `therock_amdgpu_targets.cmake` filters out all unsupported architectures for a component, it generates an empty `GPU_TARGETS` in the toolchain file. The component then falls back to its internal default architecture set instead of being excluded from the build.

**Impact**

- Packages contain unusable binaries for the target architecture
- Wasted CI resources building incompatible configurations
- Potential runtime failures for end users

**Option 1:** TheRock excludes components from build matrix when `GPU_TARGETS` would be empty
- Prevents unnecessary builds
- Requires component architecture metadata

**Option 2:** Components could be built as no-op/header-only when GPU_TARGETS is empty, but this creates potentially confusing package contents

**Example: hipSPARSELt building for gfx1151**

Generated toolchain (`_toolchain.cmake`):
```cmake
set(AMDGPU_TARGETS  CACHE STRING "From super-project" FORCE)
set(GPU_TARGETS  CACHE STRING "From super-project" FORCE)
```

Component configuration output:
```
-- Building for GPU targets: gfx942;gfx950
```

The gfx1151 package will incorrectly contain gfx942/gfx950 code objects.

Note: this will also occur in hipblaslt if we build for an unsupported architecture.
