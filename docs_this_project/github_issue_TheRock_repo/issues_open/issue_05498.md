### Problem Description

When building Debian packages from ASAN enabled ROCm artifacts, the dpkg-buildpackage process fails during the dh_dwz step. The dwz tool (DWARF debuginfo compressor) reports an error about allocatable sections appearing after non-allocatable sections in the ELF binary.

readelf output
[fft_readelf_asan.txt](https://github.com/user-attachments/files/28361707/fft_readelf_asan.txt)

  The kpack transform tool appends a .rocm_kpack_ref section to ELF binaries after all existing sections, including non-allocatable debug sections (.debug_info, .debug_line, etc.). This section is marked as allocatable (flag A) because it must be memory-mapped at runtime for GPU kernel loading.

  This results in an ELF section layout that violates standard conventions:
  [1-29]   Allocatable sections (.text, .data, .bss, etc.)     ← Flag: A
  [30-47]  Non-allocatable sections (.debug_*, .symtab, etc.)  ← Flag: (none)
  [48]     .rocm_kpack_ref                                      ← Flag: A (allocatable!)

This violates ELF section ordering conventions and causes dwz to fail.

**Why Debug Builds Fail but Release Builds Pass**

  - Release builds ( MULTI_ARCH): No .debug_* sections present → dh_dwz skips dwz entirely → build succeeds
  - ASAN builds : .debug_* sections present → dh_dwz invokes dwz → dwz detects invalid section ordering → build fails

  The bug exists in both builds, but only manifests when dwz runs.
