## Summary

30-day data (2026-04-28 â†’ 2026-05-28) for the [TheRock CI Nightly workflow](https://github.com/ROCm/rocm-libraries/actions/workflows/therock-ci-nightly.yml) (`therock-ci-nightly.yml` on `ROCm/rocm-libraries`) shows two distinct stability issues:

1. **High attempt-1 failure rate.** Of 31 scheduled runs in the window, 1 reached `success` on attempt 1 (run [207](https://github.com/ROCm/rocm-libraries/actions/runs/25721469956), 2026-05-12). 27 failed on attempt 1. Latest-attempt conclusions are 21 success / 8 failure (the gap is closed by attempt-2/3/4 reruns). Day-1 *job-level* pass rate is 88.1% (3,000 / 3,406 jobs at attempt 1) â€” a small number of failing jobs (typically 2-6 per nightly) reliably blocks the whole run, and most of those clear on subsequent attempts (suggesting environmental causes for the bulk of attempt-1 failures).

2. **Recurring multi-library test-step timeout cluster on `gfx94X-dcgpu`.** On certain nights, multiple unrelated libraries' Test steps each run far slower than their baseline, exceed the default 60-minute GitHub Actions step budget, and are killed mid-test. This subset does not recover on rerun (run [205](https://github.com/ROCm/rocm-libraries/actions/runs/25623225442) was rerun 4 times and remains failed) and accounts for the most recent unrecoverable nights.

These two issues plausibly share a root cause: environmental variability on the runner pool that produces flakes on attempt 1 in the typical case and pushes the longest-running test suites past the 60-minute step budget in the worst case. The same library code on a separate CI system (different runner pool, different orchestration) does not exhibit comparable behavior over the same window.

## Two populations of attempt-1 failures

### Group A â€” recovers on rerun (â‰ˆ19 of the 27 attempt-1 failures)

For these runs, attempt 1 has 2-6 failed jobs out of ~115 attempted. Failures are scattered across different libraries and architectures from night to night, and they pass when the same code is rerun. The pattern is consistent with environmental causes â€” runner availability, network, transient driver state, resource contention, or runner-pool acquisition timing at the nightly's start window.

### Group B â€” does not recover on rerun (8 of the 31 scheduled runs)

A subset of failures persists across multiple rerun attempts. The most distinctive shape in this group is a recurring multi-library timeout cluster on the `gfx94X-dcgpu` runner pool, detailed below. It accounts for the most recent unrecoverable nights ([222](https://github.com/ROCm/rocm-libraries/actions/runs/26440696845), [223](https://github.com/ROCm/rocm-libraries/actions/runs/26499728228), [225](https://github.com/ROCm/rocm-libraries/actions/runs/26563416208)) and the older [205](https://github.com/ROCm/rocm-libraries/actions/runs/25623225442).

## Failure shapes on `gfx94X-dcgpu`: 60-minute Test-step timeouts

Two distinct failure shapes have been observed on this workflow, both terminating as 60-minute Test-step timeouts on `gfx94X-dcgpu`. The 30-day evidence window in this issue is dominated by the first shape; the second is documented separately in internal AMD tracker `ROCM-23952`.

### Primary shape â€” degraded test throughput

Tests run at much slower-than-baseline throughput, and the test suite does not finish within the 60-minute GitHub Actions step budget. The job is then killed by the step timeout, with gtest output continuing right up to the kill.

### Secondary shape â€” silent stall

The second shape is gtest output stopping cleanly mid-run, with the rest of the 60-minute step silent. The clearest example is run [193](https://github.com/ROCm/rocm-libraries/actions/runs/25207030873) attempt 2 ([job 73943391135](https://github.com/ROCm/rocm-libraries/actions/runs/25207030873/job/73943391135), 2026-05-01) â€” `rocsolver` shard 1/3, 56-minute silent gap between `[ RUN ] checkin_lapack/GESVD.batched__float_complex/108` and the step-kill annotation. The same shard / same commit passed in 13 minutes on attempt 3 ([job 73959967301](https://github.com/ROCm/rocm-libraries/actions/runs/25207030873/job/73959967301)). The test executable from that build reproduced cleanly in under 4 minutes per run when executed outside TheRock CI on the same hardware (see internal AMD tracker `ROCM-23952` for the offline-reproduction logs).

Both shapes terminate identically as 60-minute Test-step timeouts, and both have repeatedly resolved on retry without code change. The coexistence of two distinct failure shapes on the same workflow, both resolving on retry, points at the runner environment as the shared cause.

### Variance source: per-job runner draw, not per-night cluster state

Two structural observations help frame what runner-side investigation should target.

1. **Every Linux `gfx942` runner in the 30-day window is single-use.** Across 1,001 Linux gfx942 test jobs, there are 1,001 unique runner names, all matching `linux-gfx942-2gpu-core42-ossci-rocm-x8khf-runner-<5char>` (mean appearances per runner: 1.00). This is the pod-per-job actions-runner-controller model â€” each test shard runs on a fresh ephemeral pod, and there is no persistent runner to inspect after a slow run. Anything that varies pod-to-pod â€” host placement on the underlying cluster, co-tenant load on that host, container image pull cache state, storage / artifact-network I/O, GPU thermal or power state at job start â€” is the natural candidate source for the per-shard variability shown above.

2. **Slow shards on a given night are largely independent across libraries.** Per-run mean shard durations on `gfx94X-dcgpu` correlate only weakly across libraries (Pearson r over 13 same-day pairs):

   | pair | r |
   |---|---:|
   | `rocroller` â†” `miopen` | +0.68 |
   | `rocsolver` â†” `miopen` | +0.37 |
   | `rocsolver` â†” `rocroller` | +0.26 |
   | `rocsolver` â†” `rocwmma` | âˆ’0.36 |
   | `miopen` â†” `rocwmma` | âˆ’0.18 |
   | `rocroller` â†” `rocwmma` | âˆ’0.25 |

   The pattern is consistent with each shard drawing an independent runner from a pool with non-trivial pod-to-pod variance. The long-running suites (`rocsolver`, `rocroller`, `miopen`) are most likely to draw a slow pod and tip past the 60-minute step budget on any given night, and the unrecoverable nights are those where multiple shards across multiple libraries draw slow pods together.

Together these point investigation toward host-placement and per-pod-environment telemetry on the OSS gfx942 runner pool.

## Suggested investigation

The central question: **why does the same shard, on the same workflow file, on the same runner pool, sometimes complete in 13 minutes and sometimes in 65+ minutes?** Variability of that magnitude points to the runner pool. Useful starting points:

- **Host-placement correlation across nights.** Linux `gfx942` runners are single-use ephemeral pods (see "Variance source" above), so per-runner tracking is not meaningful, but the *underlying host* a pod lands on is. Whether slow shards consistently land on a specific subset of hosts, host classes, or rack/zone placements â€” and whether those are co-located with high-utilization tenants â€” would be the cleanest way to confirm or rule out the host-placement hypothesis. This requires cluster-side telemetry that does not appear in the GitHub Actions job records.
- **Resource contention on the runner host.** CPU load, memory pressure, IO waits, network, container overhead, other tenants on the same host. A snapshot at the moment of slowest test execution would be informative.
- **GPU and driver state.** `rocm-smi` output (clock state, throttling, queue depth, ECC events), `dmesg`, recent driver reloads. Whether the slow nights correlate with any GPU-side anomaly.
- **Step-level wall-clock progress monitor.** A parallel step that records per-minute test-throughput metrics would directly distinguish "slow throughout" from "fine for 50 minutes then hangs for 10" from "stair-step slowdown" â€” the inside-the-step shape we currently can't see from the outside.
- **Cross-system corroboration.** The same library code runs cleanly on a separate CI system (different runner pool, different orchestration) over the same window. A controlled comparison (same test list, different runner configurations) would tighten the comparison.
