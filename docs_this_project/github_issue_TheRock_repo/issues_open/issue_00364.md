# #364 - Provide a compatibility path for the rocprofv3 transition

**State**: open  
**Author**: stellaraccident  
**Created**: 2025-04-08T02:21:05Z  
**Updated**: 2025-06-03T01:33:29Z

---

This issue should be kept open until we can remove the rocprof<3 compatibility fixes.

In ROCm 6.4, a decision was made to provide a non-compatible replacement library for the old `libroctx64.so`, which is used to provide profile timeline markers. Unfortunately, this library is widely used in a fairly ad-hoc way across the ecosystem (including in ROCm itself) and the new library exports the same symbols as the old, which can result in many issues, depending on load order/etc.

For TheRock, we are providing a source-level compatibility shim so that clients that use the old `roctracer/roctx.h` header get the new header from `rocprofiler-sdk-roctx`. The actual shared library, however, is another matter because the old uses versioned symbols and the new uses naked symbols. While arguably an improvement, this makes them very incompatible and resistant to the variety of shimming solutions that would otherwise exist for maintaining ABI compatibility. Since we generally expect that linking against `TheRock` by default will align with a future ROCm major version, we focus on source level interop right now and leave anyone who has binaries that link against the old `libroctx64.so` to have an unresolved library.

We provide the build time `roctracer-compat` CMake shim library which exports the header under the original path and declared the library such that it will link against the new rocprofv3 provided variant.

Note that this alone isn't enough because it is very non-standard how various clients located the libroctx64.so library. Since roctracer did not provide CMake packages, most have some kind of a `find_library(roctx64)`, which is just broken (i.e. relies on finding libraries in default/system locations). Some also just do a `-lroctx64`, which is even worse. We generally just make CMake level changes for these to instead `target_link_libraries` `roctracer-compat::roctx`. This gets everyone linking against the same thing at runtime, regardless of whether using old headers or new. Future cleanups can redirect clients to the newer package/library and avoid the shim.
