### Problem Description

Despite the fix recently merged in[ PyTorch PR #179138](https://github.com/pytorch/pytorch/pull/179138) (which fixed the missing DLL header exports), running FP8 matrix multiplication using torch.nn.functional.scaled_mm (_scaled_mm_v2) still triggers a fatal Windows access violation (0xC0000005) inside torch_hip.dll on RDNA4 (gfx120X) hardware.

Because this happens at the HIP driver/library level, the Python process is instantly killed by the OS before any exception can be caught. It appears that hipblasLt is segfaulting during the operation on Windows rather than cleanly executing or returning a not-supported status.

### Steps to Reproduce

Steps to Reproduce
Setup a Windows environment with an RDNA4 GPU.

Install PyTorch with ROCm 7.x support (via nightly) on Python 3.12.

Attempt to run FP8 inference (e.g., loading Ideogram 4 models in ComfyUI with [comfy-kitchen](https://github.com/Comfy-Org/comfy-kitchen)).

When the script invokes _scaled_mm_v2 with FP8 weights, the process immediately hard-crashes with 0xC0000005.

### Additional Information

Workaround note:
If we manually intercept the code before it hits _scaled_mm_v2 and instead cast the tensors to torch.float8_e4m3fn and route them through the older torch._scaled_mm function, the RDNA4 card handles the math perfectly without crashing. 
Found the workaround here: https://github.com/Comfy-Org/comfy-kitchen/issues/32#issuecomment-4094424801
The segfault is entirely isolated to _scaled_mm_v2.
