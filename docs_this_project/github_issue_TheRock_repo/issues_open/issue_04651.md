## Problem

In https://github.com/ROCm/TheRock/pull/4334, trying to add rocPRIM benchmark build support in TheRock.

rocPRIM benchmark builds fail with following error when building in environments where AMD SMI is not installed system-wide (e.g., TheRock super-project builds from source).

```
[rocPRIM_tests] /__w/TheRock/TheRock/rocm-libraries/projects/rocprim/benchmark/../../../shared/primbench/primbench.hpp:23:10: fatal error: 'amd_smi/amdsmi.h' file not found
[rocPRIM_tests]    23 | #include <amd_smi/amdsmi.h>
[rocPRIM_tests]       |          ^~~~~~~~~~~~~~~~~~
[rocPRIM_tests] 1 error generated when compiling for gfx942.
[rocPRIM_tests] ninja: build stopped: subcommand failed.
```
https://github.com/ROCm/TheRock/actions/runs/24379880541/job/71203849233

## Root Cause

`benchmark/CMakeLists.txt` links against `amd_smi` at [line 122](https://github.com/ROCm/rocm-libraries/blob/develop/projects/rocprim/benchmark/CMakeLists.txt#L122):

`  target_link_libraries(${BENCHMARK_TARGET} PRIVATE amd_smi)`

But never calls `find_package(amd_smi)` to create the imported target. Without the imported target, CMake treats `amd_smi` as a plain library name, it can resolve the linker flag but does not propagate include directories.

This works in standalone/system builds because `/opt/rocm/include` is already in the compiler's default search path, so `#include <amd_smi/amdsmi.h>` from primbench.hpp resolves implicitly. In from-source builds (like TheRock), amdsmi headers are in the build tree and require find_package to set up include paths.

Other projects in rocm-libraries (rocBLAS, hipBLASLt, rocWMMA) correctly call `find_package(amd_smi)` before using the target.
