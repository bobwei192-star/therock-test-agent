## Summary

Track the work needed to make TheRock CI more targeted for submodule bump PRs such as `rocm-libraries` and `rocm-systems`.

The automation should use `BUILD_TOPOLOGY.toml` to determine which build stages are affected by source changes, which stages can reuse prebuilt artifacts, and eventually which test components should run.

## Proposed Work

### 1. Build topology reverse indexes #5434 

Add reusable helpers for mappings such as:

- source set -> artifact groups
- artifact group -> artifacts
- artifact group -> build stages
- artifact -> build stages
- stage -> source sets
- source set -> stages

### 2. Changed path classification

Map changed files to affected source sets using `.gitmodules` and `BUILD_TOPOLOGY.toml`.

Also classify changes that should force a full rebuild, such as topology, build tooling, CMake, workflow, or packaging changes.

### 3. Stage impact planning

Compute:

- stages that must rebuild
- stages eligible for prebuilt artifact reuse
- reasons for each decision

### 4. Baseline run selection #5435

Select and validate a compatible successful baseline run before copying prebuilt artifacts.

Validation should cover workflow success, required artifacts, platform/family compatibility, and stale or ambiguous baselines.

### 5. Dry-run CI integration

Expose computed targeted CI decisions from `configure_multi_arch_ci.py` without changing build behavior yet.

Example outputs:

- `computed_rebuild_stages`
- `computed_prebuilt_stages`
- `computed_baseline_run_id`
- `targeted_ci_reason`

### 6. Opt-in prebuilt stage reuse

Enable stage reuse behind an explicit feature flag, workflow input, or repository variable.

Fallback to full rebuild when no valid baseline is available.

### 7. Test impact mapping

Map impacted topology artifacts/stages to test matrix components and validate the mapping against existing test configuration.

### 8. rocm-libraries and rocm-systems adapters

Connect existing submodule/project classification logic in `rocm-libraries` and `rocm-systems` to the shared TheRock targeting logic.

## Acceptance Criteria

- Targeted CI decisions are visible in dry-run mode before affecting builds.
- CI explains why each stage is rebuilt or reused.
- Prebuilt reuse only happens with a validated successful baseline.
- Missing or invalid baseline data falls back to full rebuild.
- The approach supports both `rocm-libraries` and `rocm-systems` submodule bumps.
