Following up on https://github.com/ROCm/TheRock/pull/5153#discussion_r3220301745.

We have https://github.com/ROCm/TheRock/blob/main/CMakePresets.json but not all CI/CD workflows are using presets to set their CMake options (including flag settings). For these presets to work as expected, we should use them consistently in our workflows and make it clear when each preset should be used.

See also:
* https://github.com/ROCm/TheRock/issues/1781

### Background: how presets are used today

For multi-arch CI _**and**_ releases, this is the flow today:

* CMake presets are attached to "build variants" here: https://github.com/ROCm/TheRock/blob/587268c23ad01bed66314596ddee39412f9d040e/build_tools/github_actions/amdgpu_family_matrix.py#L89-L117
* The "build variant" is selected during CI configuration: https://github.com/ROCm/TheRock/blob/587268c23ad01bed66314596ddee39412f9d040e/build_tools/github_actions/configure_multi_arch_ci.py#L179-L183
* The cmake preset is extracted from the variant: https://github.com/ROCm/TheRock/blob/587268c23ad01bed66314596ddee39412f9d040e/build_tools/github_actions/configure_multi_arch_ci.py#L911
* The cmake preset is passed from the CI configuration through to the "build artifacts" workflow: https://github.com/ROCm/TheRock/blob/587268c23ad01bed66314596ddee39412f9d040e/.github/workflows/multi_arch_release_linux.yml#L46

### Specific issues with the current approach

https://github.com/ROCm/TheRock/pull/5153 intended to set `THEROCK_FLAG_STAMP_LIBRARY_GIT_VERSIONS` off for developers and CI builds and enable in releases, but due to how CMake presets are not used throughout all such workflows it actually enabled it for multi-arch Windows CI and did not enable it for Linux releases:

Configuration | Intended flag value | Actual flag value
-- | -- | -- 
Linux local development | off | ✅ off 
Linux CI | off | ✅ off
Linux releases | on | ❌ off
Windows local development | off | ✅ off
Windows CI | off | ❌ on
Windows releases | on | ✅ on

### Proposed fixes

I had drafted two different ways to approach this:

* https://github.com/ROCm/TheRock/pull/5191
* https://github.com/ROCm/TheRock/pull/5192

agent review feedback considering each:

> • I’d lean toward the 5192 direction, but I’d tighten the model so it is clearly a “build profile” registry, not just another loose build_variant dict.
> 
>   5191 Case
> 
>   5191 is attractive if the main problem is narrowly “which CMake preset does this workflow use?” It makes preset selection visible in the workflow YAML and removes build_variant_cmake_preset from the generated build config. That is easier to audit for the current stamping bug: CI paths can say windows-dev / empty Linux dev for now, while release paths hardcode windows-release / linux-release.
> 
>   The downside is that it assumes preset choice is fixed by workflow entry point. PR 5288 and possible ASan package releases weaken that assumption. If release workflows later need release_asan, 5191’s hardcoded release inner workflows become a place you have to unwind. It also leaves CI policy split across YAML and Python: preset in workflows, family eligibility in matrix data, ASan sandbox/test behavior in script conditionals.
> 
>   5192 Case
> 
>   5192 is better if the top-level key means “what product/configuration am I asking CI to build?” In that model, ci_asan or future release_asan can map to a CMake preset, family allow-list, artifact naming, build runner policy, package behavior, and test behavior. That matches the pressure from 5288, where the technical question is not just “which preset?” but “which targets do ASan builds build, which do they test, when, and on which runner pool?”
> 
>   The risk is that 5192 can become another opaque indirection layer unless it is made explicit and typed. I would avoid letting all_build_variants grow as an unstructured bag of flags. Also, the name build_variant is now doing too much. build_profile or ci_profile would be more honest, even if compatibility keeps the old input name for a while.
> 
>   My Recommendation
> 
>   Take the 5192 architecture, but sharpen it:
> 
>   - Treat keys like ci, ci_asan, ci_tsan, release, release_asan as profile keys.
>   - Keep the workflow YAML explicit about the selected profile, not every derived field.
>   - Resolve derived fields in one Python-owned registry: CMake preset, artifact suffix/group, family allow-list, build runner class, test policy, package policy.
>   - Add tests that assert full resolved profiles, especially for ci_asan and future release_asan.
>   - Replace string heuristics like "san" in build_variant with explicit profile fields.
>   - Make CI summaries print the resolved profile so the indirection stays inspectable.
> 
>   A good compromise is: workflows choose profile: ci_asan; scripts expand that to linux-dev-asan, allowed families, sandbox/nightly test policy, and artifact naming. That keeps the choice visible while avoiding duplicated YAML policy.
