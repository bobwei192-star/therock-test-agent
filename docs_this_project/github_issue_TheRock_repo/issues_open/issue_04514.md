### Suggestion Description

Authored by me (@benvanik) + codex and reviewed for correctness. This is not a proposed final API contract. The goal is to describe the user problem, the minimum reflection data needed for correct dispatch, and the shape of runtime support that would keep HSA applications from depending on private loader details. We currently are carrying a lot of brittle code for metadata handling that would be nice to have accessible via the ROCR APIs in some form. Not only does it make it much easier to work with the COs but it allows for better versioning of the code object metadata and stops a proliferation of implementation detail leakage up into projects written against HSA.

## Summary

An HSA application can already load an AMDGPU code object, enumerate kernel
symbols, query the kernel object handle, allocate a kernarg segment, and submit
an AQL dispatch packet.

What the application cannot currently discover through the public HSA symbol
query surface is the kernel argument ABI:

- Which byte ranges in the kernarg segment correspond to pointer arguments.
- Which byte ranges correspond to by-value scalar or aggregate arguments.
- Which arguments are hidden ABI/runtime arguments.
- Which address space and access qualifiers apply to pointer-like arguments.
- Which argument kinds are unsupported by a given higher-level dispatcher.

LLVM-produced AMDGPU code objects already contain this information in the
`AMDGPU` / `NT_AMDGPU_METADATA` ELF note as AMDGPU MessagePack metadata. The
problem is not that the information is absent. The problem is that each HSA
client that wants to perform argument reflection must currently become an ELF
note walker and AMDGPU metadata parser.

The baseline request is a supported HSA/ROCR API for retrieving the raw AMDGPU
metadata note payload from the code object that the runtime accepted. The
preferred request is decoded, structured kernel argument reflection.

## User Scenario

A user has an AMDGPU kernel compiled by LLVM, clang, HIP-clang, or another
frontend that ultimately emits a normal AMDGPU HSA code object. The input may be
a `.hsaco`, `.co`, or otherwise loadable AMDGPU ELF code object.

The user wants to write a direct HSA application that does this:

1. Create an HSA code object reader from memory or a file.
2. Load the code object for an HSA agent.
3. Freeze the executable.
4. Enumerate kernel symbols.
5. Choose a kernel by symbol name.
6. Build a valid kernarg buffer for that kernel.
7. Submit an AQL dispatch packet.

Steps 1 through 5 and step 7 are covered well by existing HSA APIs. Step 6 is
where public reflection is missing.

The HSA application can query the total kernarg segment size and alignment for a
kernel symbol. That tells the application how large the buffer must be, but not
how to populate it.

For a trivial kernel, a user might know the argument list out of band and write
the correct bytes manually. For general tooling, plugin systems, external
kernel loading, language runtimes, graph executors, or dispatch adapters, that
is not enough. Those systems need to derive a packing plan from the code object
itself.

## Existing HSA Surface That Already Works

The following APIs provide the basic load and launch machinery:

- `hsa_code_object_reader_create_from_memory`
- `hsa_code_object_reader_create_from_file` or AMD loader file+offset variants
- `hsa_executable_create_alt`
- `hsa_executable_load_agent_code_object`
- `hsa_executable_freeze`
- `hsa_executable_iterate_agent_symbols`
- `hsa_executable_get_symbol_by_name`
- `hsa_executable_symbol_get_info`

For a kernel symbol, an HSA user can query:

- `HSA_EXECUTABLE_SYMBOL_INFO_TYPE`
- `HSA_EXECUTABLE_SYMBOL_INFO_NAME_LENGTH`
- `HSA_EXECUTABLE_SYMBOL_INFO_NAME`
- `HSA_EXECUTABLE_SYMBOL_INFO_KERNEL_OBJECT`
- `HSA_EXECUTABLE_SYMBOL_INFO_KERNEL_KERNARG_SEGMENT_SIZE`
- `HSA_EXECUTABLE_SYMBOL_INFO_KERNEL_KERNARG_SEGMENT_ALIGNMENT`
- `HSA_EXECUTABLE_SYMBOL_INFO_KERNEL_GROUP_SEGMENT_SIZE`
- `HSA_EXECUTABLE_SYMBOL_INFO_KERNEL_PRIVATE_SEGMENT_SIZE`
- `HSA_EXECUTABLE_SYMBOL_INFO_KERNEL_DYNAMIC_CALLSTACK`

This is enough to allocate dispatch resources and fill the mechanical AQL packet
fields. It is not enough to reflect the kernel argument layout.

The AMD loader extension also exposes useful loaded-code-object properties such
as storage memory base/size, load base/size, URI, and agent. That can help a
client find the accepted object bytes, but the client still has to parse the ELF
container and locate the AMDGPU metadata note.

## Why Kernarg Reflection Matters

The kernarg segment is a byte buffer whose layout is defined by the kernel ABI.
The HSA dispatch packet only points at that buffer. If a host application writes
the wrong bytes at the wrong offsets, the dispatch may still launch, but the
kernel observes invalid arguments.

Correct reflection lets a host application answer questions such as:

- Which arguments consume host-provided values?
- Which arguments are global-memory pointers?
- Which arguments are by-value scalars or aggregates?
- Which arguments are images, samplers, queues, pipes, or other special objects?
- Which arguments are hidden ABI fields that the runtime, compiler, or dispatch
  adapter must provide?
- What offset, size, and alignment must each argument use?

Without this information, a generic HSA dispatcher has only three bad choices:

- Require a separate sidecar manifest next to every code object.
- Ask users to provide raw kernarg bytes manually.
- Reimplement enough ELF and AMDGPU metadata parsing to recover the information
  that the code object already contains.

The first option fragments the ecosystem. The second is only usable for expert
and test-oriented workflows. The third creates duplicate parsers that can drift
from ROCR and LLVM behavior.

## Metadata Already Present In LLVM Code Objects

Modern LLVM AMDGPU HSA code objects contain an ELF note with:

- Owner/name: `AMDGPU`
- Type: `NT_AMDGPU_METADATA` (`32`)
- Descriptor payload: AMDGPU MessagePack metadata

The metadata includes top-level records such as:

- `amdhsa.version`
- `amdhsa.target`
- `amdhsa.kernels`

Each kernel record can include fields such as:

- `.name`
- `.symbol`
- `.kernarg_segment_size`
- `.kernarg_segment_align`
- `.group_segment_fixed_size`
- `.private_segment_fixed_size`
- `.reqd_workgroup_size`
- `.workgroup_size_hint`
- `.max_flat_workgroup_size`
- `.args`

The `.args` list contains the important reflection data. Argument records can
include:

- `.offset`
- `.size`
- `.value_kind`
- `.address_space`
- `.access`
- `.actual_access`
- `.type_name`
- `.name`
- additional qualifier fields depending on metadata version and frontend

From a dispatcher's perspective, `.offset`, `.size`, `.align` when available or
derivable, `.value_kind`, `.address_space`, and `.access` are the critical
fields.

## Baseline Requirement: Raw Metadata Note Access

The minimum useful API is a supported way to retrieve the descriptor payload of
the AMDGPU metadata note from a code object that ROCR has accepted.

Useful properties:

- The query works for code objects created from memory and from files.
- The query returns metadata from the same object representation accepted by the
  HSA code object manager.
- The query does not require the client to walk section headers, program
  headers, symbol tables, or note records.
- The query exposes the note owner, type, descriptor pointer, and descriptor
  size, or supports lookup by owner and type.
- The returned pointer lifetime is explicit.
- Absence of the note is distinguishable from malformed input, unsupported code
  object versions, invalid handles, or runtime failures.
- The API does not require applications to link compression libraries or parse
  higher-level HIP/offload bundle containers.

One possible lookup-style API shape is:

```c
hsa_status_t hsa_ven_amd_loader_code_object_reader_get_note(
    hsa_code_object_reader_t reader,
    const char* owner,
    uint32_t type,
    size_t* desc_size,
    const void** desc);
```

An iterator form would also work:

```c
typedef hsa_status_t (*hsa_ven_amd_loader_code_object_note_callback_t)(
    hsa_code_object_reader_t reader,
    const char* owner,
    uint32_t type,
    const void* desc,
    size_t desc_size,
    void* user_data);

hsa_status_t hsa_ven_amd_loader_code_object_reader_iterate_notes(
    hsa_code_object_reader_t reader,
    hsa_ven_amd_loader_code_object_note_callback_t callback,
    void* user_data);
```

A loaded-code-object query would also satisfy the core requirement:

```c
hsa_status_t hsa_ven_amd_loader_loaded_code_object_get_note(
    hsa_loaded_code_object_t loaded_code_object,
    const char* owner,
    uint32_t type,
    size_t* desc_size,
    const void** desc);
```

The code-object-reader form is useful before an application finalizes its own
runtime representation. The loaded-code-object form is useful because it is
unambiguously tied to the object that ROCR accepted and loaded. Either would
remove the need for every HSA application to implement ELF note discovery.

## Preferred Requirement: Decoded Kernel Argument Reflection

The better API is decoded kernel argument reflection. In that model, ROCR/HSA
exposes the kernel ABI information directly instead of forcing every client to
parse AMDGPU MessagePack metadata.

A minimal decoded record could look like this:

```c
typedef struct hsa_ven_amd_kernel_arg_info_t {
  uint32_t offset;
  uint32_t size;
  uint32_t align;
  const char* value_kind;    // "global_buffer", "by_value", "hidden_*", etc.
  const char* address_space; // Optional. For example "global", "constant".
  const char* access;        // Optional. For example "read_only".
} hsa_ven_amd_kernel_arg_info_t;
```

The most important fields are:

- `offset`: byte offset in the kernarg segment.
- `size`: byte size of the argument storage.
- `align`: required or effective alignment for the argument storage.
- `value_kind`: ABI classification such as `global_buffer`, `by_value`, or
  `hidden_*`.

The optional fields are still valuable:

- `address_space`: helps distinguish global, constant, local, private, and
  generic pointer-like values.
- `access`: helps diagnostics and validation for read-only/write-only
  arguments.

One possible symbol-attached API shape is:

```c
typedef hsa_status_t (*hsa_ven_amd_kernel_arg_callback_t)(
    hsa_executable_symbol_t symbol,
    const hsa_ven_amd_kernel_arg_info_t* arg,
    void* user_data);

hsa_status_t hsa_ven_amd_loader_executable_symbol_iterate_kernel_args(
    hsa_executable_symbol_t symbol,
    hsa_ven_amd_kernel_arg_callback_t callback,
    void* user_data);
```

Another possible loaded-code-object shape is:

```c
hsa_status_t hsa_ven_amd_loader_loaded_code_object_iterate_kernel_args(
    hsa_loaded_code_object_t loaded_code_object,
    const char* kernel_symbol_name,
    hsa_ven_amd_kernel_arg_callback_t callback,
    void* user_data);
```

The exact handle placement is less important than the contract:

- The reflection data corresponds to the code object ROCR accepted.
- The argument order matches the metadata/source order.
- Hidden arguments are exposed, not silently skipped.
- Unsupported or unknown metadata values are reported in a way that lets the
  application reject the kernel with a precise diagnostic.
- Metadata version differences are handled by ROCR where possible.

## Hidden Arguments Must Be Visible

Hidden arguments are part of the ABI contract. A generic dispatcher cannot
pretend they are absent.

Examples include hidden values for:

- Block or grid counts.
- Group sizes.
- Grid dimensions.
- Dynamic LDS size.
- Private segment buffer or related implementation details.

A host application needs to know whether hidden arguments exist so it can choose
one of three explicit behaviors:

- Populate the hidden value correctly.
- Prove that the runtime or compiler path supplies it elsewhere.
- Reject the kernel as unsupported for that dispatch mode.

Omitting hidden arguments from reflection would make invalid dispatches look
valid. A decoded API should expose them, even if the initial consumer rejects
most hidden kinds.

## Useful Kernel-Level Reflection

Argument records are the primary need. The following kernel-level fields are
also useful if the same API naturally exposes them:

- Kernel source name.
- Kernel descriptor symbol name.
- Code object metadata version.
- Target string.
- Kernarg segment size and alignment.
- Fixed group segment size.
- Fixed private segment size.
- Required workgroup size.
- Workgroup size hint.
- Maximum flat workgroup size.

Some of these are already available through HSA symbol info, but having them in
the same reflection view improves diagnostics and gives applications one stable
place to understand the kernel ABI.

## Expected Dispatcher Policies

ROCR does not need to define the policy that a higher-level dispatcher applies
to reflected metadata. ROCR only needs to expose what the code object says.

Different HSA applications may apply different policies. For example:

- A raw HSA test harness may accept only caller-provided kernarg bytes.
- A graph runtime may map pointer arguments to externally managed device
  buffers and scalar arguments to captured constants.
- A language runtime may validate argument names and type names against its own
  call boundary.
- A plugin executor may reject images, samplers, queues, pipes, or hidden
  arguments until it has explicit support for those ABI features.

All of those policies depend on the same underlying reflection data.

## Non-Goals

This request is not asking HSA to become a HIP fat binary loader for every
direct HSA client.

The reflection need applies to raw AMDGPU HSA code objects that the HSA runtime
can already accept. Applications should not have to link zstd, parse compressed
code object bundles, implement clang offload bundle selection, or emulate HIP
container loading merely to dispatch a normal LLVM-produced HSACO.

This request is also not asking ROCR to guess host-language semantics from LLVM
type names. Type names, argument names, and qualifiers are useful diagnostics
and validation aids, but the core ABI contract is still offset, size, alignment,
value kind, address space, and access.

## Why Raw Note Access Is The Baseline

Raw note access is the smallest API that fixes the worst layering problem.

It lets ROCR own:

- Accepting and validating code objects.
- Locating metadata inside the accepted code object representation.
- Providing stable access to the note descriptor bytes.

It lets applications own:

- Parsing AMDGPU metadata if they need complete custom control.
- Applying their own dispatch policy.
- Deciding which kernel ABI features they support.

This keeps application code from having to duplicate ELF note discovery and
container-specific loader behavior.

## Why Decoded Reflection Is Better

Decoded reflection is better for most users because it removes the need for
every HSA application to carry:

- ELF note walking code.
- MessagePack decoding code.
- AMDGPU metadata schema version handling.
- Hidden-argument classification logic.
- Offset, size, alignment, and qualifier extraction.

It also gives ROCR a single compatibility layer for metadata-version evolution.
Applications can consume a stable reflection ABI while LLVM and AMDGPU metadata
formats continue to evolve.

## Desired Outcome

The minimum useful outcome is:

> Given a code object reader or loaded code object that ROCR has accepted, an
> HSA client can retrieve the raw `AMDGPU` / `NT_AMDGPU_METADATA` note payload
> without walking ELF structures itself.

The preferred outcome is:

> Given a loaded kernel symbol, an HSA client can iterate decoded kernel
> argument metadata records containing offset, size, alignment, value kind, and
> optional address/access qualifiers.

Either outcome would make direct HSA use of LLVM-produced AMDGPU code objects
substantially cleaner. It would let generic dispatchers build correct kernarg
buffers from the code object ABI, avoid sidecar manifests for common cases, and
keep low-level ELF/MessagePack parsing out of every HSA application.
