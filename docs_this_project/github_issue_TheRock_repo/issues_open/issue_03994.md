### Problem Description

An AI has tracked down the complete cause:

The full chain of causation:

The amd-llvm clang compiler (built by TheRock) detects your system's ROCm installation: Found HIP installation: /opt/rocm, version 7.12.60430. This is built into clang's driver logic for HIP support.

During CMake's compiler identification, or through some related discovery mechanism, amd_comgr_DIR gets set in the ocl-clr CMakeCache to /opt/rocm/lib/cmake/amd_comgr_stub (a stub package from your system ROCm install).

When ROCclrLC.cmake calls find_package(amd_comgr 2.9 CONFIG ...), the dep provider correctly intercepts it and rewrites the call with NO_DEFAULT_PATH PATHS <build_tree>.

However, CMake's find_package in CONFIG mode always checks the <PackageName>_DIR cache variable before any search, even with NO_DEFAULT_PATH. Since amd_comgr_DIR is already cached pointing to /opt/rocm/lib/cmake/amd_comgr_stub, CMake uses that location, completely bypassing the dep provider's redirection.

The stub package at /opt/rocm provides an amd_comgr target with INTERFACE_INCLUDE_DIRECTORIES resolving to /opt/rocm/include.

This -isystem /opt/rocm/include appears before the correctly-resolved -isystem .../ROCR-Runtime/dist/include in the compile command, causing the compiler to find the old HSA headers from /opt/rocm/include/hsa/hsa_ext_amd.h instead of the fresh ones from the build tree.

The fix (line 67 of cmake/therock_subproject_dep_provider.cmake):


therock_subproject_dep_provider.cmake
Lines 63-67
    # Clear any cached <PackageName>_DIR that may point outside the
    # super-project (e.g. to a system /opt/rocm install or stub package).
    # find_package() always checks this cache variable first, even with
    # NO_DEFAULT_PATH, so a stale value here defeats the redirection.
    unset(${package_name}_DIR CACHE)
By unsetting the cache variable before the rewritten find_package call, we force CMake to actually search in the PATHS we specify (the build tree's dist directory), ensuring the super-project's built packages are always used regardless of what's installed on the system.
