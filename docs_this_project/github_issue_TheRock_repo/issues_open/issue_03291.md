## Background

Similar to https://github.com/ROCm/TheRock/issues/1559, we've had issues like https://github.com/ROCm/TheRock/issues/3042 slip through since we don't build PyTorch as part of our CI, only opt-in via manually triggered dev releases or regularly scheduled nightly releases.

## Goals

- [x] Pull requests in TheRock should be able to opt-in to building PyTorch as part of the regular CI workflow
- [x] Start with latest stable PyTorch version (currently 2.9 or 2.10), for a single Python version (3.12 or 3.13), for Linux
- [x] Add Windows too
- [ ] Add the option to build for more versions (via configure_ci, github labels, etc.)
- [ ] After building packages, we should be able to run a subset of PyTorch tests
- [ ] Add the option to trigger these jobs for jobs in rocm-libraries and rocm-systems (see https://github.com/ROCm/TheRock/issues/3177 and ongoing work to de-duplicate the workflows in use across the different repos)

## Design

My full design plan and notes are at https://github.com/ScottTodd/claude-rocm-workspace/blob/main/tasks/active/pytorch-ci.md.

### Workflow structure

At a high level, I'm currently planning on forking [`build_portable_linux_pytorch_wheels.yml`](https://github.com/ROCm/TheRock/blob/main/.github/workflows/build_portable_linux_pytorch_wheels.yml) to a new `build_portable_linux_pytorch_wheels_ci.yml` that will reuse the same underlying scripts but contain CI-specific logic (uploading to the artifacts bucket, _not_ uploading to a release bucket).

I would ideally want to have each workflow only be responsible for one thing:

* Build workflows build artifacts/packages and upload them to an "artifacts" bucket
* Test workflows download artifacts/packages
* Release workflows copy artifacts/packages into a "release" bucket based on a policy (e.g. "tests passed")

That's currently difficult to implement in the existing release workflows due to how they use `strategy.matrix` and rely on plumbing the `torch_version` output of each build job to a test job. If we move to a commit manifest approach for our workflows (see https://github.com/ROCm/TheRock/issues/1236), we could avoid the need for that plumbing. We could also unroll the job matrix as PyTorch does for workflows like https://github.com/pytorch/pytorch/blob/main/.github/workflows/generated-linux-binary-manywheel-nightly.yml.

### Package uploading

I plan on using the ROCm Python packages that CI jobs will generate from https://github.com/ROCm/TheRock/issues/1559. Those are currently uploaded along with an index page using https://github.com/ROCm/TheRock/blob/main/build_tools/github_actions/upload_python_packages.py. If we tried to upload pytorch packages (even for just a single version) to the same subdirectory, we would need to regenerate the index page to include the previous files _and_ the new files. A few options for that:

1. Generate the index page locally from a directory with _both_ the ROCm packages and the PyTorch packages and upload it (this wouldn't handle multiple parallel uploads to the same directory, e.g. for multiple pytorch versions)
2. List files in the S3 bucket and generate the index page based on that, NOT based on local files, as is done by [`build_tools/third_party/s3_management/manage.py`](https://github.com/ROCm/TheRock/blob/main/build_tools/third_party/s3_management/manage.py)
3. (Preferred) Move index file generation server-side

If we limit to a single PyTorch and Python version we might be able to get away with (1). Otherwise I'm planning on deferring package uploading (and thus testing and developer quality of life integration) until we have (3)

### Testing

This will depend on package uploading, so I'm not planning for it at the start. We probably will need a narrower set of tests to run compared to what we run nightly, to sidestep https://github.com/ROCm/TheRock/issues/2156
