### Summary

[RFC 0010](https://github.com/ROCm/TheRock/blob/main/docs/rfcs/RFC0010-Test-Scripts-Migration.md) migrates the ~35 test runner scripts from `build_tools/github_actions/test_executable_scripts/` into `rocm-systems` and `rocm-libraries`. That migration moves the scripts to the right owners, but it does not fix the underlying structural problems they carry with them:

- Each script duplicates the same 20–30 lines of setup boilerplate.
- The shared helpers scripts depend on live in `build_tools/github_actions/github_actions_utils.py` — a module whose name and location signal "GitHub Actions tooling", not "test utilities". That mismatch will be obvious once the scripts live outside TheRock.
- The scripts cannot be run locally by developers without knowing which environment variables CI sets.

Some of these concerns are also touched on in the [RFC 0010 discussion](https://github.com/ROCm/TheRock/discussions/3964).

---

### Problem 1 — Duplicated boilerplate will move with the scripts

The RFC 0010 migration seeds the owning repos from the current scripts as-is. Every migrated script will arrive carrying the same copy-pasted setup block:

```python
THEROCK_BIN_DIR = os.getenv("THEROCK_BIN_DIR")
logging.basicConfig(level=logging.INFO)

SHARD_INDEX = os.getenv("SHARD_INDEX", 1)
TOTAL_SHARDS = os.getenv("TOTAL_SHARDS", 1)
environ_vars = os.environ.copy()
environ_vars["GTEST_SHARD_INDEX"] = str(int(SHARD_INDEX) - 1)
environ_vars["GTEST_TOTAL_SHARDS"] = str(TOTAL_SHARDS)
```

The `ROCM_PATH` derivation (`Path(THEROCK_BIN_DIR).resolve().parent`) and the `++ Exec [...]$` log line are similarly repeated across ~28 scripts each. Migration moves the duplication to a new address; it does not reduce it.

There are also small inconsistencies baked into these copies: the `SHARD_INDEX` default is `1` (int) in most scripts and `"1"` (string) in others; the env dict is named `environ_vars` in ~20 scripts and `env` in ~6; `THEROCK_BIN_DIR` is sometimes a bare `str` and sometimes immediately `Path`-wrapped.

| Repeated pattern | Approx. scripts affected |
|---|---|
| `THEROCK_BIN_DIR = os.getenv(...)` boilerplate | ~28 |
| `logging.basicConfig(level=logging.INFO)` | ~28 |
| GTest sharding block (5 lines) | ~18 |
| `ROCM_PATH` derivation | ~6 |
| `sys.path` hack to import `github_actions_utils` | ~10 |
| `++ Exec [...]$` log line | ~28 |

### Problem 2 — Test-utility functions are buried in `github_actions_utils.py`

`build_tools/github_actions/github_actions_utils.py` has grown to contain two unrelated groups:

**GitHub Actions-specific** (correctly placed):
- `GitHubAPI`, `gha_send_request`, `gha_query_workflow_run_by_id`, etc.
- `gha_set_env`, `gha_set_output`, `gha_append_step_summary`, `gha_add_to_path`

**General ROCm/test utilities** (misplaced):
- `get_visible_gpu_count()` — runs `rocminfo`, counts GPU lines
- `get_first_gpu_architecture()` — runs `rocminfo`, returns first `gfxNNN`
- `find_matching_gpu_arch()` — matches a GPU arch against wildcard patterns
- `is_asan()` — checks `ARTIFACT_GROUP` for asan build
- `str2bool()` — generic string-to-boolean conversion

After migration, scripts in `rocm-systems` and `rocm-libraries` importing from `build_tools/github_actions/github_actions_utils` are importing a "GitHub Actions utils" module for `get_first_gpu_architecture()`. That name mismatch is confusing, and the path dependency on TheRock's `build_tools/github_actions/` directory needs a proper resolution.

RFC 0010 lists this as an open question: *"Resolve the `github_actions_utils` import dependency — either vendor the needed helpers, expose them as a small installable package, or accept an import from a checked-out TheRock path."* This issue is where that question gets answered.

---

### Problem 3 — Scripts are not runnable locally without CI context

Most scripts have no `argparse` interface and silently accept `None` for required values like `THEROCK_BIN_DIR`, failing later with a confusing `TypeError` from `Path(None)`. One script (`test_hipblaslt.py`) calls `.lower()` directly on `os.getenv("RUNNER_OS")` with no None guard — it raises `AttributeError` outside CI. After migration, developers in the owning repos encountering these scripts for the first time will have no clear signal about how to run them.

---

### Open Question

Where should the shared test utilities live once scripts have moved out of TheRock? The utilities need to be importable by scripts in `rocm-systems` and `rocm-libraries` without pulling in GitHub Actions-specific code. Options include vendoring helpers into each owning repo, exposing them as a small installable package, keeping a reference import from a checked-out TheRock path, or a new location within TheRock decoupled from `build_tools/github_actions/`. The right answer should inform the cleanup work in this issue.
