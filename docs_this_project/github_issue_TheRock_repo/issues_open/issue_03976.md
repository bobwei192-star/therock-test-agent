## Background

Historically, the bulk of ROCm installs were performed system-wide on Linux to a fixed `/opt/rocm/` path. Today, we support installing ROCm on Linux and Windows to a variety of paths via native packages, python packages, build directories, tarballs, and more. However, source code for many subprojects still heavily depends on `/opt/rocm/`. Some uses appear relatively safe on the surface, like as a fallback when no explicit path is chosen, but the pervasiveness of these patterns makes it difficult to spot which uses are truly dangerous.

### Why now?

Now that we've
1. Organized ROCm project sources into superrepos
2. Completed multiple Linux+Windows ROCm releases using TheRock with new packaging methods
3. Made progress turning down "legacy CI" (internal Jenkins etc.) pipelines in favor of open source CI/CD pipelines using GitHub Actions and based on TheRock

I think we're just about in the clear and can pay down this tech debt.

Note that ecosystem projects are harder to update, so we should provide solid guidance for them to follow, but that should be mostly orthogonal to fixing issues in ROCm itself. If an ecosystem project depends on `/opt/rocm/` it may not work with the new package structure / Windows / etc. regardless of whether ROCm itself still has hardcoded paths.

## Scope and auditing

I ran an audit rocm-libraries, rocm-systems, and llvm-project here: https://github.com/ScottTodd/claude-rocm-workspace/tree/main/reports/path-audit . That flagged ~371 issues flagged in rocm-libraries, ~490 in rocm-systems, and ~28 in llvm-project.

### Example classes of issues

* Default CMake install paths set to `/opt/rocm` (sometimes with `FORCE`)
* `find_package()` and `find_program()` calls that use `/opt/rocm` hints
* Data file runtime discovery that falls back to hardcoded `/opt/rocm/lib` paths
* Build/test/CI scripts like `install.sh`, `rmake.py`, etc. that hardcode `/opt/rocm`
* Scripts that hardcode `/opt/rocm/llvm/bin/clang-format` and fallback to no formatting
* Use of `ROCM_PATH`, `HIP_DIR`, and `HIP_PATH` environment variables

## Recommendations

I have some documentation drafted at https://github.com/ScottTodd/TheRock/blob/docs-path-discovery/docs/development/component_path_discovery.md with some draft recommendations / best practices to follow. We could polish that up and land it here as part of solving this.

In particular, I'd like to settle on some core principles:

1. **No hardcoded absolute paths.** References to `/opt/rocm`, `/usr/local`,
   or any other fixed filesystem location are banned. They break relocatable
   installs, containerized builds, and developer workflows where ROCm lives in
   a non-standard location.

1. **Self-contained packages.** Tools and libraries that are built and installed
   together in the same distribution tree must be able to find each other using
   only information embedded at install time — specifically, paths relative to
   their own installed location. No ambient state required.

1. **No environment-variable assumptions for intra-package discovery.**
   Environment variables like `ROCM_PATH`, `HIP_PATH`, and `ROCM_HOME` must
   never be required for components within the same install tree to find each
   other. These variables are a user-facing convenience for pointing external
   tools at a ROCm installation, not an internal wiring mechanism.

1. **Side-by-side installs must work.** Users may have multiple ROCm
   installations coexisting — e.g., different versions in separate Python
   venvs, or multiple tarball extractions on the same machine. Any reliance
   on global state (environment variables, global config files, hardcoded
   paths) breaks this. Each install tree must be fully self-describing so
   that the "active" one is determined solely by which tree's tools or
   libraries are being invoked.

1. **Fail loudly on misconfiguration; never silently fall back to a different
   install.** If a component cannot locate a required file or dependency
   through its intended discovery mechanism, it must report an error — not
   silently try a hardcoded system path. Loading files from a different ROCm
   installation (potentially a different version) produces wrong results that
   are far harder to diagnose than a clear "file not found" error. A fallback
   that silently crosses install boundaries violates every other principle
   in this list.

1. **User-facing discovery uses tool commands, not env vars.** External
   consumers (build scripts, CI pipelines, Python packages) should discover
   paths via tool commands like `hipconfig --path`, `hipconfig --rocmpath`, or
   `rocm-sdk path --root`. These tools compute the answer from their own
   installed location, so they work regardless of where the tree is rooted.

### Specific code patterns to consider

* CMake: remove inlined path hints to `find_package` calls, prefer an explicit `CMAKE_PREFIX_PATH` (perhaps setting _that_ at the top level by default as needed)
* Fail if library files can't be found, instead of falling back to a default (which is increasingly likely to point to a _different version of rocm_ -- huge risk for user issues in the field)
* Use automatic discovery with tool commands like `hipconfig --path` / `hipconfig --rocmpath` where `hipconfig` is on the existing `PATH` environment variable, avoiding custom environment variables like `ROCM_PATH`
* For tools/libraries _within rocm_, search relative paths based on established install structure or stamp tool locations into source files as part of packaging somehow
* When cascading through a set of search paths, favor the local install path _before_ environment variables (global state)
