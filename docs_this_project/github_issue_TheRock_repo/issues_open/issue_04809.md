# #4809 - Proposal: skip subprojects when their supported AMDGPU targets don't intersect THEROCK_AMDGPU_FAMILIES

**State**: open  
**Author**: FIM43-Redeye  
**Created**: 2026-04-24T17:43:53Z  
**Updated**: 2026-05-07T15:59:56Z  
**URL**: https://github.com/ROCm/TheRock/issues/4809  

---

## Summary

When `THEROCK_AMDGPU_FAMILIES` specifies arches that a subproject can't support, the subproject currently builds anyway, for its own internal default arch set. The resulting device code can't run on the user's hardware and burns significant build time. A cleaner behavior would be to auto-exclude the subproject when the intersection of its supported arches and the requested families is empty.

## Concrete symptom

User configures with `-DTHEROCK_AMDGPU_FAMILIES=gfx110X` (RDNA3 laptop dGPU + iGPU). Build includes RCCL:

```
-- Patching targets: ...; rccl_device_gfx906; rccl_device_gfx908;
   rccl_device_gfx90a; rccl_device_gfx942; rccl_device_gfx950; ...
```

RCCL's supported device targets are gfx9-family (CDNA) only; nothing in gfx110X. The user's configure selected an RDNA family, RCCL builds several CDNA variants anyway, none of them usable on the user's GPU. On a laptop this is 30+ minutes of LTO-heavy compilation that will never run.

## Current mechanism

- RCCL is declared with `USE_TEST_AMDGPU_TARGETS` in the comm-libs CMake.
- `THEROCK_TEST_AMDGPU_TARGETS` defaults to *all registered arches* (see #3698 -- intentional for CI, fixes a classic-CI artifact-upload race).
- RCCL's own CMake then filters that down to its supported gfx9 set.
- Net effect: a `gfx110X`-only user still gets gfx906/908/90a/942/950 device code compiled into their build.

Similar patterns probably exist for other subprojects with narrow arch support (hipBLASLt, MIVisionX, etc.), though I haven't audited.

## Why not "just fix the TEST_AMDGPU_TARGETS default"

Flipping `THEROCK_TEST_AMDGPU_TARGETS` to default to `THEROCK_AMDGPU_FAMILIES` would help in the general case but doesn't solve this specific pattern -- if the user's requested families are entirely outside the subproject's supported set, the "correct narrower default" would produce an empty set, and the subproject would either fail to build or fall back to its internal default again. The underlying question is *what should happen when a subproject can't support any of what you asked for.*

## Proposal directions

Leaving the design open for maintainer input, but two shapes that both feel reasonable:

1. **Declarative supported-families at the subproject declaration level.**
    ```cmake
    therock_cmake_subproject_declare(rccl
      SUPPORTED_AMDGPU_FAMILIES gfx9X  # or a list
      ...
    )
    ```
   If `THEROCK_AMDGPU_FAMILIES ∩ SUPPORTED_AMDGPU_FAMILIES = ∅`, auto-disable the subproject (optionally with a `message(STATUS ...)` noting the skip).

2. **Compute effective targets, detect empty, skip.** At whatever stage the subproject's actual arch target list is resolved (after filtering/intersecting), if it's empty, skip activation. Less declarative, but doesn't require each subproject to maintain its own supported-family list in the super-project declaration.

In either case, CI behavior is unchanged: CI already sets `THEROCK_AMDGPU_FAMILIES` broadly enough that intersections remain non-empty.

## Benefits

- Users on a single GPU family don't pay build cost for subprojects they can't use.
- Clearer user model: `THEROCK_AMDGPU_FAMILIES` becomes a reliable "only build what I can run" lever.
- Orthogonal to `THEROCK_ENABLE_RCCL=OFF` -- you don't need to know in advance which subprojects won't work on your hardware.
