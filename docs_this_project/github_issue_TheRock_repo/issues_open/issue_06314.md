**Related:** [#4424](https://github.com/ROCm/TheRock/issues/4424) (same `rocpd` RPATH bug, fixed for the pip wheel), [#4446](https://github.com/ROCm/TheRock/pull/4446) (the fix), [#5114](https://github.com/ROCm/TheRock/issues/5114) (add RPATH correctness test coverage).

## Summary

In the ROCm 7.14.0rc1 distribution tarballs, several profiler libraries that are installed into a nested `lib/<subdir>/` directory end up with a RUNPATH that cannot reach the vendored `rocm_sysdeps` libraries they depend on. As a result they fail to load in a clean environment (no `LD_LIBRARY_PATH` set).

The most user visible symptom is that `import rocpd` fails, and the `rocprof-compute` native collector cannot be dlopened.

This looks like a recurrence of #4424, which reported the same `rocpd` RPATH problem for the pip install and was fixed by #4446 for the core wheel. The tarball artifacts in 7.14.0rc1 still carry the pre-fix RPATH, so the fix does not appear to have reached the tarball packaging path. In addition, `librocprofiler-compute-tool.so` was not covered by that fix and has the same problem here.

## Environment

- Build: ROCm 7.14.0rc1 (HIP 7.14.60850)
- Tarballs checked: `therock-dist-linux-gfx94X-dcgpu-7.14.0rc1.tar.gz` and `therock-dist-linux-multiarch-7.14.0rc1.tar.gz`
- OS: Ubuntu 24.04, x86_64
- Reproduced on the host side only, so no GPU is required to hit or reproduce this.

## Reproduction (clean environment)

Extract the tarball to `install/`, then, with no `LD_LIBRARY_PATH` set:

Use the interpreter that matches the cpython tag of the binding (the example below uses the python 3.12 build with a system python 3.12). The collector example below is interpreter independent.

```
# rocpd Python bindings (python 3.12 build with system python 3.12)
env -i PATH=/usr/bin:/bin \
  PYTHONPATH="$PWD/install/lib/python3.12/site-packages" \
  /usr/bin/python3 -c "import rocpd"
# ImportError: librocm_sysdeps_elf.so.1: cannot open shared object file: No such file or directory

# rocprof-compute native collector
env -i PATH=/usr/bin:/bin /usr/bin/python3 -c \
  "import ctypes; ctypes.CDLL('$PWD/install/lib/rocprofiler-compute/librocprofiler-compute-tool.so')"
# OSError: librocm_sysdeps_dw.so.1: cannot open shared object file: No such file or directory
```

This also fails under the shipped module environment. The `rocprofiler-sdk-rocpd` modulefile prepends `${ROOT}/lib` to `LD_LIBRARY_PATH`, but the vendored libraries live in the `${ROOT}/lib/rocm_sysdeps/lib` subdirectory, which is not added, so `import rocpd` still fails after loading the module:

```
env -i PATH=/usr/bin:/bin \
  LD_LIBRARY_PATH="$PWD/install/lib" \
  PYTHONPATH="$PWD/install/lib/python3.12/site-packages" \
  /usr/bin/python3 -c "import rocpd"
# still ImportError: librocm_sysdeps_elf.so.1: cannot open shared object file
```

Adding only the sysdeps subdirectory makes it load, which isolates the missing path as the single cause:

```
env -i PATH=/usr/bin:/bin \
  LD_LIBRARY_PATH="$PWD/install/lib:$PWD/install/lib/rocm_sysdeps/lib" \
  PYTHONPATH="$PWD/install/lib/python3.12/site-packages" \
  /usr/bin/python3 -c "import rocpd"   # succeeds
```

A full clean environment `ldd` sweep of every ELF file in the tree (489 files) shows these are the only libraries whose bundled `rocm_sysdeps` dependencies are unresolved. All of them are runtime loaded profiler artifacts that live one or more directory levels below `lib/`. Every other binary and library in the tree loads without `LD_LIBRARY_PATH`, which is the relocatable design these five do not meet.

## Affected files (5)

```
lib/rocprofiler-compute/librocprofiler-compute-tool.so
lib/python3.10/site-packages/rocpd/libpyrocpd.cpython-310-x86_64-linux-gnu.so
lib/python3.11/site-packages/rocpd/libpyrocpd.cpython-311-x86_64-linux-gnu.so
lib/python3.12/site-packages/rocpd/libpyrocpd.cpython-312-x86_64-linux-gnu.so
lib/python3.13/site-packages/rocpd/libpyrocpd.cpython-313-x86_64-linux-gnu.so
```

Their RUNPATH values and the dependencies they cannot resolve:

- `rocpd` bindings (all python versions):
  - RUNPATH: `$ORIGIN:$ORIGIN/../../..:$ORIGIN/../../../rocprofiler-sdk`
  - NEEDED but unresolved: `librocm_sysdeps_elf.so.1`, `librocm_sysdeps_dw.so.1`, `librocm_sysdeps_drm.so.2`, `librocm_sysdeps_drm_amdgpu.so.1`
  - There is no `rocm_sysdeps` entry in the RUNPATH at all. The vendored libraries live at `lib/rocm_sysdeps/lib`, which from `lib/pythonX.Y/site-packages/rocpd` is `$ORIGIN/../../../rocm_sysdeps/lib`.

- `librocprofiler-compute-tool.so`:
  - RUNPATH: `$ORIGIN/rocprofiler-compute:$ORIGIN/rocm_sysdeps/lib:$ORIGIN/llvm/lib:$ORIGIN`
  - NEEDED but unresolved: `librocm_sysdeps_dw.so.1`, `librocm_sysdeps_elf.so.1`
  - The RUNPATH contains a `rocm_sysdeps/lib` entry, but it is computed as if the library sits in `lib/`. The library actually sits in `lib/rocprofiler-compute/`, so `$ORIGIN/rocm_sysdeps/lib` points at the nonexistent `lib/rocprofiler-compute/rocm_sysdeps/lib`. It should be `$ORIGIN/../rocm_sysdeps/lib`.

For comparison, libraries that live in `lib/llvm/lib/` correctly get `$ORIGIN/../../rocm_sysdeps/lib`, which shows the origin relative RUNPATH mechanism handles nesting when the origin is correct.

## Why this is a packaging issue and not a local environment issue

- The affected `rocm_sysdeps` libraries are present in the tree, they are just not reachable from these five libraries. So this is a reachability problem, not a missing file.
- It reproduces from the official tarballs using a plain `tar -xzf`, on files that were never modified. RUNPATH is embedded at build time and extraction does not change it.
- The same `rocpd` binary shipped in the official pip `rocm` wheels carries a corrected rpath that does reach the bundled sysdeps (it includes an entry resolving to `_rocm_sdk_core/lib/rocm_sysdeps/lib`), and it does not have this problem. So the correct rpath is known and applied in the wheel packaging path, but not in these tarball artifacts.
- Every other binary and library in the tarball loads in a clean environment, so the tarball is intended to be usable without setting `LD_LIBRARY_PATH`. These five are the exception.

## Likely root cause (from reading TheRock's RPATH assignment, not from a local build)

The origin relative RUNPATH is assigned by TheRock in `cmake/therock_global_post_subproject.cmake`, function `_therock_post_process_rpath_target`. It picks the target origin from `THEROCK_INSTALL_RPATH_ORIGIN`, and when that is not set it falls back to the project level default library dir (`lib`):

```cmake
get_target_property(_origin ${target} THEROCK_INSTALL_RPATH_ORIGIN)
# falls back to THEROCK_INSTALL_RPATH_LIBRARY_DIR (typically "lib") ...
foreach(rpath_dir ${THEROCK_PRIVATE_INSTALL_RPATH_DIRS})
  cmake_path(RELATIVE_PATH rpath_dir BASE_DIRECTORY ${_origin})
  ...
```

The profiler collector and the `rocpd` Python extensions install into `lib/rocprofiler-compute/` and `lib/pythonX.Y/site-packages/rocpd/`, but their origin is not overridden to that nested location, so the origin relative computation uses `lib` as the base. That produces a `rocm_sysdeps/lib` entry that is correct for `lib/` but wrong for the deeper install directories, and in the `rocpd` case it does not produce a reachable `rocm_sysdeps` entry at all.

## Proposed fix direction

For the affected targets, set the RUNPATH origin to their real install subdirectory so the origin relative path to `lib/rocm_sysdeps/lib` is computed correctly. In practice this means setting `THEROCK_INSTALL_RPATH_ORIGIN` (or the relevant project level `INSTALL_RPATH_LIBRARY_DIR`) for the rocprofiler-compute collector target and the `rocpd` extension targets to their nested install locations. An alternative is to derive the origin from the target's actual install destination inside `_therock_post_process_rpath_target` rather than the project default, which would fix this class of issue generally for anything installed below `lib/`.

I would like the rocprofiler-sdk and rocprofiler-compute owners to confirm the intended install layout before finalizing which of these two approaches is preferred.

## Verification of the fix

Correcting the RUNPATH with patchelf resolves both cases in a clean environment:

```
# rocpd: add the correct origin relative sysdeps path
patchelf --set-rpath '$ORIGIN/../../../rocm_sysdeps/lib:$ORIGIN:$ORIGIN/../../..:$ORIGIN/../../../rocprofiler-sdk' \
  lib/python3.12/site-packages/rocpd/libpyrocpd.cpython-312-x86_64-linux-gnu.so
# import rocpd now succeeds

# collector: fix each entry to account for the extra directory level
patchelf --set-rpath '$ORIGIN/..:$ORIGIN/../rocm_sysdeps/lib:$ORIGIN/../llvm/lib:$ORIGIN' \
  lib/rocprofiler-compute/librocprofiler-compute-tool.so
# dlopen now succeeds
```

## Related issues and prior fix

- #4424 reported the same `libpyrocpd` RPATH problem (missing `rocm_sysdeps` entry) for the pip install of `rocm 7.13.0a`. It was closed as fixed.
- #4446 was the fix, adding the missing RPATH entry for the `rocpd*` binaries in the core wheel. This matches what I see now: the pip wheel copy of `libpyrocpd` has the corrected rpath and loads fine, while the tarball copy still has the old rpath and fails. So the fix landed in the wheel packaging path but the tarball artifacts did not get it.
- Because the same underlying binaries end up wrong in one packaging output and right in another, it may be better to correct the RPATH at the build time assignment so every packaging output (tarball and wheel) inherits it, rather than patching a single packaging path. This also relates to #5114 (add unit test coverage for RPATH correctness), which this case is a concrete example for.
- `librocprofiler-compute-tool.so` does not appear to have been covered by #4446 and is broken here as well.
