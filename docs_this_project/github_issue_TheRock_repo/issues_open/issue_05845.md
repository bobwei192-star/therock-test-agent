### Problem Description

## Summary

On a Strix Halo (Ryzen AI Max) **gfx1151** integrated GPU under **Windows 11**, the final **tiled video VAE decode** of an LTX-2.3 (22B) video generation in ComfyUI reliably aborts the process inside the HIP/MIOpen 3D-convolution (`Conv3d`) path. It is **not** out-of-memory.

The failure reproduces across **two completely different PyTorch/ROCm stacks**, crashing at the **same instruction site** in the **same pipeline stage**, which points at the Windows ROCm/MIOpen layer rather than the application:

- **Stack A** — torch 2.9.1 + ROCm 7.2: process dies with a Windows access violation, exit code **3221225477** (`0xC0000005`), no Python traceback.
- **Stack B** — torch 2.12.0a0 + ROCm 7.13 nightly (gfx1151 build): process dies with **`Fatal Python error: Aborted`** (C++ abort) plus a full faulthandler trace pinpointing `Conv3d._conv_forward` (see Additional Information).

A clean A/B isolates the trigger to **convolution size**: with `VAEDecodeTiled` tile size **512** the decode aborts; reducing tile size to **256** completes successfully on the same stack, same workflow, same input. Smaller 3D convolutions survive; large ones abort.

## Key facts

- **Not OOM.** Peak GPU memory during a full run was ~23 GB of ~40 GB available. At decode time the VideoVAE loads fine (`loaded completely; 13607.79 MB usable, 1384.94 MB loaded, full load: True`) immediately before the abort.
- **Always the same stage.** Every crash on both stacks happens at the final video VAE decode, right after the VideoVAE loads — never during sampling. Both diffusion sampling passes complete normally.
- **Two stacks, two death signatures, one site.** Access violation (`0xC0000005`) on ROCm 7.2; C++ `Aborted` + faulthandler trace on ROCm 7.13 nightly — the same `Conv3d` call in the LTX causal video autoencoder.

## Likely related

- **#4214** — HIP "unspecified launch failure" during MIOpen conv in SDXL VAE decode on gfx1150 / Windows with ROCm 7.12; reported as not occurring with 6.4–7.2.1. Small decodes pass, large ones fail, matching the size-dependent behavior here.
- **#2684** — Strix Halo (gfx1151) VAE decode crashes; reported fixed on **Linux** via `HSA_USE_SVM=0` (a Linux-only knob, so the workaround doesn't transfer to Windows).

## What would help

Confirmation of whether this is the same MIOpen Conv3d defect as #4214, whether a Windows-side fix is planned for ROCm 8.0 / upcoming gfx1151 nightlies, and whether any Windows equivalent of `HSA_USE_SVM=0` exists.

Workload: LTX-2.3 22B video model (Q4_K_M GGUF), two-pass sampling, 1920×1088 ~5 s clip, `VAEDecodeTiled` (3D / video VAE). ComfyUI 0.24.1, Python 3.12.12.

Both affected: ROCm 7.2 (torch 2.9.1) and ROCm 7.13 nightly (torch 2.12.0a0+rocm7.13.0a20260313, gfx1151 build)

### Steps to Reproduce

1. On gfx1151 / Windows 11, run an LTX-2.3 (22B) video generation in ComfyUI (Image-to-Video, 1920×1088, ~5 s clip, Q4_K_M GGUF).
2. Both diffusion sampling passes complete normally.
3. Audio VAE loads, then VideoVAE loads.
4. At the tiled video VAE decode with **tile size 512**, the process aborts inside MIOpen/HIP `Conv3d` (Stack A: exit code 3221225477; Stack B: `Fatal Python error: Aborted`).

**A/B that isolates the trigger:** set the `VAEDecodeTiled` tile size to **256** — the identical run then completes and writes the output video. Only the tile size changed.
