Our multi-arch PyTorch build workflows currently run https://github.com/ROCm/TheRock/blob/main/external-builds/pytorch/build_prod_wheels.py and then https://github.com/ROCm/rocm-systems/blob/develop/shared/kpack/python/rocm_kpack/tools/split_python_wheels.py. For developers iterating on Python packages these steps can be difficult to follow and developers can get stuck in weird intermediate states (https://github.com/ROCm/TheRock/issues/5653).

See the code here:

https://github.com/ROCm/TheRock/blob/ee2040338f111a54d5bffa9335ee28114ec2b2f2/.github/workflows/multi_arch_build_portable_linux_pytorch_wheels.yml#L270-L323 https://github.com/ROCm/TheRock/blob/ee2040338f111a54d5bffa9335ee28114ec2b2f2/.github/workflows/multi_arch_build_windows_pytorch_wheels.yml#L292-L367

Some ideas:
- [ ] https://github.com/ROCm/TheRock/issues/4866 so users and automation outside of TheRock can `pip install rocm-kpack`. Here we'll probably still want to use the latest rocm-kpack sources from the rocm-systems submodule
- [ ] Have `build_prod_wheels.py` (optionally) run the wheel splitter itself
- [ ] Document in https://github.com/ROCm/TheRock/blob/main/external-builds/pytorch/README.md (currently no mentions of "split" in that file)

We may also want upstream pytorch to also adopt some of the multi-arch rocm build process including wheel splitting, so keep that in mind.
