### Summary
`therock_subproject.cmake` propagates `CMAKE_C_COMPILER` and `CMAKE_CXX_COMPILER` to subprojects via the generated toolchain file, but does not propagate `CMAKE_Fortran_COMPILER` or `CMAKE_POSITION_INDEPENDENT_CODE`. This causes Fortran-based subproject targets to fall back to the system gfortran without `-fPIC`, resulting in linker failures with ld.lld.

### Problem
hipBLAS Fortran samples (hipblas-example-gemmEx-fortran) fail to link:
```
[hipBLAS] ld.lld: error: relocation R_X86_64_32S cannot be used against local symbol; recompile with -fPIC
[hipBLAS] >>> defined in clients/samples/CMakeFiles/hipblas-example-gemmEx-fortran.dir/example_gemm_ex_fortran.F90.o
[hipBLAS] >>> referenced by example_gemm_ex_fortran.F90
[hipBLAS] >>> clients/samples/CMakeFiles/hipblas-example-gemmEx-fortran.dir/example_gemm_ex_fortran.F90.o:(hip_check_.part.0)
```

### Root Cause
The toolchain generation in `therock_subproject.cmake` sets `CMAKE_C_COMPILER` and `CMAKE_CXX_COMPILER` but does not set:
- `CMAKE_Fortran_COMPILER` — subprojects fall back to system gfortran instead of AMD LLVM's flang
- `CMAKE_POSITION_INDEPENDENT_CODE` — even though `linux-amdclang.cmake` sets this to ON, it never reaches subprojects

### Current Workaround
A per-project `-DCMAKE_Fortran_FLAGS=-fPIC` flag is added in `math-libs/BLAS/CMakeLists.txt`,
```
# TODO(#XXXX): Remove once CMAKE_Fortran_COMPILER and CMAKE_POSITION_INDEPENDENT_CODE
# are propagated to subprojects in therock_subproject.cmake.
# Enable -fPIC for Fortran samples (required by ld.lld linker)
"-DCMAKE_Fortran_FLAGS=-fPIC"
```

### Affected Components
- math-libs/BLAS (hipBLAS Fortran samples)
