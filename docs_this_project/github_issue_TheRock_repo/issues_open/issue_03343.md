Single stage CI scripts and workflows are being deleted (https://github.com/ROCm/TheRock/issues/3340), so other repositories must migrate or fork what they need.

---

For the strategy of migrating from legacy CI to Multi-Arch CI, we will begin with migrating multi arch CI to run on nightlies. After nightlies continue to run, we will add build / test / "last known good run" scripts to optimize. After optimizations are placed, we will use post-submit then gating pre-submit. We will follow the steps:
- Creating a Multi-Arch workflow ([rocm-libraries example](https://github.com/ROCm/rocm-libraries/blob/develop/.github/workflows/therock-multi-arch-ci.yml)) that runs nightly along with the superrepo configuration updates at [`detect_external_repo_config.py`](https://github.com/ROCm/TheRock/blob/7666949b442c6c7337e74577de103705a64ed0fa/build_tools/github_actions/detect_external_repo_config.py#L44-L61)
    - in this step, it builds everything using the superrepo source code instead of TheRock (via [`detect_external_repo_config.py`](https://github.com/ROCm/TheRock/blob/7666949b442c6c7337e74577de103705a64ed0fa/build_tools/github_actions/detect_external_repo_config.py#L53)
- Create or combine a "build selection" script where input is `projects_changed` and the output is the cmake arguments to build. Ideally, we should be able to pull the cmake arguments via [BUILD_TOPOLOGY.toml](https://github.com/ROCm/TheRock/blob/main/BUILD_TOPOLOGY.toml) instead of hard-coding cmake arguments similar to the legacy rocm-libraries [`therock-matrix.py`](https://github.com/ROCm/rocm-libraries/blob/develop/.github/scripts/therock_matrix.py) (combine with [determine_rocm_test_dependencies.py](https://github.com/ROCm/TheRock/blob/main/test_tools/determine_rocm_test_dependencies.py)?)
- Use [`determine_rocm_test_dependencies.py`](https://github.com/ROCm/TheRock/blob/main/test_tools/determine_rocm_test_dependencies.py) to determine the component and [subcomponent](https://github.com/ROCm/TheRock/blob/7666949b442c6c7337e74577de103705a64ed0fa/math-libs/BLAS/CMakeLists.txt#L234-L236) (we retrieve subcomponents via TEST_SUBPROJECTS) tests to run, where it inputs `projects_changed` and outputs `projects_to_test` which is compatible with [`fetch_test_configurations.py`](https://github.com/ROCm/TheRock/blob/main/build_tools/github_actions/fetch_test_configurations.py)
    - this script is available, just needs more component subprojects added (or at a minimum, use tests from [`therock_matrix.py`](https://github.com/ROCm/rocm-libraries/blob/develop/.github/scripts/therock_matrix.py) until we determine what the best tests to add are)
- after adding these build / test determination scripts, we need to create a "last known good run" (LKGR) determination script for the respective superrepos (Multi-Arch CI nightly for the respective superrepos will be a good start), where it pulls a run ID and stages to copy, then we can use those stages for building efficiently

Additional features:
- ability to checkout multiple external repos
- Applying the build / test / LKGR script to submodule bumps
- for pre-built artifacts, skip the testing
- once a specific component is built, begin testing while other components still build

As we are doing a rework, we only need to update TheRock's multi-arch CI workflows and scripts, then the superrepos will use call those workflows (example is [`rocm-libraries multi-arch-ci.yml`](https://github.com/ROCm/rocm-libraries/blob/develop/.github/workflows/therock-multi-arch-ci.yml). 

We can pass in inputs via [`external_repo_config`](https://github.com/ROCm/TheRock/blob/7666949b442c6c7337e74577de103705a64ed0fa/.github/workflows/setup_multi_arch.yml#L93-L95) (inputs such as `projects_changed` which is dynamic) and known constants can stay in [`detect_external_repo_config.py`](https://github.com/ROCm/TheRock/blob/main/build_tools/github_actions/detect_external_repo_config.py) (superrepo paths to checkout / submodules to ignore)
