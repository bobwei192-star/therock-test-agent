### Suggestion Description

Logging is currently inconsistent in TheRock and standardized logging with structured data, consistent timestamps, and performance time will allow for easier readability and parsing of logs. Currently a logging function is copy pasted in many python files within build_tools, a minimal logging api will be introduced to replace these copy pasted logging functions. Once the minimal logging api is in use for build_tools further design discussions and more advanced features will be added to address concerns such as:
- logs not being consistently uploaded to persistent storage in an organized way
- ROCm build workflows logs will upload but ROCm test workflows and PyTorch/JAX/etc. build/release workflows do not 
- considering existing logging tools such as https://github.com/ROCm/TheRock/blob/main/build_tools/teatime.py and how best to use them with the addition of this logging api
- poor searchability due to no standardization of how logs are written, uploaded, stored, processed
- difficultly debugging CI/CD failures due to inconsistent log formatting

### Plan for new logging feature 

1. A minimal logging api will be added to replace to copy and pasted logging functions used in many python files in build_tools
2. Once the logging api is in use further discussions can take place for more complex features to address various concerns
3. The first feature proposed will be to standardize uploading logs to persistent storage for ROCm build workflows, ROCm test workflows, PyTorch/JAX/etc. build/release workflows. Further discussion will take place for this feature but the addition of a standardized logging api will help standardize uploading logs and solve the issue of ROCm build workflows having logs uploaded while ROCm test workflows, and PyTorch/JAX/etc. build/release workflows do not have uploaded logs. 

Minimal logger proposed changes: https://github.com/ROCm/TheRock/pull/4206
Old standardized logging pr for reference: https://github.com/ROCm/TheRock/pull/2620
