### Background

Some recent changes like https://github.com/ROCm/rocm-libraries/pull/2718 have passed unit tests but failed integration tests (Python packages, PyTorch start up code, etc.) leading to expensive reverts like https://github.com/ROCm/rocm-libraries/pull/3411 and https://github.com/ROCm/rocm-libraries/pull/3416.

*We should document how to run "all the tests" or at least "build and test rocm and pytorch" for a given change to any ROCm [sub-]project.*

### Proposal

The document at https://github.com/ROCm/TheRock/blob/main/docs/development/github_actions_debugging.md#testing-release-workflows already covers how to build and test pytorch from already built ROCm packages. We can add new playbook sections there for how to build and test starting from rocm.

Caveats to just documenting the current workflows:

* While the release workflows like https://github.com/ROCm/TheRock/blob/main/.github/workflows/release_portable_linux_packages.yml do have good support for "dev" packages, they build the full matrix by default, including all python versions and all pytorch versions.
* The release workflows run PyTorch tests but they do not run ROCm tests
* Workflows have some sharp edges and require elevated permissions. Setting the wrong inputs (e.g. "nightly" instead of "dev" for the release type) risks affecting prod until permissions are reworked
* The "build packages" -> "build pytorch" workflows are disconnected from one another as the former triggers the latter via workflow_dispatch. This makes it hard to see a summary view with all build/test results in one place.
* Release workflows do not have ccache configured (?)
