# #6224 - [Issue]: rocclr "Memobj map does not have ptr" abort on 2nd Z-Image gen (Windows, dGPU + iGPU)

### Summary
On Windows, `amdhip64_7.dll` (rocclr) aborts the whole process on the **second**
image generation of a ComfyUI Z-Image workflow with:

```
:0:...\rocclr\device\device.cpp:373 : <ts> us: Memobj map does not have ptr: 0x2adf8000
```

The first generation succeeds; the second aborts. This is not a recoverable
`hipError_t` — it is a hard `guarantee()`/abort that kills the process.

### Key condition: more than one HIP device must be visible
This only reproduces when **both** the discrete GPU **and** the CPU's integrated
GPU are registered as HIP devices:

- Discrete: **AMD Radeon RX 7900 XTX** (gfx1100) → `cuda:0`
- Integrated: **Radeon 780M** in the Ryzen 8700G (gfx1103) → `cuda:1`

ComfyUI (Desktop) enumerates both by default, so it aborts on the 2nd run.

**Diagnostic:** launching with `HIP_VISIBLE_DEVICES=0` (discrete GPU only) makes
the crash disappear entirely — dozens of back-to-back generations succeed. This
points squarely at the multi-device, per-device-VA free path.

### Root cause
In the free path, `ihipFree()` resolves an allocation via `FindMemObj()`, which
matches **any** address in `[base, base + size)` and, on Windows with multiple
devices, falls back to per-device VA maps with host-pointer/base adjustment. It
then called `RemoveMemObj(ptr)`, which erased the global map only by the **exact
base key** and `guarantee()`d the result. A pointer resolved by range or adjusted
base freed correctly but missed the exact-key erase, so the guarantee aborts.
With a single device the per-device VA fallback is not exercised, so the
asymmetry never surfaces.

### Steps to reproduce
1. Windows 11, a system with a discrete AMD GPU **and** an active integrated
   Radeon GPU (e.g. Ryzen 8700G / 780M), ROCm 7.2.1 wheels, PyTorch `2.9.1+rocm7.2.1`.
2. ComfyUI with the **Z-Image (Turbo)** workflow; do **not** set
   `HIP_VISIBLE_DEVICES` (both GPUs visible — the default).
3. Generate one image (succeeds), then generate again.
4. The process aborts on the 2nd generation with `Memobj map does not have ptr: 0x...`
   (the address varies per run).

Confirmed with a minimal API driver (two `EmptySD3LatentImage` → KSampler runs):
crashes on gen 2 with both GPUs visible; never crashes with `HIP_VISIBLE_DEVICES=0`.

### Environment
- **OS:** Windows 11
- **GPUs:** RX 7900 XTX (gfx1100) + Ryzen 8700G integrated Radeon 780M (gfx1103)
- **ROCm:** 7.2.1 (TheRock `rocm-sdk` wheels), PyTorch `2.9.1+rocm7.2.1`
- **Driver:** AMD Adrenalin 26.6.2
- **App:** ComfyUI (Desktop) — Z-Image Turbo template

### Expected
A missing/absent pointer on a user-facing free should not abort the process.

### Fix
Proposed in ROCm/rocm-systems#7989 — makes `MemObjMap` removal range-aware
(symmetric with `FindMemObj`) and adds a non-fatal `TryRemoveMemObj` for
user-facing frees. Verified on this exact multi-GPU Windows/ROCm 7.2.1 setup:
`amdhip64_7.dll` rebuilt from the `rocm-7.2.1` tag + the fix → both generations
complete with no abort (A/B tested against the unpatched 7.2.1 DLL, which
crashes on generation 2).
