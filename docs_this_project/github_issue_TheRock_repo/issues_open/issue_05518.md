## Summary

ROCm has been building and running on IBM Power (ppc64le) for some time, maintained in a [branch that tracks TheRock upstream](https://github.com/ROCm/TheRock/tree/benrichard-amd/ppc64le). This issue tracks the effort to upstream those changes into TheRock proper.

## Motivation

TheRock's CMake layer currently encodes several x86-specific assumptions — hardcoded LLVM target, unconditional architecture flags, and features enabled without architecture guards. These assumptions are not wrong on x86, but they make it harder to build on any other host and obscure which features are genuinely portable.

Phase 1 addresses this at the TheRock level, independent of any particular port. Correcting these assumptions makes the build system more honest about what is and is not architecture-specific, and reduces the friction for component maintainers who want to upstream fixes: once the scaffolding is clean, component patches can focus on component-level issues rather than working around build system artifacts.

## Approach

**Phase 1 — TheRock infrastructure changes**

Corrections to x86 assumptions baked into TheRock's CMake layer:
- Use `Native` instead of `X86` when specifying LLVM targets to build — equivalent on x86 hosts, correct everywhere else
- Guard x86-specific compile flags behind architecture checks so they are not passed on non-x86 builds
- Guard x86-only features (RDC, rocprofiler-systems, ESMI) behind explicit architecture checks with clear diagnostics — these features remain enabled on x86 as before
- Refactor kpack's ELF surgery to make it straightforward to add page size and relocation constants for other architectures

**Phase 2 — Component fixes upstream**

ppc64le fixes for individual ROCm components are submitted directly to their respective projects. As they land, TheRock picks them up via normal submodule updates. No patches are carried in TheRock.

Notable work in this phase:
- **Clang**: float16 and bf16 are not currently supported on PowerPC. Patches adding this support in the ppc64le branch need to be upstreamed to LLVM/Clang before ROCm components that depend on them can build cleanly.
- Additional fixes across CLR, RCCL, ROCr, hipBLAS, and others.

## Sequencing

1. Land Phase 1 CMake changes in TheRock
2. Submit component fixes to upstream repos in parallel, with Clang float16/bf16 support as a key dependency
3. TheRock submodule bumps pick up fixes as they merge

## Success Criteria

**Phase 1**: TheRock's CMake layer passes no x86-specific flags or enables no x86-only features on non-x86 hosts, with no regressions on x86.

**Phase 2**: ROCm builds successfully on ppc64le with no out-of-tree patches.
