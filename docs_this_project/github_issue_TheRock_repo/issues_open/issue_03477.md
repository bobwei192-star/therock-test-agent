## Environment

- **OS:** Ubuntu 25.04
- **Compiler:** GCC 15 (defaults to `-std=gnu23`)
- **Python:** 3.13
- **GPU target:** `gfx1151` (AMD Strix Halo iGPU / Radeon 8060S)
- **Terminal:** Kitty (relevant for one issue below)
- **Build method:** standard cmake + ninja with `.venv`

All issues below were hit during a fresh build from source. Fixes are included for each.

---

## 1. ncurses installs terminfo to Kitty's read-only directory

**File:** `third-party/sysdeps/common/ncurses/CMakeLists.txt`

**Error:**
```
/usr/lib/kitty/terminfo: permission denied (errno 13)
```

Kitty sets `$TERMINFO=/usr/lib/kitty/terminfo` in the environment. ncurses `configure` picks this up as its `ticdir` install destination.

**Fix:** Add `--unset=TERMINFO` to the `cmake -E env` invocation so ncurses falls back to `${CMAKE_INSTALL_PREFIX}/share/terminfo`.

---

## 2. Venv tools not found by subproject cmake (`lit`, `pytest`)

**Files:**
- `math-libs/CMakeLists.txt`
- `math-libs/BLAS/CMakeLists.txt`

When building inside a `.venv`, tools like `lit` and `pytest` are in `.venv/bin/` which is not on the system PATH used by subproject cmake processes. Two separate failures:

**Error 1** (libhipcxx):
```
CMake Error at test/CMakeLists.txt:188 (find_program):
  Could not find libcudacxx_LIT using the following names: lit
```

**Fix** in `math-libs/CMakeLists.txt`: compute bin dir from `Python3_EXECUTABLE` and pass explicitly:
```cmake
cmake_path(GET Python3_EXECUTABLE PARENT_PATH _libhipcxx_python_bin_dir)
# In CMAKE_ARGS:
"-DLLVM_EXTERNAL_LIT=${_libhipcxx_python_bin_dir}/lit"
"-Dlibcudacxx_LIT=${_libhipcxx_python_bin_dir}/lit"
```

**Error 2** (rocRoller):
```
CMake Error at test/CMakeLists.txt:76 (find_package):
  Could not find a package configuration file provided by "Pytest"
```

**Fix** in `math-libs/BLAS/CMakeLists.txt`:
```cmake
cmake_path(GET Python3_EXECUTABLE PARENT_PATH _rocroller_python_bin_dir)
cmake_path(GET _rocroller_python_bin_dir PARENT_PATH _rocroller_venv_dir)
# In CMAKE_ARGS:
"-DPytest_DIR=${_rocroller_venv_dir}/share/Pytest/cmake"
"-DPYTEST_EXECUTABLE=${_rocroller_python_bin_dir}/pytest"
```

## 3. rocBLAS/Tensile: `from setuptools import setup` fails (Python 3.13)

**File:** `math-libs/BLAS/CMakeLists.txt`

**Error:**
```
ImportError: cannot import name 'setup' from 'setuptools' (unknown location)
```

rocBLAS creates a virtualenv with `--system-site-packages`. On Python 3.13 Ubuntu, the system setuptools stub doesn't expose `setup`. The venv needs pip/setuptools upgraded first.

**Fix:** Pass `-DTENSILE_VENV_UPGRADE_PIP=ON` in CMAKE_ARGS for rocBLAS. This triggers `pip install --upgrade pip packaging setuptools` in the new venv before Tensile is installed.

---

## 4. rocRoller tests fail to compile with GCC 15 STL (workaround)

**File:** `math-libs/BLAS/CMakeLists.txt`

**Error:**
```
stl_algobase.h:1191:22: error: invalid operands to binary expression
('const std::pair<const int, std::shared_ptr<std::variant<...>>>' and same type)
```

GCC 15's `std::equal` requires `operator==` for the element type. The large `std::variant` in rocRoller's `KernelGraphTest.cpp` doesn't provide one. The library itself builds fine.

**Fix:** Pass `-DROCROLLER_BUILD_TESTING=OFF` in CMAKE_ARGS for rocRoller until upstream adds the missing operator.

## 5. GCC 15 C23: PAPI build failures in rocprofiler-systems

**File:** `rocm-systems/projects/rocprofiler-systems/cmake/PAPI.cmake`
**Also:** `rocm-systems/projects/rocprofiler-systems/external/papi/src/high-level/papi_hl.c`

GCC 15 defaults to C23, where `()` in a function declaration means `(void)` (no parameters). This causes two failures in PAPI:

**Error 1** — conflicting declaration:
```
papi_hl.c:409:12: error: conflicting types for '_internal_hl_read_user_events';
have 'int(const char *)'
```
Line 170 declares `static int _internal_hl_read_user_events();` (no params) but line 409 defines it with `const char *user_events`.

**Fix:** Update the forward declaration at line 170 to `static int _internal_hl_read_user_events(const char *user_events);`

**Error 2** — incompatible pointer types:
```
papi_vector.c:221:36: error: assignment to 'int (*)(papi_mdi_t *)' from
incompatible pointer type 'int (*)(void)'
```
`vec_int_dummy()` uses old-style C "any params" declaration, which GCC 15 C23 treats as incompatible when assigned to typed function pointers.

**Fix:** Add `-std=gnu17 -Wno-incompatible-pointer-types` to all `CFLAGS=` entries in the PAPI ExternalProject in `PAPI.cmake`.

## 6. GCC 15: elfutils `-Werror=unterminated-string-initialization` in rocprofiler-systems

**File:** `rocm-systems/projects/rocprofiler-systems/cmake/DyninstElfUtils.cmake`

**Error:**
```
i386_regs.c:88:11: error: initializer-string for array of 'char' truncates NUL terminator
but destination lacks 'nonstring' attribute [-Werror=unterminated-string-initialization]
```

elfutils has `static const char baseregs[][2]` initialized with two-character string literals like `"ax"` — these are 3 bytes including NUL, which truncates under the new GCC 15 warning.

**Fix:** Add `-Wno-unterminated-string-initialization` to the elfutils `CFLAGS` in the `CONFIGURE_COMMAND` of the `DyninstElfUtils` ExternalProject.

## 7. GCC 15: missing transitive headers (multiple components)

GCC 15 no longer provides headers like `<cstdint>` or `<algorithm>` transitively. This caused errors in several bundled libraries:

| File | Error | Fix |
|------|-------|-----|
| `rocm-systems/projects/rocprofiler-sdk/external/yaml-cpp/src/emitterutils.cpp` | `unknown type name 'uint16_t'` | Add `#include <cstdint>` |
| `rocm-systems/projects/rocprofiler-sdk/external/elfio/elfio/elf_types.hpp` | `unknown type name 'uint16_t'` | Add `#include <cstdint>` |
| `rocm-systems/projects/rocprofiler-systems/source/lib/logger/logger.hpp` | `'any_of' is not a member of 'std'` | Add `#include <algorithm>` |
| `rocm-systems/projects/rocprofiler-systems/external/dyninst/common/h/dyntypes.h` | `'uint32_t' does not name a type` | Add `#include <stdint.h>` |
| `rocm-libraries/projects/hipdnn/plugins/miopen_legacy_plugin/MiopenContainer.hpp` | `unknown type name 'uint32_t'` | Add `#include <cstdint>` |

---

## Summary

Most of these fall into three categories:

1. **GCC 15 stricter C/C++ defaults** — PAPI (C23 `()` semantics, incompatible pointer types), elfutils (new `-Werror` warning)
2. **GCC 15 missing transitive headers** — five different files across rocprofiler-sdk, rocprofiler-systems, and hipDNN
3. **Python/venv environment** — `lit` and `pytest` not on PATH, Python 3.13 setuptools changes

The GCC 15 header issues are likely to recur as more of the codebase is compiled with newer compilers. A systematic grep for uses of `uint*_t`, `int*_t`, `std::any_of`, `std::sort`, etc. without direct includes might surface more instances.

Several of the fixes above touch **vendored/submodule code** (yaml-cpp, elfio, PAPI, elfutils, dyninst, hipDNN's MiopenContainer). The right long-term fix for those is to upstream the patches to each respective project rather than carry them in TheRock — the changes here are minimal and unambiguously correct, so they should be easy to get merged upstream. The TheRock-side fixes (CMake flags like `-std=gnu17`, `-Wno-*`) can serve as a temporary bridge in the meantime.
