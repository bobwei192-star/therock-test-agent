I'm seeing two related issues coming from this code:

workflow | code
-- | --
ci.yml | https://github.com/ROCm/TheRock/blob/15558f4240876c7b4eb667f20182db4e3673e4e6/.github/workflows/ci.yml#L53-L68
multi_arch_ci.yml | https://github.com/ROCm/TheRock/blob/15558f4240876c7b4eb667f20182db4e3673e4e6/.github/workflows/multi_arch_ci.yml#L45-L60

1. If a PR is created with labels already applied, the `labeled` and `opened` events will both trigger in sequence. The `opened` run will show as cancelled on the PR itself while the `labeled` run will continue in the background
2. When triggering with `workflow_dispatch` on a commit from `main` (potentially with custom inputs), the concurrency group cancels any in-progress runs on `main`

I did some co-design with an agent some time ago, notes here: https://github.com/ScottTodd/claude-rocm-workspace/blob/main/tasks/active/concurrency-groups.md

It seems like this concurrency group may at least solve the second issue:
```yml
concurrency:
  group: ${{ github.workflow }}-${{ github.head_ref || github.run_id }}
  cancel-in-progress: true
```

Not sure about the first issue. I've used another pattern before where there's a separate "job trigger" workflow that runs on `labeled` and that decides whether or not to cancel/restart the main job (it can decide if the labels have _changed_ and if unrelated labels were added/removed, for example)
