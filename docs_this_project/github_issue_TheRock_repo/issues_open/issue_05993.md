### Problem Description

The gfx1150 MES wedge described in the (closed) #1271 still reproduces on current
Ubuntu 26.04 / kernel 7.0, via the **Vulkan / Mesa RADV** path rather than the
ROCm/HIP/ComfyUI path of the original report — and with the `ttm.pages_limit`
workaround from that thread already applied. Same kernel signature
(`GCVM_L2_PROTECTION_FAULT_STATUS` → `MES failed to respond to msg=REMOVE_QUEUE`
→ `MODE2 reset` → "device wedged, but recovered through reset"), same hardware.
A kernel GPU devcoredump (the artifact requested but not captured in #1271) is
available and attached.

This is not a request to reopen #1271 for the ROCm/HIP case; it's to report that
the same MES defect is reachable through a different (Vulkan) userspace stack and
is **not** covered by the `1fb710793ce2` kernel fix or the `ttm.pages_limit` /
`ROCBLAS_USE_HIPBLASLT=0` workarounds that helped the HIP path.

### Operating System

Ubuntu 26.04 LTS (Resolute Raccoon), kernel `7.0.0-22-generic` (built 2026-05-25).

### CPU / GPU

AMD Ryzen AI 9 HX 370 w/ Radeon 890M (gfx1150), 24 GB unified memory pool
(32 GB system RAM). Single iGPU, shared system memory (APU / `uma:1`).

### ROCm / driver stack

- `libdrm 2.4.131`, `mesa-vulkan-drivers 26.0.3-1ubuntu1`, Vulkan apiVersion
  1.4.335, `libvulkan 1.4.341.0-1`.
- Triggering app: ggml `whisper-cli.bin 1.6.x` + `libggml-vulkan.so`, running on
  the **Vulkan0 / RADV STRIX1** backend (not the ROCm/HIP backend).
- Concurrent GPU consumer: ollama (`qwen3.5:9b-64k` resident at ~16 GB) + a
  qwen3-tts vocoder service (port 8202), all on the same iGPU.
- Kernel cmdline already includes:
  `ttm.pages_limit=6000000 amdgpu.gpu_recovery=1 amdgpu.cwsr_enable=0 amdgpu.dcdebugmask=0x200`
  (the `ttm.pages_limit` raise is the GTT/pool workaround that resolved the HIP
  case in #1271).

### Steps to Reproduce (the captured crash, 2026-06-19 12:52)

1. On the 24 GB APU, start the qwen3-tts service (port 8202).
2. Start a long Vulkan/RADV `whisper-cli.bin` transcription (ggml-small, ~7 h of
   16 kHz mono audio; the job runs ~14× realtime so it sustains GPU compute for
   ~30+ min). Backend forced via `LD_LIBRARY_PATH=…/whisper-vulkan/lib` so ggml
   uses `libggml-vulkan.so` (RADV), confirmed in-log: `using Vulkan0 backend` on
   `AMD Radeon 890M Graphics (RADV STRIX1) | uma:1`.
3. Concurrently, background agents drive ollama to load `qwen3.5:9b-64k` at
   `num_ctx=65536` (~16 GB resident: 16 GB KV + 6.4 GB weights) and perform
   model-swap churn (gemma ⇄ qwen, `keep_alive:0` evict/reload cycles).
4. At ~29 min / ~85 % of the audio, the whisper job dies with
   `vk::DeviceLostError` / SIGABRT (exit 134). The GPU self-recovers via MODE2
   reset ("device wedged, but recovered through reset"); the box does not reboot.

### Concurrent load at the fault (kfd-snapshot)

| pid | process | VRAM |
|---|---|---|
| 2789352 | **llama-server** (ollama: `qwen3.5:9b-64k` @ 64k ctx) | **~16.3 GB** |
| 1014927 | `whisper-cli.bin` (Vulkan/RADV) | ~1.2 GB |
| (post-reset) | llama-server reloading qwen3.5:9b-64k | climbing to ~17.1 GB |
| — | python (qwen3-tts) | — |

The faulting process is `llama-server` (ollama), faulty client `TCP` (shader /
texture cache), `PERMISSION_FAULTS: 0x5` — i.e. the ollama shader/texture pages
lost permission / were evicted under unified-memory oversubscription
(16 GB ollama + Vulkan whisper + TTS on the 24 GB pool). This is the same
oversubscription-into-eviction mechanism #1271 converged on (raising
`ttm.page_pool_size` helped the HIP path), here reached through Vulkan with the
pool already raised.

- On-demand synthetic reproduction of the wedge has so far **not** succeeded:
  `qwen3.5:9b-64k`'s 16 GB KV prefill cannot complete under concurrent Vulkan
  contention (stalls mid-prefill → static oversubscription → OOM, not a page
  fault). The real crash involved a long-lived, cache-warm 16 GB context with
  model-swap churn on top of a resident Vulkan job — a pattern one-shot synthetic
  loops do not replay. The near-reproducer (design validated, not yet fired) is
  a single high-pressure ollama context (~16 GB, `keep_alive:0` churn, no
  competing low-pressure loops so it actually loads) + one sustained Vulkan
  whisper job + TTS on the 24 GB APU.
