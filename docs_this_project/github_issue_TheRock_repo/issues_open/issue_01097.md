We have a mapping from amdgpu_family values like `gfx94x` and `gfx115x` to per-platform `test-runs-on` and `family` values in https://github.com/ROCm/TheRock/blob/main/build_tools/github_actions/amdgpu_family_matrix.py.

GitHub Actions workflows read from this file with:
* https://github.com/ROCm/TheRock/blob/main/build_tools/github_actions/configure_ci.py
* https://github.com/ROCm/TheRock/blob/main/build_tools/github_actions/fetch_package_targets.py
* https://github.com/ROCm/TheRock/blob/main/build_tools/github_actions/configure_target_run.py

This is working well enough for currently enabled workflows, but it has a few issues:

* Many workflows do not have safe or obvious defaults when triggered via workflow_dispatch. A typo can result in a job not running at all, instead of a failure.
* There is a disconnect between fetch_package_targets.py and configure_target_run.py in release workflows. A list of test runners is known at the start but then is forgotten and reconstructed later via the second script. This reconstruction is buggy, which led to churn on https://github.com/ROCm/TheRock/pull/1086.
* `test-runs-on` is currently a single string per gfx family and platform. We don't have a way to test a single package on multiple different runners (e.g. gfx1200 _and_ gfx1201)
* It is not obvious what the differences are between the "presubmit", "postsubmit", and "xfail" groups, especially when working across multiple different workflows (ci.yml that runs on pre- and post-submit, release workflows that generate rocm and pytorch python packages, etc.)

I'm brainstorming some ways to refactor now, but I could also do the minimal work to unblock myself, working within the existing structure.
