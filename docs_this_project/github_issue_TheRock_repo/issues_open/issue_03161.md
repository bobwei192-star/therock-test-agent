### Suggestion Description

As of 01/29/2016, ASAN builds of ROCm generated by [CI ASAN Workflow](https://github.com/ROCm/TheRock/actions/workflows/ci_asan.yml) can only be downloaded and flattened using install_rocm_from_artifacts.py. 

A pip installable ASAN build would enable seamless diagnosing and debugging of workflows, especially when deploying over cluster.

### Context

We are engaged with training issues at a customer site every day. For many weeks (greater than one month), they have been struggling with an illegal memory access issue which is not being caught correctly. Our team has done extensive attempts at log analysis, but they can't tell where these issues are bubbling up from. As a result, in January, we embarked on a journey to enable ASAN at all parts of the ROCm chain that's being used by the customer's training flow.
 
Address Sanitization sounds a lot to me like valgrind/Purify in that it implements/injects memory fencing, so that memory access errors are handled more gracefully than a segfault & core file. This is important because frontier labs are very careful about what files and data they will share - so we need this advanced logging / fencing behavior in order to identify which bit of code is causing the issue.
 
The goal is that any user who is experiencing illegal memory access issues will be able to deploy an alternate build of the ROCm stack (including the Frameworks) which has ASAN enabled, and run their workloads via that stack to identify what's causing the exception.

There will never (I'm pretty sure) be an expectation that ASAN-enabled builds would run in production. ASAN carries a significant performance impact. There's plenty of concern that when using ASAN, due to Heisenbug-principles, the issue may not appear. However, the team is fully blocked at moving forward, so we need to go this route.
