Description

Currently, TheRock build system generates and uploads a `therock_manifest.json` file mapping all submodules and commit SHAs. This ensures every build is traceable to a specific TheRock commit — which satisfies the first requirement (build → manifest mapping from #1748).

However, as @amd-chrissosa  highlighted, there’s a second, orthogonal problem: how we version and identify manifests across like builds (e.g., multiple nightly/cron builds on the same commit). Using just the commit SHA is not sufficiently unique or meaningful once logs and runner context expire.

Problem

1. For per-PR builds, unique manifest versioning provides little value since they’re transient and always linked to a specific PR run.
2. For post-submit / cron builds, we need persistent versioning that:
    - Uniquely identifies each run (even if on the same commit)
    - Can be referenced long-term (e.g., via a timestamp or monotonically increasing identifier)
    - Survives after GitHub logs or runner IDs expire
3. Current manifests are uploaded with only the commit SHA, making it difficult to disambiguate repeated builds of the same commit.

Proposed Direction

1. Introduce a manifest version field (manifest_version) and/or augment filenames with unique identifiers such as:
    -  UTC timestamp (e.g., therock-manifest-2025-10-21T01-35-00Z.json)
    -  A structured version schema (e.g., manifest_version: 1.0)
    -  Possibly include the workflow name, branch, or run attempt for traceability
2. Apply this versioning specifically to post-submit and cron workflows where persistence matters.
3. Leave per-PR builds as ephemeral (no need for version bumping).
