PyTorch team would like some of the 8-GPU CI hosts (`linux-gfx942-8gpu-ossci-rocm` runner) to instead be split into two independent 4-GPU runner slices, each isolated to a set like `0,1,2,3` and `4,5,6,7`, under a label such as `linux-gfx942-4gpu-ossci-rocm`. Our PyTorch distributed unit tests usually assume `world_size <= 4` so running one job across all 8 wastes half the box; 4-GPU labels let two jobs run concurrently.

This is the follow-up from https://github.com/ROCm/TheRock/pull/2705, where Prachi asked whether distributed jobs could default to 4 GPUs (https://github.com/ROCm/TheRock/pull/2705#issuecomment-4577584121) and Laura scoped it to a separate effort, then routed it to infra: "does it make sense to provide runners with just 4 gpus enabled for this?" (https://github.com/ROCm/TheRock/pull/2705#issuecomment-4578597507). It has to happen at the runner level, not in the scripts: GitHub Actions runs one job per runner, so just setting `HIP_VISIBLE_DEVICES` inside a job (tried and rejected in https://github.com/ROCm/TheRock/pull/5128 — see review at https://github.com/ROCm/TheRock/pull/5128#discussion_r3322959124) doesn't free the other 4 GPUs and can break the container's existing isolation, since `HIP_VISIBLE_DEVICES` must stay a subset of the pre-set `ROCR_VISIBLE_DEVICES` or jobs fail (https://github.com/ROCm/TheRock/issues/2208).

No test-code changes are needed: https://github.com/ROCm/TheRock/pull/2705 already adapts to whatever GPU fence the container provides, so once the 4-GPU slices exist we just point `test_runs_on_multi_gpu` at the new label.

## Our ask
- Split select 8-GPU hosts into two independent 4-GPU runner slices, each GPU-isolated to a disjoint set (e.g. `0,1,2,3` and `4,5,6,7`).
- Expose them under a new label such as `linux-gfx942-4gpu-ossci-rocm`.
- Choose the device split to follow the box's xGMI/NUMA topology so the 4-GPU group stays well-connected for collective ops.
