# #5615 - Windows: clang-cl -fgpu-rdc bundling fails (/Fo output reused as bundler input)

## Summary

On Windows, compiling a HIP translation unit with `clang-cl -x hip -fgpu-rdc -c /Fo<path>` fails at the `clang-offload-bundler` step. With the clang-cl MSVC-style output flag `/Fo`, the driver writes the **host** object to the final output path and then invokes `clang-offload-bundler` with that **same path as both an `-input` and the `-output`**. On Windows the bundler memory-maps the input object, and the OS then refuses to write the output (the same file):

```
clang-offload-bundler: error: '<...>.obj': The requested operation cannot be performed on a file with a user-mapped section open.
```

(For a small object the half-written file instead trips `llvm-objcopy: error: '<...>.obj': The file was not recognized as a valid object file`.) Either way the compile fails. This blocks any CMake-based HIP project that uses relocatable device code (`-fgpu-rdc` / `CMAKE_HIP_SEPARABLE_COMPILATION`, e.g. for a cross-TU `__constant__`/`__device__` symbol), since CMake emits `/Fo` for the clang-cl HIP compiler.

## Minimal reproduction

```bat
printf '#include <hip/hip_runtime.h>\n__global__ void k(int* p){ *p = 1; }\n' > t.cu

clang-cl -x hip --offload-arch=gfx1151 -fgpu-rdc /Fot.obj -c t.cu
```

(with `HIP_DEVICE_LIB_PATH` set to `lib/llvm/amdgcn/bitcode` and the MSVC `INCLUDE` populated). Fails 100% of the time, even at `-j1` with this trivial 2-line kernel.

`-v` shows the aliasing -- the host-object input is the same path as the output:

```
clang-offload-bundler -type=o \
  -targets=hip-amdgcn-amd-amdhsa-unknown-gfx1151,host-x86_64-pc-windows-msvc \
  -output=t.obj -input=t-hip-amdgcn-amd-amdhsa-gfx1151.bc -input=t.obj
```

## What controls it: `/Fo` vs `-o` (not object size)

The trigger is the clang-cl `/Fo` output flag. The same compile with the GNU-style `-o t.obj` instead of `/Fot.obj` **passes**, because the driver then routes the host object to a distinct temp file, so `-input != -output`:

```
... -output=t.obj -input=...\Temp\t-<hash>.bc -input=...\Temp\t-<hash>.obj   # distinct -> OK
```

Size is not the factor: a 531 KB object built with `-o` passes, while a 36 KB object built with `/Fo` fails.

## Expected

`clang-cl -fgpu-rdc /Fo<path>` should bundle correctly on Windows, the same as `-o` and the same as on Linux. The driver should not hand the bundler the same path as both an input and the output; alternatively the bundler should not keep a memory-mapping of an input open while writing an output that aliases it (POSIX tolerates replacing an mmap'd file; Windows does not -> `ERROR_USER_MAPPED_FILE` / 0x4C8).

## Workarounds

- `--offload-new-driver` -- packages offload via `clang-linker-wrapper` instead of the in-place per-TU `clang-offload-bundler`; no input==output aliasing, compiles cleanly with `/Fo`. (Apply it to the link step too.)
- Use GNU `-o` instead of clang-cl `/Fo` (not generally controllable from CMake's HIP compile rule).
