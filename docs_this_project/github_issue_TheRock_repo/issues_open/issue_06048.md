# #6048 - RFC: Custom Python bindings layering

This issue tracks discussion for a proposed RFC defining the ownership and packaging model for custom ROCm library Python bindings.

The RFC uses hipDNN as the motivating example and proposes:

- keeping native library ownership in `rocm-libraries`;
- keeping TheRock / `rocm_sdk` focused on normal native ROCm SDK artifacts;
- relocating `hip-python` into the shared bindings repository as the generated C API binding area;
- placing custom hipDNN binding projects under `rocm-bindings/hipdnn/` with `hipdnn-nanobind` producing extension artifacts and `hipdnn-frontend` producing the final wheel;
- defining Python ABI, runtime/device package, manifest, and validation responsibilities for custom binding wheels.

A draft PR will add the RFC document and link back to this issue.
