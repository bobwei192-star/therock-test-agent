## Problem

The `fft` artifact is `type = "target-specific"` in BUILD_TOPOLOGY.toml but has no splitting strategy for multi-arch CI. rocFFT has two distinct architecture-dependent data concerns:

1. **Solution maps** are compiled into `librocfft.so` but filtered by build-configured `GPU_TARGETS`, making the library binary vary across build shards
2. **Kernel cache** (`rocfft_kernel_cache.db`) is a single combined SQLite database containing pre-compiled kernels for all target architectures

## Analysis of rocFFT's installed content

### Solution maps (.dat files) — compiled into library, shard-variant

Per-architecture JSON files in `library/solution_map/` (gfx908, gfx90a, gfx942 — ~300KB total). These are host-side algorithmic tuning data (kernel selection parameters, not compiled device code).

**Problem:** Filtered by `GPU_TARGETS` at compile time:
- `src/CMakeLists.txt:217`: `set( sol_gpu_arch ${GPU_TARGETS} )`
- `solution-shipping.py:233`: only `.dat` files whose `gfx*` prefix matches are compiled into `solutions.cpp`

In multi-arch CI, each shard builds for different architectures, so `librocfft.so` will differ across shards (different embedded solution data). This breaks the split-and-merge model which requires the generic artifact content to be identical regardless of which shard built it.

**Fix (two options, recommend A):**

**(A) Always embed all solution maps (recommended):** Change `sol_gpu_arch` to `all` regardless of `GPU_TARGETS`. The `solution-shipping.py` script already supports `all` as a special value (lines 297-301). Only 3 `.dat` files exist totaling ~300KB — negligible overhead. Unused solutions are never queried at runtime.

**(B) Use `THEROCK_DIST_AMDGPU_TARGETS`:** Pass the full distribution target list instead of shard-local `GPU_TARGETS`. More precise but requires plumbing through cmake configuration.

### rocfft_kernel_cache.db — combined SQLite database

**Install path:** `lib/rocfft/rocfft_kernel_cache.db`
**Build:** `rocfft_aot_helper` compiles kernels AOT, stores all architectures in one SQLite database.

**Schema:**
```sql
CREATE TABLE cache_v1 (
    kernel_name TEXT NOT NULL,
    arch TEXT NOT NULL,          -- e.g. "gfx942"
    hip_version INTEGER NOT NULL,
    generator_sum BLOB NOT NULL,
    code BLOB NOT NULL,          -- compiled kernel binary
    timestamp INTEGER NOT NULL,
    PRIMARY KEY (kernel_name, arch, hip_version, generator_sum)
);
```

Less common architectures (gfx803, gfx900, gfx906, gfx940, gfx941, gfx1101, gfx1102, gfx1150-1153, gfx1200) are excluded from AOT — kernels for these are JIT-compiled at runtime by `rocfft_rtc_helper`.

**Runtime system cache search** (`rtc_cache.cpp:120-142`):
1. `$ROCFFT_RTC_SYS_CACHE_PATH` env var
2. `{librocfft.so dir}/rocfft_kernel_cache.db`
3. `{librocfft.so dir}/rocfft/rocfft_kernel_cache.db`

The existing `write_aot_cache()` already filters by architecture — it creates a temp table of desired arch strings and uses `JOIN` to copy only matching rows.

### Other installed content

- `librocfft.so` — fat binary with `.hip_fatbin` sections (handled by existing kpack ELF splitting)
- `rocfft_rtc_helper` — host binary for subprocess compilation (architecture-generic)
- `artifact-fft.toml` explicitly includes the cache: `include = ["**/rocfft_kernel_cache.db"]`

## Database splitting — two alternatives

Both require the **same runtime change**: modify `rtccache_db_sys_paths()` to search for per-architecture database files (e.g. `rocfft_kernel_cache_gfx942.db`) alongside the combined file. They differ in who produces the per-arch files:

### Alternative A: rocFFT produces per-arch databases at build time

Modify `rocfft_aot_helper` (or add a post-build CMake step) to call `write_aot_cache()` once per architecture, producing:
- `rocfft_kernel_cache_gfx908.db`
- `rocfft_kernel_cache_gfx90a.db`
- `rocfft_kernel_cache_gfx942.db`
- etc.

Install all per-arch files to `lib/rocfft/`. The kpack handler is then a trivial filename pattern match (same approach as rocBLAS/MIOpen handlers).

**Pro:** rocFFT owns the full lifecycle of its databases. kpack handler is trivial.
**Con:** Requires rocFFT build system and install rule changes.

### Alternative B: kpack handler splits the combined database post-build

Keep rocFFT producing a single `rocfft_kernel_cache.db`. Add a kpack handler that opens the SQLite DB, queries architectures, and creates per-arch files by copying matching rows.

**Pro:** No upstream rocFFT build changes (beyond runtime search path).
**Con:** kpack handler needs sqlite3 dependency and database manipulation logic, unlike every other handler which does simple file moves.

## Implementation checklist

Regardless of alternative chosen:

- [x] Fix solution map shard-invariance (option A or B above)
- [x] Modify `rtccache_db_sys_paths()` to search for `rocfft_kernel_cache_{arch}.db`
- [x] Produce per-architecture database files (via alt A or B)
