Parent issue tracking shipping ROCm multi-architecture packaging with Kpack based on this RFC: https://github.com/ROCm/TheRock/blob/main/docs/rfcs/RFC0008-Multi-Arch-Packaging.md.

## Overview

ROCm currently ships "fat binaries" that embed device kernels for all supported GPU architectures within each binary. This approach has several limitations:

1. **Build scalability**: Each distribution family requires a complete build pipeline, limiting parallelism
2. **Binary bloat**: Users must download and store kernels for architectures they don't use
3. **Architecture support pressure**: Lesser-used architectures face deprecation due to build time and size costs
4. **Compression inefficiency**: No cross-kernel deduplication or advanced compression

This issue tracks implementing the work outlined in the RFC which will improve each of those areas.

## Major milestones

* Integrating rocm-kpack (https://github.com/ROCm/rocm-kpack) into TheRock to enable architecture-separated packaging.
  * Kpack provides a structured format for separating device code from host code at packaging time, enabling parallel builds, selective deployment, and future compression improvements.
* Adding continuous integration (CI) coverage for multi-arch builds, including all existing project unit tests
* Build and distribute multi-arch ROCm Python packages
* Build and distribute multi-arch PyTorch Python packages
* Build and distribute multi-arch ROCm native Linux packages
