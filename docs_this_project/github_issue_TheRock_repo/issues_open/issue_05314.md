Hi, I noticed some performance gaps of PyTorch matmul on gfx1151, even though we've been improving hipBLASLt TensileLite kernels. I believe optimizing the basic matmul is a basis of optimizing more complicated ops like attention and MoE.

## Context about TensileLite kernel tuning

With the env var `TORCH_BLAS_PREFER_HIPBLASLT=1`, PyTorch ops like `torch.mm` and `F.linear` lower to hipBLASLt kernels, which are generated by TensileLite. I did some searches in the rocm-libraries repo and found that the following PRs have been merged in the last few months:
- https://github.com/ROCm/rocm-libraries/pull/3112
- https://github.com/ROCm/rocm-libraries/pull/3913
- https://github.com/ROCm/rocm-libraries/pull/5130
- https://github.com/ROCm/rocm-libraries/pull/6146
- https://github.com/ROCm/rocm-libraries/pull/5946
- https://github.com/ROCm/rocm-libraries/pull/7468

And the following have been proposed:
- https://github.com/ROCm/rocm-libraries/pull/4576
- https://github.com/ROCm/rocm-libraries/pull/6504
- https://github.com/ROCm/rocm-libraries/pull/6564
- https://github.com/ROCm/rocm-libraries/pull/6592

Note that in a kernel name like TN_HBS, T = row major (transposed in the convention of hipBLASLt, not transposed in the convention of PyTorch), N = column major, H = float16 (half precision), B = bfloat16, S = float32 (single precision) for accumulator.

TN layout is the most relevant layout for the forward pass of PyTorch linear layers. It would be good to also tune TT, NT, and NN layouts, especially because they're used in the backward pass when training models. For example:
```
forward:
--transposeA T --transposeB N -m 48 -n 32 -k 64
grad_input:
--transposeA N --transposeB N -m 64 -n 32 -k 48
grad_weight:
--transposeA N --transposeB T -m 64 -n 48 -k 32
```

These PRs generally improve the tuned kernels from 30~35 to ~40 TFLOPS for large matrices. I've confirmed the improvement on my machine. Note that the results on my machine are ~36 rather than ~40 TFLOPS because of thermal throttling and Python dispatch overhead.

## Remaining issues

In the above results, when `TORCH_BLAS_PREFER_HIPBLASLT=1`, we can see `torch.mm` and `F.linear` match the performance of hipBLASLt. TT and NT layouts are still not yet well tuned.

As a user, I would enable `TORCH_BLAS_PREFER_HIPBLASLT=1` only if it consistently outperforms `TORCH_BLAS_PREFER_HIPBLASLT=0` on commonly used input shapes and layouts. If no one is going to tune the remaining kernels in the near future, I can find some time to do it, but I hope someone can do it earlier than me.

Also, PyTorch calls `cublasLtMatmulAlgoGetHeuristic(..., requestedAlgoCount=1, ...)`, so it only uses one algorithm returned by heuristic without JIT benchmarking. I'm not sure whether it's worth returning more algorithms by heuristic and JIT benchmarking them, as in https://github.com/woct0rdho/ComfyUI-FeatherOps/blob/a26f6ff91616e907091a1e6fc0b1f6d58af5a2f6/kernel/hip/hipblaslt_kernel_fp16.cu . Maybe we should somehow do this in torch.compile max autotune.
