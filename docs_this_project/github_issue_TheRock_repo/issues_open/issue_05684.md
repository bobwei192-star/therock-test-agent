# #5684 - LLVM sub-build uses unlimited parallelism, causing OOM on systems with many cores vs RAM

## Problem

On consumer PCs with many cores relative to RAM (e.g. 14600K / 20 threads / 32 GB), a default build of TheRock OOM-kills the build process. The LLVM sub-build spawns its own internal Ninja with unlimited parallelism, ignoring the top-level `-j` flag, so each compile unit (some using 2–6 GB RSS) scales across all cores simultaneously.

## Root cause

The LLVM sub-project is built via `therock_cmake_subproject_declare` in `compiler/CMakeLists.txt`, which runs a separate CMake+ Ninja invocation. The top-level `ninja -j` does not propagate into that sub-build.

LLVM provides two mechanisms to control this:

- `LLVM_PARALLEL_COMPILE_JOBS` — hard limit on parallel compiles
- `LLVM_RAM_PER_COMPILE_JOB` — auto-computes job count from `AVAILABLE_PHYSICAL_MEMORY / RAM_PER_JOB`

But neither is exposed at TheRock's top-level CMake or forwarded to the LLVM sub-build. Only `LLVM_PARALLEL_LINK_JOBS` and `FLANG_PARALLEL_COMPILE_JOBS` are currently exposed (and both default to empty / unlimited).

## Existing infrastructure

- `compiler/CMakeLists.txt` already has an `_extra_llvm_cmake_args` mechanism used to forward `FLANG_PARALLEL_COMPILE_JOBS` and `LLVM_PARALLEL_LINK_JOBS`
- `compiler/amd-llvm/llvm/cmake/modules/HandleLLVMOptions.cmake` already implements `LLVM_RAM_PER_COMPILE_JOB` auto-detection (including `cmake_host_system_information` to query available RAM)
- `THEROCK_BACKGROUND_BUILD_JOBS` exists but only limits `BACKGROUND_BUILD` sub-projects (amd-comgr, hipcc, hipify), not the main LLVM build

## Proposed fix

Expose `LLVM_PARALLEL_COMPILE_JOBS` and `LLVM_RAM_PER_COMPILE_JOB` as top-level CMake cache variables and forward them into the LLVM sub-build via `_extra_llvm_cmake_args`, with clear precedence (RAM-based wins if both set).

Would also be good to add a "\common build pitfalls'" section in `docs/environment_setup_guide.md` documenting the OOM issue and recommended flags.
