## Description

MIOpen fails to build under the `host-asan` CI workflow on gfx942 / gfx94X-dcgpu. The root cause is in TheRock's `cmake/therock_sanitizers.cmake`: when `THEROCK_SANITIZER=HOST_ASAN`, it appends `-fsanitize=address` to `CMAKE_CXX_FLAGS_INIT` (same as full ASAN) but does **not** rewrite `GPU_TARGETS` to `gfx942:xnack+` (unlike full ASAN). As a result, every MIOpen HIP TU is compiled with both `-fsanitize=address` and `--offload-arch=gfx942` (no xnack+), which clang rejects. Because MIOpen builds with `-Werror`, the `[-Woption-ignored]` diagnostic becomes a fatal error and kills the entire build.

This is a **regression introduced by TheRock PR #6180** (commit `14b3cb73`, merged 2026-06-26 18:50 UTC), which switched bump PRs and push events from full `asan` to `host-asan`. Before that commit, math-libs on gfx942 passed (e.g. run `28214286898` from 2026-06-26 02:59). Failures started at 18:50‚Äì18:52 UTC, within minutes of the merge.

## Root Cause

In `cmake/therock_sanitizers.cmake`, the `HOST_ASAN` branch appends `-fsanitize=address` to `CMAKE_CXX_FLAGS_INIT` identically to full `ASAN`, but deliberately skips the GPU target rewrite:

```cmake
# Full ASAN: rewrites gfx942 ‚Üí gfx942:xnack+ (clang accepts -fsanitize=address)
if(_sanitizer STREQUAL "ASAN" OR _sanitizer STREQUAL "TSAN")
    list(TRANSFORM GPU_TARGETS REPLACE "^(gfx942|gfx950)$" "\\1:xnack+")
    ...
else()
    # HOST_ASAN: GPU_TARGETS unchanged ‚Äî still plain gfx942
    message(STATUS "HOST_ASAN enabled - GPU_TARGETS unchanged")
endif()
```

The `-fsanitize=address` flag ends up in the HIP device compile command for plain `gfx942`. Clang requires `xnack+` for ASAN device instrumentation and rejects the flag with `[-Woption-ignored]`, which `-Werror` promotes to a fatal error.

## Expectation

`HOST_ASAN` builds must compile successfully. Since `HOST_ASAN` is explicitly host-only, `-fsanitize=address` must not be passed to HIP device compile TUs when the GPU target lacks `xnack+`.

## Impact

Blocking `math-libs` (MIOpen) in the `host-asan` CI workflow for all gfx942 targets since 2026-06-26. Because PR #6180 routes all bump PRs through `host-asan`, **every submodule bump PR now fails the ASAN math-libs stage**. Continuous failures on `main` confirmed in runs 28270795956, 28276586324, 28309465405, 28345788558.
