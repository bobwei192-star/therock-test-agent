context: intentionally or otherwise, users can trigger parent CI workflow that triggers child builds for all gfx architecture. this can cause resource starvation specially if was triggered but not needed. this issue is to create a UI friendly automation that uses github api(s) and run_id calculation to kill all child jobs triggered via parent job. 
recommendations:
1)githuh api usage: 

```
- name: Cancel therock child workflows
  run: |
    for run_id in $(cat child_runs.txt); do
      curl -s -X POST \
        -H "Authorization: Bearer ${{ steps.app-token.outputs.token }}" \
        https://api.github.com/repos/${{ github.repository }}/actions/runs/$run_id/cancel
    done
```

run_id calculation is where most time needs to go. some initial thoughts over chat:

1) introduce run-name: child-of-${{ inputs.parent_run_id }}, 
2) can introduce a run_id.jsonl like capacity_history.jsonl for runner health. and add a button in runner health report itself to kill child jobs for queued jobs.
3) https://github.com/benc-uk/workflow-dispatch now supports getting the dispatched workflow's run id, so we could walk the workflow run graph looking for that.
