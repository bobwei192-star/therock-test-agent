## Summary
The portable Linux ROCm build's `hipblaslt-test` binary aborts in the dynamic linker **before `main()`** on glibc 2.28 (a RHEL 8-derived enterprise Linux distribution; MI308X/gfx942):

> Inconsistency detected by ld.so: rtld.c:1657: dl_main: Assertion `GL(dl_rtld_map).l_libname' failed!

- `ldd ./hipblaslt-test` resolves every library (not a missing-dependency issue).
- `file ./hipblaslt-test` → ELF 64-bit LSB ... interpreter /lib64/ld-linux-x86-64.so.2, for GNU/Linux 3.2.0, ... *too many notes (256)*.

Because the abort happens in the loader, **no tests run at all** → total loss of hipBLASLt test coverage on this OS class. This should reproduce on any glibc-2.28 (EL8-class) userspace.

## Why this is a TheRock packaging concern (not a library/source bug)
- TheRock's portable Linux artifacts are built on `quay.io/pypa/manylinux_2_28_x86_64` (glibc 2.28), so **glibc 2.28 is the intended portability floor**. The affected OS is a standard EL8-class distribution at exactly that floor — not an unsupported platform.
- The failure is in `ld.so` / ELF loading, independent of any hipBLASLt code path.
- TheRock rewrites every binary post-build with the pinned NixOS `patchelf` (`PATCHELF_GIT_REF=d0f70eea5397606c486857e0a105e53ec123904a`) for `rocm_sysdeps` bundling: SONAME prefixing (`rocm_sysdeps_*`), `$ORIGIN`-relative RPATH injection, and `AMDROCM_SYSDEPS_1.0` symbol versioning. The `rtld.c ... l_libname` assertion is a known signature of patchelf-relocated program headers tripping the glibc loader.

## Regression window (same OS, same glibc 2.28)
- **PASS:** 7.14.0-20260612 (9209 tests passed)
- **FAIL:** 7.14.0-20260618 (binary never launches; ld.so assertion)

Something in the build/packaging pipeline between 20260612 and 20260618 broke loadability on the glibc-2.28 floor.
