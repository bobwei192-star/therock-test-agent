### Problem Description

When building ROCm packages for xnack-enabled targets (e.g., gfx942:xnack+, gfx942:xnack-), the Linux packaging system fails to correctly handle xnack variant artifacts.

Current behavior:
  1. Target discovery returns full xnack suffixes (e.g., gfx942:xnack+) which causes issues with package naming
  2. Artifact collection only looks for the base artifact directory (e.g., fft_lib_gfx942), missing xnack variant directories (fft_lib_gfx942:xnack+,
  fft_lib_gfx942:xnack-)
  3. CLI inputs with xnack suffixes are not normalized, leading to inconsistent behavior

  Expected behavior:
  - Package names should use base architecture only (e.g., amdrocm-foo-gfx942)
  - All xnack variant artifacts should be bundled into the corresponding base arch package
  - xnack suffixes in CLI inputs should be stripped early for consistent processing
