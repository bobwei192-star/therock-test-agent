I don't think a direct translation is necessarily the way to go though. The design all along should have been
1. Compute manifests (see https://github.com/ROCm/TheRock/issues/1236)
2. Build based on those manifests

Instead of the current
1. Build based on floating refs (these can differ across workflows running concurrently)
2. Generate manifests based on what was build

The multi-arch release workflow structure can be seen as a forcing function here. While before we had each GPU family building in parallel for each pytorch version x python version, now we have a common build for _all_ GPU targets (also building in parallel for each pytorch version x python version). This results in a single build job --> multiple test jobs fan out where having some reliable ground truth for versions/commits results in cleaner data flow.
