### Description of errors

The recent transition to multi-arch artifacts (#5790) makes it the primary supported installation method and the path that will continue receiving new builds, where per-family artifacts are in maintenance mode and will no longer receive new releases.

However, the current structure of RELEASES.md makes the multi-arch workflow feel like an alternative installation method rather than the replacement for per-family installs.

While the documentation contains a note explaining the status of per-family artifacts, that information is only encountered after navigating to the top of the per-family section. A long-time user who goes directly to the install commands may miss this and end up install old per-family builds.

For a long-time TheRock user who is used to the per-family path, their typical User Journey looks like this:
1. Open RELEASES.md
2. Search (CTRL + F) for my GPU architecture or GPU name (i.e. gfx1103 or 780M)
3. Find the one-line command to install ROCm

This workflow was flawless when per-family artifacts were the primary, and only distribution mechanism. Now that there's two install paths, long-time users who have built a habit for this workflow can get lost. 

This can be written off as user error, but it would be beneficial to help our long-time users transition to the multi-arch path after becoming familiar with per-family for so long. The per-family section should be written and organized in a way where users would only need to use it if there is some specific debugging or development effort that requires them to use an old build.

Some suggestions I have to improve user guidance are:

1. Add a prominent notice at the top of the documentation, so that users can be informed before CTRL+F and skipping the notices placed later in the documentation
2. Clearly label the per-family artifacts as "Legacy" or move them into a "Legacy" section in the docs
3. Add warnings before in each per-family installation example
