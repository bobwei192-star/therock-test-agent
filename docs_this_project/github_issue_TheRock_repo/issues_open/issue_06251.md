## Background

Follow-up from review of #6086 (cc @ScottTodd @marbre).

`build_tools/github_actions/upload_python_packages.py` decides whether to emit a
per-family package summary by inspecting the uploaded directory layout (see
`detect_family_subdirs()` / the `multiarch` handling around
`upload_python_packages.py:277`). That detection relies on the packages being
laid out in the current **flat** structure:

```
today:
  torch.whl
  rocm.whl
  jax.whl
```

This works today, but it is not robust to a future change in publish layout. If
we move packages into per-package **subdirectories**:

```
later:
  torch/
    torch.whl
  rocm/
    rocm.whl
  jax/
    jax.whl
```

then the flat-layout assumption (and the `../` relative links that the flat
layout currently produces in pages like
`https://rocm.nightlies.amd.com/whl-multi-arch/torch/`) no longer holds, and the
family-detection heuristic may misbehave.

## Action items

- [ ] Revisit the family-detection logic in `upload_python_packages.py` once the
      package upload layout moves from flat to per-package subdirectories.
- [ ] At that point, also remove the now-unnecessary `multiarch` branch in that
      code (the subdirectory layout removes the need for the `../` indirection
      that the special-casing works around).
