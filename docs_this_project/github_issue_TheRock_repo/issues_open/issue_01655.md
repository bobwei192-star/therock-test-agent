### Problem Description

When MIOpen is built in TheRock using `--preset==linux-release-package` I am seeing issues where the logging prints as if it was built in debug (all logging enabled).

Additionally, I see crashing when trying to use the libMIOpen.so built in this way.
It seems to crash when creating a MIOpen handle in the hipDNN samples, example [here](https://github.com/ROCm/rocm-libraries/blob/develop/projects/hipdnn/samples/batchnorm/BnBackward.cpp).

Segfault occurs during:
```
    hipdnnHandle_t handle;
    HIPDNN_CHECK(hipdnnCreate(&handle));
```

The call stack is segfaulting inside MIOpen when the handle is being created (hipDNN loads the MIOpen plugin. and tries to create the handle which explodes)

The segfault issue might be related to an undefined symbol issue we were seeing with rocblas when built using this mode, but it requires further investigation to see if it's related. 

Issues aren't seen when using the release build config.

### Steps to Reproduce

I have a docker file [here](https://github.com/ROCm/rocm-libraries/blob/develop/projects/hipdnn/dockerfiles/Dockerfile.ubuntu24) that can build TheRock from source.

Example run command to replicate::
`docker build -f Dockerfile.ubuntu24 --build-arg BUILD_TYPE=fullbuild --build-arg THEROCK_ASIC=gfx90a -t registry-sc-harbor.amd.com/miopen-images/hipdnn_env:fullbuild_20251001_gfx90a .`

I believe this also can be replicated by building TheRock manaually in a Ubuntu 24.04 environment by providing the cmake commands:

```
cmake .. \
    --preset=linux-release-package \
    -DCMAKE_INSTALL_PREFIX=/opt/rocm \
    -DTHEROCK_ENABLE_ALL=OFF \
    -DTHEROCK_ENABLE_MIOPEN=ON \
    -DTHEROCK_AMDGPU_FAMILIES=gfx90a

cmake --build . && cmake --install .
````
