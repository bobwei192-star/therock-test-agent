### ROCm ASAN Packaging for TheRock

Parent ticket for tracking Shipping ROCm ASAN packages. 

## Summary

Add AddressSanitizer (ASAN)–instrumented build variants to the ROCm release pipeline in TheRock, and publish them as  release artifacts alongside the existing non-instrumented packages.

## Background & Motivation

ROCm is currently released as multi-architecture artifacts in three forms: tarballs, Python wheels, and native Linux packages (RPM and DEB). None of these are built with sanitizer instrumentation.

AddressSanitizer is a compile-time instrumentation pass (`-fsanitize=address`) that detects memory-safety defects at runtime — heap/stack/global buffer overflows, use-after-free, use-after-return, and memory leaks. Shipping ASAN-enabled ROCm builds lets developers and partners link their applications against an instrumented ROCm stack to catch memory errors that originate in or interact with ROCm libraries, which is not possible with the current release artifacts.

Because ASAN changes the ABI/runtime behavior and requires the ASAN runtime to be present, these must be shipped as **separate, clearly-labeled package variants** rather than replacing the standard packages.

## Scope

**In scope** — ASAN variants for:
- Tarball
- Native Linux packages (RPM, DEB)

**Out of scope (for now):**
- ASAN-instrumented Python wheels (can be tracked as a follow-up if there is demand)
- Coexistence with standard package is not a requirement for now. 

## Technical Considerations

- **Build flags:** ASAN builds require `-fsanitize=address`  propagated through the relevant CMake/HIP toolchain configuration. Confirm how this is plumbed for both host and device code in TheRock.
- **Packaging/naming:** ASAN artifacts must be distinguishable from standard artifacts 
      - For tar ball adding a -asan suffix . For eg `rocm-7.14-asan-<arch>.tar.gz` or similar
      - For native packages it should be in a seperate folder as defined in RFC12(https://github.com/ROCm/TheRock/blob/main/docs/rfcs/RFC0012-Repo-Structure.md). Like repo.amd.com/rocm/packages/$OS/asan
      - Install location of packages should be /opt/rocm/core-7.14
- **Runtime dependency:** The ASAN runtime must be declared as a dependency. Document the expected `LD_PRELOAD` / runtime setup for consumers.
- **Multi-arch:** ASAN variants must be produced for all Mi series of architecure, like gfx908, gfx90a, gfx942, gfx950.
- **CI cost:** ASAN builds increase build time, artifact size, and storage. So we start with weekly builds for ASAN.

## Major Milestones
- [ ] **Pipeline integration** — Create a ASAN release build into the ROCm release pipeline similar to ([multi_arch_release.yml](https://github.com/ROCm/rockrel/blob/main/.github/workflows/multi_arch_release.yml)) with weekly schedule 
- [ ] **Weekly tarballs** — Generate ASAN tarballs in ROCm nightly builds
- [ ] **Weekly native packages** — Generate ASAN native packages (RPM/DEB) in ROCm nightly builds
- [ ] **Pre-release/Release builds** — Generate ASAN packages in pre-release builds

## Acceptance Criteria

- ASAN tarball and native packages are produced automatically by the weekly and pre-release pipelines for all supported architectures.
- Artifacts are clearly named and documented so they cannot be confused with standard packages.
- A documented procedure exists for installing and running against an ASAN ROCm build, with at least one validated end-to-end smoke test.
