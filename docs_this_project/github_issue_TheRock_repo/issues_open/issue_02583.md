### Suggestion Description

There are certain times where rocm-libraries teams must submit a PR with urgency in order to meet a customer deadline. However at times of high contention, TheRock CI can be waiting for 12+ hours for a GPU executor. Combined with infrastructure failures that cause failures, a re-run is often needed which goes back into the long queue.

We would like to have some confidence in the code change with automated TheRock CI testing but sometimes cannot wait for multiple cycles of long queues. While system contention remains an issue, can a queue priority system be implemented to give high priority PRs a fast track to testing and merging?
