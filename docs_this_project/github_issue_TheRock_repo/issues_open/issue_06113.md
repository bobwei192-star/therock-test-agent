## Problem Description

`rocm-sdk-core-7.14.0a20260624` introduced a new `clang++.cfg` → `rocm.cfg` configuration file that unconditionally injects `-Wl,--enable-new-dtags` into every `clang++` invocation. This breaks PyTorch inductor's precompiled header (PCH) compilation with:

```
clang++: error: cannot specify -o when generating multiple output files
```

When compiling a PCH (`-x c++-header ... -o file.pch`), clang++ treats `-Wl,--enable-new-dtags` as implying an additional linker output, making total outputs > 1, which conflicts with the explicit `-o flag`.

This causes all inductor shards to fail immediately at the CPU sanity check — before any GPU tests run — in PyTorch CI: https://github.com/ROCm/TheRock/actions/runs/28110484343

### Root cause bisection

Reproduced on `banff-s80-2` inside `ghcr.io/rocm/no_rocm_image_ubuntu24_04@sha256:405945a40deaff9db90b9839c0f41d4cba4a383c1a7459b28627047bf6302a26`.

**Jun12 SDK (passes):** `rocm-sdk-core-7.14.0a20260612` — no `clang++.cfg`, `clang++` commit `52226beb248f`

**Jun24 SDK (fails):** `rocm-sdk-core-7.14.0a20260624` — has `clang++.cfg` referencing `rocm.cfg`, `clang++` commit `46fcb339fb61`

Contents of new `clang++.cfg`:
```
@rocm.cfg
```

Contents of `rocm.cfg`:
```
-Wl,--enable-new-dtags
--rocm-path='<CFGDIR>/../../..'
-frtlib-add-rpath
```

Minimal repro confirming `-Wl,--enable-new-dtags` is the sole trigger:

```bash
# Fails (Jun24, cfg active):
clang++ -x c++-header test.h \
  -fno-trapping-math -funsafe-math-optimizations -ffinite-math-only \
  -fno-signed-zeros -fno-finite-math-only -fno-unsafe-math-optimizations \
  -fmath-errno -ffp-contract=off -march=native -fPIC -std=c++20 \
  -o test.h.pch
# clang++: error: cannot specify -o when generating multiple output files

# Passes (Jun24, cfg disabled):
clang++ --no-default-config -x c++-header test.h [same flags] -o test.h.pch
# SUCCESS

# Passes (Jun12, no cfg):
clang++ -x c++-header test.h [same flags] -o test.h.pch
# SUCCESS

# Isolated: -Wl,--enable-new-dtags alone reproduces the failure:
clang++ --no-default-config -Wl,--enable-new-dtags -x c++-header test.h [same flags] -o test.h.pch
# FAILED

# Isolated: other cfg flags (-frtlib-add-rpath, --rocm-path) are fine alone:
clang++ --no-default-config -frtlib-add-rpath -x c++-header test.h [same flags] -o test.h.pch
# SUCCESS
```

## Steps to Reproduce

```bash
# Pull the container used in CI
docker run -it ghcr.io/rocm/no_rocm_image_ubuntu24_04@sha256:405945a40deaff9db90b9839c0f41d4cba4a383c1a7459b28627047bf6302a26 bash

python3 -m venv /tmp/venv && source /tmp/venv/bin/activate
pip install --index-url=https://rocm.nightlies.amd.com/whl-multi-arch/ "rocm[devel]==7.14.0a20260624"

CLANGPP=$(find /tmp/venv -name clang++ | head -1)
echo "#include <cstdio>" > /tmp/test.h

# Reproduces the CI failure:
$CLANGPP -x c++-header /tmp/test.h \
  -fno-trapping-math -funsafe-math-optimizations -ffinite-math-only \
  -fno-signed-zeros -fno-finite-math-only -fno-unsafe-math-optimizations \
  -fmath-errno -ffp-contract=off -march=native -fPIC -std=c++20 \
  -o /tmp/test.h.pch
```

Expected: PCH file created successfully.
Actual: `clang++: error: cannot specify -o when generating multiple output files`

## Suggested Fix

`-Wl,--enable-new-dtags` is a linker-only flag and should not be injected unconditionally for all compilation modes. The `rocm.cfg` / `clang++.cfg` should either:
- Scope linker flags so they are not passed during header-only or compile-only invocations, or
- Use a driver config that conditionally applies `-Wl,...` flags only when linking
