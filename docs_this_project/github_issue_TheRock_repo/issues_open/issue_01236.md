At the moment, our PyTorch release workflows use floating branch or tag names like
* `release/2.7`: https://github.com/ROCm/TheRock/blob/11739e9aed1c07d4b3ceba65c78dcaecb37eaf0c/.github/workflows/build_portable_linux_pytorch_wheels.yml#L65-L69
* `main`: https://github.com/ROCm/TheRock/blob/11739e9aed1c07d4b3ceba65c78dcaecb37eaf0c/.github/workflows/build_windows_pytorch_wheels.yml#L105-L116

These are the _versions_ that we want, but if a workflow fails and we re-run it, the workflow can pick a different commit. Furthermore, if multiple workflows are each schedule at slightly different times, each can use a different commit even without any re-runs - and that is what happens with one run of [`release_portable_linux_packages.yml`](https://github.com/ROCm/TheRock/actions/workflows/release_portable_linux_packages.yml) -> multiple runs of [`release_portable_linux_pytorch_wheels.yml`](https://github.com/ROCm/TheRock/actions/workflows/release_portable_linux_pytorch_wheels.yml). Windows is especially prone to this because it uses pytorch `main`, which receives 10s of commits per day.

See for example:
* https://github.com/ROCm/TheRock/actions/runs/16909060869/job/47905678539#step:7:70 `HEAD is now at f33ce40bc0 [bucketing] Bucket only adjacent collectives to prevent reordering (#159983)`
* (two hours later, for the same nightly release) https://github.com/ROCm/TheRock/actions/runs/16910148775/job/47909418218#step:7:70 `HEAD is now at 7fbc22855c Update triton xpu commit to support python 3.14 (#160183)`

I think we should introduce a workflow like https://github.com/iree-org/iree/blob/main/.github/workflows/schedule_candidate_release.yml that chooses specific commits to release from and then triggers independent workflow runs using workflow_dispatch. We would be intentionally breaking the link between workflows that way, such that we could re-run the child workflows and have them retain the frozen commits determined during the "schedule" workflow run. In the child workflows, we would checkout with commands like `pytorch_torch_repo.py checkout --repo-hashtag EXACT_COMMIT --patchset main`

---

We can also _remove_ the workflow_dispatch from the `release_*_packages.yml` workflows so they keep all actions relating to a single release grouped together: https://github.com/ROCm/TheRock/blob/11739e9aed1c07d4b3ceba65c78dcaecb37eaf0c/.github/workflows/release_portable_linux_packages.yml#L208-L210

I do like having separate status badges for ROCm packages and PyTorch packages at https://github.com/ROCm/TheRock?tab=readme-ov-file#nightly-release-status though, so we could keep the trigger but have the PyTorch workflow use a matrix across gfx targets _and_ Python versions, instead of just Python versions.

However we refactor the ROCm and PyTorch workflows, we should keep https://github.com/ROCm/TheRock/issues/1072 in mind though - if we want to only upload ROCm releases if PyTorch tests pass, having separate workflows makes that trickier.
