# #2084 - Add support for uploading test reports to S3 from fork-based PR CI runs

Summary

Currently, our CI workflows only upload build artifacts and test reports to S3 for runs triggered from internal branches (e.g., main, develop, or feature branches within the ROCm organization).

When a pull request is opened from a fork, the CI run completes successfully but does not copy test reports or logs to the S3 artifact buckets (therock-artifacts or therock-artifacts-external). This makes it difficult to review test results for fork-based contributions.

Goal

Enable the upload of test logs and reports to S3 for PRs originating from forks, while preserving security and preventing untrusted forks from gaining direct write access to internal S3 buckets.

Proposed Solutions

To safely support this, we are evaluating two approaches:

Use OIDC (OpenID Connect) to grant temporary scoped access for the test runner to upload artifacts to designated S3 buckets, restricted by repository context and event type.

Implement conditional upload logic in the workflow to determine the correct destination bucket based on the source of the PR (internal branch vs. external fork).

Next Steps

 1. Define and validate IAM/OIDC policies for fork-based workflow runs.

 2. Update the workflow logic to support conditional S3 upload paths.

3.  Test uploads from both internal and fork-based PRs to ensure consistent behavior.


This issue tracks the work to enable secure artifact and test log uploads from fork-based PR runs, ensuring contributors outside the ROCm organization can still produce and share test results for review.
