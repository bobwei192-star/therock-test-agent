# #4202 - Generic artifacts can silently overwrite existing artifacts with different SHA in S3 uploads

## Summary

Generic (`_generic`) artifacts uploaded to S3 can silently overwrite existing artifacts without any verification, even when the new artifact has a different SHA256 checksum. This can mask build system problems where a subproject was incorrectly marked as `target-neutral` but actually produces target-specific binaries.

## Problem

The current upload logic in `artifact_manager.py` and `storage_backend.py` performs uploads without:

1. Checking if the S3 object already exists
2. Comparing checksums between existing and new artifacts
3. Warning or failing when different content would overwrite an existing artifact

### Affected Code Paths

**`storage_backend.py:265-279`** - `S3StorageBackend.upload_file()`:
```python
def upload_file(self, source: Path, dest: StorageLocation) -> None:
    # No existence check, no SHA comparison
    _s3_retry(
        "upload",
        dest.s3_uri,
        self.s3_client.upload_file,
        str(source),
        dest.bucket,
        dest.relative_path,
        ExtraArgs={"ContentType": content_type},
    )
```

**`artifact_backend.py:302-305`** - `S3Backend.upload_artifact()`:
```python
def upload_artifact(self, source_path: Path, artifact_key: str) -> None:
    loc = self.output_root.artifact(artifact_key)
    self.s3_client.upload_file(str(source_path), self.bucket, loc.relative_path)
```

### Risk Scenario

1. Stage A (generic) builds `foo_lib_generic.tar.zst` on Job 1 with SHA `abc123`
2. Stage A (generic) runs again on Job 2 in parallel (different runner) with different content (e.g., due to non-deterministic build or incorrectly-marked target-specific artifact), producing SHA `def456`
3. Both jobs call `artifact_manager.py push` which uploads to the same S3 path
4. The second upload silently overwrites the first - **no warning, no error**

This is particularly concerning because:
- If an artifact is incorrectly marked as `target-neutral` in `BUILD_TOPOLOGY.toml` but actually contains target-specific binaries, multiple parallel jobs could produce different content
- The "last writer wins" with no audit trail
- SHA256sum files *are* uploaded alongside artifacts, but there's no pre-upload check against existing checksums

## Proposed Solution

For generic artifacts, add SHA-based conflict detection:

1. Before uploading, check if the artifact already exists in S3
2. If it exists, download or fetch the existing `.sha256sum` file
3. Compare with the local artifact's SHA256
4. If they differ, **fail the upload with an error** - this surfaces incorrectly-marked artifacts
5. If they match, skip the upload (artifact is already there with correct content)

This ensures:
- Genuinely generic/deterministic artifacts can be safely uploaded from any parallel job
- Non-deterministic or incorrectly-marked artifacts are detected and flagged
- No silent data corruption

## Additional Context

The artifact type (`target-neutral` vs `target-specific`) is defined in `BUILD_TOPOLOGY.toml` and propagated to CMake via `topology_to_cmake.py`. The CMake logic in `therock_artifacts.cmake` uses the `TARGET_NEUTRAL` flag to determine whether to use `_generic` suffix.

Target-specific artifacts are split by `split_artifacts.py` into per-architecture artifacts, so they don't have this issue (each arch gets its own S3 key).
