## Problem

With `THEROCK_ENABLE_ROCPROFSYS=ON`, the build fails on any host whose system compiler is **GCC 15** (Ubuntu 25.10, Fedora 42, Arch, …).

`rocprofiler-systems` and its bundled dyninst are compiled with the **host system compiler**, not TheRock's amd-llvm toolchain — unlike most third-party subprojects, which were moved onto amd-llvm in #4032 (re-applying #3474). `profiler/CMakeLists.txt` encodes the assumption directly:

```cmake
# rocprofiler-systems requires GCC, but sanitizer builds use Clang.
# Skip enabling rocprofiler-systems when building with sanitizers.
if (THEROCK_ENABLE_ROCPROFSYS AND THEROCK_SANITIZER STREQUAL "")
```

So on a GCC 15 host the whole stack runs through GCC 15's stricter defaults and fails to compile. (As a side effect, rocprofsys is also silently dropped from sanitizer builds.)

## The failures fall into two classes

1. **GCC 15 `-Werror`-by-default dialect rules** — `incompatible-pointer-types`, `discarded-qualifiers`, `unterminated-string-initialization`. These are GCC frontend defaults; they do not occur under clang. Reported piecemeal in #3515 (PAPI) and #5120 (elfutils riscv_disasm), and fixed in rocm-systems#5574 (elfutils register-name).
2. **libstdc++ 15 dropping transitive `<cstdint>`** (#3517) — a standard-library change, so it affects **any** compiler using libstdc++ 15, clang included. Example: `dyninst/stackwalk/src/framestepper.C` uses `uint64_t` (lines 359/370/393) with no direct `<cstdint>` include and fails to compile without one.

## Observation

Building rocprofiler-systems with the amd-llvm toolchain (`COMPILER_TOOLCHAIN amd-hip`, the same pattern #4032 applies to its siblings) and dropping the GCC-only guard produces a clean full build on Ubuntu 25.10 / GCC 15.2.0 / amd-llvm clang (gfx1102+gfx1103), rocprofsys staged and linkable. Failure class (1) disappears; only the class-(2) `<cstdint>` issue (#3517) remains and needs a source-level fix regardless of which compiler is used.

Recording this as the root-cause view behind the individual GCC-15 break reports (#3517, #3515, #3477). Any approach that builds rocprofsys on a GCC 15 host resolves the user-facing failure.

(dyninst's earlier `std::hash<MachRegister>` failure is *not* part of this — not reproducible at the current pin `8d6f2bf30`; resolved by the boost removal and closed as #5460.)

## Environment

Ubuntu 25.10, GCC 15.2.0, amd-llvm clang, `THEROCK_AMDGPU_FAMILIES=gfx1102;gfx1103`, `THEROCK_ENABLE_ROCPROFSYS=ON`, rocm-systems at a tip including #6419.
