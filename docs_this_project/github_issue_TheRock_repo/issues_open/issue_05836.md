### Problem Description

## Summary

We are observing a benchmark performance regression at **batch size 32** in vLLM. The regression correlates with a change in the vLLM build version.

## Build Versions

| | Build |
|---|---|
| **Baseline (no regression)** | `v0.9.2rc2…d20260317` (March build) |
| **Regressed** | `v0.21.1.dev0…d20260610` (June build) |

## Symptoms

- Performance regression is observed at **batch size 32**
- The June build (`v0.21.1.dev0…d20260610`) shows degraded benchmark results compared to the March build (`v0.9.2rc2…d20260317`)

## Root Cause Hypothesis

The primary variable that changed between the two data points is the vLLM build itself — from a March release candidate (`v0.9.2rc2`) to a June development build (`v0.21.1.dev0`). No other significant environmental changes have been identified.

## Steps to Reproduce

1. Run vLLM benchmark with batch size 32 using the March build (`v0.9.2rc2…d20260317`)
2. Run the same benchmark with the June build (`v0.21.1.dev0…d20260610`)
3. Compare throughput/latency results

## Additional Context

- Batch sizes other than 32 may or may not be affected — further investigation needed
- This may be related to changes introduced between vLLM v0.9.x and v0.21.x dev

Please investigate the performance delta between these two builds and identify which commit or change introduced the regression.
