## Summary

ccache hit rate caps near 38% on full TheRock rebuilds. Root cause: each subproject's stage/install step rewrites `dist/include/*` and `stage/include/*` headers with content that drifts between builds even when source content is unchanged. Downstream subprojects with a wide header surface (composable_kernel, MIOpen) bear the worst impact, because each compile pulls in hundreds of these volatile headers.

Possibly related to #1224 (a placeholder ticket from 2025-08 with no body) and #4419 (Windows-specific). I have not seen this Linux scenario discussed elsewhere.

## Empirical evidence

Tested on `integrate/local` rebased on `main` as of 2026-05-02, single-arch `gfx1102`, ccache 4.11.2, Linux, gcc 15. Standard ccache config: `depend_mode=true`, `compiler_check=content`, `base_dir=$THEROCK`, `sloppiness=include_file_mtime,include_file_ctime,time_macros,file_stat_matches`.

Cumulative ccache stats from a partial integration build:

```
Hits:    159781 / 429320 (37.22%)
Misses:  269538 / 429320 (62.78%)
```

Inspecting `ccache --inspect` on the manifest for `composable_kernel/library/src/utility/device_memory.cpp`, the manifest stored two Result entries with distinct fingerprint sets. About 50 of 408 indexed headers had been rewritten between the two compile sessions, all under staged paths, including:

- `build/core/clr/dist/include/hip/amd_detail/amd_hip_fp16.h`
- `build/profiler/rocprofiler-sdk/stage/include/...`
- `build/base/half/stage/include/half.hpp`
- `build/third-party/sysdeps/linux/{libdrm,numactl}/build/stage/lib/rocm_sysdeps/include/...`
- `build/compiler/amd-llvm/dist/include/...`

The `core/clr/dist/lib/llvm/bin/clang++` symlink also had a fresh mtime each build cycle. Combined with `compiler_check=content`, this further widens the invalidation blast radius for any session that follows an amd-llvm re-stage.

## Reproducer

Pick CK objects compiled in a previous session, delete and rebuild:

```bash
cd build/ml-libs/composable_kernel/build
TEST_FILES=$(find . -name '*.cpp.o' -not -newermt '<previous-session-cutoff>' -size -5M | head -5)
for f in $TEST_FILES; do rm -f "$f"; done
time ninja $TEST_FILES
```

In my test: 5 misses, 0 hits, 10.85s wall time across 5 small CK source files.

## Workaround (lossy, not safe to recommend)

Adding to `~/.config/ccache/ccache.conf`:

```
ignore_headers_in_manifest = $THEROCK/build/core/clr/dist/include:$THEROCK/build/compiler/amd-llvm/dist/include:$THEROCK/build/profiler/rocprofiler-sdk/stage/include:$THEROCK/build/profiler/roctracer/stage/include:$THEROCK/build/base/half/stage/include:$THEROCK/build/third-party/sysdeps/linux/libdrm/build/stage/lib/rocm_sysdeps/include:$THEROCK/build/third-party/sysdeps/linux/numactl/build/stage/lib/rocm_sysdeps/include
```

Same 5 CK objects after this change: 5 hits, 0 misses, 0.14s wall time. Roughly 77x speedup, projecting multi-hour CK rebuilds into a sub-10-minute warm-cache window.

The trade-off, and the reason this is a workaround rather than a recommendation: a real upstream HIP header change (e.g., a bug fix pulled via `fetch_sources.py`) would not invalidate cache and could yield silently wrong code. The mitigation is `ccache -C` after every fetch, which is easy to forget.

## Proposed fix

Make staged headers byte-identical to source where possible, so ccache's content-based invalidation can do its job correctly.

1. Audit headers under `dist/include/*` and `stage/include/*` for sources of nondeterminism: build timestamps, absolute paths embedded by `configure_file()`, version strings that change between minor builds.
2. For headers that need substitution, ensure substitutions are deterministic (same inputs produce identical bytes).
3. For headers that do not need substitution, prefer symlinks or `cmake -E copy_if_different`-equivalent semantics so the destination is not rewritten when content is unchanged.
4. Investigate why `core/clr/dist/lib/llvm/bin/clang++` symlink is recreated on each install cycle even when the underlying clang binary is unchanged.

This would benefit incremental local builds and CI machines that re-fetch sources, without forcing users to manage cache hygiene manually.

- TheRock branch: `integrate/local` (rebased on `main` as of 2026-05-02)
