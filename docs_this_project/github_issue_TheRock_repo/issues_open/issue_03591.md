### Suggestion Description

Concerns from  https://github.com/ROCm/TheRock/pull/3567

Raw notes are at https://github.com/ScottTodd/claude-rocm-workspace/blob/main/reviews/pr_3567.md

Each build takes 50-70 minutes, with a breakdown like this:

Phase	Time	% of Total
Docker LLVM 18 build from source	9-13 min	~18%
Bazel: jax-rocm-pjrt (cub_sort_kernel_rocm.cu.cc dominates)	21-31 min	~42%
Bazel: jaxlib	12-17 min	~23%
Bazel: jax-rocm-plugin	3-4 min	~6%
Docker setup, auditwheel, S3 upload	~5 min	~9%
We already build LLVM as part of ROCm... can rocm-jax use the same LLVM? Or install from a package / prebuilt location? Or at least build once and share across each python version build, since the compiler is independent of the python version?

The single largest bottleneck is cub_sort_kernel_rocm.cu.cc which takes 10-14 minutes alone (compiles CUB sort kernels for 9 GPU targets). I see this in the logs too:

20:56:50,335 - DEBUG - ROCm tookit path: /opt/rocm-7.12.0a20260217
20:56:50,340 - DEBUG - ROCm AMD GPU targets: gfx908,gfx90a,gfx942,gfx950,gfx1030,gfx1100,gfx1101,gfx1200,gfx1201
Is there an existing issue to properly plumb through the list of targets to build for?

Logs of log spam (to be fair, PyTorch has way more I think...)

Some concerning warnings during build/test, including some deprecations we should get ahead of
