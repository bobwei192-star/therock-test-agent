## Problem

The CK enablement logic in `ml-libs/CMakeLists.txt:8-20` overrides the topology-provided `THEROCK_ENABLE_COMPOSABLE_KERNEL=ON` by scanning `THEROCK_AMDGPU_TARGETS` at configure time. If **any** target is not in CK's hardcoded supported list, CK is disabled entirely for that shard:

```cmake
# ml-libs/CMakeLists.txt:8-20
if(THEROCK_ENABLE_COMPOSABLE_KERNEL)
  set(_ck_supported_gfx_targets gfx908 gfx90a gfx942 gfx950 gfx1150 gfx1151 gfx1200 gfx1201)
  foreach(_gfx_target ${THEROCK_AMDGPU_TARGETS})
    if(NOT ${_gfx_target} IN_LIST _ck_supported_gfx_targets)
      message(WARNING
        "${_gfx_target} (included in THEROCK_AMDGPU_TARGETS) is not supported by composable kernel. "
        "Disabling composable kernel and it's use in MIOpen.")
      set(THEROCK_ENABLE_COMPOSABLE_KERNEL OFF)
      set(THEROCK_MIOPEN_USE_COMPOSABLE_KERNEL OFF)
    endif()
  endforeach()
endif()
```

In multi-arch CI, the `gfx110X-all` shard targets gfx1100, gfx1101, gfx1102, and gfx1103 — **none** of which are in CK's supported list. This disables CK for that shard, causing MIOpen to be built without CK.

## Impact

CK is statically linked into `libMIOpen.so` (via `libdevice_conv_operations.a`, ~1.8 GB). When CK is enabled, `libMIOpen.so` contains `.hip_fatbin` sections with pre-compiled CK device code (~35 MB per architecture) and significant additional `.text` for CK host template instantiations. The resulting library is ~739 MB.

When CK is disabled, `libMIOpen.so` is structurally different and much smaller — it lacks the CK code objects, host instantiations, and fat binary sections entirely.

Multi-arch CI requires that the generic (non-architecture-specific) portion of each artifact is identical across all shards so that split artifacts can be merged. Structurally different library binaries break this assumption.

## CI evidence

From [run 22164483079](https://github.com/ROCm/TheRock/actions/runs/22164483079):

| Shard | Targets | CK builds? |
|-------|---------|------------|
| gfx94X-dcgpu | gfx942 | ✅ Yes |
| gfx1151 | gfx1151 | ✅ Yes |
| gfx120X-all | gfx1200, gfx1201 | ✅ Yes |
| **gfx110X-all** | gfx1100, gfx1101, gfx1102, gfx1103 | ❌ **No** |

The gfx110X-all shard logs show:
```
gfx1100 (included in THEROCK_AMDGPU_TARGETS) is not supported by composable kernel. Disabling composable kernel and it's use in MIOpen.
gfx1101 (included in THEROCK_AMDGPU_TARGETS) is not supported by composable kernel. Disabling composable kernel and it's use in MIOpen.
gfx1102 (included in THEROCK_AMDGPU_TARGETS) is not supported by composable kernel. Disabling composable kernel and it's use in MIOpen.
gfx1103 (included in THEROCK_AMDGPU_TARGETS) is not supported by composable kernel. Disabling composable kernel and it's use in MIOpen.
```

All other shards build CK successfully.

## Required fix

CK must be buildable for all architectures included in the distribution so that every shard produces a structurally identical `libMIOpen.so` (differing only in fat binary device code content, which the kpack ELF splitter handles).

The missing architectures are: **gfx1100, gfx1101, gfx1102, gfx1103** (the RDNA 3 family).

Alternatively, the filtering logic could be restructured to skip unsupported targets rather than disabling CK entirely — but this would still leave CK operations unavailable on those GPUs at runtime, so expanding support is the preferred path.
