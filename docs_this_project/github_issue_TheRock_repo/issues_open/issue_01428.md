### Suggestion Description

Thanks for the work on this project, it makes installing ROCm libraries and PyTorch faster and helps development work. I have two questions / feature suggestions to further improve the developer experience.

# Finding versions of ROCm libraries in a specific nightly release

If a bug or issue is discovered in a ROCm library, typically one needs to provide precise version information as part of any bug ticket. If the ROCm stack has been installed via TheRock releases, the best method I have found so far is to proceed as follows:
1) Track down the nightly release YYYYMMDD version installed.
2) Find the corresponding build in the [workflow view filtered to scheduled CI actions from main branch](https://github.com/ROCm/TheRock/actions/workflows/release_portable_linux_packages.yml?query=branch%3Amain+event%3Aschedule)
3) Find the commit to which the rocm-libraries submodule is pinned to in this build.

Is there a better way? Especially steps 2 and 3 take quite a bit of manual work. 

Suggestions:
- To ease step 2, perhaps it could be considered to name the runs with the YYYYMMDD tag to make locating them easier? Or make the release information searchable in another way? 
- Could the commit information from 3 be saved somewhere in the created artifacts, or is there another way to have this searchable? 

An ideal solution would probably give an automated way to look up all relevant libraries' version information to make filing tickets and fixing issues more streamlined.

# Recording accurate PyTorch version information

We regularly build specific commits of PyTorch, triton, and torchvision from source. I notice that after installing PyTorch with the help of [`external-builds/pytorch/build_prod_wheels.py`](external-builds/pytorch/build_prod_wheels.py), `pip list` produces a version string such as: `torch==2.9.0a0+rocmsdk20250826`

While this is helpful especially for torch wheel releases, I am wondering is there a way to set the version information for local/custom builds to show the exact commit of PyTorch instead of the generic `rocmsdkYYYYMMDD` string? This would again make it easier to track the specific commit/version used.
