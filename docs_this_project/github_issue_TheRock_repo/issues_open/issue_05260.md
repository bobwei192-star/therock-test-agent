**Problem**

  TheRock CI needs to dynamically configure:
  1. Build runners - Which runner labels to use for builds, with weighted distribution across pools
  (e.g., 90% Azure, 10% AWS)
  2. GPU family matrix - Which GPU families to build/test for different trigger types (presubmit,
  postsubmit, nightly), including test runner labels, target architectures, and build variants

  Currently this configuration is hardcoded in workflow files and Python scripts within TheRock. This
  creates problems:
  - Configuration changes require PRs to TheRock, rocm-libraries, rocm-systems
  - Difficult to update runner weights or add new GPU families quickly
  - No centralized view of CI infrastructure configuration

---

  **Approaches Considered**

  **Option 1: S3-hosted JSON config**

  - Store runner-config.json in an S3 bucket
  - Fetch at workflow runtime via curl or boto3

  Pros:
  - Fast updates (no git workflow)
  - Can be updated by automation/scripts easily

  Cons:
  - No version history or audit trail
  - Harder to review changes before deployment
  - Requires AWS credentials for updates
  - No PR-based review process

  **Option 2: GitHub repository**

  - Create dedicated repo ROCm/therock-ci-config with JSON config
  - Checkout during workflow setup step
  - Config changes go through normal PR process

  Pros:
  - Full git history for traceability
  - PR-based review for config changes
  - Public accessibility (no credentials needed to read)
  - Easy rollback via git revert
  - Can validate schema in CI before merge

  Cons:
  - Slightly slower to update (requires PR merge)
  - Additional repo to maintain

 **Option 3: Repository Variables/Secrets**

  - Store config as GitHub repository variables or environment variables
  - Access via ${{ vars.RUNNER_CONFIG }} in workflows

  Pros:
  - Native GitHub integration
  - No external dependencies
  - Instant updates (no checkout needed)

  Cons:
  - Does not work for forked repos
  - Size limits on variables
  - No version history
  - Awkward for complex nested JSON
  - Changes not auditable via git
 
**Option 4: Git submodule in TheRock**

  - Add therock-ci-config as a submodule
  - Pin to specific commit, update via bump PRs

  Pros:
  - Explicit versioning (pinned commit)
  - Clear dependency relationship
  - Changes require TheRock PR (deliberate)

  Cons:
  - Slower iteration (two PRs: config repo + bump)
  - Submodule complexity

---

  **Proposed Implementation**

  Use Option 2 with ROCm/therock-ci-config repository:

  1. Config structure (runner-config.json):
    - build_runners: Runner labels with weights per platform
    - gpu_families: Per-trigger-type GPU configurations (test runners, targets, variants)
  2. Workflow integration:
    - Checkout therock-ci-config at main during setup
    - Load config via ci_config_loader.py
    - Pass config through to configure_multi_arch_ci.py for matrix expansion
  3. Traceability:
    - Log config repo checkout SHA in workflow
    - Config changes tracked via git history

  This keeps infrastructure config separate from application code while maintaining full auditability.
