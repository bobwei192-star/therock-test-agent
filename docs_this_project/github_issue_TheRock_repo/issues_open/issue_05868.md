## Summary

[ROCm/rocm-systems#6664](https://github.com/ROCm/rocm-systems/pull/6664) removed `rocprofiler-sdk::rocprofiler-sdk-sqlite3` from `rocprofiler-sdk-output-library` to avoid linking `libsqlite3.so` directly into the rocpd Python extension. That fixed the mixed-SQLite runtime collision, but it also dropped the SQLite include path used when compiling rocprofiler-sdk SQL output sources that still include `<sqlite3.h>`.

The issue is hidden in TheRock CI because the [manylinux base image](https://github.com/ROCm/TheRock/blob/main/dockerfiles/build_manylinux_x86_64.Dockerfile#L12) already has a from-source SQLite header at `/usr/local/include/sqlite3.h`, and `/usr/local/include` is on GCC's built-in default include search path. Clean Ubuntu 24 environments do not have that fallback unless `libsqlite3-dev` is installed.

## Regression

Introduced by:

- [ROCm/rocm-systems#6664](https://github.com/ROCm/rocm-systems/pull/6664)
- Commit: `27674006c1`
- File: `projects/rocprofiler-sdk/source/lib/output/CMakeLists.txt`

The relevant change removed:

```cmake
rocprofiler-sdk::rocprofiler-sdk-sqlite3
```

from:

```cmake
target_link_libraries(rocprofiler-sdk-output-library ...)
```

Before that change, `rocprofiler-sdk-output-library` received the SQLite include directory through:

```text
find_package(SQLite3)
  -> SQLite::SQLite3
  -> rocprofiler-sdk::rocprofiler-sdk-sqlite3
  -> rocprofiler-sdk-output-library
```

After the link was removed, the bundled TheRock SQLite sysdep is still built and staged, but its include directory no longer reaches the `rocprofiler-sdk-output-library` compile.

## Why TheRock CI stays green

TheRock CI does not appear to be relying on bundled SQLite include propagation after [ROCm/rocm-systems#6664](https://github.com/ROCm/rocm-systems/pull/6664).

The manylinux build image already contains:

```text
/usr/local/include/sqlite3.h
```

and GCC searches `/usr/local/include` by default:

```text
#include <...> search starts here:
  /opt/rh/gcc-toolset-14/.../include
  /usr/local/include
  /opt/rh/gcc-toolset-14/root/usr/include
  /usr/include
```

So TheRock CI still finds `<sqlite3.h>` even when rocprofiler-sdk no longer propagates the bundled SQLite include directory to `rocprofiler-sdk-output-library`.

## Observed downstream failure

The hipDNN Ubuntu 24 Docker image is built through TheRock, but its clean Ubuntu base does not have `sqlite3.h` on a compiler default include path.

Affected downstream Dockerfile:

- [`projects/hipdnn/dockerfiles/Dockerfile.ubuntu24`](https://github.com/ROCm/rocm-libraries/blob/develop/projects/hipdnn/dockerfiles/Dockerfile.ubuntu24)

The rocprofiler-sdk build fails with:

```text
source/lib/output/sql/extract_data_type.hpp:27:10:
fatal error: 'sqlite3.h' file not found
```

Downstream workaround:

- [ROCm/rocm-libraries#8444](https://github.com/ROCm/rocm-libraries/pull/8444) adds `libsqlite3-dev` to the hipDNN Ubuntu 24 Dockerfile.

That workaround puts `sqlite3.h` in `/usr/include`, which makes the hipDNN Ubuntu image behave like TheRock's manylinux CI image: SQLite headers are available on a default compiler include path.

## Expected behavior

rocprofiler-sdk should not necessarily depend on incidental host/default-path SQLite headers when compiling sources that include `<sqlite3.h>`.

The rocpd Python extension should still avoid directly linking a separate `libsqlite3.so`, preserving the runtime collision fix from [ROCm/rocm-systems#6664](https://github.com/ROCm/rocm-systems/pull/6664).

## Suggested fix

`rocprofiler-sdk-output-library` is a **STATIC** library that compiles the `<sqlite3.h>` sources directly. After  [ROCm/rocm-systems#6664](https://github.com/ROCm/rocm-systems/pull/6664) it no longer receives the SQLite include directory, so it only compiles when `sqlite3.h` happens to be on the  compiler's default include path.

The target needs the SQLite include directories at compile time, but **must not link** `libsqlite3.so` (to preserve the runtime collision fix from #6664). Add the includes without linking:

```cmake
# rocprofiler-sdk-output-library compiles the sql/* sources that #include <sqlite3.h>,
# so it needs the SQLite3 include path at its own compile time. Do NOT link
# SQLite::SQLite3 here: the Python bindings target must not directly link libsqlite3.so
# (see #6664). Add include directories only.
target_include_directories(rocprofiler-sdk-output-library SYSTEM PRIVATE
    $<TARGET_PROPERTY:SQLite::SQLite3,INTERFACE_INCLUDE_DIRECTORIES>)
```
