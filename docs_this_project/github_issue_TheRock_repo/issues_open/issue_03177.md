## Background

Our CI (per-commit) and CD (dev/nightly/stable release) workflows diverged a while ago. They are building and testing different sets of jobs.

Coverage currently looks like (✨ indicates "new since the issue was filed"):

Job description | CI in rocm-libraries<br>and rocm-systems | CI in TheRock | Releases in TheRock and rockrel
-- | -- | -- | --

Test subset of ROCm artifacts | ✅ Yes | ✅ Yes ([see docs](https://github.com/ROCm/TheRock/blob/main/docs/development/ci_behavior_manipulation.md)) | ⚪ N/A

Notes:
* When coverage is only included in release workflows, commits that pass presubmit can break releases
* When coverage is only included in CI workflows, cherry-picks onto release branches carry additional risk
* When coverage is only included in TheRock, changes can land in rocm-libraries/rocm-systems/etc. that are only caught when we update those submodules in TheRock (~once a day, sometimes several days lag)
* We should also plan on more jobs being added over time, such as "build vLLM", "test vLLM", "build ROCm dockerfiles", etc.

## Design thinking

I'd like to aim for a setup where CI and CD workflows reuse the same building blocks. See for example how these workflows both build artifacts:
* https://github.com/ROCm/TheRock/blob/main/.github/workflows/build_portable_linux_artifacts.yml
* https://github.com/ROCm/TheRock/blob/main/.github/workflows/release_portable_linux_packages.yml

I'm focusing on our CI workflows first, as the release workflows already include many jobs. One key component I think we need is a central output directory in our cloud storage for each workflow run. We already sort of have that for some files:
* Sample artifacts: https://therock-ci-artifacts.s3.amazonaws.com/21415127692-linux/index-gfx94X-dcgpu.html
* Sample artifact build logs: https://therock-ci-artifacts.s3.amazonaws.com/21415127692-linux/logs/gfx94X-dcgpu/index.html

I want to build off of that and also upload Python packages and more files to that style of `21415127692-linux/` subdirectory. I think we should ultimately have a single subdirectory in just one cloud bucket that contains all outputs of a particular GitHub Actions workflow run, for both CI and release workflows. Then, for release workflows we can _also_ copy a subset of those outputs to the dedicated release cloud buckets (e.g. https://therock-nightly-python.s3.amazonaws.com/ for https://rocm.nightlies.amd.com/).

Once we have more jobs connected to the CI workflows we can continue to expand on https://github.com/ROCm/TheRock/blob/main/build_tools/github_actions/configure_ci.py where we decide for each workflow run:
* What subset of artifact groups (GPU families, sanitizer configurations, etc.) to build
* Which tests to run (components, types like "smoke" / "full")
* What jobs to run (see table above)
* Which versions of packages to build/test (e.g. Torch 2.9, Python 3.12)
