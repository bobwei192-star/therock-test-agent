The instructions at https://github.com/ROCm/TheRock/blob/main/RELEASES.md#installing-pytorch-python-packages are installing old torch versions on Linux instead of our latest nightly builds.

Looking at our index pages like
* https://d2awnip2yjpvqn.cloudfront.net/v2/gfx94X-dcgpu/torch/
* https://d2awnip2yjpvqn.cloudfront.net/v2/gfx110X-dgpu/torch/

We have wheels following two version schemes, with a switch on 2025-07-10:
```
torch-2.7.0a0+rocm7.0.0rc20250715-cp312-cp312-linux_x86_64.whl
torch-2.7.0a0+rocm7.0.0rc20250715-cp313-cp313-linux_x86_64.whl
torch-2.7.0a0+rocmsdk20250603-cp312-cp312-linux_x86_64.whl
torch-2.7.0a0+rocmsdk20250604-cp312-cp312-linux_x86_64.whl
```

Unfortunately, the older dates are considered to be newer versions due to how versions are parsed per https://packaging.pypa.io/en/stable/version.html:
```python
>>> from packaging.version import Version, parse
>>> v_rocmsdk = parse("2.7.0a0+rocmsdk20250709")
>>> v_rocm = parse("2.7.0a0+rocm7.0.0rc20250715")
>>> v_rocm < v_rocmsdk
True
```

So if a user tries `python -m pip install torch torchaudio torchvision --force-reinstall --index-url https://d2awnip2yjpvqn.cloudfront.net/v2/gfx94X-dcgpu`, they will get `torch==2.7.0a0+rocmsdk20250709` instead of the more recent `torch==2.7.0a0+rocm7.0.0rc20250715`.

### Possible solutions

1. Yank the old releases
2. Roll forward with new releases that have newer versions (`2.7.1`, `2.7.0a1`, `2.8.0`, etc.)
