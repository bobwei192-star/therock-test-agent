### Suggestion Description

Heya :wave: 

Following the large amount of news for TheRock in the past few months and the talk by JP Lehr at FOSDEM 2026, I wanted to give TheRock a spin to try to build some components in the context of [EasyBuild](https://easybuild.io/). For those unaware, EasyBuild is an software building and installation framework, similar to Spack, mainly targeted at HPC sites. Other projects, like [EESSI](https://github.com/EESSI), build on top of this framework to provide archictecture optimized builds via [CernVM-FS](https://cernvm.cern.ch/fs/).

Similar to other packaging options, EasyBuild needs to get its sources somehow.
I've specified these options:
```python
sources = [
    {
        'filename': SOURCE_TAR_XZ,
        'git_config': {
            'url': 'https://github.com/ROCm/',
            'repo_name': '%(name)s',
            'tag': '%(namelower)s-%(version)s',
            'clone_into': '%(name)s',
            'recursive': True,
            'keep_git_dir': False,
        },
    }
]
checksums = ['9a06ae164cd276b78172105faafc2c7ba7e4b9c1237dc999db541081b9ae8d22']
```

Which basically translates to something like:
```bash
$ git clone https://github.com/ROCm/TheRock.git
$ cd TheRock
$ git checkout therock-7.10
$ git submodule update --init --recursive
```

To get a consistent checksum, EasyBuild uses `keep_git_dir = False`, _removing_ the `.git` folder before packing a `.tar.xz` archive.

This causes an issue when building TheRock later on.

----

```

-DTHEROCK_ENABLE_COMPILER=ON \
-DTHEROCK_ENABLE_HIP_RUNTIME=ON \
[...]/TheRock/7.10/GCCcore-14.3.0/TheRock
```

The system dependencies listed in the [top level CMakeLists.txt](https://github.com/ROCm/TheRock/blob/505f68498dc1d2150b11e9d8d11af848d9f404ca/CMakeLists.txt#L361) are all provided via module dependencies, and should be picked up accordingly.

TheRock tries to write a manifest file, which includes information about the versions of the built components. Importantly, this [tries to find the commit used to build TheRock](https://github.com/ROCm/TheRock/blob/505f68498dc1d2150b11e9d8d11af848d9f404ca/build_tools/generate_therock_manifest.py#L168), using `git rev-parse`. Since we removed the `.git` folder, again, to get a consistent checksum, this command fails, hence aborting the whole build.

---

I fully understand that this workflow is a very good idea for development builds, where component versions regularly change and such. However, I feel like building releases should not rely on users cloning the repository with `.git`. This should also break building from the assets in the GitHub releases page, though I'm not sure if the submodules are even in there...

There are valid reasons to not use the provided build artifacts and instead build from source, e.g. providing optimized builds per architecture, having a very specific selection of accelerators and so on.

Therefore, it would be great if builds, at least from tags, would be supported even if generating a manifest with `.git` is not possible. Looking through the repository for a bit, I haven't found any mechanism to disable this particular part of the build.
