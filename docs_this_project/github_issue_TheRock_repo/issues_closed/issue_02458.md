### Problem Description

I initially encountered GPU driver timeout issues during YOLO inference on Windows. After further investigation, I found that the root cause is not YOLO itself, but that all Conv2D operations using the `GemmFwdRest` produce incorrect numerical results on recent Windows ROCm Pytorch nightly builds. The outputs contain NaNs or extremely large incorrect values, while switching to the naive convolution path produces correct results.

I observed that the same convolution configuration may select either `GemmFwdRest` or `ConvDirectNaiveConvFwd`  on different ROCm versions. When I force MIOpen to disable the direct convolution path using `MIOPEN_DEBUG_CONV_DIRECT=0`, MIOpen consistently selects `GemmFwdRest`, and the result is incorrect.

However, on latest linux nightly build(I installed it in v2-staging) and the `gfx120x-all 20251105` windows nightly build , using `GemmFwdRest` does not produce incorrect results.

It only requires generating random input and weights on the CPU, computing the reference result with F.conv2d on the CPU, and then running the same F.conv2d call on the GPU for comparison.
```
import torch
import torch.nn.functional as F

# Example tensor shapes
N, C_in, H, W = 1, 16, 64, 64
C_out = 32
kernel_size = 3
stride = 1
padding = 1
dtype = torch.float32
device = "cuda"

# Generate random input and weight on CPU
x_cpu = torch.randn(N, C_in, H, W, dtype=dtype)
w_cpu = torch.randn(C_out, C_in, kernel_size, kernel_size, dtype=dtype)

# CPU reference
out_cpu = F.conv2d(x_cpu, w_cpu, stride=stride, padding=padding)

# GPU computation
x_gpu = x_cpu.to(device)
w_gpu = w_cpu.to(device)
out_gpu = F.conv2d(x_gpu, w_gpu, stride=stride, padding=padding)

# Compare results
max_diff = (out_gpu.cpu() - out_cpu).abs().max()
print("max diff:", max_diff)
```
