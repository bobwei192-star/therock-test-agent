## Summary

The job fails during artifact extraction — no test is ever executed. `fetch_artifacts.py` uses a `ThreadPoolExecutor` to flatten multiple `.tar.zst` archives concurrently into the shared `build/` directory. Several archives all contain `hsa-runtime64.lib`. Two threads hit the same destination file simultaneously: one has the file open for writing while the other calls `os.unlink()`, which is forbidden on Windows when the file handle is still live.

## Error

```
File "B:\runner\_work\TheRock\TheRock\build_tools\_therock_utils\artifacts.py", line 217, in __call__
    os.unlink(dest_path)
PermissionError: [WinError 32] The process cannot access the file because it is being used by another process: 'build\\lib\\hsa-runtime64.lib'
##[error]Process completed with exit code 1.
```

## Root cause

`ArtifactPopulator.__call__()` (flatten mode) does:

```python
if dest_path.exists() and not dest_path.is_dir():
    os.unlink(dest_path)   # line 217 — race target
# ...
with open(dest_path, "wb") as out_file:
    out_file.write(...)
```

On Linux this is safe (open handles survive unlink). On Windows, `os.unlink` raises `PermissionError` (WinError 32) if any other thread has the file open. The PR itself (`rocm-systems` submodule bump only) did not introduce this bug — it is a latent race in the flattener.
