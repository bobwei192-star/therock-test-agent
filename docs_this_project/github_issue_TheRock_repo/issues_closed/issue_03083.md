### Suggestion Description

Hi ROCm-TheRock Team,

I would like to start by thanking the community and the developers for the ongoing efforts to bring ROCm to Windows. It is a significant milestone for many of us in The-Rock community.

I am writing to report a performance observation regarding the recent Windows Nightly builds for **gfx1100 (RDNA3)**. In my benchmarks, I noticed a substantial performance difference between the `dgpu` flavor and the `all` flavor.

### Benchmark Data:

*   **Environment:** 
    *   `gfx1100-dgpu==7.10.0a20251120-cp312-cp312-win_amd64`
    *   `gfx1100-all==7.10.0a20251121-cp312-cp312-win_amd64`
*   **Driver:** Verified on both **Adrenalin Edition 25.12.1** and **26.1.1**. (The performance gap persists across both driver versions). 
*   **Hardware:** gfx1100 (RX 7900 XTX)
*   **Data Type:** `torch.bfloat16`
*   **Shape:** `[B=1, H=16, L=16384, D=128]` (Loop=100)
*   **Test Script:** [benchmark_compare_100.py](https://github.com/user-attachments/files/24853849/benchmark_compare_100.py)

### Results:

| Build Flavor | Latency (ms) | Status |
| :--- | :--- | :--- |
| **gfx1100-dgpu** | **49.1 ms** | Optimized |
| **gfx1100-all** | **75.8 ms** | Performance Loss |

**Performance Gap:** 
The `all` build is approximately **54.4% slower** (in terms of latency) than the `dgpu` build, which translates to a **35.2% performance loss**.

### Discussion & Request:

While I am not entirely certain of the root cause, I suspect the disparity might be related to differences in integrated components (such as **pytorch_triton_rocm**) or specific kernel optimizations present in the `dgpu` build but missing in the `all` build.

As a developer, I prefer using the `all` build for its broader compatibility and convenience. However, the current performance loss makes it difficult to use as a primary environment for development and stress testing.

**Could you please look into aligning the performance of the `gfx1100-all` build with the `dgpu` version?** 

Ensuring that the `all` build is as performant as the specialized builds will allow developers like myself to:
1. Conduct more thorough and accurate testing.
2. Provide high-quality feedback and constructive suggestions to help improve the ROCm ecosystem on Windows.

I am more than happy to provide additional logs, run specific diagnostic scripts, or test experimental wheels to help identify the cause.

Thank you for your time and consideration!

Best regards.
