# #5175 - [Issue] Windows PyTorch `nightly` wheel build fails when HIP objects are compiled with clang++ using MSVC-style flags

## Summary

The `Release Windows PyTorch Wheels` workflow fails during PyTorch HIP compilation because the generated HIP compile commands invoke `clang++.exe` while passing MSVC-style options such as `/DWIN32`, `/D_WINDOWS`, `/O2`, `/Ob2`, and `/DNDEBUG`.

This causes the compiler to treat those flags as file paths and abort before the wheel build completes.

##  Environment

- Workflow: `Release Windows PyTorch Wheels`
- Workflow file: `.github/workflows/release_windows_pytorch_wheels.yml`
- Impacted Archs: `gfx103x, gfx906, gfx908, gfx101, gfx900`
- Pytorch Version: `nightly`

## Error signature

The first hard failures in the log are repeated HIP compile failures like:

```text
clang++: error: no such file or directory: '/DWIN32'
clang++: error: no such file or directory: '/D_WINDOWS'
clang++: error: no such file or directory: '/O2'
clang++: error: no such file or directory: '/Ob2'
clang++: error: no such file or directory: '/DNDEBUG'
```

Example failing targets include:

- `caffe2/CMakeFiles/torch_hip.dir/__/aten/src/ATen/hip/cub-RadixSortKeys.hip.obj`
- `caffe2/CMakeFiles/torch_hip.dir/__/aten/src/ATen/hip/Sleep.hip.obj`
- `caffe2/CMakeFiles/torch_hip.dir/__/aten/src/ATen/hip/cub-RadixSortPairs-f16-8.hip.obj`

The log shows the failing compiler binary is:

```text
C:\home\runner\_work\_tool\Python\3.14.5\x64\Lib\site-packages\_rocm_sdk_devel\lib\llvm\bin\clang++.exe
```
## Impact

This blocks the `Release Windows PyTorch Wheels` workflow from producing Windows PyTorch wheel artifacts.
