Follow-up from @ScottTodd's review on #5780.

Today the full PyTorch test dispatch (`dispatch_pytorch_wheels_full_test` in `multi_arch_build_portable_linux_pytorch_wheels.yml`) is scoped with hardcoded `if` conditions: `gfx94X-dcgpu` built and `python_version == '3.12'`.

Instead, teach `configure_pytorch_test_matrix.py` to be aware of the **test type (short vs full)** and generate matrix entries along that axis too. The build/test workflow would then select short/full per matrix entry rather than relying on hardcoded conditions.

This generalizes cleanly as full tests expand beyond gfx942 — e.g. Windows gfx1151 and opt-in Linux gfx950.

### Also fold in (from @ScottTodd's 2026-06-18 review)
The dispatch job also hardcodes the runner labels:
- `test_runs_on: linux-gfx942-1gpu-ossci-rocm`
- `test_runs_on_multi_gpu: linux-gfx942-8gpu-ossci-rocm`

These should be sourced dynamically from `configure_pytorch_test_matrix.py` (`test_runs_on: ${{ matrix.test_runs_on }}`, as already done for the short-test matrix). Otherwise, if runners migrate from the `linux-gfx942-*-ossci-rocm` labels to another provider, this code can silently go stale if we miss updating it.

Refs #5780, #5110.
