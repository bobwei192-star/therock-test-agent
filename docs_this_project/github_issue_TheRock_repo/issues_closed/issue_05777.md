Follow-up from #4499 (review feedback from @WBobby).

## Problem

The full PyTorch test dispatch (`dispatch_pytorch_wheels_full_test` in `.github/workflows/multi_arch_release_linux_pytorch_wheels.yml`) gates on the **aggregate** `needs.build_pytorch_wheels.result`. The build is a matrix over `python_version` × `pytorch_git_ref` (`release/2.9`, `release/2.10`, `release/2.11`, ..., `nightly`), but the dispatch job cannot tell which refs individually succeeded.

This forces a bad either/or:

- `needs.build_pytorch_wheels.result == 'success'` (current, after #4499) — **underreports**: a single failed leg (even an unrelated ref/python) skips the entire full-test dispatch for that night.
- `always() && result != 'cancelled'` — **misreports**: dispatch runs even when the leg we test failed, and the resolver picks up a stale wheel from the live index, reporting a misleading green.

Both are stuck on the same root cause: only aggregate status is visible.

## Proposed fix

- Surface **per-ref build status** (e.g. per-matrix-cell job outputs or a status artifact) so the dispatch can decide independently for each tested ref.
- Test the **exact version built this run**, consuming the `torch_version` / `package_index_url` build-job outputs added in #5661 instead of re-resolving via `pip install --dry-run` against the live index. This also removes the index-propagation-lag staleness window.
