### Suggestion Description

We need to have a tool to quickly able to identify the set of commit hashes of all sub-modules at the time of the build. This will be critical in releases to identify which commits are part of it.

This information is stored in:
https://github.com/ROCm/TheRock/blob/main/.gitmodules

One thing to think about:
- Data format for the manifest. Needs to be human parseable but would also like it to be machine parseable. We use the [repo manifest file](https://gerrit.googlesource.com/git-repo/+/master/docs/manifest-format.md) in other places but it probably doesn't make sense here. Suggest any format that stores both the commit hash of the Rock Repo + content of the .gitsubmodules file. 

Easy solution:
- When we produce a build, generate an additional artifact that stores this information as file. Even just uploading a file with the commit hash of the rock may be sufficient. Upload with our artifacts. Give it an easily discoverable name. Caveat - this will only work with forward looking builds
