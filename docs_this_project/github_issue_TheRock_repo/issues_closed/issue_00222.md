## Background

We have several release workflows right now, each driven by manual triggers:

Workflow file | Description
-- | --
[`portable_linux_package_matrix.yml`](https://github.com/ROCm/TheRock/blob/main/.github/workflows/portable_linux_package_matrix.yml) | Portable "dist archive", built under manylinux
[`publish_pytorch_dev_docker.yml`](https://github.com/ROCm/TheRock/blob/main/.github/workflows/publish_pytorch_dev_docker.yml) | Ubuntu 24.04 Dockerfile with PyTorch and ROCm for specific AMDGPU_TARGETS

This list will grow to include other package types (debs, rpms, wheels, dockerfiles, etc.), more targets (e.g. gfx942), and more platforms (e.g. Windows).

## Continuous releases

For "continuous delivery" (CD) of releases, we need these actions to run at some regular cadence, ideally without manual triggering, at least for nightly builds.

### Release channels

* Development: produced at every commit, or on-demand (e.g. on a pull request)
* Nightly: produced once a day, for some reasonable subset of configurations
* Stable: produced from a release branch and published to more official channels

## Requirements

We can start by looking at just nightly releases, then expand to also include dev, stable, and possibly other channels.

- [x] Nightly releases need some version identifier(s) so they do not overwrite one another. Some available options:

    * Commit hashes
    * Dates
    * Some other monotonically increasing identifier, like github build numbers

- [x] Version identifiers must be included in package file names
- [ ] Version identifiers should be stamped into binaries within the release too
- [x] Nightly releases could default to the latest passing build of TheRock, or the latest passing build across all sub-projects (ideally these converge, with https://github.com/ROCm/TheRock/issues/223)
- [x] Manually triggered releases should still be possible, given a branch to build from and any extra metadata needed
- [x] Releases must be uploaded to some index

    * We could start with https://github.com/ROCm/TheRock/releases and/or some of our S3 buckets for storage
    * We'll be directing users to these releases so some level of polish will be expected (release notes, install guides, known issues, level of testing performed, etc.)
- [ ] Releases can be untested, but we should strive to run at least smoketests before advertising them.
