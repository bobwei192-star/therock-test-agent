## Context

Azure runners are being deprecated. All CI workloads are migrating to AWS EKS-based ARC runners. As part of this migration, we should establish best security practices for AWS runners from the start — rather than carrying over patterns designed for the Azure environment.

## Problem

The prod Azure runners mount AWS credentials from the host via a Docker volume:
```
-v /runner/config:/home/awsconfig/
AWS_SHARED_CREDENTIALS_FILE=/home/awsconfig/credentials
```

This does not work on the NPI EKS-based ARC runners — container `options` Docker flags are ignored in k8s mode (ARC uses the k8s exec API, not Docker), and there is no `/runner/config` host path on EKS nodes. More importantly, static credential file mounts are not a secure or sustainable pattern for cloud-native workloads.

## Solution: IRSA (IAM Roles for Service Accounts)

IRSA uses the EKS OIDC provider to automatically inject temporary STS credentials into pods via the pod identity webhook. No credential files, no manual rotation, no secrets stored in GitHub.

This is **already implemented for prod and dev runners** in [TheRock-Infra](https://github.com/AMD-ROCm-Internal/TheRock-Infra) and serves as the model for NPI:

| Environment | File | IAM Role | Account |
|---|---|---|---|
| Prod | `projects/therock-runners/prod/irsa.tf` | `therock-runners-prod-runner-s3-external` | 324352301041 (AMD-therock-aws-prod) |
| Dev | `projects/therock-runners-dev-useast/dev/irsa.tf` | `therock-runners-dev-useast-runner-s3-external` | 317668459450 (customer-dev) |

Both follow the same pattern: OIDC trust policy scoped to the runner service account, S3 permissions for CI artifacts, SA annotated with `eks.amazonaws.com/role-arn`.

See also: `docs/irsa-runner-s3-external.md` in TheRock-Infra for the credential resolution flow, verification commands, and cross-account access setup.

## Required Changes for NPI

### Infrastructure (TheRock-Infra)
1. Create `projects/therock-runners-npi/irsa.tf` with an IAM role scoped to:
   - Service account: `aws-linux-scale-rocm-npi-gha-rs-kube-mode` (namespace `aws-linux-scale-rocm-npi`)
   - S3 buckets: `if neccessary`
2. Annotate the service account with the new role ARN:
   ```bash
   kubectl annotate serviceaccount -n aws-linux-scale-rocm-npi \
     aws-linux-scale-rocm-npi-gha-rs-kube-mode \
     eks.amazonaws.com/role-arn=arn:aws:iam::<account>:role/<role-name>
   ```
3. Remove `AWS_SHARED_CREDENTIALS_FILE` references
4. Remove `-v /runner/config:/home/awsconfig/` from `container.options`

AWS SDK/CLI will automatically pick up credentials from the injected env vars (`AWS_ROLE_ARN` + `AWS_WEB_IDENTITY_TOKEN_FILE`).
