Some changes on `pull_request` and `push` events don't need to run all workflows. We also want to enable "try bots" to run selected builds/tests and similar triggers for releases.

Concrete examples:

* A PR only modified markdown files so only run lint checks, no source builds or tests
* A PR only affects Windows so don't run Linux builds or tests
* A PR adds a new GPU family. Run those builds/tests but skip others
* I want to trigger a release build for a specific GPU family
* I want to trigger a release build for _all_ GPU families, even those that are known to be broken

## Possible solutions

The low tech solutions for configuring which workflows run are the `paths` and `paths-ignore` filters provided by GitHub Actions natively: https://docs.github.com/en/actions/writing-workflows/workflow-syntax-for-github-actions#onpushpull_requestpull_request_targetpathspaths-ignore . This is not very flexible and is incompatible with required status checks (https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/managing-protected-branches/about-protected-branches#require-status-checks-before-merging).

Another approach is to have a script control which jobs are enabled. I've used this approach on other projects before:

* https://github.com/iree-org/iree/blob/main/.github/workflows/setup.yml
* https://github.com/iree-org/iree/blob/main/build_tools/github_actions/configure_ci.py
* https://github.com/iree-org/iree/blob/main/build_tools/github_actions/configure_ci_test.py

I expect that as we add more target families to the build and test matrices that we'll need an even more sophisticated approach.

We'll probably end up with a set of heuristics like "If only `*.md` files are modified, skip builds". Whatever scripting/tooling we build should make it clear which jobs were selected and why.
