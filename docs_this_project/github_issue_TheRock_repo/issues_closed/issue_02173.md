## Overview

For issues like https://github.com/ROCm/TheRock/issues/2156, as well as for developers and users that want to install and test nightly releases, we should have these workflows summarize how to reproduce their environment.

* We have some documentation for ROCm tests at https://github.com/ROCm/TheRock/blob/main/docs/development/test_environment_reproduction.md for ideas (and we can improve those too).
* See also https://rocm.docs.amd.com/projects/install-on-linux/en/latest/install/3rd-party/pytorch-install.html#test-the-pytorch-installation

## Analysis

Here's a sample workflow run: https://github.com/ROCm/TheRock/actions/runs/19422651443/job/55568013840 . Note what it does:
1. Have a system with the GPU you want to test on (e.g. mi300, w7900)
1. Run in a container (`ghcr.io/rocm/no_rocm_image_ubuntu24_04@sha256:405945a40deaff9db90b9839c0f41d4cba4a383c1a7459b28627047bf6302a26`): https://github.com/ROCm/TheRock/blob/5afb2d02a5cb18f39816fbc36ba3c619a9c5c7d7/.github/workflows/test_pytorch_wheels.yml#L68-L76

    _We can provide a `docker run` command for this using either a pinned version or simply `latest`._

1. Checkout TheRock sources (_these should not be needed for many issues_)
1. Checkout pytorch sources
    ```bash
    python external-builds/pytorch/pytorch_torch_repo.py checkout \
      --gitrepo-origin https://github.com/ROCm/pytorch.git \
      --repo-hashtag release/2.8 \
      --no-hipify \
      --no-patch
    ```
    
    _This could be simplified to_
    ```bash
    git clone -b release/2.8 https://github.com/ROCm/pytorch.git
    ```

1. Create a venv and install packages

    ```bash
    VENV_DIR: /home/nod/actions-runner-7900/_work/TheRock/TheRock/.venv
    TORCH_VERSION: 2.8.0+rocm7.10.0a20251117

    python build_tools/setup_venv.py ${VENV_DIR} \
      --packages torch==${TORCH_VERSION} \
      --index-url https://rocm.nightlies.amd.com/v2-staging \
      --index-subdir gfx110X-dgpu \
      --activate-in-future-github-actions-steps
    ```

    _This could be simplified to_
    
    ```bash
    python3 -m venv .venv && source .venv/bin/activate
    python -m pip install --index-url=https://rocm.nightlies.amd.com/v2-staging/gfx110X-dgpu torch==2.8.0+rocm7.10.0a20251117
    ```

1. Run pytorch tests

    ```bash
    bash external-builds/pytorch/run_linux_pytorch_tests.sh
    ```

    Note that https://github.com/ROCm/TheRock/pull/1656 will change the filtering and test entry point, **but we should also have a self-contained `python test/run_test.py --include test_nn` style reproducer for individual test case failures. If tests only fail when run together with other tests (as _may_ be the case for https://github.com/ROCm/TheRock/issues/2118), we need to know that information too.

## Summary

I think we can produce steps like this:

```bash
# Start with a fresh container that does not contain ROCm or PyTorch
sudo docker run -it \
  --ipc host --group-add video --group-add render --device /dev/kfd --device /dev/dri \
  ghcr.io/rocm/no_rocm_image_ubuntu24_04@latest

# Install torch (and rocm) into a venv
python3 -m venv .venv && source .venv/bin/activate
python -m pip install --index-url=https://rocm.nightlies.amd.com/v2-staging/gfx110X-dgpu torch==2.8.0+rocm7.10.0a20251117

# Fetch pytorch test sources and run a test
git clone -b release/2.8 https://github.com/ROCm/pytorch.git
python pytorch/test/run_test.py --verbose --include test_nn
```

We can include that information in an easy to copy-paste format in workflow step summaries and then include it in issue reports for easier triage/reproduction/debugging/fixing.
