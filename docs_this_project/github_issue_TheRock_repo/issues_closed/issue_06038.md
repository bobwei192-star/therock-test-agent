# #6038 - install_rocm_from_artifacts.py / fetch_artifacts.py drop xnack-suffixed artifacts when only base arch is requested (breaks ASAN test runs)

### Summary
When `--amdgpu-targets gfx942` is passed, the artifact fetch matches only the literal `gfx942` and silently drops xnack-suffixed variants such as `gfx942:xnack+`. ASAN builds emit their test artifacts with the `:xnack+` suffix, so those test artifacts are never downloaded, and downstream test jobs fail with missing artifacts.

### Reproduction
```
python ./build_tools/install_rocm_from_artifacts.py --amdgpu-targets gfx942
```
pulls only base-arch artifacts (`blas_lib_gfx942`, `fft_lib_gfx942`, ...), whereas
```
python ./build_tools/install_rocm_from_artifacts.py --amdgpu-targets gfx942,gfx942:xnack+
```
additionally pulls `blas_lib_gfx942:xnack+`, `blas_test_gfx942:xnack+`, `rccl_test_gfx942:xnack+`, etc.

Example artifact that is **not** fetched with base arch alone:
`https://therock-ci-artifacts.s3.amazonaws.com/27891627402-linux/rccl_test_gfx942%3Axnack%2B.tar.zst`

### Root cause
- `build_tools/fetch_artifacts.py` builds `targets_to_match = {"generic"} ∪ amdgpu_targets` and includes an artifact only if `ArtifactName.target_family in targets_to_match` (exact equality).
- `build_tools/_therock_utils/artifacts.py` (`ArtifactName.from_filename`) parses `rccl_test_gfx942:xnack+.tar.zst` into `target_family = "gfx942:xnack+"`.
- `"gfx942:xnack+" != "gfx942"`, so the artifact is excluded. The xnack suffix is never normalized or treated as a variant of the base arch.

### Context
- Nightly builds currently do **not** produce separate xnack artifacts (e.g. https://therock-nightly-artifacts.s3.amazonaws.com/27922350966-linux/index.html ), so this is invisible on the nightly path.
- **ASAN** CI builds **do** emit `:xnack+`-suffixed artifacts (e.g. `rccl_test_gfx950:xnack+.tar.zst` at https://therock-ci-artifacts.s3.amazonaws.com/27891627402-linux/index.html ), and this has not been plumbed through the build/CI/CD scripts.
- Believed to be the root cause of ROCM-25535.

### Proposed direction (to discuss)
Either (a) match base arch inclusively so a requested `gfx942` also pulls its `gfx942:xnack+` variants, or (b) have the test/CI workflow explicitly request the xnack variants. Preference seems to be pulling all variants of a requested base arch. Noted separately that this script is overly static and slated for replacement.
