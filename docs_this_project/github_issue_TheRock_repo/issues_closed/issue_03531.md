## Summary

Validated kpack-based multi-arch artifacts on consumer GPUs (gfx1201, gfx1100) using CI run [#22208507839](https://github.com/ROCm/TheRock/actions/runs/22208507839) from the `multi_arch/integration-kpack` branch (PRs #3527, #3528, #3529).

### Failing — Libraries with external kernel databases

| Library | Test | Result | Notes |
|---------|------|--------|-------|
| rocblas | axpy_ex | FAILED | `rocblas_status_internal_error` on both GPUs |
| rocsparse | csr2dense | FAILED | `hipErrorInvalidImage` on both GPUs |

**Root cause:** These libraries depend on Tensile-generated kernels that are loaded at runtime from an external database, not compiled into the library's ELF as standard HIP code objects. The kpack splitter processes only HIP code objects embedded in ELF `.hipk` sections; Tensile's `.co` files are loaded through a separate runtime path that kpack does not intercept.

Debug trace showing the failure (rocsparse → rocblas dependency):
```
kpack: parsed HIPK metadata: kernel_name='math-libs/BLAS/rocSPARSE/stage/lib/librocsparse.so.1.0', co_index=5
kpack: expanded pattern: ../.kpack/blas_lib_@GFXARCH@.kpack -> .../blas_lib_gfx1201.kpack
kpack: opened and cached archive: .../blas_lib_gfx1201.kpack
kpack:   archive blas_lib_gfx1201.kpack has architecture
kpack:   kernel not found in this archive    <-- Tensile kernel not in archive
kpack: no matching architecture found in any archive
```

The archive exists and has the correct architecture, but the specific code object index (a Tensile-generated kernel) was never packed into it.

## Remaining Work

### 1. gfx942 fetch matching (blocked — #3444)

Datacenter GPUs use feature-flagged ISA strings (e.g., `gfx942:sramecc+:xnack-`). The fetch filter in `fetch_artifacts.py` does exact set membership, so `"gfx942"` in the oracle doesn't match `"gfx942:sramecc+:xnack-"` in S3.

**Fix implemented locally** (not yet committed): A prefix-aware `target_matches()` utility in `build_tools/_therock_utils/gpu_target_match.py` with matching rules:
- Filter entries containing `:` → verbatim match only
- Filter entries without `:` → prefix match (bare arch matches itself + any `:`-suffixed variant)

This handles both the common case (`gfx942` matches all feature variants) and future explicit filtering (`gfx942:sramecc+:xnack+` for ASAN-only). Unit tests at `build_tools/tests/gpu_target_match_test.py` (18 tests, all passing).

Files changed:
- `build_tools/_therock_utils/gpu_target_match.py` (new)
- `build_tools/tests/gpu_target_match_test.py` (new)
- `build_tools/fetch_artifacts.py` line 94: `an.target_family in targets_to_match` → `target_matches(an.target_family, targets_to_match)`
- `build_tools/github_actions/amdgpu_family_matrix.py` line 63: updated comment

### 2. Tensile external kernel database support

rocblas and hipblaslt use Tensile to generate optimized GEMM kernels as external `.co` (code object) files loaded at runtime via `hipModuleLoad`. These are NOT embedded in the library ELF and therefore not captured by the kpack splitter.

**Affected libraries:** rocblas, hipblaslt, and any library that links them (rocsparse, rocsolver, hipDNN plugins, etc.).

**Options to investigate:**
- Intercept Tensile's kernel loading in the kpack runtime and redirect to `.kpack` archives
- Include Tensile's `.co` database files in the split artifact alongside `.kpack` archives (requires changes to artifact_splitter to recognize and handle Tensile's file layout)
- Hybrid approach: kpack for HIP fatbin, pass-through for Tensile `.co` files

### 3. Broader test coverage on consumer GPUs

Libraries confirmed working (rocrand, hipfft) should be added to a consumer-GPU smoke test matrix. Additional libraries to validate once Tensile support is resolved:
- rocsparse (blocked on Tensile)
- rocsolver (blocked on Tensile + needs correct test invocation)
- rccl (needs multi-GPU test setup)
- MIOpen (kpack exists, needs test)
- rocWMMA (kpack exists, needs test)
