Our current build/test/release support matrix for PyTorch covers 2.7 for Linux and "main"/"nightly" for both Linux and Windows: https://github.com/ROCm/TheRock/tree/main/external-builds/pytorch#supported-pytorch-versions . We should generalize this to [latest, N, N -1] where N is the current (or soon to be upcoming) stable release version. Concretely we should add 2.8 and 2.9 now.

* Linux matrix is defined here: https://github.com/ROCm/TheRock/blob/472254848e7a6499547c39ae11294b1ddb7ee7d2/.github/workflows/release_portable_linux_pytorch_wheels.yml#L82-L92

* Windows does not have multiple versions in the matrix yet: https://github.com/ROCm/TheRock/blob/472254848e7a6499547c39ae11294b1ddb7ee7d2/.github/workflows/release_windows_pytorch_wheels.yml#L82-L86

We may need to land commits/patches onto our release branches at https://github.com/ROCm/pytorch for build or test failures. We should _not_ plan on backporting _features_ to release branches (for example, aotriton support for Windows landed after the 2.9 branch cut).

TBD how much of the matrix we should run nightly or eventually on every commit. These build and test jobs take a considerable time to run across Python 3.11, 3.12, and 3.13 and we don't yet have a cache configured for the builds. We'll also want to add Python 3.14 soon. We could decide to run just "latest" and/or "latest stable" on individual commits (or nightly jobs) and then run older release builds on TheRock's release branches (perhaps in a fork).
