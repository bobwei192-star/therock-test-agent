### Problem Description

On gfx1151 (Strix Halo, RDNA 3.5), the MIOpen Winograd Sp3 assembly kernels for gfx11 cause a complete hard system lockup (power cycle required) during MIOpen's solver benchmark sweep when no find-db cache exists. The lockup is a compute-unit hang — **no VM fault is generated** even with `amdgpu.vm_fault_stop=1`, `amdgpu.vm_debug=1`, and `ignore_loglevel` active. Kernel logs captured via netconsole to a remote machine confirm zero amdgpu fault messages.

**Two independent crashes captured on different shapes, both hitting the same solver family:**

**Crash 1** (full MuseTalk inference pipeline, empty cache):
- Solver: `miopenSp3AsmConv_v30_3_1_gfx11_fp32_f3x2_stride2`
- Shape: N=8, C=1280, H=8, W=8, K=1280, R=3, S=3, pad=1, stride=2, groups=1
- Context: GenericSearch benchmark, iterating `n_groups` values (~iteration 21–23 of 40)

**Crash 2** (UNet2DConditionModel batch=8 forward pass, empty cache):
- Solver: `miopenSp3AsmConv_v30_3_1_gfx11_fp32_f3x2_stride1`
- Shape: N=8, C=780, H=20, W=20, K=280, R=3, S=3, pad=1, stride=1, groups=1
- Context: EvaluateInvokers benchmark for `ConvBinWinogradRxSf3x2`, after `GemmFwdRest` completed successfully for a prior shape

**Assembly binaries involved:**
- `Conv_Winograd_v30_3_1_fp32_f3x2_stride1.s.o`
- `Conv_Winograd_v30_3_1_fp32_f3x2_stride2.s.o`
- Both compiled with: `-Wa,-defsym,ROCM_METADATA_VERSION=5 -Wa,-defsym,FORCE_CACHE_BYPASS_ON_STORE=0 -mcumode -mwavefrontsize64 -mcpu=gfx1151`

The crash is **not shape-specific** — it is solver-family-specific. It requires cumulative back-to-back Winograd Sp3 dispatches across multiple shapes during the benchmark sweep, suggesting a resource leak, missing pipeline drain, or gfx1151-specific hardware errata that the assembly does not account for.

### Steps to Reproduce

**Hardware:** AMD Ryzen AI MAX+ 395 (gfx1151 / Strix Halo / RDNA 3.5), 128 GB unified memory
**OS:** Ubuntu 26.04 LTS, kernel 7.0
**ROCm:** 7.2.2 (system) / torch 2.10.0+rocm7.13.0a20260513 (TheRock nightly)

**Minimal reproducer — UNet batch=8 forward pass with empty MIOpen cache:**

1. Install the 0513 TheRock nightly for gfx1151:
```bash
pip install --index-url https://rocm.nightlies.amd.com/v2/gfx1151/ torch torchaudio torchvision
pip install diffusers
```

2. Obtain MuseTalk v1.5 UNet model files (freely available):
   - Config: `models/musetalkV15/musetalk.json`
   - Weights: `models/musetalkV15/unet.pth`
   - Repository: https://github.com/TMElyralab/MuseTalk

3. Clear MIOpen cache to force GenericSearch:
```bash
rm -rf ~/.cache/miopen/*
```

4. Save and run this script:
```python
#!/usr/bin/env python3
"""Minimal reproducer: gfx1151 Winograd Sp3 CU hang during MIOpen benchmark.
Requires: MuseTalk v1.5 UNet files, torch 2.10.0+rocm7.13.0a20260513, diffusers.
Crashes during MIOpen GenericSearch after cumulative Winograd Sp3 dispatches.
Does NOT crash at batch=1 — only batch=8 (different conv shapes trigger longer benchmark)."""
import sys, os, json, torch
sys.path.insert(0, os.path.expanduser("~/MuseTalk"))

from diffusers import UNet2DConditionModel

config_path = os.path.expanduser("~/MuseTalk/models/musetalkV15/musetalk.json")
weight_path = os.path.expanduser("~/MuseTalk/models/musetalkV15/unet.pth")

with open(config_path) as f:
    config = {k: v for k, v in json.load(f).items() if not k.startswith("_")}

model = UNet2DConditionModel(**config)
state_dict = torch.load(weight_path, map_location="cpu", weights_only=False)
model.load_state_dict(state_dict)
model = model.cuda().eval()

B = 8  # batch=1 survives; batch=8 crashes
C = config["in_channels"]
S = config.get("sample_size", 64)
cross_dim = config.get("cross_attention_dim", 768)

x = torch.randn(B, C, S, S, device="cuda")
t = torch.tensor([1.0] * B, device="cuda")
enc = torch.randn(B, 1, cross_dim, device="cuda")

print(f"Batch={B}, input={tuple(x.shape)}")
print("MIOpen cache empty — GenericSearch will benchmark Winograd Sp3 for every layer.")
print("System will hard-lock during the Winograd Sp3 gfx11 benchmark sweep.")
sys.stdout.flush()
os.sync()

with torch.no_grad():
    out = model(x, t, encoder_hidden_states=enc).sample

torch.cuda.synchronize()
print(f"SURVIVED — output: {tuple(out.shape)}")
```

5. Run with gfx1151 environment:
```bash
export HSA_OVERRIDE_GFX_VERSION=11.5.1
export HSA_ENABLE_SDMA=0
export PYTORCH_HIP_ALLOC_CONF="backend:native,expandable_segments:True"
python reproducer.py
```

**Expected:** Forward pass completes.
**Actual:** System hard-locks during MIOpen's Winograd Sp3 benchmark, requiring power cycle.

### Additional Information

**Conclusions from elimination:**
1. The crash is in the **Winograd Sp3 gfx11 assembly family**, not the CK grouped-conv library (which was the original theory in #5259).
2. The crash is **not shape-specific** — two different shapes, two different stride variants, same solver family.
3. The crash requires **cumulative dispatches** — the benchmark sweep for a UNet at batch=8 produces enough back-to-back Winograd Sp3 kernel launches across many shapes to trigger the hang. A single shape or batch=1 (fewer/different shapes) does not.
4. MIOpenDriver's clean HIP context does not trigger it even with the same shape and exhaustive search — the crash requires PyTorch's in-process HIP context with accumulated state from prior dispatches.
5. The crash is a **CU hang, not a page fault** — `amdgpu.vm_fault_stop=1`, `amdgpu.vm_debug=1`, and `ignore_loglevel` produce no fault messages on the kernel log (verified via netconsole to remote machine). This suggests the assembly kernel enters an infinite loop, exhausts a hardware resource, or triggers a gfx1151-specific errata condition.

**Capture methodology:**
Diagnostics captured via `netconsole` (kernel log over UDP to a remote machine, surviving the lockup) and `MIOPEN_ENABLE_LOGGING=1 MIOPEN_ENABLE_LOGGING_CMD=1 MIOPEN_LOG_LEVEL=6 AMD_LOG_LEVEL=4` (MIOpen + HIP runtime logging streamed over UDP). Full MIOpen logs for both crashes available on request.
