Hello TheRock development team,

First of all, thank you for your incredible work on providing ROCm packages for Windows. It's a huge help for the community and allows users with the latest AMD hardware to get started with AI.

I am writing to report a repeatable crash that I am experiencing on a new AMD Strix Halo APU.

Environment
* OS: Windows 11
* GPU: AMD Strix Halo APU (Radeon 8060S / GFX1151)
* Packages: Pre-built packages from TheRock's GitHub releases.
* PyTorch Version: 2.7.0a0+git3f903c3
* ROCm Version: (6, 5) (as reported by PyTorch)
* Application: ComfyUI

Problem Description
When running a workflow in ComfyUI that involves a video-capable VAE (e.g., WanVAE), the process fails with an miopenStatusUnknownError.

This appears to be triggered specifically when a 3D convolution (conv3d) operation is called by the VAE, particularly when using fp16 precision (by launching ComfyUI with the --force-fp16 flag). Standard models using only 2D convolutions work correctly.

Minimal Reproducible Example (MRE)
While I don't have the exact parameters of the failing convolution, a minimal script to reproduce this type of error would look like this. The key is using .cuda().half() to force the operation onto the GPU with fp16 precision.

Python

```python
import torch

# Check if ROCm device is available
if not torch.cuda.is_available():
    print("ROCm device not found.")
    exit()

print(f"Device: {torch.cuda.get_device_name(0)}")

# Example parameters for a 3D convolution
# These may need to be adjusted to match the exact failing case
try:
    conv_layer = torch.nn.Conv3d(
        in_channels=320,
        out_channels=320,
        kernel_size=3,
        padding=1
    ).cuda().half()

    input_tensor = torch.randn(
        1, 320, 16, 32, 32
    ).cuda().half()

    print("Attempting to perform 3D convolution with fp16...")
    output = conv_layer(input_tensor)
    print("Success! The convolution worked.")

except Exception as e:
    print("\n--- ERROR ---")
    print(f"Failed as expected: {e}")
```

Analysis
My hypothesis is that the MIOpen library included in the pre-built package is missing an optimized kernel/algorithm for this specific conv3d configuration on the new gfx1151 architecture, especially for the fp16 data type. The problem does not occur when using fp32.
