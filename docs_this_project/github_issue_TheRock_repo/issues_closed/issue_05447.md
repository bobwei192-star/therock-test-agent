### Problem Description

After a pip multi-arch install, no mechanism registers the ROCm library directories with
  the system dynamic linker. Users must manually export `LD_LIBRARY_PATH` before any ROCm
  application can load `libhiprand.so`, `librocsparse.so`, and other libraries. Without it,
  any ROCm binary fails immediately:

  error while loading shared libraries: libhiprand.so.1: cannot open shared object file: No such file or directory

  The required workaround is:

  ```bash
  SITE=.venv/lib/python3.12/site-packages
  export LD_LIBRARY_PATH=${SITE}/_rocm_sdk_core/lib:${SITE}/_rocm_sdk_libraries/lib:${SITE}/_rocm_sdk_devel/lib
```
  Note that _rocm_sdk_libraries/lib must be included specifically, not just
  _rocm_sdk_devel/lib. On the multi-arch layout, _rocm_sdk_devel/lib contains stub
  libraries (~41 KB each) whose GPU kernels are externalized to .kpack/ archives stored
  under _rocm_sdk_libraries/.kpack/. If the stub is loaded without _rocm_sdk_libraries/lib
  on the path, kpack searches _rocm_sdk_devel/.kpack/ — a directory that does not exist —
  and every GPU kernel call fails silently.

  The single-arch install has the same gap but is less exposed: some binaries carry a
  build-time RUNPATH into _rocm_sdk_devel/lib/, which on single-arch contains full
  (non-stub) libraries with device code compiled in.

  The _rocm_sdk_core package already ships etc/ld.so.conf.d/10-rocm-opencl.conf with a
  placeholder /opt/rocm/lib path — the ldconfig mechanism exists but is not wired to the
  actual pip install paths or triggered post-install.

### (Optional for Linux users) Output of /opt/rocm/bin/rocminfo --support

Install ROCm (multi-arch pip)
```bash
  python3 -m venv .venv
  source .venv/bin/activate
  pip install --index-url https://rocm.nightlies.amd.com/whl-multi-arch/ "rocm[libraries,device-all]"
  rocm-sdk init
```
  Build and run any ROCm example without setting LD_LIBRARY_PATH
```bash
  SITE=.venv/lib/python3.12/site-packages
  git clone https://github.com/ROCm/rocm-examples.git
  cd rocm-examples
  cmake -B build \
    -DROCM_PATH=${SITE}/_rocm_sdk_devel \
    -DROCM_EXAMPLES_ENABLE_ROCDECODE=OFF \
    -Wno-dev
  cmake --build build -j$(nproc)
  cd build
  ctest -j1 --output-on-failure
```
  Most tests fail with the shared library error shown above.

  Proposed fix

  Add a post-install hook (pip entry_points console script or post_install step) that
  runs:
```bash
  ldconfig /path/to/_rocm_sdk_core/lib /path/to/_rocm_sdk_libraries/lib
```
  or installs a file into /etc/ld.so.conf.d/ pointing at those directories using the
  actual install-time paths (not a hardcoded /opt/rocm/lib).

  Alternatively, provide a rocm-sdk activate shell script analogous to venv/bin/activate
  that exports the correct LD_LIBRARY_PATH for the detected install layout.
