## Analysis of MIOpen's installed content

MIOpen installs ~480MB under `ml-libs/MIOpen/stage/`. Here's how the content breaks down:

### libMIOpen.so.1.0 (~460MB) — architecture-GENERIC

The shared library is architecture-generic. It contains **no** `.hip_fatbin` ELF sections and no compiled device code. The 435MB `.rodata` section consists of HIP/OpenCL kernel **source code** inlined as hex-encoded byte arrays via MIOpen's `addkernels` build tool (`src/CMakeLists.txt` line 824). At runtime, MIOpen JIT-compiles these sources for the target GPU via COMGR/HIPRTC.

Key build flags confirming this:
- `MIOPEN_EMBED_DB` = empty (databases not embedded in binary)
- `MIOPEN_EMBED_BINCACHE` = OFF (binary caches not embedded)
- No `.hip_fatbin` ELF sections (`readelf -S` confirms)

### Composable Kernel (CK) — separate artifact

When `MIOPEN_USE_COMPOSABLEKERNEL=ON` (enabled for supported GPU families), MIOpen dynamically links `composable_kernel::device_conv_operations` (a shared library from the separate `composable-kernel` artifact). The CK device code (`.hip_fatbin` sections) lives in CK's own `.so` files, **not** in libMIOpen.so. CK already has its own topology entry (`[artifacts.composable-kernel]`, `type = "target-specific"`).

### share/miopen/db/ (~17MB) — architecture-SPECIFIC

61 tuning model files with explicit `gfx<ARCH>` filename prefixes:

| Architecture | Files | Example |
|---|---|---|
| gfx908 | 5 | `gfx908.tn.model`, `gfx908_ConvAsm1x1U_decoder.ktn.model` |
| gfx90a | 8 | `gfx90a.tn.model`, `gfx90a_ConvHipIgemmGroupFwdXdlops_decoder.ktn.model` |
| gfx942 | 25 | `gfx942.tn.model`, `gfx942_3d.tn.model`, `gfx942_ConvHipImplicitGemm3DGroup*.tn.model` |
| gfx950 | 23 | same pattern as gfx942 |

Extensions: `.tn.model`, `.ktn.model`. All files start with `gfx<ARCH>_` or `gfx<ARCH>.`. No generic (architecture-neutral) files exist in this directory.

Which architectures get installed is controlled by `-DMIOPEN_INSTALL_GPU_DATABASES="${THEROCK_AMDGPU_TARGETS}"`, but MIOpen only ships model files for architectures where training data exists (currently gfx908, gfx90a, gfx942, gfx950).

### Other installed content

- `bin/MIOpenDriver` — host executable (architecture-generic)
- `bin/miopen_gtest*` — test binaries (architecture-generic)
- Headers, cmake config, docs — architecture-generic

## Splitting strategy

The only architecture-specific content in the `miopen` artifact is `share/miopen/db/`. A `MIOpenHandler` database handler will move these files to per-architecture split artifacts.

After splitting, the generic artifact will contain everything except the db files (libMIOpen.so, headers, cmake, etc.). Per-architecture artifacts will contain just `share/miopen/db/gfx<ARCH>*` files.

### 1. Add `MIOpenHandler` to `database_handlers.py`

**File:** `base/rocm-kpack/python/rocm_kpack/database_handlers.py`

Follow the existing pattern (RocBLASHandler, HipBLASLtHandler, AotritonHandler):

```python
class MIOpenHandler(DatabaseHandler):
    """Handler for MIOpen performance database/model files."""

    def name(self) -> str:
        return "miopen"

    def detect(self, path: Path, prefix_root: Path) -> Optional[str]:
        try:
            rel_path = path.relative_to(prefix_root)
            path_str = rel_path.as_posix()

            if "miopen/db" not in path_str:
                return None

            if not path.name.endswith(".model"):
                return None

            match = _GFX_ARCH_PATTERN.search(path.name)
            if match:
                return f"gfx{match.group(1)}"

            return None
        except (ValueError, AttributeError):
            return None
```

Register in `AVAILABLE_HANDLERS` dict.
