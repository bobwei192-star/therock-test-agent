## Summary

The `Build PyTorch Wheels` step fails on all Windows nightly wheel builds (observed starting 2026-05-23) with a compile error in `torch/csrc/autograd/profiler_kineto.cpp`.

## Environment

- **Workflow:** `release_windows_pytorch_wheels.yml`
- **Platform:** `Windows`
- **Effected GPU arch:**  `all the supported archs`
- **Pytorch Version:** `nightly`

## Error

```
B:/src/torch/torch/csrc/autograd/profiler_kineto.cpp(1312,27): error: no member named 'activities' in 'torch::profiler::impl::kineto::DummyTraceBuffer'
 1312 |     return trace_->get()->activities();
      |            ~~~~~~~~~~~~~~~^
5 warnings and 1 error generated.
ninja: build stopped: subcommand failed.
```

## Root Cause

Upstream PyTorch PR [#184273](https://github.com/pytorch/pytorch/pull/184273) ("Expose ITraceActivity to Python") added `ProfilerResult::traceActivities()` — which calls `trace_->get()->activities()` — without wrapping it in `#ifdef USE_KINETO`. TheRock Windows builds set `USE_KINETO=OFF` ([`build_prod_wheels.py` L475](https://github.com/ROCm/TheRock/blob/main/external-builds/pytorch/build_prod_wheels.py#L475)), so the compiler resolves `trace_` to `DummyTraceBuffer` (the no-Kineto stub), which has no `activities()` member.

The upstream fix ([pytorch/pytorch commit 4819caaa](https://github.com/pytorch/pytorch/commit/4819caaa0d), PR #184916, merged 2026-05-22) guards the function behind `#ifdef USE_KINETO`, but the nightly branch snapshot pulled by TheRock CI still contained the un-guarded version at time of failure.
