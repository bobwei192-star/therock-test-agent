New release workflows using multi-arch should upload tarballs to e.g. https://rocm.nightlies.amd.com/tarball/.

Existing release pipelines build `therock-dist` then run `tar cfz` and upload:
* https://github.com/ROCm/TheRock/blob/5250110dbb4e9fbe3e2062f6145734785ff0ea83/.github/workflows/release_portable_linux_packages.yml#L179
* https://github.com/ROCm/TheRock/blob/5250110dbb4e9fbe3e2062f6145734785ff0ea83/.github/workflows/release_portable_linux_packages.yml#L240-L246
* https://github.com/ROCm/TheRock/blob/5250110dbb4e9fbe3e2062f6145734785ff0ea83/.github/workflows/release_portable_linux_packages.yml#L307-L310

I'm planning on adding dedicated scripts and workflows that will repackage artifacts into tarballs, similar to what we do for python packages. Details:
* Without kpack split, produce one tarball per GPU family
* With kpack split, produce one tarball with ALL GPU families per platform (and optionally also include the per-GPU tarballs)
* Later we may want to produce tarballs that include test artifacts and tarballs that don't

We can either include this step just for releases or also include for CI runs (at potentially 10GB+ of extra storage used per run, due to how inefficiency these files are compressed and how they overlap). Building JAX packages on CI will currently need tarballs, as JAX packages do not build against the ROCm Python packages.
