### Problem Description

## Summary

On Windows ROCm with an RDNA 4 / gfx120X GPU (RX 9070 / 9070 XT / Radeon 890M), calling
`torch.ops.aten._grouped_mm` with a batched (3-D) weight tensor crashes the process with a
`0xC0000005` access violation (null pointer dereference inside the HIP kernel).

**This is fixed in the ROCm 7.13 wheel (`torch==2.11.0+rocm7.13.0`).
If you are on ROCm 7.12, please update — see the install instructions below.**

---

## Affected versions

| torch wheel | HIP version (`torch.version.hip`) | Status |
|---|---|---|
| `torch==2.10.0+rocm7.12.0` | `7.2.0` | ❌ Crashes with 0xC0000005 |
| `torch==2.11.0+rocm7.13.0` | `7.13.99004` | ✅ Fixed |

Tested on: Windows 11, RX 9070 XT (gfx1201), ROCm Windows driver.

---

## Steps to reproduce (ROCm 7.12 only)

```python
import torch

# Verify you are on the affected version
print(torch.__version__)          # torch==2.10.0+rocm7.12.0
print(torch.version.hip)          # 7.2.0

a = torch.randn(4, 8,    device="cuda", dtype=torch.float16)
b = torch.randn(2, 8, 4, device="cuda", dtype=torch.float16)   # 3-D batch

# Process exits with 0xC0000005 — no exception is raised, the process just dies
torch.ops.aten._grouped_mm(a, b)
```

The crash also occurs during training via PyTorch's JitDecomp autograd decomposition
system (not `torch.compile`) — the training process silently exits mid-run.

---

## Root cause

The HIP kernel registered for `aten::_grouped_mm` on gfx120X in ROCm 7.12 is a null
function pointer. Any dispatch to that kernel path dereferences null and kills the process.
This affects both the 3-D batched variant and the grouped variant with `offs`
(MoE-style variable-size batches).

---

## Fix / How to update

Install the ROCm 7.13 wheel from AMD's Windows package index:

```
pip install torch==2.11.0+rocm7.13.0 torchvision torchaudio --index-url https://repo.radeon.com/rocm/manylinux/rocm-rel-7.13/gfx120X-all/
```

> **Note:** AMD uses a dual versioning scheme. The wheel filename says `rocm7.13.0` but
> `torch.version.hip` will report `7.13.99004` — this is expected and both strings refer
> to the same release.

---

## Verification

After updating, the following script should complete without crashing:

```python
import torch
print(torch.__version__)   # torch==2.11.0+rocm7.13.0
print(torch.version.hip)   # 7.13.99004

a    = torch.randn(4, 8,    device="cuda", dtype=torch.float16)
b    = torch.randn(2, 8, 4, device="cuda", dtype=torch.float16)
offs = torch.tensor([2, 4], device="cuda", dtype=torch.int32)

# 3-D batched variant
result = torch.ops.aten._grouped_mm(a, b)
print("3-D variant OK:", result.shape)      # torch.Size([4, 4])

# Grouped variant with offs
result = torch.ops.aten._grouped_mm(a, b, offs)
print("grouped variant OK:", result.shape)  # torch.Size([4, 4])
```
