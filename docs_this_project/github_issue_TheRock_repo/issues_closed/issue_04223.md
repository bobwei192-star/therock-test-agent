rocgdb fails to build on systems where the compiler defaults to LLD and `libiberty-dev` is installed. GDB's bundled `libiberty.a` conflicts with the system copy -- LLD rejects the duplicate symbols, GNU ld silently resolves them.

```
[rocgdb] ld.lld: error: duplicate symbol: get_DW_TAG_name
>>> defined at dwarfnames.o in archive /usr/lib/x86_64-linux-gnu/libiberty.a
>>> defined at dwarfnames.c in archive ../libiberty/libiberty.a
```

(Same error for `get_DW_FORM_name`, `get_DW_AT_name`, `get_DW_OP_name`, `get_DW_ATE_name`, `get_DW_CFA_name`, `get_DW_IDX_name`, `get_DW_UT_name`.)

## Root cause

The libpython directory gets added to LDFLAGS in `debug-tools/rocgdb/CMakeLists.txt:198-199`:

```cmake
set(LIBPYTHON_LDFLAGS "-L${libpython_dir} -Wl,-rpath,${libpython_dir}")
```

When libpython lives in `/usr/lib/x86_64-linux-gnu`, this puts the entire system lib directory on the search path. GDB's BFD configure then resolves `-liberty` against the system `libiberty.a` in addition to GDB's own bundled copy, causing the duplicate.

Related to #1780 and #2149 -- this will affect anyone building rocgdb once LLD becomes the default system linker.

## Environment

- TheRock main (59d98622), Clang 20.1.8 defaulting to LLD, `libiberty-dev` installed, Linux x86_64

## Reproduction

```bash
cmake -B build -GNinja -DTHEROCK_AMDGPU_FAMILIES=gfx1100 -DTHEROCK_ENABLE_ROCGDB=ON
ninja -C build rocgdb
```

Requires: Clang defaulting to LLD (`clang -Wl,--version` shows LLD) and `libiberty-dev` installed.
