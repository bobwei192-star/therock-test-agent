### Problem Description

## Description

When TheRock-based PyTorch package is installed along with the ROCm's standard `amdsmi`, just importing `torch` causes a segmentation fault (double free) on exit, causing configuration issues on certain packages (e.g. while installation of programs like vLLM).

```sh
uv pip install --index-url https://repo.amd.com/rocm/whl/gfx1151/ torch{,vision,audio} --prerelease=allow
uv pip install PATH/TO/_rocm_sdk_core/share/amd_smi
```

```console
(testenv) root@ubuntu-container:~# python
Python 3.12.3 (main, Aug 14 2025, 17:47:21) [GCC 13.3.0] on linux
Type "help", "copyright", "credits" or "license" for more information.
>>> import torch
>>> [reprod: Press Ctrl-D to quit]
double free or corruption (!prev)
Aborted (core dumped)
```

Note that, if the `amdsmi` package is missing in the Python environment, this problem will not occur.

## Direct Cause

In this configuration, there are two shared libraries (with very similar codebase) in question loaded here:

1.  `librocm_smi64.so`
2.  `libamd_smi.so`

and each has a C++ map `amd::smi::Device::devInfoTypesStrings` (both `static` variable constructed / destructed along with the shared library life cycle) (c.f. [`librocm_smi64.so`](https://github.com/ROCm/rocm-systems/blob/7313c3752a22e7c2f628b39ea3de09c347705402/projects/rocm-smi-lib/src/rocm_smi_device.cc#L382-L458) / [`libamd_smi.so`](https://github.com/ROCm/rocm-systems/blob/7313c3752a22e7c2f628b39ea3de09c347705402/projects/amdsmi/rocm_smi/src/rocm_smi_device.cc#L426-L532)).

However `libamd_smi.so` (loaded later through the `amdsmi` package, which is imported by `torch`) in this configuration prefers `amd::smi::Device::devInfoTypesStrings` (symbol with the same name) in `librocm_smi64.so`.  This will result in serious consequences.

1. First, `librocm_smi64.so` is loaded, and `amd::smi::Device::devInfoTypesStrings` is registered for destruction at library unloading.
2. Next, `libamd_smi.so` is loaded; however, if `librocm_smi64.so` (or any shared library loaded with the `RTLD_GLOBAL` option) contains the same symbol name, some internal (yet global) symbols are *incorrectly* resolved as external symbols.
3. Specifically, `libamd_smi.so` registers `amd::smi::Device::devInfoTypesStrings` for destruction at unloading, but the symbol with the same name **from `librocm_smi64.so`** ends up being registered instead [^skipped-details].
4. When the program exits, the `amd::smi::Device::devInfoTypesStrings` from `libamd_smi.so` is not freed. Instead, the `amd::smi::Device::devInfoTypesStrings` from `librocm_smi64.so` is attempted to be freed twice, leading to heap corruption.

[^skipped-details]: `amd::smi::Device::devInfoTypesStrings` is not the only symbol incorrectly resolved (all global names are affected).  In this double free scenario, the destructor of `std::map<...>` in `libamd_smi.so` is resolved into the one of `librocm_smi64.so`.  Fortunately (and, as it happened by chance), this incorrect resolution of the destructor did not cause problems by itself but instead, made the problem easier to spot.

## Possible Root Cause

Clearly, something's wrong with the method how to load shared libraries.

The possible root cause might be... `_rocm_init.py` (bundled in [TheRock-based PyTorch package](https://github.com/ROCm/TheRock/blob/ab5cc820c9aa1f8af96c81131dec36d5d9937e83/external-builds/pytorch/build_prod_wheels.py#L705-L706)) and initialization functions of the module `rocm_sdk`.

The reason of incorrect symbol resolution is the `RTLD_GLOBAL` flag (enabled by default as in [`build_tools/packaging/python/templates/rocm/src/rocm_sdk/__init__.py`](https://github.com/ROCm/TheRock/blob/ab5cc820c9aa1f8af96c81131dec36d5d9937e83/build_tools/packaging/python/templates/rocm/src/rocm_sdk/__init__.py)) which makes all global symbol names *truly* global (used on subsequently-loaded shared libraries; c.f. [(1)](https://github.com/ROCm/TheRock/blob/ab5cc820c9aa1f8af96c81131dec36d5d9937e83/build_tools/packaging/python/templates/rocm/src/rocm_sdk/__init__.py#L85-L98), [(2)](https://github.com/ROCm/TheRock/blob/ab5cc820c9aa1f8af96c81131dec36d5d9937e83/build_tools/packaging/python/templates/rocm/src/rocm_sdk/__init__.py#L106-L114)).

So, I think this behavior should be disabled (use `RTLD_LOCAL` as possible).  If possible, we may need to remove `rtld_global` argument to force using `RTLD_LOCAL`.

## Workaround (applying the suggested fix above)

Locate `rocm_sdk/__init__.py` in TheRock-based ROCm SDK distribution and patch it like:

```py
def initialize_process(
    *,
    preload_shortnames: Optional[List[str]] = None,
    rtld_global: bool = False,  # [PATCHED] originally True!
    env_override: bool = True,
    check_version: Optional[str | re.Pattern] = None,
    fail_on_version_mismatch: bool = False,
    **kwargs,
):
```

## Why is this *not* PR but an Issue?

Because:

1.  I don't have access to various ROCm-based systems with various AMD hardware and
2.  I am not sure whether disabling `RTLD_GLOBAL` would cause other (unintended) side-effects.

### Steps to Reproduce

For reproduction, I used `uv` with a venv (`testenv` here).

Install a TheRock-based PyTorch package plus ROCm's standard `amdsmi` package together.

Replace the index URL appropriately for your environment (this is non-Nightly TheRock-based builds for Strix Halo `gfx1151`).

```sh
uv pip install --index-url https://repo.amd.com/rocm/whl/gfx1151/ torch{,vision,audio} --prerelease=allow
uv pip install PATH/TO/_rocm_sdk_core/share/amd_smi
```

If the entire ROCm SDK (`rocm[libraries,devel]`) is installed, the second line may be replaced with:

```sh
uv pip install $(rocm-sdk path --root)/share/amd_smi
```

Importing `torch` then exiting Python will cause a segmentation fault.

```console
(testenv) root@ubuntu-container:~# python
Python 3.12.3 (main, Aug 14 2025, 17:47:21) [GCC 13.3.0] on linux
Type "help", "copyright", "credits" or "license" for more information.
>>> import torch
>>> [reprod: Press Ctrl-D to quit]
double free or corruption (!prev)
Aborted (core dumped)
```

As you can see, this is the equivalent Python code.

```python
import torch  # Alone!
```

Depending on your environment, you might want to add path to ROCm (the example below assumes that `rocm[libraries,devel]` is installed).

```console
(testenv) root@ubuntu-container:~# python
Python 3.12.3 (main, Nov  6 2025, 13:44:16) [GCC 13.3.0] on linux
Type "help", "copyright", "credits" or "license" for more information.
>>> import torch
/root/libamd_smi.so: cannot open shared object file: No such file or directory
Unable to find libamd_smi.so library try installing amd-smi-lib from your package manager
... (snip) ...
>>> [reprod: Press Ctrl-D to quit]
(testenv) root@ubuntu-container:~# : Because libamd_smi.so is not loaded, double free did not occur.
(testenv) root@ubuntu-container:~# export ROCM_PATH=$(rocm-sdk path --root)
(testenv) root@ubuntu-container:~# python
Python 3.12.3 (main, Nov  6 2025, 13:44:16) [GCC 13.3.0] on linux
Type "help", "copyright", "credits" or "license" for more information.
```
