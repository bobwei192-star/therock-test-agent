## Problem
Marking aqlprofiler-tests and hip-tests target-neutral is needed for multi-arch CI but break the classic CI. In classic CI, packages get only build for a single GPU-architecture and then uploaded to the artifacts buckets. Whichever run for a specific GPU-architecture is latest wins the race and the previously uploaded artifacts will be overridden. Tests therefore fail if they download a package built for the wrong architecture. This is what is reported in #3636.

In multi-arch CI this isn't an issue as the stage builds not for a single but for all the specified GPU-architectures. PR #3654 reverts marking aqlprofiler-tests and hip-tests target neutral.

## Options
### 1. Build for all supported gfx-architectures

The initial PR #3348 set to build for all supported architectures which however broke PyTorch builds (which also tried to build for all architectures). If enabling for all supported architectures, we need to make sure to not break PyTorch again (and hide the architectures).

### 2. Introduce target specific multi-arch build stages for aqlprofiler-tests and hip-tests

Another, not desirable option, is to introduce target-specific build stages in multi-arch CI after a revert. This might need some work on the build topology and further adds the overhead of further build stages.

### 3. Upload to a different subfolder

In classic CI, artifacts could be uploaded to gfx-specific subfolders. This would require to touch every upload- and download related script. Furthermore, it would require branching for artifacts generated _after_ a certain date as well as for multi-arch artifacts where it is convenient (and correct) that they all go into the same subfolder.

### 4. Keep target neutral on a multi-arch integration branch only

Until fully switching over to multi-arch CI, we could revert on main without applying the revert on a multi-arch integration branch. This has the downside of manual integration from main to the integration branch.

### 5. Keep target neutral only for multi-arch, branch in code

Similar to above target neutral artifacts could only be build in multi-arch CI whereas classic CI builds as before. Instead of using different branches, branching in the CMake config and the topology file can be an alternative.

### 6. Add a special case to harcode the acceptable thing

Another option is to hardcoding the gfx-architectures (limited to CI) to hip-tests and other target neutral artifacts.

## Plan

We are going to land the revert #3654 to fix CI while breaking multi-arch CI. A proper fix is to aim for option 1 (or 6) which we will work on next week.
