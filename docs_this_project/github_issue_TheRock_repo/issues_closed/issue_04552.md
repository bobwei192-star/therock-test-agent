## Environment

- **GPU:** AMD Radeon 8060S (Strix Halo, gfx1151)
- **ROCm:** 7.12.0 (bundled in vllm-rocm portable package from lemonade-sdk/vllm-rocm)
- **Triton:** 3.6.0
- **PyTorch HIP:** 7.12.60610
- **Kernel:** Linux 7.0.0-1-mainline (CachyOS / Arch)
- **RAM:** 128GB unified

## Bug

Triton's AMD HIP driver crashes with `invalid device ordinal` when executing kernels in a **forked/spawned child process** on gfx1151 (Strix Halo iGPU).

```
RuntimeError: Triton Error [HIP]: Code: 101, Messsage: invalid device ordinal
```

The error comes from `triton/backends/amd/driver.py` during kernel launch in vLLM's EngineCore subprocess.

## Key Observation

**Main process:** PyTorch HIP works perfectly.
```python
import torch
print(torch.cuda.device_count())   # 1
print(torch.cuda.get_device_name(0))  # Radeon 8060S Graphics
x = torch.randn(100, 100, device="cuda:0", dtype=torch.float16)
y = torch.matmul(x, x)  # OK
```

**Child process (fork/spawn):** Triton kernel launch fails. The HIP device ordinal becomes invalid after the process fork. Both `ROCM_ATTN` and `TRITON_ATTN` backends crash the same way since both use Triton kernels.

## Reproduction

```bash
# Raw vLLM launch — crashes
python -m vllm.entrypoints.openai.api_server \
  --model any-model \
  --dtype float16 \
  --enforce-eager \
  --port 8000
```

Tried: `HIP_VISIBLE_DEVICES=0`, `VLLM_USE_TRITON_FLASH_ATTN=0`, `VLLM_WORKER_MULTIPROC_METHOD=spawn`, `--enforce-eager`, `--disable-custom-all-reduce`. All fail with the same Triton error.

## Workaround

Lemonade SDK's wrapped server backend (lemonade-sdk/lemonade#1537) launches vLLM through a `ProcessManager` that sets up the environment correctly (`PYTHONNOUSERSITE=1`, `FLASH_ATTENTION_TRITON_AMD_ENABLE=TRUE`, proper `LD_LIBRARY_PATH` via a launcher script). Through this path, **all 14 tested models work including a 72B dense model**.

Benchmark results through the Lemonade wrapper:
```
Qwen3-0.6B          116.7 tok/s
Llama-3.2-1B-AWQ    110.4 tok/s
Llama-3.2-3B-AWQ     50.5 tok/s
Qwen3-4B-AWQ         42.8 tok/s
Qwen3-8B-AWQ         22.8 tok/s
Qwen2.5-72B-AWQ       2.3 tok/s (72B dense on 128GB unified)
```

## Root Cause Hypothesis

The gfx1151 iGPU uses unified memory (no discrete VRAM). The HIP device ordinal may not propagate correctly through process fork/spawn when the GPU is an integrated device rather than a discrete PCIe card. The Lemonade workaround suggests the issue is in environment setup (LD_LIBRARY_PATH, device visibility) rather than a fundamental HIP limitation.

## Impact

gfx1151 (Strix Halo) has 128GB unified memory — it's an ideal target for large model inference. The workaround exists but standalone Triton/vLLM users on gfx1151 cannot use the HIP backend without it.
