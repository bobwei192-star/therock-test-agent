### Problem Description

Pytorch will allocate VRAM before loading a large model for inference or training, but then it loads the model in shared GPU memory, causing an OOM. 

Seems to only occur when I have 96GB of VRAM assigned, but is fine when set to 64GB, but this reduces how much VRAM I can use. 

This was occurring with the 7.11 Pytorch specific Adrenalin driver, but I updated to 26.1.1 today and the same behavior is experienced. 

### Operating System

10.0.26220

### CPU

AMD RYZEN AI MAX+ 395 w/ Radeon 8060S

### GPU

AMD Radeon(TM) 8060S Graphics

### ROCm Version

7.11.0a20260120

### ROCm Component

_No response_

### Steps to Reproduce

Example 1: 
1. Load Qwen3-Next 80B in 4-bit with bitsandbytes
2. 55gb of VRAM is immediately allocated
3. Model starts loading, Shared GPU memory usage slowly increases
4. Script crashes, returns this error: torch.OutOfMemoryError: HIP out of memory. Tried to allocate 2.00 MiB. GPU 0 has a total capacity of 107.87 GiB of which 33.06 GiB is free. Of the allocated memory 11.46 GiB is allocated by PyTorch, and 52.46 GiB is reserved by PyTorch but unallocated. If reserved but unallocated memory is large try setting PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True to avoid fragmentation.  See documentation for Memory Management  (https://pytorch.org/docs/stable/notes/cuda.html#environment-variables)

Example 2: 
1. Load Qwen-3-30B-A3B-Instruct-2507 in BF16 for training
2. 57GB of VRAM is immediately allocated 
3. Model loads successfully
4. Dataset and training LoRA data is loaded into Shared GPU memory, despite there being 32+GB of free VRAM
5. Script crashes with an OOM (I can reduce batch sizes and use 8bit optimizers to prevent the OOM, but there should be plenty VRAM to increase batch size)


Sometimes it gives a more generic error instead of the OOM error

torch.AcceleratorError: HIP error: unspecified launch failure
Search for `hipErrorLaunchFailure' in https://docs.nvidia.com/cuda/cuda-runtime-api/group__HIPRT__TYPES.html for more information.
HIP kernel errors might be asynchronously reported at some other API call, so the stacktrace below might be incorrect.
For debugging consider passing AMD_SERIALIZE_KERNEL=3
Compile with `TORCH_USE_HIP_DSA` to enable device-side assertions.

Setting PYTORCH_NO_CUDA_MEMORY_CACHING=1 improves things slightly in Example 1. It loads the model in around 34GB of VRAM, overflows slightly into Shared GPU memory, but then inference works. For example 2 it also only loads 34GB of the model into VRAM, starts loading the rest into Shared GPU Memory, then immediately crashes when that is full with this (partial) error:
