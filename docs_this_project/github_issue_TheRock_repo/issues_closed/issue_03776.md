## Update (2026-03-06) — Error profile changed after fp8/hiprtc revert

After analyzing the 4 logs referenced in this issue (jobs 66060909053, 66060909098, 66060909297, 66060909328 from run 22752257739 attempt 2), the error profile has **significantly changed** compared to the original filing:

### Resolved by fp8/hiprtc revert (`be0675a1a6`)
- **`__habs` ambiguous**: ~~1,178+ occurrences per shard~~ → **0**. Completely fixed.
- **HIPRTC compilation failures for BatchNorm/Neuron/reduction source files**: ~~5,339+ occurrences per shard~~ → **0**. All 14 originally listed source files (MIOpenBatchNormBwdSpatial.cpp, MIOpenBatchNormFwdTrainSpatial.cpp, etc.) no longer fail. The only remaining HIPRTC errors are for `error_hip.cpp`/`error_ocl.cl` which are MIOpen's intentional error test files (test PASSES).

### Still present
- **`named symbol not found`**: 1,791 total (66 + 1,618 + 52 + 55 across 4 shards). Now the **primary failure mode**. Affected kernels are all precompiled/assembly: `MIOpenActiveFwdLite`, `miopenSp3AsmConv_v30_3_1_gfx9_fp32_f2x3_stride1/stride2`, `batched_transpose_*` (22 variants), `check_numerics_bf16`. This points to a **code object loading / symbol lookup regression**, not HIPRTC. Potential culprit: `0b42a7f472 clr: Eliminate unnecessary kernel name string copies (#3774)`.
- **60-minute timeout**: All 4 shards. Consequence of the symbol-not-found cascade.
