# #4808 - Proposal: auto-derive super-project-owned libraries in therock_explicit_finders.cmake

## Summary

`cmake/therock_explicit_finders.cmake` overrides `find_library` / `find_path` for sub-projects that still use naked finders instead of `find_package` / imported targets. The file's own comment correctly frames this as an interim fix while those sub-projects migrate. However, the override's current design -- a hand-curated 3-item allowlist of names that MUST resolve from super-project paths -- leaves most naked `find_library` calls unprotected, and they silently fall through to system paths like `/opt/rocm/lib`.

This is routinely a problem on dev hosts where `/opt/rocm` is installed alongside a TheRock build.

## Current behavior

[cmake/therock_explicit_finders.cmake#L23-L27](https://github.com/ROCm/TheRock/blob/main/cmake/therock_explicit_finders.cmake#L23-L27):

```cmake
set(THEROCK_SUPER_PROJECT_FIND_LIBRARY_NAMES
  hsa-amd-aqlprofile64
  hsa-amd-aqlprofile
  roctx64
)
```

- Names **in** the list: resolved from `CMAKE_LIBRARY_PATH` only; `FATAL_ERROR` if not found.
- Names **not in** the list: fall through to CMake's default search (system paths included).

The list has had 3 entries since it was introduced in #752 (2023). Seems to just be a list of names people have hit bugs with instead of a set designed for completeness.

## Concrete symptom

On a host with `/opt/rocm` installed, a full build with `libopenmpi-dev` (which enables the `jacobi-hip` example in rocprofiler-systems) produces:

```
CMake Warning at examples/hpc/jacobi-hip/CMakeLists.txt:56 (add_executable):
  Cannot generate a safe runtime search path for target jacobi-hip because
  there is a cycle in the constraint graph:

    dir 0 is [.../profiler/roctracer/stage/lib]
      dir 1 must precede it due to runtime library [libroctracer64.so.4]
    dir 1 is [/opt/rocm/lib]
      dir 0 must precede it due to runtime library [libroctx64.so.4]
```

`roctx64` is in the allowlist → resolves to TheRock's staged `libroctx64.so`. `roctracer64` is not → falls through to `/opt/rocm/lib/libroctracer64.so.4`. The resulting rpath cycle is unresolvable and the linked binary mixes two different ROCm versions.

## Broader scope

Other ROCm-adjacent names that naked `find_library` calls search for across `rocm-systems/projects/*/` and that are **not** in the current allowlist:

- `roctracer64` -- rocprofiler-systems jacobi-hip
- `rocblas` -- rocprofiler-systems matrix-exponential-streams example
- `omptarget` -- rocprofiler-systems openmp/target example
- `rocprofiler` / `rocprofiler64` -- rdc's `Findrocprofiler.cmake` (legacy name)
- `rocm-core` -- rdc's `Findrvs.cmake`
- `ROCSHMEM` -- rccl

Any of these will hit the same fall-through-to-system pattern whenever `/opt/rocm` is present.

## Proposal

Change the override's semantics from *hand-curated allowlist* to *auto-derived from super-project stage paths*:

1. **First pass (super-project-first)**: For every `find_library` call, try to resolve from `CMAKE_LIBRARY_PATH` only (`NO_DEFAULT_PATH`). If found, return that.
2. **Second pass (system fallback)**: If not found in super-project stage dirs, do the default `find_library` (system search).
3. **Critical-library allowlist (trimmed)**: Keep a small allowlist only for libraries where system-fallback would be actively dangerous (e.g. `hsa-amd-aqlprofile64`, where version-mixing causes silent runtime failures). These retain FATAL-on-missing semantics.

`CMAKE_LIBRARY_PATH` is already constructed from transitive RUNTIME_DEPS stage dirs, so "auto-derived" requires no new machinery -- it's using information the super-project already has.

## Trade-offs

**Gained**:
- No more manual allowlist maintenance as new libraries get added.
- Every library a super-project subproject stages is automatically preferred over system equivalents.
- Closes the whole class of rpath-cycle / version-mix bugs on dev hosts with `/opt/rocm`.

**Lost**:
- FATAL-on-missing safety for non-critical libs. If a subproject accidentally omits a library from its RUNTIME_DEPS, old behavior errored loudly; new behavior would silently resolve to system. **Mitigation**: keep the FATAL for the small critical-library allowlist; emit a `message(STATUS ...)` for other libs showing which path was actually chosen, so unexpected resolutions are visible in configure logs.

Happy to open a PR if this direction sounds right.
