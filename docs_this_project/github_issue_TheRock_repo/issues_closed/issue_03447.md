## Problem

**Severity:** P1 — requires CI retries (intermittent build failures)

There is an intermittent failure during the build phase where split artifact outputs appear to race with their consumption within the same ninja build. The dependency chain is:

```
stage.stamp → unsplit manifest → split manifest (generic only tracked)
```

Only the generic manifest is tracked as a CMake custom command OUTPUT (`cmake/therock_artifacts.cmake:222-223`). The per-arch artifacts are produced as side effects. If another target within the same stage depends on split artifacts (through the dist flatten), there may be a race where ninja schedules consumption before the side-effect outputs are fully written.

## Evidence

- [Run 21854651990](https://github.com/ROCm/TheRock/actions/runs/21854651990) required attempt 2 (first attempt had build failures)
- Logs suggested a missing dependency edge in the ninja build graph
- The comment at `cmake/therock_artifacts.cmake:248-258` acknowledges that only the generic manifest is tracked as output

## Investigation Plan

Do a local configure with `THEROCK_KPACK_SPLIT_ARTIFACTS=ON` and analyze the generated `build.ninja` rules to find the missing dependency edge:

1. Configure locally with split artifacts enabled
2. Inspect the ninja dependency graph around split artifact targets
3. Look for targets that consume split outputs without depending on the split command
4. The flatten-to-dist step is a likely suspect (it runs as part of the unsplit command, before split runs)

## Key Files

- `cmake/therock_artifacts.cmake:208-261` — split custom commands and dependency tracking
