## Problem

[`ml-libs/CMakeLists.txt:11-23`](https://github.com/ROCm/TheRock/blob/main/ml-libs/CMakeLists.txt#L11-L23) currently flips **both** `THEROCK_ENABLE_COMPOSABLE_KERNEL` *and* `THEROCK_MIOPEN_USE_COMPOSABLE_KERNEL` to `OFF` if *any* arch in `THEROCK_AMDGPU_TARGETS` is missing from the CK allowlist:

```cmake
set(_ck_supported_gfx_targets gfx908 gfx90a gfx942 gfx950 gfx1150 gfx1151 gfx1152 gfx1153 gfx1200 gfx1201)
foreach(_gfx_target ${THEROCK_AMDGPU_TARGETS})
  if(NOT ${_gfx_target} IN_LIST _ck_supported_gfx_targets)
    message(WARNING "...Disabling composable kernel and it's use in MIOpen.")
    set(THEROCK_ENABLE_COMPOSABLE_KERNEL OFF)
    set(THEROCK_MIOPEN_USE_COMPOSABLE_KERNEL OFF)
  endif()
endforeach()
```

A single unsupported arch in a multi-arch family disables CK *for the entire MIOpen build* AND skips building the CK subproject altogether — silently dropping CK kernels on supported arches in the same family.

**Both flag flips are wrong:**

- `THEROCK_MIOPEN_USE_COMPOSABLE_KERNEL OFF` → tells MIOpen "don't use CK at all," dropping CK kernels even on supported arches.
- `THEROCK_ENABLE_COMPOSABLE_KERNEL OFF` → skips the CK subproject build entirely, even though CK already filters its own input via `rocm_check_target_ids`.

## Concrete impact

| `THEROCK_AMDGPU_FAMILIES` | Supported subset | Current outcome |
|---|---|---|
| `gfx94X-all` | gfx942 | CK on (correct) |
| `gfx110X-all` | none | CK off (correct) |
| **`dgpu-all`** | gfx942/gfx950/gfx1200/gfx1201 | **CK silently off everywhere** |
| **User-set `gfx942;gfx1100`** | gfx942 | **CK silently off for gfx942** |

The `dgpu-all` case is the most realistic in-tree hazard.

## Why no per-arch partition is needed in TheRock

CK already filters its own input. [`projects/composablekernel/CMakeLists.txt:257-258`](https://github.com/ROCm/rocm-libraries/blob/develop/projects/composablekernel/CMakeLists.txt#L257-L258) calls `rocm_check_target_ids` against CK's internal allowlist, and per-family `if(SUPPORTED_GPU_TARGETS MATCHES "gfx9|gfx11|gfx12")` gates produce mixed XDL+WMMA builds. Passing the full `THEROCK_AMDGPU_TARGETS` list to CK is the correct behavior — TheRock does not need to filter the list before forwarding it.

After ROCm/rocm-libraries#6836 lands, MIOpen will own per-arch CK shared-library filtering internally and the `MIOPEN_USE_COMPOSABLEKERNEL` flag becomes a narrow escape hatch for skipping the per-arch CK shared libraries entirely. At that point neither side needs TheRock's intervention.

## Proposed end state

1. **Delete the entire L11-23 block.** Stop auto-flipping either flag based on arch content. Both flags become user-controlled (`THEROCK_ENABLE_COMPOSABLE_KERNEL` defaults ON; users can still opt out explicitly).
2. **Remove `THEROCK_MIOPEN_USE_COMPOSABLE_KERNEL` entirely.** Don't forward `-DMIOPEN_USE_COMPOSABLEKERNEL=...` to MIOpen at all (delete L93). MIOpen defaults ON and self-filters.
3. **Remove the `optional_miopen_build_deps` conditional at L74-76**; just always add `composable_kernel` as an MIOpen build dep when CK is enabled.
4. **Keep L25-69 (CK subproject build) and L96 (install GPU databases)** unchanged.

## Acceptance criteria

- `THEROCK_MIOPEN_USE_COMPOSABLE_KERNEL` is removed from TheRock's CMake.
- `THEROCK_ENABLE_COMPOSABLE_KERNEL` is no longer auto-flipped based on `THEROCK_AMDGPU_TARGETS` content.
- Building `dgpu-all` produces a MIOpen package where CK kernels remain available on gfx942/gfx950/gfx1200/gfx1201.
- A user-specified multi-arch list mixing supported and unsupported arches keeps CK on the supported subset.
- Existing single-family configurations (`gfx94X-all`, `gfx110X-all`) continue to work unchanged.
