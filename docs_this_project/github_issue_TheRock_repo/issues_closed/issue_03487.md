### Problem Description

After PR #3305 ("Reland - Move amdsmi subproject from `base/` to `core/` directory"), `import torch; import ray` crashes with SIGBUS on gfx1151 (Strix Halo). This breaks vLLM, which imports both torch and ray during startup.

This is the same class of bug as #2269 (`RTLD_GLOBAL` symbol collision between `libamd_smi.so` and `librocm_smi64.so`), re-triggered by the amdsmi relocation in #3305.

**Last good nightly:** `7.12.0a20260211`
**First bad nightly:** `7.12.0a20260212` (includes PR #3305, commit 219ffa3)

### Reproducer

```bash
uv venv .venv && source .venv/bin/activate
uv pip install \
  "rocm[devel]==7.12.0a20260212" \
  "torch==2.10.0+rocm7.12.0a20260218" \
  "triton==3.6.0+rocm7.12.0a20260218" \
  ray \
  --extra-index-url https://rocm.nightlies.amd.com/v2-staging/gfx1151 \
  --index-strategy unsafe-best-match
rocm-sdk init
export ROCM_PATH=$(python -c "import _rocm_sdk_devel; import os; print(os.path.dirname(_rocm_sdk_devel.__file__))")
export PYTHONPATH=$(python -c "import _rocm_sdk_core; import os; print(os.path.join(os.path.dirname(_rocm_sdk_core.__file__), 'share', 'amd_smi'))")

python -c "import torch; import ray"
# SIGBUS (core dumped)
```

Minimal ctypes-only reproducer (no torch/ray needed):
```python
import ctypes, os
core = os.environ["VIRTUAL_ENV"] + "/lib/python3.12/site-packages/_rocm_sdk_core/lib"
devel = os.environ["VIRTUAL_ENV"] + "/lib/python3.12/site-packages/_rocm_sdk_devel/lib"
ctypes.CDLL(f"{core}/libamd_smi.so.26", mode=ctypes.RTLD_GLOBAL)
ctypes.CDLL(f"{devel}/librocm_smi64.so")  # SIGBUS
```

### Analysis

The crash occurs when `librocm_smi64.so` (from `_rocm_sdk_devel`) is `dlopen`'d after `libamd_smi.so.26` (from `_rocm_sdk_core`) has been loaded with `RTLD_GLOBAL`. Backtrace:

```
#0  amd::smi::GpuMetricsBase_v18_t::~GpuMetricsBase_v18_t() from libamd_smi.so.26
#1  ?? () from librocm_smi64.so
#2  call_init at ./elf/dl-init.c:74
...
#9  _dl_open (file=".../librocm_smi64.so")
```

The two libraries share ~250 `amd::smi::*` symbols (including destructors like `GpuMetricsBase_v*_t::~GpuMetricsBase_v*_t()`). When `libamd_smi.so.26` is loaded with `RTLD_GLOBAL` (as `rocm_sdk.initialize_process()` does), its symbols become globally visible. When `librocm_smi64.so` is subsequently loaded, the dynamic linker resolves its internal symbols to the ones from `libamd_smi.so.26`, causing ABI-incompatible cross-library calls during constructor/destructor execution.

In practice, the loading chain is:
1. `import torch` → `rocm_sdk.initialize_process(rtld_global=True)` → preloads ROCm libs with `RTLD_GLOBAL`, which transitively loads `libamd_smi.so.26`
2. `import ray` → ray's bundled `pyamdsmi` → `ctypes.cdll.LoadLibrary($ROCM_PATH/lib/librocm_smi64.so)` → SIGBUS
