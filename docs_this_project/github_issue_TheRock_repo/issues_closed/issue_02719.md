### Problem Description

Projects with build systems based on autotools are susceptible to timestamp changes to the source files. If a timestamp is changed in a particular way, it may trigger calls to regenerate files via tools that won't be available in the host system doing the build. This can be autoconf, automake or makeinfo.

This is the case for the third-party/sysdeps libraries we download/extract/patch, of which gmp is one. For those libraries we usually do a copy of the source directory and build off of that copy.

For example, here's the original file, extracted from the tarball:

```
ls -lah --full-time /therock/output/build/third-party/sysdeps/linux/gmp/source/doc/gmp.texi
-rw-r--r-- 1 root root 446K 2025-12-30 01:22:40.116677079 +0000 /therock/output/build/third-party/sysdeps/linux/gmp/source/doc/gmp.texi
```

And here's the copy of that file:

```
ls -lah --full-time /therock/output/build/third-party/sysdeps/linux/gmp/build/build/s/doc/gmp.texi
-rw-r--r-- 1 root root 446K 2025-12-30 01:22:40.837679081 +0000 /therock/output/build/third-party/sysdeps/linux/gmp/build/build/s/doc/gmp.texi
```

As we can see the timestamps are not the same. Depending on the order of the copy, files might end up with timestamps that will signal a need to call regeneration tools.

For now I've proposed a workaround for GMP (https://github.com/ROCm/TheRock/pull/2718), but it would be nice to have stable timestamps to avoid any issues in the future.

Just an additional piece of information. The downloading/extracting phase *can* have the original timestamp preserved via the "DOWNLOAD_EXTRACT_TIMESTAMP TRUE" option. But those timestamps will still be lost after a directory copy.
