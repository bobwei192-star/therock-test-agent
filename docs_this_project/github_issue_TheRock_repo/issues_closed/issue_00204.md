We have a comment about not wanting to use a cache on release branches here: https://github.com/ROCm/TheRock/blob/939ab298a2ea2cd77fbe7a204814876c963c81c3/.github/workflows/portable_linux_package_matrix.yml#L61-L69

The workflow is not using ccache, so this seems to just be a cache for pip... which isn't working.

## Questions and ideas

* Do we want to use a cache (CMake ccache) for snapshot (mainline) builds? Then no cache for proper release branches?
* If we do want a cache to work (at least for some types of runs), should we delete the `pip/` subdir prior to saving the cache? Or force pip to use a different directory? Maybe some other fix?
* If we _never_ want to use a cache for release builds, we should remove the code that doesn't currently work
* On other projects we've had writing to the cache controlled by the CI configuration. For example, `pull_request` events only _read_ from the cache while `push`, `schedule`, and possibly `workflow_dispatch` events can also _write_ to the cache. We could design something similar as part of https://github.com/ROCm/TheRock/issues/199.
