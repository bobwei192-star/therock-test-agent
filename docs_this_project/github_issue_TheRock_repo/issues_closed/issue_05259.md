### Problem Description

Several GPU kernels produce fatal hardware exceptions (HSA_STATUS_ERROR_EXCEPTION, code 0x1016) on gfx1151 during neural network inference using PyTorch/ROCm. The crashes occur in torch.multinomial, torch.gather/index_select, and scatter_gather_elementwise_kernel. These operations are triggered during autoregressive token generation in XTTS (Coqui TTS v2), but the crashes are in core PyTorch GPU operations and are not specific to XTTS.

The three kernel names that crash:
- `_ZN2at6native25_assert_async_cuda_kernelIbEEvPKT_NS0_3MsgE` (multinomial)
- `_ZN2at6native24vectorized_gather_kernelILi16ElEEvPcS2_PT0_illllb` (gather / index_select)
- `_ZN2at6native34_scatter_gather_elementwise_kernelILi256ELi4EZNS0_36_cuda_scatter_gather_internal_kernelILb0ENS0_10OpaqueTypeILi4EEElEclINS0_12TensorAssignEEEvRNS_14TensorIteratorElllRKT_EUliE_EEviT1_` (scatter_gather_elementwise)

Moving torch.multinomial and torch.nn.functional.embedding to CPU via monkey-patching prevents the hard crashes for shorter input sequences, but for longer sequences (~20+ words of input text) the GPU still produces inf/nan values in probability tensors, indicating upstream numerical corruption in the GPU computation even when individual operations do not crash.

Setting MIOPEN_CONV_PRECISE_ROCBLAS_FLOWS=1 and MIOPEN_DEBUG_CONV_GEMM=0 improved convolution correctness and resolved some crashes, but the kernel crashes listed above persist regardless.

This is related to a previously filed issue regarding missing gfx1151 MIOpen files (gfx1151_20.HIP.fdb.txt and libMIOpenCKGroupedConv_gfx1151.so). Even with workarounds for those missing files in place, these kernel crashes occur independently.

> **Update:** The crashes were traced to a missing timeout in the reverse proxy layer. The proxy's `requests.post()` call to AllTalk had no timeout set. For long generations (30-60+ seconds), Flask's default connection timeout killed the proxy's connection to AllTalk while the GPU was still mid-inference. AllTalk's GPU kernels crashed when the connection they were reporting back through was abruptly dropped.
>
> Adding `timeout=300` to the proxy's POST request resolved the crashes. Direct calls to AllTalk (port 7851) never exhibited the crashes — only requests routed through the proxy. This suggests the kernel crashes are triggered by an interrupted in-flight GPU computation rather than by the operations themselves being intrinsically broken.
>
> This may be relevant to other users running AllTalk or similar TTS models behind any reverse proxy or load balancer without adequate timeout configuration. The underlying question of whether these kernels handle interrupted computations gracefully on gfx1151 remains open.

### Steps to Reproduce

**Steps to Reproduce:**
1. Hardware: AMD Ryzen AI Max+ 395 (gfx1151 / Strix Halo), 128GB unified memory, Ubuntu 26.04 LTS, ROCm 7.2.2, torch 2.11.0+rocm7.13.0a20260425
2. Install AllTalk TTS V2 with XTTS v2 model (Coqui TTS 0.22.0)
3. Set HSA_OVERRIDE_GFX_VERSION=11.5.1, HSA_ENABLE_SDMA=0, MIOPEN_CONV_PRECISE_ROCBLAS_FLOWS=1, MIOPEN_DEBUG_CONV_GEMM=0
4. Run XTTS inference with any input text of approximately 15+ words
5. The multinomial crash occurs on virtually all inputs. The gather/scatter crashes occur on medium to long inputs.

Minimal reproduction without XTTS — run this directly in Python to confirm the multinomial crash in isolation:

```python
import torch
# Simulate the probability tensor shape XTTS produces during token generation
probs = torch.softmax(torch.randn(1, 8194, device='cuda'), dim=-1)
# This crashes with HSA_STATUS_ERROR_EXCEPTION on gfx1151
result = torch.multinomial(probs, num_samples=1)
print(result)
```
> **Reliable reproduction path (updated):** Route TTS requests through a reverse proxy that has a shorter timeout than the generation time. With a proxy timeout shorter than the GPU computation time, the connection drops mid-inference and the kernel crashes reliably. Direct calls to AllTalk with no proxy do not reproduce the crashes.

### Additional Information

**Additional Information:**
- The crashes occur in the GPU kernels themselves, not in Python. The HSA queue abort message appears in stderr from rocdevice.cpp line 3966.
- HSA_SERIALIZE_KERNEL=3 was suggested by the error output but does not prevent the crashes — it only changes the reporting timing.
- Workaround currently in use: monkey-patching torch.multinomial and torch.nn.functional.embedding to run on CPU. This prevents the hard crashes but does not resolve the underlying numerical corruption for longer sequences.
- The inf/nan values appearing in probability tensors for longer inputs suggest that upstream GPU convolution operations (likely in the GPT attention layers) are producing incorrect results, causing the probability distributions to become degenerate before multinomial is even called. This points to the MIOpen workspace allocation failures (GemmFwdRest returning zero-size workspace) as the root cause of the numerical corruption, even when the missing fdb/CK library issues are worked around.
- Previous related issue: [https://github.com/ROCm/TheRock/issues/5105]

> **Update:** The CPU fallback patches for multinomial, embedding, and torchaudio.load, and the additional MIOpen environment variables (MIOPEN_FIND_ENFORCE, MIOPEN_DEBUG_CONV_GEMM, MIOPEN_CONV_PRECISE_ROCBLAS_FLOWS) were all removed after the root cause was identified. AllTalk operates correctly on gfx1151 with standard configuration when requests are not interrupted mid-computation.
