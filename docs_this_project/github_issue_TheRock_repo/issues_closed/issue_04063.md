## Summary

Many subprojects in TheRock contain hardcoded references to `/opt/rocm` in their CMake files, build scripts, and Python tooling. While the majority are overridable defaults (CMake CACHE variables, env var fallbacks), a significant number are **truly hardcoded with no override mechanism**.

The core problem: when building TheRock on a system that already has a `/opt/rocm` installation, some subproject builds and tests pick up the **system ROCm** components instead of the ones built by TheRock. This leads to subtle link/runtime failures that are difficult to diagnose. This is a common scenario — developers frequently build TheRock from source on machines that already have a system ROCm install, particularly when debugging compiler issues or doing bisections.

This was discovered while building TheRock from source for an LLVM compiler bisection on such a machine.

## Overridable Defaults (Lower Priority)

The most common patterns below are technically correct (they provide defaults) but can still cause issues when a system `/opt/rocm` exists and shadows the intended build:

- **CMake CACHE**: `set(CMAKE_INSTALL_PREFIX "/opt/rocm" CACHE PATH ...)` — affects rocThrust, MIOpen, hipSPARSE, half, and others
- **CMake search paths**: `find_package(... PATHS /opt/rocm)` — widespread across CLR, rocBLAS, MIOpen, RCCL, etc.
- **Python env fallback**: `os.getenv('ROCM_PATH', "/opt/rocm")` — in most `rmake.py` files
- **Bash fallback**: `ROCM_PATH="${ROCM_PATH:=/opt/rocm}"` — in most `install.sh` files

~150+ files across ~50 subprojects use these patterns.
