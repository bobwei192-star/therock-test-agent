## Environment

- TheRock at origin/main + local cherry-picks (2026-05-26), gfx1102+gfx1103
- `THEROCK_ENABLE_ROCPROFSYS=ON`
- rocm-systems at origin/develop tip (includes #6419, which bumped the dyninst pin to `8d6f2bf30` — picking up dyninst#16 "Remove boost dependency")

## Failure

`rocm-systems/projects/rocprofiler-systems/external/dyninst/dyninstAPI/src/RegisterConversion-x86.C:301` declares:

```cpp
map<MachRegister, Register> reverseRegisterMap = map_list_of...;
```

(`map` is `std::unordered_map` per dyninst's typedef). Compiling under libstdc++ 15:

```
hashtable.h:210:21: error: static assertion failed due to requirement
'std::is_copy_constructible<std::hash<Dyninst::MachRegister>>::value':
hash function must be copy constructible
```

Followed by ~12 cascade errors (no `allocator_type`, `pointer`, `iterator`, etc. in the failed `_Hashtable` instantiation).

## Mechanism

dyninst does not provide a `std::hash<MachRegister>` specialization. Under libstdc++ < 15, the primary template was sufficient (resolved to an undefined/incomplete type that wasn't instantiated). Under libstdc++ 15, `<bits/hashtable.h>` added a `static_assert(is_copy_constructible<_Hash>::value)`, and the primary `std::hash<T>` template for non-specialized `T` is now defined as non-copy-constructible by design (to catch invalid uses). The static_assert fires.

Same template-instantiation-strictness pattern as #3514 (rocRoller GCC 15 STL); same libstdc++ tightening, different invalid-since-forever-but-only-now-caught usage.

## Proposed fix

Either:
1. Add `std::hash<MachRegister>` specialization to `dyninst/common/h/registers/MachRegister.h` (using MachRegister's existing identity field as the hash input)
2. Pass an explicit hasher at each `std::unordered_map<MachRegister, ...>` declaration site

Option 1 is centralized and safer. Small upstream PR to ROCm/dyninst.
