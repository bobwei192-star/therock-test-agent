### Problem Description

I believe this is a bug as it's changes the clang behavior related to CMAKE_PREFIX_PATH compared to older versions, details below. I have found a way the bug when building the triton with therock provided newerst clang instead of gcc. (older clang 20 versions from therock works ok)

After TheRock clang 22 update by commit 78afbd25005690b1d2b404c7c1557e87d5d4b6a5, clang has some problems if CMAKE_PREFIX_PATH is set by triggering a following kind of error message when clang is used to build triton for example:

#CMake Error at /therock/build/dist/rocm/lib/llvm/lib/cmake/mlir/MLIRTargets.cmake:2862 (message):
#The imported target "obj.MLIRCAPIDebug" references the file
#"/therock/build/dist/rocm/lib/llvm/lib/objects-Release/obj.MLIRCAPIDebug/Debug.cpp.o"

If CMAKE_PREFIX_PATH is not set, then error does not happen. Older therock/clang versions before the clang 22 update from 20251031 will work just fine even if the CMAKE_PREFIX_PATH is set.

I have verified that the problem is not rocm-sdk python wheel related as the same error can be demonstrated if building the therock sdk manually and using it then from the build/dist/rocm directory.

### Steps to Reproduce

Bug can be triggered by running a following bash script that will download therock to setup python venv, then installs the rocm-sdk from python wheels and finally downloads and starts the triton build.

Bug does not show up if CMAKE_PREFIX_LINE is commented out or older rocm-sdk is installed. (Both can be tested by commenting/uncommenting lines from the script below)

    git clone https://github.com/ROCm/TheRock.git therock

else

fi

# adjust environment variables

export TRITON_BUILD_WITH_CLANG_LLD=true

# fix 2, instead of installing the older rocm-sdk, comment out CMAKE_PREFIX_PATH env setup

echo "-------------------env start --------------------"

python setup.py bdist_wheel
