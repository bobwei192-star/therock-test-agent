### Summary

TheRock CI is failing in rocBLAS on Windows while running `gemmt_batched` pre-checkin tests. The failures are not numeric mismatches; instead the test process reports `rocblas_status_internal_error` and then hits a Windows SEH access violation (`0xc0000005`) in the test body. Some failing cases are preceded by small HIP allocation failures, suggesting an allocation-handling or edge-case-path bug rather than a flaky test.

Observed in TheRock workflow run `26483975375`, job `78062611884`.

```text
Received uncaught exception: rocblas_status_internal_error
```

In several cases, the crash is preceded by HIP allocation warnings:


### Suspected root cause

This looks more like a real rocBLAS bug than test flakiness.

The failure pattern suggests one of:
- incorrect handling of `gemmt_batched` edge cases, especially `k=0`
- continuing after failed HIP workspace / pointer-array allocation
- dereferencing null/invalid device memory after allocation failure
- returning `rocblas_status_internal_error` where `rocblas_status_memory_error` or a clean quick-return should happen

### Expected behavior

`gemmt_batched` should either:
- quick-return safely for no-op / zero-K cases, or
- return a clean rocBLAS error code without crashing the test process

It should not trigger `rocblas_status_internal_error` followed by SEH `0xc0000005`.
