## Description

`rocm-sdk test` fails one check (`testSharedLibrariesLoad`, `ROCmDevelTest`) on Linux because a prebuilt hipDNN frontend Python extension is shipped inside the **`rocm-sdk-devel`** wheel and cannot be loaded:

```
OSError: .../_rocm_sdk_devel/share/hipdnn/python/hipdnn_frontend/hipdnn_frontend_python.cpython-312-x86_64-linux-gnu.so:
        undefined symbol: hipdnnBackendGetSerializedJsonGraph_ext
```

This appears to be a packaging leak rather than a code defect: per #5429, the hipDNN bindings' Python tree was intended to ship *"only via wheel, not the shared-library tarball"* and was excluded from the **lib** component. It is indeed absent from `rocm-sdk-libraries`, but the `share/hipdnn/{python,tests}` tree — including the prebuilt `.so` — still ships in **`rocm-sdk-devel`**, where `rocm-sdk test` discovers and tries to load it.

## Environment

- Linux, Python **3.13**, installed from the nightly index:
  `pip install --pre --index-url https://rocm.nightlies.amd.com/v2/gfx110X-all/ "rocm[libraries,devel]"`
- ROCm nightly `7.14.0a20260607` (`rocm-sdk-devel`), gfx110X-all.

## Reproduce

```bash
python3.13 -m venv v && . v/bin/activate
pip install --pre --index-url https://rocm.nightlies.amd.com/v2/gfx110X-all/ "rocm[libraries,devel]"
rocm-sdk test     # -> Ran 26 tests ... FAILED (errors=1), the hipdnn_frontend_python load
```

## Root cause — two compounding factors

**(a) Dead CI RPATH.** The shipped `.so` declares `NEEDED: libhipdnn_backend.so` but its RPATH is a GitHub Actions build path that exists only on the runner:

```
$ readelf -d hipdnn_frontend_python.cpython-312-*.so | grep -E 'RPATH|NEEDED'
 (RPATH)   Library rpath: [/home/runner/_work/TheRock/TheRock/output/build/core/clr/dist/lib]
 (NEEDED)  libhipdnn_backend.so
```

So on any non-CI machine the backend lib is unfindable and its symbols (e.g. `hipdnnBackendGetSerializedJsonGraph_ext`) resolve as undefined. Proof it's a *findability* issue, not a missing symbol:

```bash
# bare load: undefined symbol hipdnnBackendGetSerializedJsonGraph_ext
# with backend dir on the path: that symbol resolves
LD_LIBRARY_PATH=.../_rocm_sdk_devel/lib python -c "import ctypes; ctypes.CDLL('.../hipdnn_frontend_python.cpython-312-*.so')"
```

**(b) cp312 ABI lock.** With (a) worked around, the load then fails on `_PyObject_LookupAttr` — a private CPython symbol present in 3.12 and **removed in 3.13**. The extension is a nanobind `.so` tagged `cpython-312` and is the *only* Python version shipped, whereas sibling extensions ship for all supported versions:

```
hipdnn_frontend_python.cpython-312-...so          # cp312 ONLY
libpyroctx.cpython-{310,311,312,313}-...so         # all four
libpyrocpd.cpython-{310,311,312,313}-...so         # all four
```

So even after the RPATH is fixed, the shipped artifact can only load on Python 3.12.

## Suggested direction

The clean fix is the one #5429 already intended: **don't ship `share/hipdnn/python/**` (the prebuilt ext + test suite) in `rocm-sdk-devel`** — extend the lib-component exclusion to the devel component (or exclude this tree from `rocm-sdk test`'s shared-library discovery). That removes both symptoms at once, since the artifact isn't meant to be a user-loadable library. (If the ext *is* meant to ship in devel for some consumer, then it needs a relocatable `$ORIGIN`-relative RPATH and an all-Python build matrix like pyroctx/pyrocpd — but the simpler exclusion seems aligned with #5429.)

## Broader context (optional, not the core ask)

The dead `/home/runner` RPATH isn't unique to hipDNN: **40 of 404** shipped `.so` in the `rocm-sdk-*` wheels carry a `/home/runner/...` RPATH (incl. `librocrand.so.1`, `libmiopen_plugin.so`, `libhipblaslt_plugin.so`, the hipDNN test plugins). Most still resolve their deps via co-location, so only the standalone-`dlopen`ed hipDNN frontend actually breaks — but it suggests CI build paths are leaking into shipped binaries instead of `$ORIGIN`-relative RPATHs, which is the same root-cause family as #4853.
