The `BUILD_TOPOLOGY.toml` already defines `iree-libs` as a generic build stage (added in the initial fusilli integration and refined in #3264). Once #3264 lands, this stage needs a corresponding job in the multi-arch CI workflow so it actually runs.

Everything in the topology and source-set plumbing is already wired up — the only missing piece is the workflow job itself.

## What's already in place

The full chain from stage → source sets → submodules is configured:

```
BUILD_TOPOLOGY.toml:
  [build_stages.iree-libs]           →  artifact_groups = ["iree-libs"]
  [artifact_groups.iree-libs]        →  type = "generic", source_sets = ["iree-libs"]
  [source_sets.iree-libs]            →  submodules = ["iree", "fusilli"]
  [artifacts.iree-compiler]          →  artifact_group = "iree-libs", type = "target-neutral"
  [artifacts.fusilli-plugin]         →  artifact_group = "iree-libs", type = "target-neutral"
```

Source fetching also works: `fetch_sources.py --stage iree-libs` resolves the topology chain and fetches `iree` + `fusilli` submodules, including IREE's nested submodules (`third_party/llvm-project`, `third_party/stablehlo`, etc.) via the `--nested-submodules` defaults (updated in #3264).

## What needs to change

**File:** `.github/workflows/multi_arch_build_portable_linux.yml`

Add an `iree-libs` job following the same pattern as `dctools-core` or `profiler-apps` (both are generic stages that depend on `compiler-runtime` and run in parallel with `math-libs`):

```yaml
  # ==========================================================================
  # STAGE: iree-libs (generic, parallel to math-libs)
  # ==========================================================================
  iree-libs:
    needs: compiler-runtime
    uses: ./.github/workflows/multi_arch_build_portable_linux_artifacts.yml
    secrets: inherit
    with:
      stage_name: iree-libs
      stage_display_name: "Stage - IREE Libs"
      timeout_minutes: 180  # 3 hours - IREE compiler build is substantial
      dist_amdgpu_families: ${{ inputs.dist_amdgpu_families }}
      rocm_package_version: ${{ inputs.rocm_package_version }}
    permissions:
      contents: read
      id-token: write
```

Key points:
- **`needs: compiler-runtime`** — iree-libs depends on `hip-runtime` artifacts which are produced by the `compiler-runtime` stage
- **No `strategy.matrix`** — this is a `generic` stage (not `per-arch`), so it runs once, not per GPU family
- **No `amdgpu_family` input** — generic stages omit this
- **Runs in parallel** with math-libs, comm-libs, dctools-core, profiler-apps, etc.

Also update the header comment (lines 1-11) to include iree-libs in the stage list.

## How the pipeline resolves this at runtime

The reusable workflow (`.github/workflows/multi_arch_build_portable_linux_artifacts.yml`) handles everything automatically for a stage:

1. **Artifact fetch**: `artifact_manager.py fetch --stage iree-libs` computes inbound artifacts by walking `artifact_group_deps` (`hip-runtime` → provides `core-hip`, etc.) and `artifact_deps` on each artifact
2. **Source fetch**: `fetch_sources.py --stage iree-libs` resolves the topology chain above, fetching `iree` + `fusilli` submodules + IREE's nested submodules (`third_party/llvm-project`, `third_party/flatcc`, `third_party/benchmark`, `third_party/stablehlo`, `third_party/torch-mlir`)
3. **Configure**: `configure_stage.py --stage iree-libs` generates the CMake args to enable `THEROCK_ENABLE_IREE_COMPILER` and `THEROCK_ENABLE_FUSILLI_PLUGIN`
4. **Build**: `cmake --build build --target stage-iree-libs therock-artifacts`
5. **Push**: Uploads `iree-compiler` and `fusilli-plugin` artifacts to S3

## Why this matters

Without this job, iree-libs won't build in the multi-arch pipeline even though `THEROCK_ENABLE_IREE_LIBS` is flipped to ON by default in #3264. The IREE compiler build is heavy (includes its own captive LLVM), so running it as a parallel stage means it doesn't add to the critical path.
