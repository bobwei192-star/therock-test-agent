## 描述

Using ROCm 5.2, RadeonVII, Ubuntu 21.10, Linux kernel 5.18.8.
For this OpenCL kernel:
```
kernel void testKernel(global uint *io) {
  uint me = get_local_id(0);
  ulong x = io[me] & 0x00ffffffu;
  io[me] = (x * x) >> 32;
}
```
this ISA is generated:
```
	s_load_dwordx2 s[0:1], s[4:5], 0x0
	v_lshlrev_b32_e32 v0, 2, v0
	v_mov_b32_e32 v2, 0
	s_waitcnt lgkmcnt(0)
	v_mov_b32_e32 v1, s1
	v_add_co_u32_e32 v0, vcc, s0, v0
	v_addc_co_u32_e32 v1, vcc, 0, v1, vcc
	global_store_dword v[0:1], v2, off
	s_endpgm
```
As you see, the generated code disregards the input value, and simply writes 0 (v2). This is not correct, the expected behavior would be to write the high dword of the multiplication, as would be produced by e.g. v_mul_hi_u32_u24

### 评论 #11 — dmikushin (2022-07-06T10:49:31Z)

Also we could note that [mul24 intrinsic emission](https://github.com/llvm/llvm-project/blob/main/llvm/lib/Target/AMDGPU/AMDGPUCodeGenPrepare.cpp#L558) is performed on a type read from the instruction: 

```c++
bool AMDGPUCodeGenPrepare::replaceMulWithMul24(BinaryOperator &I) const {
...
  Type *Ty = I.getType();
  unsigned Size = Ty->getScalarSizeInBits();
...
Value *Result =
        getMul24(Builder, LHS, RHS, Size, LHSBits + RHSBits, IsSigned);
```

It means at the time when mul is replaced with mul24, the return type has been already changed from 64-bit to 32-bit, which suggests that the cause is not the mul24 itself.
