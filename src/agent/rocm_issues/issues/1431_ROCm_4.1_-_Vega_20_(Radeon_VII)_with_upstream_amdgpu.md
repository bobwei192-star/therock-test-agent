## 描述

As documentation of ROCm 4.1 release mentions Vega20 based devices seem to be temporarily disabled when upstream AMDGPU kernel module is used. As expected, after upgrading ROCm 4.0 to 4.1 `rocminfo` reports:

> HSA Error:  Incompatible kernel and userspace, Vega 20 [Radeon VII] disabled. Upgrade amdgpu.

(OpenSUSE Tumbleweed with kernel 5.11.6-1)

What version of kernel/amdgpu is required to re-enable Vega20 with ROCm 4.1?

### 评论 #12 — FilipVaverka (2021-05-13T21:00:33Z)

Its buried deep down in [AMD ROCm™ Release Notes v4.2](https://rocmdocs.amd.com/en/latest/Current_Release_Notes/Current-Release-Notes.html#driver-compability-issue-in-rocm-v4-1) as

> Driver Compability Issue in ROCm v4.1
>
> In certain scenarios, the ROCm v4.1 or higher run-time and userspace environment are not compatible with ROCm v4.0 and older driver implementations for 7nm-based (Vega 20) hardware (MI50 and MI60).
>
> To mitigate issues, the ROCm v4.1 or newer userspace prevents running older drivers for these GPUs.

Considering that the issue persists with ROCm 4.2 with kernel 5.12.0 it seems to me like there is no way to update "amdgpu" kernel module in upstream sooner than with 5.13 release in about two months. Having one of two officially supported architectures (considering only desktop products) disabled for so long with so little information doesn't seem very good for ROCm.

Only option seems to be to downgrade all the way to ROCm 4, but then there are other issues with that.
