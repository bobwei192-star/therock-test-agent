### 评论 #13 — jfshorter (2021-02-05T00:29:53Z)

Understandable problems to encounter trying to debug this remotely, I somewhat sympathise.
I see there being two problems.

### VNC/X-forwarding isn't enough on its own
Normally I would suggest simply running via sudo in a VNC session with your window manager running, rather than root directly as you have presented us. Root cannot export windows to a user VNC session without an authentication.

However, I also know this will not be enough to run the application once installed anyway.
Whilst VirtualGL might work if you also have the OpenCL interposer working too, along with the ROCm modules, but we cannot guarantee that will work.

Simplest method is going to be to leverage a real X11 display with OpenCL+GL support for the application to also have a chance of running after installation.

X forwarding will also likely not work unless you are actively logged in on display :0 - I believe, I might be wrong.

### Mirroring an existing physical screen via VNC server

The solution I propose may not be in line with AMD internal security policy, but I suggest you use **x11vnc** to mirror display :0 as if it were another VNC session.
This is also unencrypted by default but I believe this can be enabled if you are not using a VPN for some reason - Which I highly doubt.
 
Since it's technically not as secure if there's an attached display that isn't off and some bad-actor happening to be in the same room...
We cannot _force_ you to do this, but I am assuming like us over at arm you are holed up at home and security are on the premises/monitoring.
Make sure you remember to lock the screen/have an inactivity timeout lock policy on your window manager, then you are as good to go and covered as anyone can reasonably expect.

### Setting up the VNC mirror

Now, hopefully with legalities and risk mitigation out of the way:

- I am assuming the lab machine is in runlevel 5
- With a session manager running and alive

If not, ensure the machine is in runlevel 5 and check the Xorg log first for signs of death before proceeding.

1. If you have any firewall running by default: Allocate a small port range to open up temporarily using the distributions' admin tools/command line as appropriate.
2. Launch an VNC server to mirror display :0 using an open port with **x11vnc** - This is the magic to allow you to use display :0 as if you were physically there: https://wiki.archlinux.org/index.php/x11vnc
3. Connect with your VNC viewer of choice to the port you allocated and authentication set up.
4. Use as normal, run as if you were a normal user and click through the set up.
5. Kill the VNC server and close the ports once you are finished.

Example command call:
`x11vnc -display :0 -usepw -forever -o ~/x11vnc_x0.log`

The "-forever" switch disables exiting of the VNC server after the user disconnects, as I assume you will need to connect many times.
This argument "-usepw" will inherit the current users' ~/.vnc/passwd password.
If you want to set a temporary launch-time password I would suggest creating a separate authentication file:
`vncpasswd ~/.vnc/vncx0passwd"
And specify with "-auth ~/.vnc/vncx0passwd \"

To kill the server once you are done, I believe, it is just like any other VNC server.
Failing that sending the middle finger with "kill -9 PID" will achieve the same goal, just more rudely.

Drop a reply if you need some specific guidance on this.

There are other potential options but I believe they would also require X :0 sessions for them to work anyway - i.e. parsec (https://parsec.app/downloads/) Which _might_ be easier and better for latency but I have never used it myself to know and would require a commercial license prior to use in this case.

One thing I should mention is that the commitment to support workstation apps on ROCm compute is fairly recent, and before that the PAL-based OpenCL implementation we were shipping with AMDGPU-PRO was the primary solution for applications like Resolve and Blender. We are now at the point where ROCm-based OpenCL is a good alternative to the previous PAL-based solution, and have made it the default on Linux for Vega, Navi and onwards. 

The work we did for ROCm OpenCL in AMDGPU-PRO has not yet made it into a regular ROCm release but it should get there soon - at that point you should see the same improvement in the ROCm releases with one caveat.

The caveat concerns GL/CL interop - as far as I know the testing focus for GL/CL interop has been against the workstation GL driver in AMDGPU-PRO rather than the upstream Mesa GL driver (radeonSI). I don't know the exact state of GL/CL interop at the moment but will make sure we find out and get that documented publicly. 

The reason I'm saying all this is that in the short term you might still get directed to the AMDGPU-PRO driver packages (with ROCm-based OpenCL plus workstation OpenGL) since that is what has been tested and fixed with applications using GL/CL interop. If you want to use ROCm stack releases along with a sufficiently new upstream graphics stack that should work in general, but you may still get pushback in the short term in the specific case where the problem seems related to GL/CL interop between the ROCm OpenCL driver and the upstream OpenGL driver.
