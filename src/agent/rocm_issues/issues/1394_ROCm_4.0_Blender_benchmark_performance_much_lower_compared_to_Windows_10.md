# ROCm 4.0 Blender benchmark performance much lower compared to Windows 10

> **状态**: closed

## 描述

Blender's benchmark with the bmw27 scene takes about 4x longer (35s vs 1m 54s) to run using ROCm on Linux using the mainline kernel driver (i.e. not rocm-dkms), compared to Windows 10. Also, I'm observing strange artefacts on screen while the benchmark runs, and my GUI (GNOME 3.38 Wayland) becomes laggy.

In comparison, other applications and packaged benchmarks such as Luxmark 4.0 alpha and Ethminer run about the same or even faster using ROCm compared to Windows 10. AMDGPU-PRO drivers exhibit the same issue, considering apparently for Navi cards the OpenCL stack in AMDGPU-PRO is just ROCm's OpenCL component.

Finally, I observed that while running Blender there'd be huge fluctuations in VRAM usage (hitting up to the limit of 16GB on the card and then going back down to 2-3GB) and that the 'Texture Addresser' and 'Shader Interpolator' loads showed by Radeontop would constantly fluctuate up and down, hitting lows around 20-30%, despite high utilization being reported by rocm-smi as well as the 'Graphics Pipe' indicator in Radeontop. In contrast, Ethminer and Luxmark show steady utilization for 'Texture Addresser' and 'Shader Interpolator' as well as VRAM. I do not know if this is because of a bug or due to the nature of the Blender workload.
