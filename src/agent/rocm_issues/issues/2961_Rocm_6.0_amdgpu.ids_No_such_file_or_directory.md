## 描述

### Problem Description

When pytorch 2.3.0.dev20240312+rocm6.0 is used, only the Laptop Graphic card is active:
`>>> torch.cuda.is_available()
True
>>> torch.cuda.device_count()
1
>>> torch.cuda.device(0)
<torch.cuda.device object at 0x7f0ff2000830>
>>> torch.cuda.get_device_name(0)
'AMD Radeon Graphics'`

whereas on the same machine, on the same linux, with the same user when pytorch 2.2.1+rocm5.7 is used, then the proper GPU is found:
`>>> torch.cuda.is_available()
True
>>> torch.cuda.device_count()
1
>>> torch.cuda.current_device()
0
>>> torch.cuda.device(0)
<torch.cuda.device object at 0x7f913c3b2360>
>>> torch.cuda.get_device_name(0)
'AMD Radeon RX 7700S'`

### 评论 #17 — teleprint-me (2025-12-13T21:13:49Z)

You can use `mlocate` to see if the GPU ids are available. This is a 3rd party package supported by most distros.

I run into this issue every time I do a clean install even though I have everything I need.

```sh
locate amdgpu.ids
/usr/share/libdrm/amdgpu.ids
```

But PyTorch looks elsewhere for this.

```sh
python -c "import torch; print(torch.cuda.get_arch_list())"
/opt/amdgpu/share/libdrm/amdgpu.ids: No such file or directory
```

Just as a sanity check, I usually just check for the files existance.

```sh
file /opt/amdgpu/share/libdrm/amdgpu.ids
/opt/amdgpu/share/libdrm/amdgpu.ids: cannot open `/opt/amdgpu/share/libdrm/amdgpu.ids' (No such file or directory)
```

Because ROCm is installed into a non-conventional path, the binaries, libraries, etc. are never registered even though everything that is required already exists.

This means usually having to dig through a bunch of docs, references, posts, issues, etc. just to figure out which environment variables I need. This is non-trivial even though it should be.

```sh
# Only enable if using ROCm
export ROCM_PATH=/opt/rocm
export ROCM_HOME=/opt/rocm

# ROCm device visibility
export HIP_VISIBLE_DEVICES=0
export ROCR_VISIBLE_DEVICES=0
export TRITON_USE_ROCM=1

# ROCm device override
export AMDGPU_TARGETS="gfx1100"
export HCC_AMDGPU_TARGET="gfx1100"
export PYTORCH_ROCM_ARCH="gfx1100"
export HSA_OVERRIDE_GFX_VERSION="11.0.0"  # Ensure correct RDNA3 detection

# Optional: Disable hipBLASLt if issues occur
export USE_HIPBLASLT=0
export TORCH_BLAS_PREFER_HIPBLASLT=0

# PyTorch ROCm Memory Management
export PYTORCH_ALLOC_CONF="expandable_segments:False,garbage_collection_threshold:0.8"

# ROCm path export
export LD_LIBRARY_PATH=$LD_LIBRARY_PATH:/opt/rocm/lib
export PATH=$PATH:/opt/rocm:/opt/rocm/lib:/opt/rocm/share:/opt/rocm/bin
```

The only way to figure out which environment variables are valid are to use `rocminfo` and `rocm_agent_enumerator`. `rocm_agent_enumerator` tells me which version I have registered. 

The versioning is unclear and usually I just brute force my way through it until I figure it out since there's no consistent schema for it.

I'm not sure if there's a variable I can set to remedy where the device ids are set.

e.g.

```sh
LIBAMDGPU_ID_PATH=/usr/share/libdrm/amdgpu.ids
```

versus just doing

```sh
# Create the directory that torch expects
sudo mkdir -p /opt/amdgpu/share/libdrm

# Point that directory to the real file
sudo ln -s /usr/share/libdrm/amdgpu.ids /opt/amdgpu/share/libdrm/amdgpu.ids
```

Symlinking is probably the easiest method, but I'm not sure how reliable or flexible that is.

As far as I can tell, this is unresolved issue. I'm posting this here for others that encounter similar issues. I wouldn't mind any feedback regarding my solutions to resolving these issues.
