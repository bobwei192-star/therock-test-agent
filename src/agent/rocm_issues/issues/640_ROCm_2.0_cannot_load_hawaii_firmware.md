### 评论 #6 — dmikushin (2019-01-09T23:44:20Z)

Just to clarify the previous comments, as they were misleading for me: after providing Hawaii firmwares under correct paths, simple kernel module reload or reboot won't make it work! These files are not read from disk - `MODULE_INFO` gets them from the depth of ELF sections inside the .ko file, where they are packaged upon compilation. __So, you have to rebuild dkms module first and then do the modprobe.__

### 评论 #9 — jlgreathouse (2019-01-12T04:14:08Z)

Hi @dmikushin and @Hobbes921 

I apologize that Hawaii seems to be broken in ROCm 2.0. As noted in our [hardware support list](https://rocm.github.io/hardware.html), Hawaii *should* be enabled in ROCm but is not officially supported or tested by AMD at this time. Such problems can happen when code is updated and is not tested against hardware.

I understand that this does not help you with your card, and I understand your frustration at its lack of  functionality in ROCm 2.0. I personally have multiple Hawaii FirePro cards in use and it frustrates me when they don't work. Please don't think that this is intentional or that this happened because AMD "wants to sell new cards". We are working to fix this problem for future releases.

Please check my response in #668 and see if that fixes your problem with OpenCL. I would like your feedback here, as it will help us understand if such a patch is sufficient for future releases.

In addition, if you would instead prefer to roll back to ROCm 1.9.2, please see the [roc-1.9.2 branch of the Experimental ROC repository](https://github.com/RadeonOpenCompute/Experimental_ROC/tree/roc-1.9.2) for scripts that can be used to install this older version from the repository archives located on repo.radeon.com

OK, it appears that there are three main problems with Hawaii in ROCm 2.0.

1. The driver does not load firmware from the proper location
2. The OpenCL runtime does not properly pass `gfx701` to the compiler when it attempts to build kernels
3. The driver has a possible segfault due to some refactoring, and this is sensitized primarily by Hawaii GPUs running some workaround code.

Issue number one is what was originally raised in this thread.
Issue number two is what @dmikushin points out in #668 
Issue number three is what causes the problem seen by @rdemaria [in the third post post in this thread](https://github.com/RadeonOpenCompute/ROCm/issues/640#issuecomment-449040058).

I believe there are workarounds for each of these that should not cause too much difficulty (though they may take a while to recompile the code):

1. After installing `rock-dkms` for ROCm 2.0, go to `/usr/src/amdgpu-2.0-89/firmware/amdgpu/` (`/usr/src/2.0-89.el7/firmware/amdgpu/` on CentOS/RHEL), and run:
   1. `sudo cp ../radeon/hawaii* .`
   1. `sudo dkms remove amdgpu/2.0-89 --all` (`sudo dkms remove amdgpu/2.0-89.el7 --all` on CentOS/RHEL)
   1. `sudo dkms add amdgpu/2.0-89` (`sudo dkms add amdgpu/2.0-89.el7` on CentOS/RHEL)
   1. `sudo dkms install amdgpu/2.0-89` (`sudo dkms install amdgpu/2.0-89` on CentOS/RHEL)
1. Make the changes [I describe here](https://github.com/RadeonOpenCompute/ROCm/issues/668#issuecomment-453717654) to patch and recompile the OpenCL runtime.
