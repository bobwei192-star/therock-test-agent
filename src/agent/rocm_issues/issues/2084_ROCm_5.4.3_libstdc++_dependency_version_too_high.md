## 描述

On SLES 15 SP4, using ROCm 5.4.3 is so difficult compared to 5.4.0 due to extremely tight libstdc++ version requirement.
amdclang and hip libraries both requires at least gcc12 equivalent libstdc++.
When I use gcc 11 as the host compiler and hip runtime library.
```
/soft/compilers/gcc/11.1.0/x86_64-suse-linux/bin/g++ main.cpp /soft/compilers/rocm/rocm-5.4.3/lib/libamdhip64.so.5.4.50403
/usr/bin/ld: /soft/compilers/rocm/rocm-5.4.3/lib/libamd_comgr.so.2: undefined reference to `std::condition_variable::wait(std::unique_lock<std::mutex>&)@GLIBCXX_3.4.30'
collect2: error: ld returned 1 exit status
```
main.cpp
```
#include <memory_resource>
int main()
{}
```
Please minimize the dependency version.

PS: no issue with amdclang due to its using ld.lld as the linker instead of ld and probably no symbol checks.

### 评论 #9 — ye-luo (2024-02-27T02:53:26Z)

@vstempen here is what I found.

You are using g++11 installed via the OS package manager. It relies on libstdc++ from OS which is GLIBCXX_3.4.32 equivalent to GCC 13.2.0. This is achieved by softlinking `/usr/lib64/gcc/x86_64-suse-linux/11/libstdc++.so -> ../../../libstdc++.so.6` and your g++11 picks it up via `LIBRARY_PATH=/usr/lib64/gcc/x86_64-suse-linux/11`.

In my case, I have a standalone installation of gcc 11. This is common in HPC as we don't install compilers from the OS package manager. My gcc 11 finds the true gcc 11 shipped libstdc++ in my standalone gcc 11 installation.

When the libamdhip64 library changed dependency in the 5.4.3 minor release. My g++-11 complains but your g++-11 doesn't.

### 评论 #12 — vstempen (2024-03-13T17:14:44Z)

I was able to reproduce the issue by compiling gcc11 to isolated location. In this case gcc is using GLIBCXX_3.4.30. When gcc is installed with package manager, it uses GLIBCXX_3.4.32.

Difference between those two versions GLIBCXX_3.4.30 has __cpp_lib_jthread undefined for condition_variable class which makes std::condition_variable::wait(std::unique_lock<std::mutex>&) unavailable.

Simply defining __cpp_lib_jthread in condition_variable to the value defined in stop_token or version, for example, causes many build errors. So, this does not seem to be a solution.

ROCm LLVM uses parallel processes in many occasions and synchronization between them is done using std::condition_variable::wait. But using multithreading is gated by LLVM_ENABLE_THREADS compilation flag. So, the problem can be resolved by rebuilding LLVM with LLVM_ENABLE_THREADS=OFF.
