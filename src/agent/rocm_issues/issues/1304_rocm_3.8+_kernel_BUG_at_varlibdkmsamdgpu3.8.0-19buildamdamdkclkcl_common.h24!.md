### 评论 #2 — ROCmSupport (2020-11-26T05:45:16Z)

Hi @tymoteuszrogalewski 
Thanks for reaching out.

As you are saying that kernel driver is loaded perfect and GPUs are visible when you use ROCm 3.7 but not the same case with 3.8 and 3.9, can you try installing ROCm 3.7 once again after uninstalling ROCm 3.8/3.9 now and check whether GPUs are visible now?

You no need to copy any files or compiling sources manually.
Once you install ROCm using **sudo apt install rocm-dkms** command, rock-dkms takes care of moving kernel component to respective folders automatically. All user space will land @ /opt/rocm.
One important thing to be noted is: before installing rocm-dkms for any specific version, make sure that your system is always clean(I mean old ROCm should be uninstalled well)

### 评论 #9 — kentrussell (2020-11-26T14:34:40Z)

OK that would be the issue there. There's no reservation_ww_class function. Here's what my system returns:
$ cat /proc/kallsyms |grep reservation_ww
0000000000000000 r __ksymtab_reservation_ww_class
0000000000000000 r __kcrctab_reservation_ww_class
0000000000000000 r __kstrtab_reservation_ww_class
0000000000000000 D reservation_ww_class
0000000000000000 r __ksymtab__kcl_reservation_ww_class  [amdkcl]

0000000000000000 r __kcrctab__kcl_reservation_ww_class  [amdkcl]
0000000000000000 B _kcl_reservation_ww_class    [amdkcl]

So it's erroring out because there's no reservation_ww_class function that can be used by the KCL (Kernel Compatibility Layer, what ROCm/amdgpu-pro use to support DKMS packages on different kernels). This is normally provided by drivers/dma-buf/dma-resv.c:
dma-buf/dma-resv.c:DEFINE_WD_CLASS(reservation_ww_class);
dma-buf/dma-resv.c:EXPORT_SYMBOL(reservation_ww_class);
Is that present on the machine where you're building the kernel?
