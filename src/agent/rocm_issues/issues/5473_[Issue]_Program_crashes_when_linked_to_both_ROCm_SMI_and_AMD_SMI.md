## 描述

### Problem Description

When an application links to both librocm_smi64.so (ROCm SMI) and libamd_smi.so (AMD SMI) libraries in the same process, memory corruption error occurs.

### Operating System

Red Hat Enterprise Linux 8.10 (0otpa)

### CPU

AMD EPYC 7402 24-Core Processor

### GPU

AMD Instinct MI210

### ROCm Version

ROCm 7.0.1

### ROCm Component

amdsmi

### Steps to Reproduce

source.cpp:

```
#include <hip/hip_runtime.h>


#define HIP_CHECK(condition)                                                                \
    {                                                                                       \
        const hipError_t error = condition;                                                 \
        if(error != hipSuccess)                                                             \
        {                                                                                   \
            printf("An error encountered: \"%s\" at %s:%d  \n",                             \
                    hipGetErrorString(error),__FILE__, __LINE__);                           \
            return -1;                                                                      \
        }                                                                                   \
    }


int main(int argc, char **argv)
{
    HIP_CHECK(hipStreamSynchronize(0));
    return 0;
}
```

compilation:

```
hipcc source.cpp -lamd_smi -lrocm_smi64 -o source
```

error:

```
corrupted size vs. prev_size in fastbins
Aborted (core dumped)
```

Is there a particular reason you are trying to link both libraries? amd-smi is intended as a successor to rocm_smi_lib, so I'd recommend sticking with one or the other, thanks!

### 评论 #3 — DanielVelicka (2025-10-08T20:56:41Z)

Hello @darren-amd ,

We are developing a tool called [MERIC](https://code.it4i.cz/energy-efficiency/meric-suite/meric). The development of the tool started years ago, when ROCm SMI was the only option to interact with AMD GPUs.

Since the tool is intended to be used with various HPC applications, it happens regularly that such an application has a dependency on some older version of ROCm. Thus we intended to offer support for both libraries.

However, sticking with just one of the libraries should be perfectly manageable for our case.

---

### 评论 #4 — darren-amd (2025-10-16T17:58:54Z)

Thanks @DanielVelicka, glad to hear it isn't blocking you. I had a chat with the amd-smi team, and because the two libraries share external symbols, we don't support linking both libraries.
