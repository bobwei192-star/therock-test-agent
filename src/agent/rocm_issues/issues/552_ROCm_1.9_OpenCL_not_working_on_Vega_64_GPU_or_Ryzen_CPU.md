### 评论 #7 — jlgreathouse (2018-12-05T01:23:39Z)

Hi @fshi98 

The ROCm OpenCL runtime only claims OpenCL 1.2 support because we do not yet fully support all required OpenCL 2.0 features. In particular, pipes and device enqueue are not fully supported as of ROCm 1.9.x. As such, we only fully support OpenCL 1.2. That said, we do support all other OpenCL 2.0 features, and we fully support the OpenCL 2.0 device-side language.

Our amdgpu-pro line does not use the same underlying HSA technology as our ROCm runtime. For instance, sending work to the GPU requires going through the GPU kernel driver, which can add extra latency. However, I believe the AMD OpenCL runtime that runs on top of the amdgpu-pro stack does support all of the OpenCL 2.0 features. As such, it claims a different level of support that our ROCm OpenCL runtime.

You may see different performance between these two stacks because the software layers sitting between your OpenCL code and the GPU hardware are different. This includes the layer between the bottom of the OpenCL runtime and the GPU (the user-level ROCr runtime for ROCm, various user-land and kernel-land layers for amdgpu-pro).

This also includes some of the compiler path. I believe amdgpu-pro defaults to the HSAIL intermediate language between your OpenCL device code and the eventual GCN assembly. ROCm defaults to going from your OpenCL device code solely through the compilers intermediate representation before going to GCN assembly. If you see in the "Driver version:" the "LC" characters -- this stands for "Lightning Compiler", which is our compilation technique that skips the HSAIL intermediate language. I believe you may say "HSAIL" in that location on the amdgpu-pro drivers.

In the end, the resulting kernels between these two systems may be different and you may see different performance.

Note that I say "different" performance. I can't tell you which will perform better for your applications of interest. That said, the "OpenCL 1.2" line in your output does not imply that your performance will be worse. If you look at the "[OpenCL] driver version" (which is the version of the OpenCL runtime) you'll see that the one on your ROCm system is quite a bit newer (v 2679.0 vs. 1800.5), and thus may include some performance optimizations.

### 评论 #11 — jlgreathouse (2018-12-11T19:44:13Z)

ROCm is officially supported on Ubuntu 16.04.5 and Ubuntu 18.04.1, both of which run kernel 4.15 with some backported patches. It should also work on e.g. 4.12. However, the ROCm kernel driver is not supported on kernel 4.18 at this time.

That being said, if you are running kernel 4.18, you should be able to skip the step of installing the `rock-dkms` package (which is installed as part of the `rocm-dkms` meta-package). Instead, you can install `rocm-dev` and run the udev setup command [listed here](https://github.com/RadeonOpenCompute/ROCm/tree/roc-1.9.1#rocm-19-is-abi-compatible-with-kfd-in-upstream-linux-kernels). This is not an AMD-supported configuration, but it should work nonetheless. Note, however, that there are no guarantees that the user-land software will work if you are not running Ubuntu. Our libraries and binaries are compiled against certain system library versions and thus have not been tested on e.g. Debian.
