### 评论 #8 — jlgreathouse (2018-03-05T21:57:15Z)

Hi @sampie ,

Looking through your code and bug report, it appears you're trying to use OpenCL 2.0 device enqueue. However, support for device enqueue is currently a work in progress for the ROCm OpenCL runtime. As such, we do not guarantee that we can compile such kernel, nor can we run them at this time.

For example, if you try to [run the following](https://github.com/ROCm-Developer-Tools/clARMOR/blob/master/tests/good_complex_svm/test_good_complex_svm.c#L31) test in clARMOR and bypass [its check for device enqueue support](https://github.com/ROCm-Developer-Tools/clARMOR/blob/master/tests/good_complex_svm/test_good_complex_svm.c#L324), the compilation will fail. It won't segfault, but clBuildProgram() will still not succeed.

I would recommend doing like we do in clARMOR and [testing whether the runtime can support device enqueue](https://github.com/ROCm-Developer-Tools/clARMOR/blob/master/tests/common_include/common_test_functions.c#L84) before attempting to compile a kernel that uses this feature. We do this by checking CL_DEVICE_MAX_ON_DEVICE_QUEUES from clGetDeviceInfo(). If there are no device queues (even the default queue), then you likely won't be able to compile a kernel that tries to do device enqueue.
