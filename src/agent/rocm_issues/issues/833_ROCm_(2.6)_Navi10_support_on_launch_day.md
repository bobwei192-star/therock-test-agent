### 评论 #10 — Degerz (2019-07-10T19:50:20Z)

@smartbitcoin You still don't know what you're talking about ... 

And no Pascal doesn't support PCIe atomics, Nvidia uses another interconnect interface known as NVLink to support system-wide atomics. Pascal **needs** NVLink which is only available on select GPUs otherwise atomics within CUDA kernals between host/device and device memory isn't going to be visible as stated in the manual. 

> On GPU architectures with compute capability lower than 6.x, **atomics operations done from the GPU are atomic only with respect to that GPU**. If the GPU attempts an atomic operation to a peer GPU’s memory, the operation appears as a regular read followed by a write to the peer GPU, and the two operations are not done as one single atomic operation. Similarly, atomic operations from the GPU to CPU memory will not be atomic with respect to CPU initiated atomic operations.

This means that atomics are only coherent when executed from the **same** device or host.

As far as your philosophy is concerned, how on earth would that apply to ROCm if there's no open source CUDA runtime available to copy from ? 

ROCm may be flawed from a raw performance standpoint but it pioneered some of the features like unified memory first before CUDA even had the equivalent functionality ... 

Yes, it's sad that AMD is deprecating HCC and their support for gfx803 has fallen behind but this is why the project was open source to begin with so that **YOU**, the community can continue developing the kernel/runtime even if AMD decides to eventually ditch the project ...

@smartbitcoin I do understand but ROCm has different design goals to CUDA so they aren't comparable ... 

ROCm is [intentionally](https://github.com/RadeonOpenCompute/ROCm_Documentation/blob/master/Installation_Guide/More-about-how-ROCm-uses-PCIe-Atomics.rst) designed with heterogeneous programming in mind where interactions between host and device memory is seamless.
