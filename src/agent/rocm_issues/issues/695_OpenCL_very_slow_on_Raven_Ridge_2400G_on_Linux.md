### 评论 #2 — macpete (2019-02-10T16:22:40Z)

Thanks @jlgreathouse,

an OpenCL test set me on the right track: The culprit is Halide's runtime, which calls `clCreateBuffer()` & `clReleaseMemObject()` once for each intermediate result during the execution of the generated pipeline. These two functions were the cause of the high CPU load and extremely low frame rate (about 200ms per pair).

Getting rid of the intermediate allocations (by changing the Halide runtime to reuse the buffers in subsequent frames) brought the frame rate up and CPU load down.

Now, this indicates that my initial problem was not caused by ROCm.

But the OpenCL pipeline is still a bit slower than targetting the CPU alone, probably due to additional copies between "host" and "device" memory. Also, the frame rate is very inconsistent with the OpenCL back-end. I see big fluctuations in the execution times of the host-device buffer copies (and texture uploads).

Can you give some general advice on how to get consistent frame rates when doing OpenCL and OpenGL in the same application on Raven?

Thanks,
Marc
