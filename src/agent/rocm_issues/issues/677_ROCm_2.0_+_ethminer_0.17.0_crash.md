### 评论 #1 — jlgreathouse (2019-01-17T18:24:51Z)

Please note that the CUDA and OpenCL paths of applications are not guaranteed to be the same at all, so whether or not CUDA works for you on an Nvidia GPU doesn't really mean a lot for this kind of report.

This appears to be a buffer overflow within one of the kernels of this application. Even if the CUDA and OpenCL code paths were the same (or if the OpenCL version did not crash on another GPU) that does not mean the application is correct. The ROCm software stack can use pages as small as 4 KB, and accessing past the end of one of those pages can cause a segfault. Other GPUs or other software stacks may use larger pages (e.g. 64 KB), and this may hide some buffer overflows.

Nearly every time I've seen such an error, the problem has ended up being incorrect code in an application kernel (and, to note, I've [written tools specifically to try to find these problems](https://github.com/ROCm-Developer-Tools/clARMOR)). As such, I am going to close this issue for now, because we do not have enough proof that it is a "ROCm" problem.

AMD, unfortunately, does not have enough resources to debug every user application. If the developer can demonstrate that the error is caused by a problem with the ROCm stack (e.g. "this piece of code is compiled incorrectly"), please submit an issue on that. However, simply having an application crash does not sufficiently demonstrate that something is wrong in ROCm.
