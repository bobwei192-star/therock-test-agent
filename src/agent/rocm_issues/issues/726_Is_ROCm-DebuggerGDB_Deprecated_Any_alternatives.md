## 描述

I was recently looking at how I might easily disassemble a GPU kernel created by a HC Program I have been working on. I found that the view ISA function in CodeXL doesn't support gfx906 yet and then found [ROCm-Debugger which seems to be able to do this.](https://github.com/RadeonOpenCompute/ROCm-Debugger/blob/master/TUTORIAL.md#how-do-i-view-the-gpu-isa-disassembly)
However it hasn't been updated since 2017.
It's instructions state that:
> Note that both rocm-gpudebugsdk and rocm-gdb debian packages are included as part of the ROCm repo install.

But this is **no longer the case**, [others users have discovered this already but their issue has not received any attention.](https://github.com/RadeonOpenCompute/ROCm-Debugger/issues/10)
The [current ROCM documentation states](https://rocm-documentation.readthedocs.io/en/latest/ROCm_Tools/ROCm-Tools.html#rocm-gdb):
> ROCm-GDB
> The ROCm-GDB is being revised to work with the ROCr Debug Agent to support debugging GPU kernels on Radeon Open Compute platforms (ROCm) and will be available in an upcoming release.

This seems unlikely given that development stopped two years ago and had been regular before that.
Are there plans to resurrect ROCm-GDB? **Any other ways to extract and disassemble a kernel from a HC Program?** I will try using the [rocr_debug_agent](https://github.com/ROCm-Developer-Tools/rocr_debug_agent#code-object-saving) but it looks like a step backwards in usability and functionality compared to ROCM-gdb which looked to have had similar usage to gdb.

Also, I think that all the ROCm and GPUOpen repos could use a spring clean more generally, because there seem to be loads of _other_ dead/zombie repos similar to rocm-gdb, where development stopped a while ago or moved elsewhere. However **google still ranks these repos high up in search results** (which is how I found them).
It would be useful if outdated repos were marked as _Deprecated_ so that nobody is confused by why the instructions in them don't work or conflict with the main documentation.

### 评论 #1 — jlgreathouse (2019-03-08T16:23:44Z)

You can ask the HCC compiler to emit assembly code by setting the environment variable  `KMDUMPISA=1` You can control other compile-time mechanisms [using similar environment variables](https://github.com/RadeonOpenCompute/hcc/blob/roc-2.1.0/lib/clamp-device.in#L15). For example, `KMDUMPDIR` lets you configure where these files are saved to.

Set the environment variables before compiling your HCC or HIP kernels.

If you are running OpenCL, you can pass [related flags in at kernel compilation time](https://github.com/RadeonOpenCompute/ROCm-OpenCL-Runtime/blob/roc-2.1.0/compiler/lib/utils/OPTIONS.def#L948), or you can set them in the environment variable `AMD_OCL_BUILD_OPTIONS_APPEND`:
- `-save-temps` will save ISA files for your kernels into the current working directory. You can set `-save-temps=foo` to append "foo" to all of the files that are dumped out.
- You can also set `-g` to turn enable debug symbols.

You may want to match assembly lines back to source code lines. Our [debug agent](https://github.com/ROCm-Developer-Tools/rocr_debug_agent/blob/roc-2.1.0/src/HSADebugInfo.cpp#L409) shows how you can use our included `llvm-objdump` to perform this. 

To get your GPUs' graphics generation numbers, you can run:
```bash
rocminfo | grep "Name:\ \+gfx" | awk {'print $2'}
```

Then, to get an ISA file annotated with line numbers:
- For OpenCL, run the following command on the .so file generated by `-save-temps`
  - `/opt/rocm/opencl/bin/x86_64/llvm-objdump -mcpu=${GFX_NUM} -source -line-numbers _temp_${N}_${GFX_NUM}.so`
    - Make sure you pass in `-g` when building the kernel. See the above note about `AMD_OCL_BUILD_OPTIONS_APPEND` for an easy way to do that.
    - `${N}` will change depending on how many OpenCL strings were passed into `clBuildProgram`.
    - You should change `${GFX_NUM}` depending on the GPU graphics generation number of your target device.
- For HIP and HCC, run this on the .isabin file generated by `KMDUMPISA`
  - `/opt/rocm/hcc/bin/llvm-objdump -mcpu=${GFX_NUM} -source -line-numbers dump-${GFX_NUM}.isabin`
    - You should build your kernel with `-gline-tables-only` passed in to the HCC compiler.
    - You should change `${GFX_NUM}` depending on the GPU graphics generation number of your target device.

### 评论 #4 — jlgreathouse (2019-03-21T22:05:26Z)

Hi @JMadgwick 

For the moment, the ROCr debug agent is our mechanism for accessing such debug information on GPUs within the ROCm stack. We are still in the process of making a new GDB port that uses the ROCr debug agent as its underlying mechanism. While the documentation hasn't been updated recently, rest assured we have engineers hard at work getting a new version of GDB built on this more robust underlying debug layer.

To help avoid confusion, we are going through the process of updating old repos that contain deprecated software, archiving them, and moving them to [ROCmArchive](https://github.com/ROCmArchive/). This should help avoid some problems in the future where folks find old, no-longer-maintained repos and use them even if they've been replaced or no longer maintained.

I'll also note that you can get the code objects run by your application [using the ROCr debug agent](https://github.com/ROCm-Developer-Tools/rocr_debug_agent/blob/roc-2.2.0/README.md#code-object-saving). If you use this mechanism, you no longer need to set any HCC environment variables at compile time. You can instead save off the files at runtime and then run objdump on them later.
