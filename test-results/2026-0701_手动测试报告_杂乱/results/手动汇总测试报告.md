> ## 🔍 失败分析总结（2026-07-08）
>
> ### 一、20 个已确认失败组合 × 6 大根因分类
>
> | 根因类别 | 涉及组合数 | 涉及组件 | 严重程度 |
> |----------|:----------:|----------|:--------:|
> | **① gfx1151 架构/硬件限制** | 7 | hipsparselt(3)、rocroller(1)、rocprofiler_compute(1)、rocalution(2 稳定精度) | 🔴 **不改源码不可修复** |
> | **② Python 3.14 工具链不兼容** | 4 | rocprofiler_systems(3)、tensilelite(1) | 🔴 **CI 全部阻塞** |
> | **③ Comprehensive 模式新增失败** | 6 | rocblas、libhipcxx_hipcc、libhipcxx_hiprtc、rocprim、rocsolver、rocrtst | 🟡 **Nightly 级测试暴露** |
> | **④ Unified Memory 平台不兼容** | 2 | rocsparse(2) | 🟡 **Full 模式阻塞** |
> | **⑤ HIP Graph 死锁** | 2 | hiptests(2) | 🟡 **3 个 graph 测试真实 hang** |
> | **⑥ GPU Ring Timeout** | 1 | rocsparse(full) | 🟠 **Full 负载触发 GPU reset** |
>
> ### 二、跨组件共同规律
>
> 1. **gfx1151 APU 统一内存架构是核心矛盾** — rocalution 的 CG/BiCGSTAB/FGMRES FP32 精度失败、rocsparse 的 memory guard 检查失败、rocalution IDR NaN 崩溃，都根源相同：APU 的 FP32 数值行为与独立 GPU 不同，rocBLAS/rocSPARSE/rocALUTION 的精度断言是面向 dGPU 设计的。
>
> 2. **Python 3.14 升级导致 2 个组件的 pytest 全面瘫痪** — rocprofiler_systems（conftest.py system_version=None）和 tensilelite（路径分隔符反斜杠 vs 正斜杠）。这是环境升级带来的全局性阻塞，与 gfx1151 无关。
>
> 3. **Comprehensive 模式暴露了 5 个 unique 失败** — 对比 quick/standard，comprehensive 独有的 nightly/weekly 级测试发现 rocblas f16 kernel SIGABRT、rocprim 1/85 FAIL、rocsolver subprocess abort、rocrtst SIGKILL、libhipcxx cmake 构建。说明测试覆盖越深，暴露问题越多。
>
> ### 三、与 ROCm GitHub Issue 的对应关系
>
> | 测试发现问题 | 对应 GitHub Issue | 吻合度 |
> |-------------|-------------------|:------:|
> | rocsparse GPU ring timeout → GPU reset | **#5151** GPU hang on gfx1151 / MES REMOVE_QUEUE 失败 | ✅ **完全吻合** — 同一 queue eviction cascade 链条 |
> | rocalution FP32 精度失败 | **#6034** 93 ML 实验 5 个 bf16 精度 bug / **#5807** fp32 只有 fp16 1/10 性能 | ✅ **强相关** — gfx1151 FP32 精度/性能双重问题 |
> | rocblas f16 kernel SIGABRT | **#6271** TORCH_BLAS_PREFER_HIPBLASLT 内核 crash | ✅ **强相关** — rocblas f16 相关 kernel 异常 |
> | rocsparse unified mem guard 失败 | **#6004** VRAM/GTT 语义混乱 + **#6186** constant PERMISSION_FAULTS | ✅ **强相关** — 统一内存分配语义问题 |
> | hiptests graph 死锁 | **#6372** hipStreamEndCapture SIGSEGV | ✅ **强相关** — HIP Graph capture 缺陷 |
> | rocblas performance | **#4748** rocBLAS/hipBLAS vs gfx1100 2x 慢 | ✅ **性能基准确认** |
> | amd-smi N/A | **#6035** amd-smi ALL N/A on Strix Halo | ✅ **完全吻合** |
> | Python 3.14 兼容性 | 上游问题，Issue 未覆盖新 OS | 🔶 **预见到但尚未报告** |
>
> **结论**：20 个已确认失败组合中，多数可在现有 ROCm GitHub Issue 中找到对应或关联问题，大部分为 gfx1151 (Strix Halo) APU 的已知架构限制。Python 3.14 兼容性问题是 Ubuntu 26.04 + ROCm 较新组合独有的新问题。Profiler 集成测试结论暂不纳入本摘要，待确认是否存在稳定、对应的测试用例后再评估。
>
> ### 四、对 gfx1151 实际使用的影响
>
> | 使用场景 | 受影响组件 | 实际影响 |
> |----------|-----------|----------|
> | **深度学习训练 (PyTorch/TF)** | rocalution（CG solver）、rocblas（f16）、rocprim | ⚠️ CG 求解器不能用；f16 dot 可能产生 NaN；rocprim 99% 通过，基本可用 |
> | **推理 (llama.cpp/vLLM)** | rocsparse（memory guard）、hipsparselt（exclude） | ⚠️ 稀疏矩阵加速不可用；推理主路径（GEMM/Attention）走 rocblas/hipblas 正常 |
> | **数值计算 / HPC** | rocalution（IDR/CG/FGMRES NaN）、rocsolver（abort） | ⚠️ 稀疏线性求解器全部失败；稠密求解器 comprehensive 模式 abort |
> | **HIP 原生开发** | hiptests（graph deadlock）、hipsparselt（exclude） | ⚠️ HIP Graph capture 在 gfx1151 上不稳定；hipsparselt kernel 不可用 |
> | **稳定性** | 多个组件 | ⚠️ GPU ring timeout 在 full 负载下可能触发 display freeze/GUI crash |
>
> **总体评估**：gfx1151 上 ROCm 对 **标准 BLAS/FFT/RAND/PRIM 基础计算功能可用**（32/39 组件通过，82.1%）。**稀疏求解器（rocalution/rocsparse）、comprehensive/full 深度测试存在重大限制**。影响最大的实际场景是：稀疏矩阵计算完全不可用，f16 混合精度训练有 NaN 风险，long-running 任务可能触发 GPU ring timeout 导致系统不稳定。以上问题不改源码无法修复，需等待上游 gfx1151 APU 支持和 Python 3.14 适配。

### 覆盖率说明

- **理论最大值**：39 × 4 = 156 种组合（核心），含 ffm 变体 39 × 8 = 312
- **实际有意义**：约 111 种（部分组件不支持所有 test_type）
- **当前覆盖**：39 种 quick（100%） + 32 种 standard（82%） + **28 种 comprehensive（72%）** + 0 种 full = **99 种 / 111 种 ≈ 89%**
- **剩余待跑**：comprehensive 约 8 种（后台执行中） + full 约 36 种（跳过 3 个排除组件）≈ 44 种

所以 156 是理论最大值，实际约 111 种有意义组合。当前累计已执行 87 种组合（quick 39 + standard 32 + comprehensive 16）。如果把 39 个组件 × 4 个类型全跑完，按 rocblas quick 就花 752s 算，总耗时会是**天数级别**——这也解释了为什么官方 CI 用 48 个组件 × 6 分片并行，还要设 240 分钟超时。

### Sanity 和 amdsmi 失败的具体原因

#### Sanity 失败（路径配置）

**原判**：`pytest tests/` 执行时报 `ModuleNotFoundError: pytest_check`，被错误归因为缺少 Python 依赖。

**实际根因（2026-07-02 重新验证）**：`test_sanity.py` 通过环境变量 `THEROCK_BIN_DIR` 定位 `rocminfo`/`hipcc` 等二进制。在没有 `.env` 文件或环境变量指向不正确时，默认路径 `./build/bin/` 为空目录，导致 `subprocess.run(rocminfo)` 报错。而 `pytest-check` 在系统 Python 3.14 中已正常安装。

**验证**：设置 `THEROCK_BIN_DIR` 为 `output-linux-portable/build/dist/rocm/` 后，5/5 测试全部通过（0.92s）。

**修复**：在 `test_sanity.py` 或 `.env` 中显式设置 `THEROCK_BIN_DIR` 指向正确的 ROCm 产物目录。

#### amdsmi 失败（部分可用）

**原判**：无 GPU 设备访问权限。

**实际结果（2026-07-02 重新验证）**：`/dev/kfd` 和 `/dev/dri` 均正常映射。`amdsmitst` 二进制运行结果：

| ❌ FAILED | 1 | `TestMemoryReadWrite` 内存读写失败 |

**验证命令**：


**结论**：amdsmi 功能**大部分正常**（45/60）。14 个 skip 是权限限制（预期行为），仅 `TestMemoryReadWrite` 需进一步排查是否为 gfx1151 兼容性问题。

## 22a. GPU Ring Timeout 事件记录（2026-07-07）

### 事件经过

2026-07-07 约 **16:45** 发生 GPU ring timeout。当时的测试状态：`run_resume.sh` 正在执行 comprehensive 批次（已连续运行约 2.5 小时），已完成 rocblas（687s）、rocprofiler_systems（3,533s/59min）、rocalution、rocsparse 等组件。

**现象**：
1. 桌面突然黑屏 → 锁屏
2. 输入密码登录后，所有后台进程消失（`run_resume.sh` 及其子进程全部被终止）
3. 系统 GPU 自动恢复，内核日志显示 `GPU reset(3) succeeded!`

**内核日志关键信息**：
```
amdgpu 0000:c5:00.0: GPU reset(3) succeeded!
amdgpu 0000:c5:00.0: [drm] device wedged, but recovered through reset
```

**受影响的进程**（crash dump 文件证实）：
| 进程 | 崩溃时间 | 文件 |
|------|----------|------|
| trae-cn | 15:19 | `/var/crash/_usr_share_trae-cn_trae-cn.1000.crash` |
| chrome | 15:27 | `/var/crash/_opt_google_chrome_chrome.1000.crash` |
| gnome-shell | 16:45 | `/var/crash/_usr_bin_gnome-shell.1000.crash` |

### 根因

与 AGENTS.md 中记录的 4 次 GPU ring timeout 完全一致：

- Strix Halo（gfx1151）为 **APU 统一内存架构**，GPU 计算环（gfx ring）与显示输出共用同一个物理 GPU
- ROCm 测试长时间占用 GPU 计算资源（rocprofiler_systems 连续运行 59 分钟），超过 gfx ring 超时阈值
- ring timeout → AMDGPU 驱动触发 GPU reset → 显示引擎冻结 → Wayland/GNOME 崩溃 → 用户重新登录
- **第 5 次发生**（此前 4 次分别于 10:33、15:30、16:00、16:18）

### 影响

| 影响项 | 说明 |
|--------|------|
| 测试中断 | comprehensive 批次剩余 10 个组件未执行 |
| 结果丢失 | CSV 已持久化到磁盘，未丢失 |
| 桌面恢复 | GPU reset 成功后 systemd 自动重启 gnome-shell |
| 硬件安全 | 未造成永久性硬件损坏（已验证 5 次 recover） |

### 建议

1. **任务调度**：避免在桌面环境下长时间运行 GPU 密集型测试（如 rocprofiler_systems 可跑 59 分钟）。长测试安排在夜间或通过 SSH 会话执行
2. **GPU 隔离**：考虑在独立的 X11/Wayland 会话中运行 ROCm 测试，或使用虚拟显示设备
3. **分批执行**：每 2-3 个 GPU 密集组件后插入冷却间隔（sleep 60s），降低连续 GPU 负载
4. **监控预警**：监控 `/sys/class/drm/card1/device/gpu_busy_percent`，超过 90% 持续 5 分钟时主动插入休眠

该脚本通过解析上述 CMakeLists.txt 中的 `therock_cmake_subproject_declare(...)` 块，提取 `TEST_SUBPROJECTS` 依赖关系，实现**按改动范围智能选择测试子集**。

```
        ↓
```

## 附录 E：test_component.yml 的调用机制（Sanity 闸门 vs 并行组件测试）

这是理解 CI 调用链路最关键的部分。调用关系如下：

```
test_artifacts.yml
 │
 ├─ [Job: generate-matrix]          # 无 GPU，生成 JSON 配置
 │   └─ fetch_test_configurations.py
 │
 ├─ [Job: sanity]                   # 第 1 个 test_component.yml 调用
 │   └─ uses: ./.github/workflows/test_component.yml
 │       with:
 │         test_config: ${{ needs.generate-matrix.outputs.sanity_component }}
 │       # ↑ 传入的是 sanity 的 JSON 对象（单组件）
 │
 └─ [Job: component-tests]          # 第 N 个 test_component.yml 调用（矩阵展开后）
    └─ strategy:
         matrix:
           component: ${{ fromJson(needs.generate-matrix.outputs.components) }}
       uses: ./.github/workflows/test_component.yml
         with:
           test_config: ${{ matrix.component }}
         # ↑ 传入的是每个组件的 JSON 对象（如 rocsparse、hipsparse……）
```

### 关键理解：uses 是 GitHub Actions 的「可复用工作流调用」

- `uses:` 是 GitHub Actions 的语法，调用另一个工作流文件作为**子工作流（reusable workflow）**
- `test_component.yml` 是一个**模板**，定义了"在容器内执行一个测试的全套步骤"（清理 → 装环境 → GPU 体检 → 跑测试脚本 → 失败复现）
- Sanity 调用 `test_component.yml`：传的是 **sanity** 的 JSON 配置 → 跑 `test_sanity.py`
- 组件测试也调用 `test_component.yml`：传的是各个**组件**的 JSON 配置 → 跑对应的 `test_script`
- **不是两个不同的 test_component.yml 文件，而是同一个文件被调用了多次，每次传不同的配置**

```
              ┌── sanity ──→ test_component.yml (with sanity config) ──→ test_sanity.py
test_artifacts.yml
              └── rocsparse ──→ test_component.yml (with rocsparse config) ──→ test_runner.py
              └── hipsparse ──→ test_component.yml (with hipsparse config) ──→ test_runner.py
              └── rocblas   ──→ test_component.yml (with rocblas config)   ──→ test_runner.py
              └── ...（每个组件各一个并行 job）
```

### 执行顺序

```
时间 →
├─ [generate-matrix] ──────────→ 完成后输出 JSON
│
├─ [sanity] ───────────────────→ 必须通过（depends on: generate-matrix）
│                               ↓ 通过后
├─ [rocsparse] [hipsparse] [rocblas] ...  ← 并行启动（depends on: sanity）
```

每个 `test_component.yml` 内部执行：

```
test_component.yml（容器内）
 ├─ 1. cleanup_processes.sh        # 清理残留 GPU 进程
 ├─ 2. setup_venv.py + install_rocm_from_artifacts.py
 │      └─ fetch_artifacts.py      # 下载并安装 ROCm 构件
 ├─ 3. health_status.py / print_driver_gpu_info.py  # GPU 健康检查
 ├─ 4. ★ 执行 component.test_script ★  # 核心：调用 test_runner.py / test_*.py
 └─ 5. reproduce_test_failure.py   # 失败时输出本地复现命令
```

1-5 步对 sanity 和普通组件**完全一样**，唯一的区别是第 4 步执行的 `test_script` 不同（sanity 跑 `test_sanity.py`，组件跑 `test_runner.py` 或专用脚本）。

### 附录 E 补充：Step 2「装环境」具体装什么？

Step 2 由三个脚本组成，职责分离如下：

#### 2a. `setup_venv.py` — Python 虚拟环境

- 在容器内创建 `.venv`（`python -m venv`）
- 升级 pip、setuptools、wheel
- 可选从 ROCm pip 源安装 Python 包（如 `rocm[libraries,devel]`）
- **不涉及任何系统级组件**

#### 2b. `install_rocm_from_artifacts.py` — 安装 ROCm 预编译产物（核心）

这是"装环境"的核心。它从 **S3 或 GitHub CI artifact** 下载并解压 **预编译的二进制 tarball**：

| 类别 | 包含内容 | 示例 |
|------|----------|------|
| 运行时库 | `.so` 共享库 | `libhip_hcc.so`, `librocblas.so` |
| 测试可执行 | 编译好的测试二进制 | `rocblas-test`, `hipsparse-test` |
| CTest 配置 | `CTestTestfile.cmake` | 注册了所有测试用例和标签 |
| 头文件 | `.h` 开发头文件 | `hip/hip_runtime.h` |
| 数据文件 | MIOpen 调优 DB 等 | `share/miopen/db/*.db` |
| 工具链 | LLVM/clang 等 | `bin/hipcc`, `bin/clang` |

#### 2c. `fetch_artifacts.py` — 底层下载引擎

- 与 S3 通信，列出可用 artifact
- 按 include/exclude 正则过滤
- 并行下载并按 `--flatten` 模式解压到统一目录（`therock-build/dist/`）

#### 需要重装内核吗？**不需要。**

**一句话：artifact 只装用户态，不碰内核态。**

| 层 | 内容 | 来源 | 安装方式 |
|-----|------|------|---------|
| **用户态 ROCm 库** | HIP runtime、rocBLAS、MIOpen 等 `.so` | artifact 下载 | 解压到 `therock-build/` |
| **测试可执行** | `rocblas-test`、`test_runner.py` | artifact 下载 | 解压到 `therock-build/` |
| **内核模块** | `amdgpu.ko`、`amdkfd.ko` | **宿主 OS 预装** | **不需要重装/更新** |
| **GPU 驱动** | 内核态驱动 + firmware | **宿主 OS 预装** | **不需要重装/更新** |

CI 的 GPU runner 在启动时就已具备正确的内核驱动版本。artifact 中的 ROCm 用户态库与宿主内核驱动通过 `/dev/kfd` 和 `/dev/dri` 通信——只要内核接口兼容，就无需重新编译或加载内核模块。

```
                 容器（test_component.yml）
 ┌─────────────────────────────────────────────────┐
 │  therock-build/dist/                             │
 │   ├── lib/     ← librocblas.so（用户态库）        │
 │   ├── bin/     ← rocblas-test（测试可执行）        │
 │   └── share/   ← CTestTestfile.cmake             │
 │                                                  │
 │   setup_venv.py → .venv（Python 依赖）             │
 └─────────────────────────────────────────────────┘
                          │
                          ▼ 通过 /dev/kfd, /dev/dri 调用
 ┌─────────────────────────────────────────────────┐
 │  宿主 OS 内核                                      │
 │   ├── amdgpu.ko（GPU 驱动）  ← 预装，不动           │
 │   └── amdkfd.ko（KFD 接口）  ← 预装，不动           │
 └─────────────────────────────────────────────────┘
```

因此，Step 2 **不重装内核、不加载内核模块、不影响宿主 OS 的驱动状态**。它只下载并解压 ROCm 用户态二进制产物，与本地 `make -j` 编译产物的唯一区别是**预编译好直接从 S3 下载**，无需在测试机上重新编译。

### test_component.yml 的执行环境：容器

`test_component.yml` 的 `container` 字段指定了测试在容器内执行：

```yaml
container:
  image: "ghcr.io/rocm/no_rocm_image_ubuntu24_04@sha256:..."
  options: "..."   # --device /dev/kfd --device /dev/dri ...
```

注意镜像名：**`no_rocm_image_ubuntu24_04`** —— 这是一个**纯净的 Ubuntu 24.04**，里面没有预装任何 ROCm 组件。

整个测试执行是一个"从零搭建"的过程：

```
启动容器（纯净 Ubuntu 24.04，无 ROCm）
  │
  ├─ checkout build_tools（只拉取 build_tools/ 目录）
  │
  ├─ cleanup_processes.sh（清理宿主残留 GPU 进程）
  │
  ├─ checkout 完整仓库（拉取 TheRock 完整源码，不含 submodule）
  │
  ├─ setup_test_environment（核心 Step 2）
  │   ├─ setup_venv.py                          → 创建 .venv
  │   └─ install_rocm_from_artifacts.py          → 从 S3 下载预编译产物
  │       └─ fetch_artifacts.py                  → 解压到 ./build/
  │            ├── build/bin/        ← 测试可执行（如 rocblas-test）
  │            ├── build/lib/        ← 共享库（如 librocblas.so）
  │            └── build/share/      ← CTestTestfile.cmake
  │
  ├─ health_status.py / print_driver_gpu_info.py（GPU 健康检查）
```

**关键事实**：

## 附录 F：Sanity 闸门的含义

### Sanity 测试测什么？

**Sanity 不是把全部组件都测一遍。** 它只测最基础的"安装是否可用"，由 `test_sanity.py` 驱动，内部调用 `pytest tests/` 运行 `tests/test_rocm_sanity.py` 中的几个快速检查：

| 测试项 | 命令 | 验证内容 |
|--------|------|---------|
| GPU 检测 | `rocminfo` | 能找到 GPU（`Device Type: GPU`）、GFX 架构名在输出中 |
| HIP 编译器 | `hipcc` 编译 `hipcc_check.cpp` | 能用 hipcc 编译一个 HIP 程序并运行成功 |
| GPU 枚举 | `rocm_agent_enumerator` | 能列出 GPU agent |

**它不跑任何组件的功能测试**（不跑 rocBLAS、不跑 MIOpen、不跑 RCCL）。它只验证三件事：
1. ROCm 安装正确（产物下载+解压成功）
2. GPU 可访问（`/dev/kfd`, `/dev/dri` 映射正常，驱动响应）
3. HIP 工具链可用（hipcc 能编译、链接、运行）

### 「闸门」（Gate）是什么意思？

闸门是一个**流水线比喻**：

```
                    ┌──────────────────────┐
                    │  Sanity 闸门           │
                    │  (pytest tests/)      │
                    │  耗时 ~30 秒           │
                    └─────────┬────────────┘
                              │ 通过?
                     ┌───────┴───────┐
                     ▼               ▼
                   是               否
              ┌────────────┐   ┌──────────────┐
              │ 放行        │   │ 拦截          │
              │ 所有组件并行  │   │ 跳过后续测试    │
              │ 开始测试     │   │ 报告基础设施问题 │
              └────────────┘   └──────────────┘
```

它的作用：

| 特性 | 说明 |
|------|------|
| **前置依赖** | `test_artifacts.yml` 中 Sanity job 设置了 `needs: generate-matrix`，而所有组件测试 job 又设置了 `needs: sanity` |
| **快速失败** | Sanity 耗时约 30 秒。如果连这个都过不了（如 artifact 下载出错、GPU 没映射上），50 个组件测试 100% 全挂，没必要浪费资源 |
| **区分故障类型** | Sanity 失败 → 基础设施/安装问题；Sanity 通过但组件失败 → 组件自身问题。排障时直接二分 |
| **节省成本** | 避免在 bad build 或坏掉的 runner 上跑大量无用测试 |

**简单说：Sanity 是"这轮测试值不值得跑"的快速验证。它不是组件测试的一部分，而是组件测试的前提条件。**

## 26. 风险与建议

1. **GPU ring timeout 风险** — rocBLAS/hipBLAS 等大规模 GPU 测试执行时可能引发 ring timeout，导致桌面应用崩溃。建议使用 GPU 隔离或 headless 模式运行。
2. **gfx1151 profiling 不可用** — rocprofiler-sdk/rocprofiler-compute/rocprofiler-systems 等 profiling 工具因 KFD 驱动限制无法使用，需等待内核驱动更新或使用 workstation GPU。
3. **Python 3.14 版本兼容性风险** — 影响 **tensilelite** 和 **rocprofiler_systems** 两个组件：
   - **tensilelite**：pytest 9.1.1 内部 `checksubpath()` 将 Linux 路径转换为 `WindowsPath` 对象导致路径比较失败。1/1588 测试受此影响（INTERNALERROR 而非测试 FAIL）。
   - **rocprofiler_systems**：`conftest.py` 中 `system_version` 变量解析失败返回 `None`，导致 pytest collection 阶段报 `TypeError: '<' not supported between instances of 'NoneType' and 'tuple'`，23/90 baseline 测试全部阻塞。
   - **建议**：统一使用 Python 3.12 作为测试环境，或在测试脚本中增加 Python 版本检查和兼容处理。
4. **CI 路径固化问题（/therock/）** — 构建产物中大量硬编码 `/therock/` 路径（CI 环境源码根目录），影响 `rocprofiler_sdk`、`hipkernelprovider`、`hipdnn` 三个组件。需创建 `/therock → /home/zs/TheRock` 符号链接（需 root 权限）或使用 `sed` 批量替换路径。
5. **rocjpeg 测试脚本 vs 真实测试结果不一致** — `test_rocjpeg.py` 因 cmake FetchContent 重建失败，但直接 `ctest` 执行预构建测试全部通过（22/22）。建议测试脚本增加跳过重建步骤的选项。
6. **Python 依赖链** — `sanity` 和 `rocprofiler_compute` 均因缺少 Python 包失败。建议预装 `pandas`, `prettytable`, `matplotlib` 等常用数据分析包。
7. **测试规模过大** — 部分组件测试集庞大，需分片或分段运行。hipblas 已通过直接运行原生测试二进制解决（636s 全量通过 47,367 tests）。

## 28. hipblas 超时问题修复（2026-07-03）

### 问题描述

Phase 2 报告中 hipblas 组件标记为"超时"——CTest 框架下 44,757 个测试需 900s+ 超时，300s 默认不够。实际上 `test_runner.py` 通过 CTest 调用的配置仅识别了 44,757 个测试，且 CTest 的 900s 超时设置不足以完成全量测试。

### 解决方法的思路

**绕过 CTest 框架，直接运行原生测试二进制 `hipblas-test`。** 根本原因有二：
1. **CTest 开销**：CTest 对每个测试套件（suite）单独启动进程、采集输出、管理超时，引入额外调度开销
2. **环境变量传递**：`test_runner.py` 通过 CTest 传递环境变量时，`ROCM_PATH` 和 `LD_LIBRARY_PATH` 可能未正确透传到子进程，导致部分测试行为异常

### 具体方法

```bash
cd /home/zs/TheRock/output-linux-portable/build/dist/rocm/bin && \
ROCM_PATH=/home/zs/TheRock/output-linux-portable/build/dist/rocm \
LD_LIBRARY_PATH=/home/zs/TheRock/output-linux-portable/build/dist/rocm/lib:$LD_LIBRARY_PATH \
timeout 3600 ./hipblas-test 2>&1
```

关键点：
- **直接调用 `hipblas-test`**：这是 hipBLAS 的原生 GoogleTest 二进制，包含全部 47,367 个测试用例（比 CTest 识别的 44,757 多出 2,610 个）
- **设置 `ROCM_PATH`**：指向编译产物目录，确保 HIP runtime 能正确加载库文件
- **设置 `LD_LIBRARY_PATH`**：包含 `dist/rocm/lib`，确保动态链接器能找到 `libhipblas.so` 等共享库
- **`timeout 3600`**：设置 1 小时超时（实际仅需 636s），远大于 CTest 的 900s 限制
